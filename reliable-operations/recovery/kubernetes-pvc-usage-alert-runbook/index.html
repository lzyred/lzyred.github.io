    Kubernetes Reliability Runbook

    PVC 使用率告警 Runbook¶

    目的¶

    本 Runbook 定义当 Kubernetes PersistentVolumeClaim 接近或达到容量阈值时，如何响应。

    核心判断是：

    PVC 使用率告警是数据风险告警，不是磁盘清理工单。

    响应必须识别所属工作负载、确认真实文件系统使用量、理解数据是可丢弃还是业务关键，并选择不会悄悄删除状态的恢复路径。

    触发条件¶

    以下告警适用本 Runbook：

    • KubePersistentVolumeFillingUp
    • KubePersistentVolumeUsageHigh
    • KubeletVolumeStatsUsedBytes 阈值告警
    • 与 Kubernetes PVC 关联的 NFS 或共享文件系统容量告警
    • 表示写入因存储满而失败的应用告警

    推荐阈值模型：

    严重级别 建议阈值 预期响应
    Warning 已使用 75%，或预计 7 天内写满 评审负责人和增长模式。
    High 已使用 85%，或预计 72 小时内写满 准备清理或扩容。
    Critical 已使用 90% 到 95%，或写入失败 立即行动，负责人不可用时升级。

    阈值应按增长速率调整。增长很慢的归档卷和快速增长的上传卷，不应共享同一种运营紧急度。

    前置条件¶

    行动前确认：

    • 命名空间、PVC 名、PV 名、StorageClass 和所属工作负载。
    • 卷类型是块存储、文件存储、本地盘、云盘还是 NFS。
    • 数据是可丢弃、缓存类、可重建还是业务关键。
    • StorageClass 和后端是否支持扩容。
    • 业务关键数据是否有备份或快照。
    • 应用负责人是否在线。

    如果所有权未知且告警为 critical，在删除任何内容前升级给平台负责人和事故指挥。

    影响检查¶

    检查应用是否已经受影响：

    kubectl -n <namespace> get pvc <pvc>
kubectl -n <namespace> describe pvc <pvc>
kubectl -n <namespace> get pods -o wide | grep <workload-or-pvc-hint>
kubectl -n <namespace> get events --sort-by=.lastTimestamp

    关注：

    • 写入失败。
    • Pod 重启。
    • readiness 失败。
    • 应用日志中的 no space left on device。
    • 由该工作负载引起的 Ingress 或 API 错误。
    • 队列积压或任务延迟。

    如果已经有用户影响，把告警当作事故处理，而不是例行维护。

    确认真实使用量¶

    Kubernetes PVC 容量和真实文件系统使用量并不总是同一个信号。

    查看 kubelet volume stats 相关对象：

    kubectl -n <namespace> get pvc <pvc> -o wide
kubectl -n <namespace> describe pod <pod-using-pvc>

    从批准的诊断 Pod 或应用 Pod 内查看：

    df -h <mount-path>
du -xh --max-depth=1 <mount-path> 2>/dev/null | sort -h

    对于 NFS-backed PVC，还要在 NFS 服务器或平台监控路径确认使用量。PVC 可能只请求 100Gi，而真实共享文件系统是承载多个 claim 目录的更大容量池。

    记录：

    • PVC 申请容量。
    • 文件系统总量、已用、可用和使用率。
    • 最大目录或文件类别。
    • 最近增长速率。
    • 所属工作负载和负责人。

    决策树¶

    根据数据类型和增长行为选择响应：

    条件 首选动作
    可丢弃缓存或临时文件占用空间。 通过应用批准路径清理。
    业务关键数据按预期增长。 扩容卷或后端容量。
    业务关键数据异常增长。 为安全先扩容（如需要），再调查增长来源。
    日志或导出文件写入持久数据路径。 停止写入者或改路径，再清理批准文件。
    NFS 后端池被多个 PVC 共同写满。 作为共享存储事故处理；容量动作属于 NFS 层。
    StorageClass 不支持扩容。 迁移到更大卷或新的存储画像。
    负责人未知。 不删除；升级，并只做非破坏性缓解。

    操作流程¶

    1. 识别负责人和数据类别¶

    找到使用 PVC 的工作负载：

    kubectl -n <namespace> get pods -o jsonpath='{range .items[*]}{.metadata.name}{"\t"}{range .spec.volumes[*]}{.persistentVolumeClaim.claimName}{" "}{end}{"\n"}{end}' | grep <pvc>

    识别标签、Deployment、StatefulSet、CronJob 或 Helm release：

    kubectl -n <namespace> get pod <pod> --show-labels
kubectl -n <namespace> get ownerreferences <pod>

    如果当前 kubectl 版本不支持第二条命令，使用：

    kubectl -n <namespace> get pod <pod> -o jsonpath='{.metadata.ownerReferences}'

    2. 必要时停止不安全增长¶

    如果容量快速耗尽，先停止来源再清理：

    • 暂停写入数据的 Job 或 CronJob。
    • 关闭产生文件的 feature flag。
    • 经批准后缩容非关键写入者。
    • 将日志或导出从持久路径重定向。

    不要盲目缩容有状态系统。数据库、消息队列和队列类系统必须遵循服务专属 Runbook。

    3. 只清理被批准的数据¶

    只有理解数据类别后才允许清理。

    安全示例：

    • 应用拥有的过期临时文件。
    • 服务负责人批准的可重建缓存目录。
    • 超过保留期的已完成导出文件。
    • 保留事故证据后的重复诊断 dump。

    不安全示例：

    • 数据库目录。
    • 队列 segment。
    • 用户上传文件。
    • 对象元数据。
    • 所有权未知的文件。

    删除前，如应用负责人需要证据，优先将小样本移动到隔离路径。

    4. 数据必须保留时扩容¶

    确认扩容支持：

    kubectl get storageclass <storageclass> -o yaml
kubectl -n <namespace> get pvc <pvc> -o yaml

    如支持，申请更大容量：

    kubectl -n <namespace> patch pvc <pvc> \
  --type merge \
  -p '{"spec":{"resources":{"requests":{"storage":"<new-size>"}}}}'

    然后验证：

    kubectl -n <namespace> get pvc <pvc> -w
kubectl -n <namespace> describe pvc <pvc>

    扩容后检查挂载文件系统。有些驱动只有在 Pod 重启或重新挂载后才完成文件系统扩容。

    5. 不能扩容时迁移¶

    如果 StorageClass 不能扩容，或后端不适合该工作负载：

    • 使用批准的存储画像创建新 PVC。
    • 停止写入。
    • 使用批准工具复制数据，并做 checksum 或应用级验证。
    • 让工作负载使用新 PVC 启动。
    • 在验证和回滚窗口结束前保留旧 PVC。

    除非数据明确可丢弃，否则迁移后不要立刻删除旧 PVC。

    验证¶

    只有全部满足，响应才算完成：

    检查 通过条件
    使用率 PVC 或后端使用率低于恢复阈值，且不再异常增长。
    应用 写入成功，readiness 稳定。
    告警 Critical 告警清除，或降级并有负责人和截止时间。
    数据安全 没有未经负责人批准删除业务关键数据。
    监控 仪表盘展示的容量现实与应用看到的一致。
    跟进 增长原因、负责人和下一次评审日期已记录。

    回滚与升级¶

    只有在删除前移动或备份了文件，清理才可能回滚。如果数据已经删除，恢复依赖备份或应用重建能力。

    以下情况升级：

    • 用户可见写入失败。
    • PVC 超过 critical 阈值且负责人不可用。
    • 数据类别未知。
    • 扩容失败或持续 pending。
    • PVC 层处理后，后端存储仍然满。
    • 同一 NFS 或共享存储后端上的多个 PVC 同时增长。

    升级信息应包含命名空间、PVC、PV、StorageClass、工作负载、使用率、增长速率和疑似最大路径。

    事故后复盘¶

    恢复后回答：

    • 为什么数据增长没有更早被预测？
    • 阈值或预测告警是否需要调整？
    • 工作负载是否需要保留策略？
    • 日志、导出或临时文件是否应迁移到对象存储？
    • 当前 StorageClass 是否适合该工作负载？
    • 是否需要把负责人映射加入平台 inventory？
