    OpenTelemetry 自动埋点标准¶

    目的¶

    本标准定义如何在 Kubernetes 工作负载上启用 OpenTelemetry 自动埋点，而不把它当成一个对整个命名空间盲目生效的开关。

    OpenTelemetry 自动埋点通过 OpenTelemetry Operator 和 Kubernetes 准入变更（admission mutation）实现。它会给新 Pod 加入 init 容器、agent、环境变量和运行期配置。这意味着它会改变应用启动、内存占用、网络出站，以及离开集群的遥测数据。

    本标准的核心判断：

    自动埋点是一次生产变更，不是一个可观测性开关。一次推一个工作负载，带验证和回滚，就像你发布任何会改动运行中 Pod 的变更一样。

    适用场景¶

    在以下情况使用本标准：

    • 你在 Kubernetes 上运行 OpenTelemetry Operator，或计划安装它。
    • 你想从那些暂时无法手动埋点的服务获取链路追踪，以及可选的指标或日志。
    • 你需要在多个服务间保持一致的运行期 agent 配置。
    • 你想要一座从“没有链路”到“关键路径手动埋点”的安全过渡桥梁。

    它对那些负责可观测性落地、但并不拥有每一个应用代码库的平台、SRE 和 DevOps 团队最有用。

    非目标¶

    本标准不覆盖：

    • 应用代码内部的手动 SDK 埋点。
    • 链路后端本身的设计。后端选型见 SigNoz 可观测性标准。
    • 超出下文“推广安全基线”之外的完整采样策略设计。
    • 用于虚拟机或裸金属的非 Kubernetes agent。
    • 厂商特定的 OpenTelemetry 发行版。

    前置条件¶

    启用注入前，确认：

    • 存在一个可达的 OTLP 兼容 collector 或后端。
    • OpenTelemetry Operator 及其准入 webhook 已安装且健康。
    • 目标命名空间在目标应用 Pod 创建之前已有一个 Instrumentation 资源。
    • 第一个目标工作负载是非关键的，并有明确的回滚负责人。
    • 该工作负载有注入前的延迟、错误率、重启和资源基线。
    • 运行时语言已知：Java、Node.js、Python、.NET、Go、Deno 或仅 SDK 注入。

    本标准聚焦应用运行期埋点。Apache HTTPD 和 NGINX 模块注入需要单独的 Web 服务器模块评审，因为它们的兼容性和配置风险与应用 agent 不同。

    不要从高流量、延迟敏感或事故频发的服务开始。自动埋点作为分阶段推广，比作为全平台运动更安全。

    推荐标准¶

    把 collector 作为默认边界¶

    除非有刻意的例外，应用遥测都发往集群内的 OpenTelemetry Collector。

    collector 给平台团队一个统一控制点来管理：

    • 后端凭据。
    • 重试与队列上限。
    • 内存限制。
    • 属性归一化。
    • 过滤不安全的 span、指标和日志属性。
    • 向一个或多个后端的导出路由。

    不要让工作负载指向 localhost、kubectl port-forward、节点 IP 或开发者笔记本端点。生产环境的 Instrumentation 资源必须引用一个稳定的服务名。

    端点示例：

    http://otel-collector.observability.svc.cluster.local:4318

    当 collector 与应用不在同一命名空间时，使用完全限定的服务名。

    显式设置 OTLP 协议；并让 collector 端口匹配¶

    不要依赖某个语言的默认 OTLP 协议。每个自动埋点运行时的默认协议都与版本相关，并且在语言 SDK 和 Operator 两侧都变过。把“Node.js 用 gRPC，其它都用 HTTP”当成固定规则，迟早会让某个工作负载指向一个 collector 并未监听的接收端口，于是链路悄无声息地消失。

    可靠的模型是一份你刻意设定、而非推断的三方契约：

    1. 协议——在 Instrumentation 资源的运行时段里显式设置 OTEL_EXPORTER_OTLP_PROTOCOL（grpc、http/protobuf 或 http/json）。
    2. 端点端口——必须与所选协议匹配：
    OTLP 协议 Collector 接收端口 端点示例
    grpc 4317 http://otel-collector.observability.svc.cluster.local:4317
    http/protobuf 或 http/json 4318 http://otel-collector.observability.svc.cluster.local:4318
    1. Collector 接收器——collector 必须为你所选的端口暴露匹配的 OTLP 接收器（4317 走 gRPC，4318 走 HTTP）。

    这里端点上的 http:// 前缀对两种协议都是正确的；它就是 agent 从 OTEL_EXPORTER_OTLP_ENDPOINT 读取的值。必须一致的是 协议 ↔ 端口 ↔ collector 接收器，而不是这个前缀。

    仍然重要的运行时说明：

    • Python：较旧的 Operator 版本可能需要显式设置 OTEL_EXPORTER_OTLP_PROTOCOL=http/protobuf；升级时核对。
    • .NET：对 musl 基础镜像检查 runtime identifier。
    • Go：处于特性开关后，需要 OTEL_GO_AUTO_TARGET_EXE，使用 eBPF sidecar，且不支持多容器 Pod（见下文 Go 安全例外）。
    • Deno：其支持可能需要运行时标志；未经测试不要当作稳定默认。
    • 仅 SDK：协议和端点来自你注入的 SDK 配置；让 collector 接收器与你所设的一致。

    每次 Operator 升级后，对照你所运行的接收器重新确认注入的协议。协议是你设定的值，不是你继承的架构边界。

    让注入保持显式开启（opt-in）¶

    使用按工作负载的 Pod 模板注解。不要把整命名空间的注入当成默认运营模式。

    对 Deployment，注解应放在：

    spec.template.metadata.annotations

    放在 Deployment 对象自身 metadata 上的注解不会改动 Pod。

    按语言的开启注解：

    运行时 注入注解
    Java instrumentation.opentelemetry.io/inject-java: "true"
    Node.js instrumentation.opentelemetry.io/inject-nodejs: "true"
    Python instrumentation.opentelemetry.io/inject-python: "true"
    .NET instrumentation.opentelemetry.io/inject-dotnet: "true"
    Go instrumentation.opentelemetry.io/inject-go: "true"
    Deno 或仅 SDK instrumentation.opentelemetry.io/inject-sdk: "true"

    当命名空间里存在多个 Instrumentation 对象时，用资源名代替 "true"：

    instrumentation.opentelemetry.io/inject-java: "java-standard"

    显式指定目标容器¶

    多于一个容器的 Pod 需要显式指定目标。

    单语言埋点时使用：

    instrumentation.opentelemetry.io/container-names: "example-app"

    同一 Pod 内多语言时，Operator 需要多埋点特性开关以及按语言的容器名注解：

    instrumentation.opentelemetry.io/inject-java: "true"
instrumentation.opentelemetry.io/java-container-names: "example-api"
instrumentation.opentelemetry.io/inject-python: "true"
instrumentation.opentelemetry.io/python-container-names: "example-worker"

    不要让 webhook 在多容器 Pod 里猜目标容器。这会造成一个常见失败模式：Pod 健康、注解存在，但你真正关心的服务没有产出有用的遥测。

    让第一个 Instrumentation 资源保持最小¶

    在运营模式稳定之前，每个运行时、每个命名空间用一个 Instrumentation 资源。

    Java 基线示例：

    apiVersion: opentelemetry.io/v1alpha1
kind: Instrumentation
metadata:
  name: java-standard
  namespace: example-namespace
spec:
  exporter:
    endpoint: http://otel-collector.observability.svc.cluster.local:4318
  propagators:
    - tracecontext
    - baggage
  sampler:
    type: parentbased_traceidratio
    argument: "0.05"
  java:
    env:
      # 显式设置协议，使其与上面的 4318 HTTP 接收器匹配。
      - name: OTEL_EXPORTER_OTLP_PROTOCOL
        value: http/protobuf

    这个示例刻意从一个较低的采样比例开始。采样值 "1" 会追踪所有流量，作为高吞吐服务的默认值并不安全。

    协议设在运行时专属段（这里是 spec.java.env）而非交给 SDK 默认，于是 4318 端点、http/protobuf 协议和 collector 的 HTTP 接收器三者一致。其它运行时用 spec.nodejs.env、spec.python.env 等。

    只有当某个运行时环境覆盖表达一个刻意的决定（例如关闭某个吵闹的库或设置一个兼容性标志）时才添加它。不要把厂商的大段示例不加裁剪地粘进生产；先把它裁到你真正想要的行为。

    把 Go 当作安全例外¶

    Go 自动埋点不同于 Java、Python、Node.js 和 .NET。它使用 eBPF，需要一个具备提升权限的注入 sidecar。它还需要通过 OTEL_GO_AUTO_TARGET_EXE 或对应注解给出目标可执行文件路径。

    除非以下全部为真，否则不要启用 Go 自动埋点：

    • 目标 Pod 只有你打算埋点的那个 Go 容器。
    • 可执行文件路径已知且稳定。
    • 安全负责人接受特权 sidecar 的要求。
    • 推广窗口内有回滚负责人在场。

    对许多 Go 服务，手动埋点是更安全的长期标准。

    推广步骤¶

    1. 选定一个非关键工作负载和一个运行时。
    2. 记录基线：请求延迟、错误率、Pod 重启次数、CPU、内存、日志量，以及当前的链路量（若有）。
    3. 创建或更新该运行时专属的 Instrumentation 资源。
    4. 如果目标是 Go，在继续之前为特权 sidecar 取得明确的安全批准。
    5. 在动工作负载之前，验证 operator、webhook、collector 服务和 Instrumentation 资源。
    6. 给工作负载的 Pod 模板加上注入注解。
    7. 让正常的部署机制创建新 Pod，然后观察 rollout 状态。
    8. 验证 Pod 变更、遥测链路、应用健康和后端负载。
    9. 在约定的观察窗口结束后，再给下一个工作负载加注解。

    工作负载注解示例：

    apiVersion: apps/v1
kind: Deployment
metadata:
  name: example-service
  namespace: example-namespace
spec:
  template:
    metadata:
      annotations:
        instrumentation.opentelemetry.io/inject-java: "java-standard"
        instrumentation.opentelemetry.io/container-names: "example-app"

    Instrumentation 资源必须在应用 Pod 创建之前存在。如果应用 Pod 已经存在，webhook 无法事后改动它，必须通过一次 rollout 创建新 Pod。

    验证¶

    分层验证，并在第一层失败处停下。

    第 1 层：Operator 与 webhook¶

    kubectl -n opentelemetry-operator-system get pods
kubectl get mutatingwebhookconfigurations
kubectl -n opentelemetry-operator-system logs -l app.kubernetes.io/name=opentelemetry-operator --container manager --since=10m

    预期结果：

    • Operator Pod 处于 Ready。
    • mutating webhook 配置存在。
    • Operator 日志中没有针对目标命名空间的注入、证书或准入错误。

    第 2 层：collector 可达性¶

    kubectl -n observability get svc otel-collector
kubectl -n observability get pods -l app.kubernetes.io/name=otel-collector
kubectl -n observability logs deploy/otel-collector --since=10m

    预期结果：

    • collector 服务暴露了目标运行时所需的协议和端口。
    • collector Pod 处于 Ready。
    • 测试窗口内 collector 日志没有接收器绑定失败、导出认证错误、队列饱和或内存限制器丢弃。

    第 3 层：Instrumentation 资源¶

    kubectl -n example-namespace describe otelinst java-standard

    预期结果：

    • 端点指向 collector 服务，而不是 localhost。
    • propagators 和 sampler 是显式的。
    • 需要运行时覆盖时，运行时专属段存在。
    • Events 为空，或解释了一个非阻塞状况。

    第 4 层：工作负载变更¶

    kubectl -n example-namespace rollout status deploy/example-service
kubectl -n example-namespace get events --sort-by=.metadata.creationTimestamp
kubectl -n example-namespace describe pod -l app.kubernetes.io/name=example-service

    预期结果：

    • 注解施加之后创建了新 Pod。
    • 适用时，Events 显示自动埋点 init 容器或注入 agent 的行为。
    • 目标容器（而非某个 sidecar 或不相关的 init 容器）收到了埋点。
    • Pod 在没有异常启动延迟的情况下 Ready。

    第 5 层：遥测与应用健康¶

    同时验证可观测性与服务行为：

    • 后端在约定的测试窗口内收到来自 example-service 的 span。
    • 链路的服务名、命名空间、环境和版本属性可用于过滤。
    • 请求延迟、错误率、重启次数和内存仍在推广阈值内。
    • 日志量没有因 agent 调试输出而激增。
    • collector 队列、丢弃的 span 和后端摄入延迟都低于告警阈值。

    如果应用健康退化，回滚。除非服务负责人明确接受风险，否则不要“先开着观察观察”。

    回滚¶

    注入发生在 Pod 创建时，因此回滚必须创建干净的替换 Pod。

    1. 从工作负载 Pod 模板移除注入注解。
    2. 从同一 Pod 模板移除运行时专属的容器名注解。
    3. 通过正常部署机制对工作负载做一次 rollout。
    4. 确认新 Pod 不再包含自动埋点 init 容器、注入 agent 或注入的 OpenTelemetry 环境变量。
    5. 确认应用延迟、错误率、重启次数和内存回到注入前基线。
    6. 在没有任何已注解工作负载依赖它之前，保留 Instrumentation 资源。

    只有在已注解工作负载清理完毕之后，才移除 Operator 或 Instrumentation 资源。先移除平台组件，可能让之后的 Pod 创建变成准入或启动失败。

    已知失败模式¶

    失败模式 症状 可能原因 正确处置
    注解加错对象 Deployment 有注解，但新 Pod 不变 注解放在了 metadata.annotations 而非 spec.template.metadata.annotations 把注解移到 Pod 模板并创建新 Pod。
    资源创建太晚 注解存在，但注入没发生 Instrumentation 在 Pod 创建之后才创建 先建 Instrumentation，再 rollout 工作负载。
    collector 不可达 agent 日志报连接失败，后端没有 span 服务名、命名空间、端口、协议错误，或端点默认到了 localhost 修正 spec.exporter.endpoint；校验 collector 接收端口。
    运行时注解不对 Pod 变更不正确或根本没变 运行时注解与应用语言不符 换成正确的语言注解。
    目标容器选错 Pod 健康，但没有有用的 span 多容器 Pod 未显式指定容器 加 container-names 或按语言的容器名注解。
    后端过载 推广后后端延迟、丢 span 或存储激增 采样过高，或一次启用了太多工作负载 降低采样、暂停推广、校验 collector 队列上限。
    启动退化 Pod 变 Ready 更慢或启动崩溃 agent 镜像/运行时不兼容或资源压力 回滚，并在预发环境测试该运行时镜像/版本。
    敏感属性泄露 链路里出现令牌、请求体、头、用户数据或查询串 默认埋点采集了本应过滤的字段 关闭相关埋点，或在 collector 和应用层过滤。
    Go 多容器失败 Go 工作负载未如期埋点 Go 自动埋点不支持多容器 Pod 该 Pod 形态不用 Go 自动埋点；改手动埋点或拆分工作负载。

    安全说明¶

    • 把遥测出站当作一条数据导出路径。大规模推广前评审目的地、认证、TLS 和保留期。
    • 默认不要采集请求体、完整头、查询串、authorization 头、cookie 或用户标识。
    • 固定或审批自动埋点镜像。被注入的 agent 会在每个已注解 Pod 内运行。
    • 没有针对其 eBPF 和特权 sidecar 模型的安全例外，不要启用 Go 自动埋点。
    • 尽量把后端凭据放在 collector，而不是每个应用工作负载里。
    • 限制谁能修改 Instrumentation 资源和工作负载注入注解。
    • 给 collector 设资源上限并监控队列饱和，使后端故障不会演变成节点压力。
    • 使用安全的资源属性，如服务名、命名空间、环境、团队和版本；避免客户、租户、邮箱或请求级标识。

    复审周期¶

    • 负责人：平台或可观测性团队。
    • 每次 OpenTelemetry Operator 升级时复审。
    • 每次 Kubernetes 小版本升级时复审。
    • 每次启用新的运行时语言时复审。
    • 当链路量、后端成本或 collector 丢弃超过约定阈值时复审。
    • 每当加入高流量服务时重新校验采样。
