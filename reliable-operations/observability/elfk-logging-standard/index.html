    ELFK 日志标准¶

    目的¶

    本标准定义如何运行 ELFK（Elasticsearch、Logstash、Filebeat、Kibana）日志栈，使其保持可用、可负担，且自身不成为故障或数据泄漏的来源。

    本标准的核心判断：

    日志栈以两种可预见的方式失败：它撑满自己的磁盘并转为只读（恰在你最需要时让日志下线），或它悄悄把密钥与个人数据索引进一个可搜索的存储。留存/生命周期与"索引什么"不是调优——它们就是设计。

    适用场景¶

    在以下情况使用本标准：

    • 运行自管的基于 Elasticsearch 的日志栈（ELFK/ELK 或等价的 OpenSearch）。
    • 需要受控的留存、索引生命周期与日志访问。
    • 决定采集什么，以及如何防止存储被撑满。

    非目标¶

    本标准不涵盖：

    • 在 ELFK 与指标式日志系统（Loki）之间的选择——该权衡见 Loki 日志标准。
    • 全文搜索相关性调优或 APM。
    • 超出下述容量安全原则之外的集群容量计算。

    核心原则¶

    1. 使用结构化日志（字段一致的 JSON），使日志可查询，而非只能 grep。
    2. 用索引生命周期管理（ILM）控制留存——每个索引都有明确的滚动与删除策略。
    3. 防范磁盘水位：磁盘撑满时 Elasticsearch 会转为只读，因此容量与留存必须使你保持在水位以下。
    4. 限制 Kibana 与 Elasticsearch API；日志是敏感的。
    5. 不要索引密钥、token、凭据、完整请求体或个人数据。

    推荐标准¶

    结构化日志与一致结构¶

    输出字段稳定的 JSON 日志（时间戳、级别、service、env、trace id）。在 Filebeat 或 Logstash 做解析/富化，而非索引自由文本——非结构化日志会撑爆存储且难以查询。

    用 ILM 控制留存¶

    定义 ILM 策略，使索引按大小/年龄滚动并按计划删除。没有它，索引会一直增长直到触发磁盘水位、集群转为只读。

    PUT _ilm/policy/logs-policy
{
  "policy": {
    "phases": {
      "hot":    { "actions": { "rollover": { "max_size": "50gb", "max_age": "1d" } } },
      "delete": { "min_age": "14d", "actions": { "delete": {} } }
    }
  }
}

    留存来自刻意的决策（成本 vs 你实际回溯多久），而非"全都留着"。

    容量与磁盘水位¶

    Elasticsearch 有磁盘水位（low/high/flood-stage）。到 flood 阶段它会把索引设为只读——一次往往与你正在排查的事故同时发生的日志故障。留出余量：按峰值写入 × 留存规划存储，在远早于水位时对磁盘使用告警，并确保 ILM 删除确实在执行。

    访问控制¶

    • 将 Kibana 置于认证与最小权限角色之后；不要公开暴露。
    • 将 Elasticsearch HTTP API 限制到可信网络/客户端。
    • 日志常含敏感数据——把读取访问当作一种特权。

    校验¶

    curl -sS http://<es-host>:9200/_cluster/health?pretty
curl -sS http://<es-host>:9200/_cat/indices?v
curl -sS http://<es-host>:9200/_cat/allocation?v   # 每节点磁盘使用

    确认：

    • 集群健康为 green（或有正当理由的 yellow），而非 red。
    • ILM 正在按计划滚动并删除旧索引（_cat/indices 显示增长有界）。
    • 磁盘使用在 high 水位之下并有余量。
    • Filebeat/Logstash 正在投递；无大量写入积压。
    • 抽查日志不含密钥或完整敏感载荷。

    已知失败模式¶

    失败模式 症状 可能原因 正确处置
    集群只读 写入停止；出现 read_only_allow_delete 标志 磁盘触发 flood 水位 释放空间/删旧索引；清除该 block；修复留存。
    无界增长 磁盘稳步撑满 无 ILM，或 ILM 删除未运行 应用带滚动+删除的 ILM；验证其执行。
    存储暴涨 成本/磁盘飙升 索引了非结构化/高基数字段 结构化日志；在 Filebeat/Logstash 丢弃噪声字段。
    索引中含密钥 敏感数据可被搜索 索引了请求体/header/token 在投递端过滤；清除受影响索引；修源头。
    Kibana 暴露 未授权访问日志 无认证/开放网络 将 Kibana 置于认证后；限制网络。
    集群红 分片未分配 节点丢失、磁盘或分片分配问题 查 _cat/allocation 与 allocation explain；恢复容量。

    回滚/恢复¶

    1. 恢复只读集群：释放磁盘（删除或迁移旧索引），再清除 read_only_allow_delete block；随后修复 ILM/留存以防复发。
    2. 回滚 ILM/策略变更：恢复上一策略；ILM 向前生效，故按预期留存核对索引年龄。
    3. 对必须留存以合规的索引保留快照备份；从快照恢复，而非仅依赖在线集群。

    安全注记¶

    • 切勿索引密钥、token、凭据、authorization header、完整请求体或个人数据——在到达存储前于 Filebeat/Logstash 过滤。
    • 限制 Kibana 与 ES API；使用 TLS 与认证。
    • 切勿公开真实主机名、endpoint 或索引内容。
    • 把快照当作敏感物（含日志数据）。

    评审周期¶

    • 负责人：平台/可观测性团队。
    • 定期重新确认留存 vs 成本与排查需求。
    • 验证 ILM 滚动/删除确实在运行、磁盘余量稳定。
    • 每接入新日志源时复核索引了哪些字段。
