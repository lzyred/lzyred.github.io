    Prometheus Exporter 监控标准¶

    目的¶

    本标准定义如何加入 Prometheus exporter（Node Exporter、数据库 exporter、Nginx、JVM、blackbox 等），使指标可信、exporter 自身不成为风险，且每个 exporter 都随附让它真正有用的告警。

    本标准的核心判断：

    "能抓到的 exporter"不等于"可信的监控"。没有告警的 exporter 只是装饰；而"exporter 宕了"与"目标宕了"的区别，就是"盲区"与"真实故障"的区别——你的告警必须能区分二者。

    适用场景¶

    在以下情况使用本标准：

    • 为主机、数据库、Web 服务器或应用运行时添加 Prometheus 监控。
    • 跨服务统一 exporter 的部署、打标与加固方式。
    • 需要一条最小告警基线，使新 exporter 真正可处置。

    非目标¶

    本标准不涵盖：

    • Prometheus 服务端容量、联邦或长期存储设计。
    • 仪表盘设计（仅要求存在一个）。
    • 应用级自定义指标埋点（请用 OpenTelemetry/SDK 埋点）。
    • 告警路由与值班策略（见 Prometheus 告警标准）。

    Exporter 原则¶

    1. 让 exporter 靠近目标服务（同主机或 sidecar），使 exporter 健康在本地反映目标可达性。
    2. 数据库/服务 exporter 使用最小权限指标账号——只读统计访问，绝不用管理账号。
    3. 不公开暴露 exporter；将指标端口限制到 Prometheus。
    4. 打上标识 service、env、team、owner 的标签，使告警可路由。
    5. 每个 exporter 都随附仪表盘和告警规则，否则不算完成。

    Exporter 覆盖检查清单¶

    Exporter 需抓取的关键信号
    Node Exporter CPU 饱和、内存压力、磁盘容量与 inode、文件系统/只读错误、网络错误、负载
    MySQL Exporter up、连接数 vs max_connections、复制是否运行与延迟、慢查询、buffer pool
    Redis Exporter up、内存 vs maxmemory、淘汰、客户端、持久化状态、复制
    MongoDB Exporter 连接、锁/争用、复制延迟、存储
    Nginx Exporter up、请求、活动连接、5xx 比例（经 status 端点）
    JVM Exporter 堆使用、GC 暂停/频率、线程、类加载
    Blackbox Exporter HTTP/TCP/DNS 探测成功率，及 TLS 证书到期

    Node Exporter 要点¶

    Node Exporter 是规范的主机 exporter；在每台 Linux 主机运行，并将其端口限制到 Prometheus。最常抓住真实事故的信号：

    • 磁盘容量与 inode（inode 表满会导致写失败，而 df 仍显示有空间）。
    • 只读/故障的文件系统。
    • 内存压力与 OOM 指标。
    • CPU 饱和与持续负载。

    抓取配置模式¶

    在抓取时附上标识标签，使每条 series 可路由、可归属。

    scrape_configs:
  - job_name: node
    static_configs:
      - targets:
          - <exporter-host>:9100
        labels:
          service: example-service
          env: prod
          team: platform

    不要公开真实内部目标或主机名。规模化时优先用服务发现，而非手工维护的静态目标列表。

    必备告警覆盖¶

    在以下告警（按适用）就位之前，新 exporter 不算"已监控"：

    • Exporter down —— exporter 任务的 up == 0。表示你失明了，而非服务宕了。据此 page 或 warn。
    • Target down —— exporter 所观测的服务不可达/不健康。这才是真实故障信号。
    • 磁盘容量与 inode 耗尽。
    • 内存压力。
    • 错误率（5xx、连接失败）超阈值。
    • 复制延迟（数据库）。
    • TLS 证书到期（blackbox）。
    • JVM GC 压力/堆耗尽（JVM 应用）。

    区分 exporter down 与 target down 是最重要的设计点。若二者合成一条告警，你将无法分辨"我们失去了可见性"与"服务正在故障"——并最终会信任一块只因 exporter 死掉而变绿的仪表盘。

    校验¶

    # exporter 在本地提供指标：
curl -sS http://<exporter-host>:9100/metrics | head

    在 Prometheus 中：

    up{job="node"}                      # 1 = 抓取正常，0 = exporter 不可达
rate(node_filesystem_avail_bytes[5m]) # 信号存在且在变化

    确认：

    • 目标在 Prometheus targets 中为 UP 且带预期标签。
    • 该 exporter 类型的关键指标存在且不陈旧。
    • 仪表盘可渲染，告警规则无错加载（promtool check rules）。
    • 故意制造一次抓取失败（在预发短暂停掉 exporter）会触发"exporter down"告警。

    已知失败模式¶

    失败模式 症状 可能原因 正确处置
    仪表盘绿但服务已死 看着正常但用户报错 "exporter down"与"target down"未分离 拆分告警；对 up == 0 单独告警。
    缺标签 告警触发却无人认领 抓取时无 service/team/owner 标签 在 scrape_configs 加标识标签。
    Exporter 成为安全漏洞 指标可从不可信网络访问 exporter 端口公开暴露 绑定私有网卡；端口限制到 Prometheus。
    Exporter 权限过大 数据库 exporter 有写/管理权限 复用了管理账号 创建最小权限只读统计用户。
    指标陈旧 数值冻结、无告警 exporter 在线但未采集 查 exporter 日志与目标连接；对陈旧告警。
    基数爆炸 Prometheus 内存/CPU 飙升 高基数标签（逐请求 ID） 抓取时丢弃/聚合高基数标签。

    回滚/移除¶

    1. 移除 exporter 即移除可见性——移除前确认没有东西依赖它的告警。
    2. 同时移除抓取任务与关联告警规则；引用已消失任务的孤立告警规则是噪声。
    3. 若某 exporter 造成负载或安全问题，先限制其端口或停掉，再干净移除。

    安全注记¶

    • exporter 暴露服务元数据；把指标端点当作内部的，并限制到 Prometheus。
    • 数据库/服务 exporter 使用最小权限只读凭据；存于密钥管理系统。
    • 当指标敏感时使用 TLS 或私有网络。
    • 避免可能把标识（用户 ID、请求 ID）泄漏进指标的高基数标签。

    评审周期¶

    • 负责人：平台/可观测性团队。
    • 每引入一种新 exporter 类型时复核告警基线。
    • 在 Prometheus 或 exporter 升级后重新校验 exporter-down 与 target-down 的拆分。
    • 添加或轮换数据库 exporter 时复核指标账号权限。
