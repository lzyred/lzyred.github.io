    DevOps 能力地图与水平自评¶

    DevOps 不是“会写脚本的人”，也不是“会用 Kubernetes 的运维”。它是一套把软件交付、生产运行、可靠性、安全治理和组织反馈连接起来的工程能力系统。

    这个知识库从三个问题开始：

    1. DevOps 到底由哪些层级能力组成？
    2. DevOps 的价值如何定义，成熟度如何评级？
    3. 如何判断自己现在只是 Ops、初级 DevOps，还是已经接近架构师或专家？

    1. DevOps 多层级技能拓扑¶

    DevOps 能力不是工具清单，而是从基础设施到组织反馈的多层能力栈。工具只是实现手段，真正有价值的是每一层能否产生可验证的工程结果。

    flowchart TB A["基础层<br/>Linux / 网络 / 存储 / 安全基线"] --> B["运行层<br/>容器 / Kubernetes / 云资源 / 中间件"] B --> C["交付层<br/>Git / CI / CD / 制品 / 环境治理"] C --> D["可靠性层<br/>SLO / 容量 / 变更风险 / 灾备 / 回滚"] D --> E["可观测层<br/>指标 / 日志 / 链路 / 告警 / 复盘"] E --> F["平台层<br/>自助化 / 标准化 / 多团队复用 / 成本治理"] F --> G["组织反馈层<br/>度量 / 责任边界 / 决策复盘 / 工程文化"] A -.支撑.-> D C -.约束.-> D E -.反馈.-> C G -.改进.-> F
    能力域 核心问题 典型产物 能力证据
    基础设施 系统是否可安装、可加固、可恢复？ 基线镜像、网络规划、存储策略、访问控制 能解释一次主机或网络故障如何定位和恢复
    容器与云原生 服务是否可移植、可调度、可扩缩？ Dockerfile、Helm Chart、Kubernetes 集群、云资源模块 能设计生产级发布、扩容、回滚和隔离策略
    交付工程 变更是否可重复、可审计、可回滚？ CI/CD Pipeline、制品仓库、环境 promotion、发布门禁 能降低发布失败率，而不是只把命令自动化
    可靠性工程 系统失败时是否有边界、有预案、有恢复路径？ SLO、容量模型、降级方案、故障演练、复盘机制 能把事故转化为结构性改进
    可观测性 出问题时能否快速定位，而不是只有“监控”？ 指标、日志、链路、告警分级、值班面板 能减少噪音告警，提高定位速度
    平台工程 多团队是否能用同一套标准交付和运维？ 自助平台、黄金路径、模板、权限模型、成本报表 能让团队少找人、多自助、少犯错
    安全与治理 速度是否建立在可控风险上？ Secrets 管理、审计、供应链扫描、最小权限、合规检查 能把安全前置到交付流程，而不是上线前补锅

    一个可靠的 DevOps 工程师不一定每层都最强，但必须知道每层之间的依赖关系。只会工具、不懂交付风险；只会交付、不懂生产可靠性；只会平台、不懂团队反馈，都会在生产环境里暴露短板。

    2. DevOps 的价值、定义与成熟度评级¶

    定义：DevOps 是一种工程化协作模式，用自动化、标准化、可观测性和反馈机制，把“开发交付”与“生产运行”连接成一个持续改进系统。

    它的价值不在于“用了多少工具”，而在于是否稳定改善下面四类结果：

    flowchart LR A["更小批量变更"] --> B["更快交付"] B --> C["更低失败率"] C --> D["更快恢复"] D --> E["更可信的业务节奏"] E --> A
    价值维度 低成熟度表现 高成熟度表现 可量化指标
    交付速度 手工发布、靠人记步骤 Pipeline 标准化、环境晋级清晰 部署频率、交付周期
    变更安全 发布靠经验，失败后临场处理 灰度、回滚、变更冻结、发布门禁 变更失败率、回滚耗时
    恢复能力 故障靠个人英雄主义 有 SLO、告警分级、演练和复盘 MTTR、故障影响时长
    系统透明度 有监控但定位慢 指标、日志、链路和业务上下文联动 MTTD、告警有效率
    成本效率 资源长期闲置，无归属 容量模型、成本归因、弹性策略 单位业务成本、闲置资源比例
    安全治理 凭证散落、上线后补扫描 Secrets、审计、供应链、安全门禁前置 高危漏洞修复周期、权限漂移数量

    成熟度五级¶

    级别 状态 特征 最大风险
    L1 工具化 有脚本和工具 能减少重复操作，但依赖个人 自动化了错误流程
    L2 流程化 有标准流程 发布、回滚、审批、排障开始可复用 流程存在，但执行质量不稳定
    L3 平台化 有共享能力 多团队能用统一模板和平台交付 平台变成新的工单入口
    L4 度量化 有工程指标 能用数据评估交付、可靠性、成本和安全 只看指标，不看真实风险
    L5 自适应 有反馈闭环 事故、成本、交付数据会反向改进平台和流程 组织复杂度上升，需要治理能力

    真正的 DevOps 成熟度不是“有没有平台”，而是“平台、流程和团队行为是否持续降低生产风险”。

    3. 如何量化评价自己的 DevOps 水平¶

    先区分角色边界：不是所有运维工作都叫 DevOps，也不是所有 DevOps 都必须写平台。关键在于你是否能把“系统运行”与“变更交付”连接起来，并用工程方法持续改进。

    水平分层¶

    水平 常见状态 能独立完成什么 还缺什么
    不属于 DevOps 只接触应用使用或简单部署 按文档操作少量工具 缺系统、网络、发布、故障处理基础
    未达 Ops 会执行命令，但不知道影响面 重启服务、复制文件、看简单日志 缺 Linux、网络、权限、备份恢复能力
    Ops 能维护生产系统 主机、服务、日志、备份、常规故障处理 缺交付自动化、平台化、可靠性设计
    初级 DevOps 能把重复操作脚本化 基础 CI/CD、容器构建、简单部署 缺生产风险意识和系统性回滚设计
    中级 DevOps 能负责一条交付链路 Pipeline、Kubernetes 部署、监控告警、常规排障 缺跨团队标准化、容量、安全、成本治理
    高级 DevOps 能设计生产级方案 高可用架构、灰度发布、SLO、事故复盘、平台模板 缺组织级架构取舍和长期演进能力
    DevOps 架构师 能定义多团队标准 平台架构、治理模型、黄金路径、迁移策略 需要更强业务约束、成本和组织影响判断
    DevOps 专家 能处理复杂系统与组织问题 识别错误路线、建立反馈闭环、用复盘推动体系升级 风险是脱离一线细节，变成只讲原则

    自评分矩阵¶

    给每个维度打 0 到 4 分：

    • 0：不了解，无法独立判断。
    • 1：能按文档操作。
    • 2：能独立交付常规任务。
    • 3：能设计方案并处理异常。
    • 4：能定义标准、指导他人，并用数据持续改进。
    维度 0-1 分表现 2 分表现 3-4 分表现
    Linux / 网络 / 存储 只会常用命令 能排查常见系统问题 能做容量、隔离、恢复和安全基线
    容器 / Kubernetes / 云 会部署示例 能维护常规集群和服务 能设计生产级高可用、升级和故障恢复
    CI/CD / 制品 / 环境 能跑流水线 能维护发布流程 能设计安全、可回滚、可审计的交付体系
    可观测性 / 故障处理 会看日志和图表 能定位常见故障 能设计告警分级、SLO 和复盘闭环
    安全 / 合规 / 成本 知道基本概念 能执行扫描和权限配置 能把治理嵌入平台与交付流程
    平台化 / 标准化 / 影响力 只服务自己 能沉淀脚本和模板 能服务多团队并量化平台价值

    分数解释¶

    总分 判断 建议
    0-6 未达 Ops 或刚入门 先补 Linux、网络、权限、日志、备份和故障处理
    7-12 Ops 到初级 DevOps 把重复操作脚本化，建立基础 CI/CD 和容器化能力
    13-18 中级 DevOps 重点补可靠性、回滚、可观测性和安全前置
    19-22 高级 DevOps 开始做跨团队标准、平台抽象、容量和成本治理
    23-24 架构师 / 专家候选 用真实案例、事故复盘和指标证明体系性影响

    判断自己是否在进步，不要只看“学了几个工具”。更硬的证据是：你是否让发布更安全、恢复更快、故障更少、成本更清楚、团队更少依赖个人经验。
