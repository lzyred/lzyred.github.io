    EKS and Kubernetes Upgrade Standard¶

    Purpose¶

    This standard describes a safe process for upgrading a managed or self-managed Kubernetes cluster. It is written for public reuse and does not depend on any private environment.

    When To Use¶

    Use this procedure when:

    • Upgrading Amazon EKS, self-managed Kubernetes, or a Kubernetes-compatible platform.
    • Moving workloads from an older minor version to a newer supported minor version.
    • Replacing worker node groups during a platform upgrade.
    • Upgrading core add-ons such as CNI, CoreDNS, kube-proxy, ingress controllers, CSI drivers, monitoring, or logging agents.

    Upgrade Principles¶

    1. Upgrade one minor version at a time unless the platform explicitly supports a larger jump.
    2. Treat add-ons as part of the cluster, not as optional extras.
    3. Upgrade control plane, add-ons, and nodes in a controlled sequence.
    4. Use new node groups for workload migration instead of in-place mutation when possible.
    5. Validate both Kubernetes health and application behavior before declaring the upgrade complete.
    6. Keep rollback paths available until post-upgrade monitoring is stable.

    Pre-Upgrade Assessment¶

    Collect this baseline before making changes:

    kubectl version --short
kubectl get nodes -o wide
kubectl get pods -A
kubectl get deployment,statefulset,daemonset -A
kubectl get ingress -A
kubectl get storageclass
kubectl get pvc -A
kubectl top nodes
kubectl top pods -A

    Review these compatibility areas:

    Area Check
    Kubernetes APIs Deprecated APIs removed in the target version.
    Add-ons CNI, CoreDNS, kube-proxy, CSI drivers, ingress controller, cert-manager, metrics-server.
    Workloads Pod disruption budgets, probes, resource requests, priority classes, security contexts.
    Storage CSI driver compatibility, PVC binding behavior, snapshot and backup support.
    Networking Ingress class behavior, load balancer annotations, service type changes, network policies.
    Observability Prometheus scrape targets, alert rules, dashboards, log collectors.
    Access IAM roles, RBAC, service accounts, cluster admin access, break-glass access.

    Standard Upgrade Sequence¶

    1. Freeze Risky Changes¶

    Pause unrelated application deployments and infrastructure changes during the upgrade window. Communicate the maintenance window, expected impact, rollback criteria, and owner contacts.

    2. Back Up Critical State¶

    For managed clusters, export manifests and configuration:

    kubectl get all -A -o yaml > cluster-workloads.snapshot.yaml
kubectl get configmap,ingress,svc,pvc,storageclass -A -o yaml > cluster-config.snapshot.yaml

    Do not export raw Kubernetes Secret values into plain text files. Back up secrets through the platform-approved encrypted backup process or secret management system.

    For self-managed clusters, back up etcd using the platform-recommended encrypted method.

    3. Upgrade Control Plane¶

    Upgrade the control plane through the cloud provider or cluster management tool. Wait until the API server is healthy and all core control plane checks pass.

    Validation:

    kubectl get --raw='/readyz?verbose'
kubectl get componentstatuses
kubectl get nodes

    4. Upgrade Core Add-ons¶

    Upgrade add-ons to versions compatible with the target Kubernetes version:

    • CNI plugin.
    • CoreDNS.
    • kube-proxy.
    • CSI drivers.
    • Metrics server.
    • Ingress controller.
    • Observability agents.

    Validation:

    kubectl get pods -n kube-system
kubectl rollout status deployment/coredns -n kube-system
kubectl get daemonset -n kube-system

    5. Create New Worker Node Group¶

    Create a new node group with the target Kubernetes-compatible image, runtime, IAM role, labels, taints, and scaling configuration.

    Validation:

    kubectl get nodes -L kubernetes.io/os,kubernetes.io/arch
kubectl describe node <new-node-name>

    6. Drain Old Nodes Gradually¶

    Drain old nodes in small batches:

    kubectl cordon <old-node-name>
kubectl drain <old-node-name> --ignore-daemonsets --delete-emptydir-data --timeout=10m

    After each batch, validate application health before continuing.

    7. Validate Workloads¶

    Check:

    kubectl get pods -A
kubectl get events -A --sort-by='.lastTimestamp'
kubectl get ingress -A
kubectl get pvc -A

    Application validation should include:

    • HTTP health checks.
    • Login or core API smoke tests.
    • Background job processing.
    • Database connection checks.
    • External callback or webhook checks.
    • Alerting and dashboard visibility.

    8. Decommission Old Node Group¶

    After the observation window passes, scale down and remove the old node group. Keep snapshots and upgrade notes for review.

    Rollback Strategy¶

    Managed Kubernetes control plane upgrades are often not directly reversible. Practical rollback usually means:

    • Stop node migration.
    • Keep or recreate old node groups if still supported.
    • Roll back add-ons to compatible versions when safe.
    • Roll back application deployments.
    • Restore from backups if stateful components are affected.

    Define rollback triggers before starting:

    • Core business smoke test fails.
    • Error rate exceeds threshold for more than 10 minutes.
    • Critical pods cannot schedule.
    • Storage attach or mount errors affect stateful workloads.
    • Ingress or load balancer behavior breaks external traffic.

    Post-Upgrade Review¶

    Record:

    • Source and target versions.
    • Add-on versions before and after.
    • Node group changes.
    • Incidents and mitigations.
    • Deprecated API findings.
    • Follow-up platform work.
