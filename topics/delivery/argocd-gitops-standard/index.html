    Argo CD GitOps 标准¶

    目的¶

    本标准定义如何将 Argo CD 作为 Kubernetes 的 GitOps 部署控制器运行：什么应放入 Git，让 Argo CD 以多强的力度进行协调，以及如何同时恢复应用和 Argo CD 自身。

    本标准的核心判断：

    GitOps 把风险从"谁执行了命令"转移到"Git 里是什么、以多快协调"。selfHeal 与 prune 不是便利开关——它们决定了一次错误提交或一次手工热修是自愈，还是在你不知情时被回退。要按风险分级、刻意地选择它们。

    健康的 Argo CD 体系有两个时钟：Git 里的人工评审时钟，以及集群里的协调时钟。多数 GitOps 事故都发生在把这两个时钟当成一回事的时候。被评审过的提交仍然可能是错的，而一个快速协调器能在任何人看完 diff 之前，把一行错误扩散成整组服务故障。

    适用场景¶

    在以下情况使用本标准：

    • 部署到 Kubernetes，并希望 Git 成为集群状态的唯一可信源。
    • 已运行或计划运行 Argo CD 作为协调控制器。
    • 需要一套站得住脚的同步策略、密钥处理模型与 GitOps 恢复方案。

    非目标¶

    本标准不涵盖：

    • CI（构建与测试制品）。GitOps 是 CD 的一半；CI 的一半见 Jenkins-to-Kubernetes 流水线标准。
    • 清单模板选型（Helm vs Kustomize vs 纯 YAML），仅涉及其与同步的交互。
    • 超出此处单 Application 模型的多集群机群设计与 ApplicationSets 生成器。

    GitOps 原则¶

    1. Git 是期望状态。集群变更通过评审过的提交流动，而非 kubectl edit。
    2. 密钥通过密钥管理流程引用，绝不明文提交。
    3. 每个 Application 都有与其风险分级匹配的刻意同步策略。
    4. 漂移可见并按策略处理，而非到处静默覆盖。
    5. 通知暴露状态，而非 webhook 密钥或 token。

    Application 发布契约¶

    一个 Application 被 Argo CD 管理为生产应用之前，先写清它隐含的发布契约：

    契约项 必须决策
    事实源 哪个 repo、branch、path、chart 或生成清单可以定义生产状态。
    评审权限 谁可以批准该路径的变更，以及批准者是否承担生产影响。
    目标边界 这个 Application 可触达哪个集群、命名空间、资源类型和 AppProject 白名单。
    协调速度 同步是手动、自动、自愈，还是允许 prune 删除资源。
    应急路径 如何停止协调、revert Git，或做临时修复而不是与控制器对抗。
    密钥边界 哪套外部密钥机制是权威，哪些内容绝不能进入 Git。
    恢复证据 哪些状态、健康、指标和用户路径检查能证明新的期望状态是安全的。

    只要有一行说不清，这个 Application 就还不适合生产自动同步。GitOps 不会消除发布责任；它只会更快暴露责任缺口。

    安装模型¶

    使用官方清单或 Helm chart。生产环境：

    • 以高可用模式运行 Argo CD。
    • 将 Argo CD 放在独立命名空间，与应用命名空间分离。
    • 通过 RBAC 限制谁可创建/修改 Application 与 AppProject 资源。
    • 将 AppProject 限定到各团队可部署的仓库、目标与资源类型——这才是真正的爆炸半径控制。
    • 集群凭据、仓库凭据和管理员 token 必须放在批准的密钥管理路径中，不要进入导出示例或共享 runbook。
    • 把 Argo CD 升级当成生产交付系统变更：先备份配置、检查 release notes、避开高风险发布窗口，并在控制器重启后验证现有 Applications。

    校验：

    kubectl get pods -n argocd
kubectl get applications -n argocd
argocd app list

    按风险分级的同步策略¶

    这是核心决策。按服务匹配策略，而非按操作者的便利。

    分级 automated selfHeal prune 适用
    生产/有状态 关（手动同步） 关 关 高风险服务；每次同步由人审批。
    标准服务 开 开 关 多数无状态服务；自动纠正漂移，删除保持手动。
    成熟、充分验证 开 开 开 评审与回滚很强；Git 完全掌管生命周期含删除。

    手动同步不是不成熟的标志；对有数据状态或共享基础设施的服务，它往往是正确做法，因为人工审批时钟必须慢于控制器时钟。自动同步只有在团队已经在非顺风场景中证明了验证和回滚闭环后才合适。

    apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: example-service
  namespace: argocd
spec:
  project: example-team
  source:
    repoURL: https://git.example.com/example/example-repo.git
    targetRevision: main
    path: deploy/kubernetes
  destination:
    server: https://kubernetes.default.svc
    namespace: example-namespace
  syncPolicy:
    automated:
      prune: false      # 该分级下删除保持手动
      selfHeal: true    # 回退带外漂移

    启用前需理解的两个开关：

    • selfHeal: true 会回退任何直接对集群所做的变更。这正是其目的——但也意味着一次应急 kubectl 热修会在几分钟内被撤销。正确的应急路径是提交，或临时关闭自动同步，而不是与控制器对抗。
    • prune: true 允许 Argo CD 删除从 Git 中消失的资源。对生命周期整洁很有用，但若有人误删清单则很危险。仅用于评审很强的服务。

    密钥处理¶

    明文密钥绝不入 Git。选定一种模型并一致应用：

    • Sealed Secrets（在 Git 中加密，集群内解密），或
    • 从密钥管理系统拉取的外部 secret operator，或
    • 从 vault 挂载的 CSI secrets 驱动。

    Argo CD 的 Application 清单本身包含仓库 URL、项目名与集群引用；把导出物当作敏感拓扑对待。

    漂移、回滚与恢复¶

    漂移不一定总是坏事。一次直接集群变更可能是应急补丁、攻击行为、控制器故障，也可能是人为绕过。运行问题在于：谁有权决定 Git 和在线集群谁赢。

    对生产 Application：

    • 只忽略已知由控制器持有、噪声较大或可安全外部化的字段。
    • 即使 selfHeal 会纠正，也要对生产意外漂移告警。
    • 记录谁关闭了自动同步、为什么关闭、什么时候必须恢复。
    • 保留比正常访问更窄的 break-glass 路径；应急访问不能变成第二套部署系统。

    回滚应用¶

    由于 Git 是唯一可信源，回滚是一次 Git 操作：

    # 首选：revert 该提交，让 Argo CD 协调。
git revert <bad-commit> && git push

    对于启用了自动同步或 self-heal 的 Application，不要从 argocd app rollback 开始。Argo CD 不能在自动同步启用时执行 Application 回滚，而 self-healing 控制器可能把在线集群重新拉回坏的 Git 状态。先选择以下路径之一：

    • revert 或修正 Git，让 Argo CD 协调到恢复状态，或
    • 通过平台批准路径暂停自动同步，并记录 Owner、原因和恢复期限。

    只有事实源已被控制后，才把直接回滚作为应急桥接：

    argocd app history example-service
argocd app rollback example-service <revision>

    直接 argocd app rollback 是应急桥接，不是最终状态。只有 Git 与恢复状态一致，并且应用通过服务级验证后，才重新启用自动化。

    Argo CD 自身的灾难恢复¶

    Argo CD 是生产依赖；要为失去它做准备。

    导出物包含敏感部署拓扑。只能导出到批准的加密/内部备份路径，并使用受限权限；不要把原始导出物附到工单、提交到仓库或随意共享。

    umask 077
argocd admin export > <encrypted-internal-backup-path>/argocd-export-<yyyy-mm-dd>.yaml
# 恢复：
argocd admin import - < <encrypted-internal-backup-path>/argocd-export-<yyyy-mm-dd>.yaml

    由于期望状态在 Git 中，重建的 Argo CD 加上 Git 仓库即可重建集群——但前提是 Application/AppProject 定义也被捕获（在 Git 或导出物中）。

    不要把导出物当成普通备份文件。它可能暴露仓库、项目、集群目标和运行拓扑。需要在隔离环境中测试 import，再把它作为事故恢复依据。

    校验¶

    argocd app get example-service
kubectl get applications -n argocd

    确认：

    • 变更的应用同步状态为 Synced、健康为 Healthy。
    • 已同步版本与目标 Git 提交一致。
    • 同步后没有意外的 OutOfSync 资源（漂移）。
    • 对启用 prune 的应用，没有删除你不打算删除的内容。
    • AppProject 白名单仍然符合目标团队边界。
    • 服务级冒烟测试或合成检查在同步后通过，而不只是 Argo CD health 绿色。
    • 通知路由到拥有发布的团队，以及拥有平台控制器的团队。

    已知失败模式¶

    症状 可能原因 检查/处置
    应用卡在 OutOfSync 在线漂移、被忽略的差异或生成清单不匹配 argocd app diff；协调，或刻意添加 ignoreDifferences。
    同步失败 RBAC、缺命名空间、缺 CRD、准入策略 读同步错误；创建前置条件；检查 AppProject 白名单。
    应急修复反复被回退 selfHeal 在回退带外变更 提交修复，或临时关闭自动同步；不要与控制器对抗。
    健康一直 Unknown 自定义资源缺健康评估 添加资源健康自定义。
    仓库连接失败 凭据、网络或未知 host key 重新检查仓库凭据与连通性。
    意外删除 prune: true 加上被删除/误生成的清单 通过 Git revert 恢复；重新评估该分级是否该用 prune。
    手工回滚反复被撤销 Git 仍指向坏的期望状态 先 revert 或修正 Git，再恢复自动化。
    控制器故障阻塞交付 Argo CD 依赖未纳入发布连续性计划 使用已文档化的应急路径，然后恢复控制器状态并重新协调 Git。

    回滚（操作小结）¶

    1. 优先对问题提交执行 git revert；让 Argo CD 协调回良好状态。
    2. 如果启用了自动同步或 self-heal，先 revert 或修正 Git，或带 Owner、原因和期限暂停自动化，再执行任何直接回滚。
    3. 在事实源受控后，为求应急速度，可 argocd app rollback 到已知良好版本，随后让 Git 与之对齐。
    4. 若问题出在 Argo CD 本身，关闭关键应用的自动同步以冻结集群，同时恢复控制器。
    5. 从受保护导出物恢复 Argo CD；验证应用回到 Synced/Healthy。

    安全注记¶

    • 切勿明文提交密钥、webhook URL 或聊天 token。
    • 将 AppProject 限定到允许的仓库、目标与资源类型——以界定爆炸半径。
    • 使用 SSO 与最小权限 RBAC；限制谁可同步生产级应用。
    • 把 argocd admin export 输出当作敏感内部拓扑；加密存放。
    • 固定 Argo CD 版本并及时打补丁；它拥有集群级协调能力。
    • 把 Application 清单当成生产变更授权材料评审，而不是无害的部署元数据。

    评审周期¶

    • 负责人：平台/交付团队。
    • 在重大变更时重新评审每个 Application 的风险分级与同步开关。
    • 按周期演练 Argo CD 灾难恢复（export/import），而非停留在纸面。
    • 当团队的部署范围变化时，重新检查 AppProject 白名单。
    • 把关闭自动同步和 drift exception 作为周期性运行债务清单评审。
