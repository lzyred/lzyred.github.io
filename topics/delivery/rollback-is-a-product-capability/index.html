    回滚是一种产品恢复能力¶

    问题：镜像回退不等于业务恢复¶

    kubectl rollout undo 能把工作负载切回历史 Pod template，但坏变更可能已经越过了容器边界。数据库已经写入新结构、队列中已有新格式消息、外部通知已经发送、GitOps 仍声明新版本时，回退镜像甚至可能制造第二次故障。

    mindmap root((产品恢复能力)) 应用产物 镜像与二进制 Chart 与依赖 配置 ConfigMap 与 Secret Feature Flag 数据库 Schema 已写入数据 消息与异步任务 事件格式 队列与消费者偏移 缓存与索引 缓存键结构 搜索索引 外部副作用 支付 通知 Webhook 客户端兼容 API 契约 移动端版本 GitOps 事实源 Git 流水线 IaC 状态 业务结果 关键用户路径 数据完整性 SLO

    图中的每个分支都可能成为权威状态。只有镜像恢复而其他分支仍停留在新状态，系统就处于“代码旧、事实新”的分叉状态。

    核心定义¶

    回滚能力，是系统在坏变更已经部分生效后，仍能选择最小风险动作，把关键用户路径恢复到已知可接受状态的能力。

    可选动作不是一个，而是五类：

    动作 解决什么 典型前提
    Rollback 恢复旧产物或旧配置 旧版本仍兼容当前数据和协议
    Disable 停止新功能继续写入或触发副作用 Feature Flag 同时控制读写和外部调用
    Recover 从健康副本或备份恢复权威状态 RPO/RTO 与恢复点已获批准
    Compensate 纠正已经发生的外部或业务事实 操作幂等、可审计、可重放
    Forward Fix 保留当前状态并用最小修复恢复服务 反向迁移风险高于向前修复

    决策机制¶

    flowchart TD A[坏变更被确认] --> B{能立即停止继续扩散吗} B -- 可以 --> C[关闭写入 Flag 或摘流] B -- 不可以 --> D[冻结变更并升级事故等级] C --> E{旧版本能读取当前数据与消息吗} E -- 能 --> F{是否已发生外部副作用} E -- 不能 --> G{是否有满足 RPO 的健康副本或备份} F -- 没有 --> H[Rollback] F -- 有 --> I[Rollback 或 Disable + Compensate] G -- 有 --> J[Recover] G -- 没有 --> K[Forward Fix] H --> L[同步修正 GitOps 事实源] I --> L J --> L K --> L L --> M[以用户路径和数据检查验证]

    判断顺序有三个重点：

    1. 先止损：关闭写入、摘流或暂停消费者，避免状态继续变化。
    2. 再判断兼容性和权威状态：旧代码是否能读新数据，哪个副本或事实源最可信。
    3. 最后选择动作：恢复动作必须同时修改运行时和声明式事实源，并以业务结果验证。

    完整案例：为什么只回滚镜像会失败¶

    假设订单服务 v2 已发布：

    • 数据库采用 expand-only Schema，新增字段仍与 v1 兼容；
    • v2 已向队列发送新格式消息；
    • 新消费者处理后向客户发送通知；
    • GitOps 仓库仍声明 v2；
    • 支付未发生，但通知已经发送。
    sequenceDiagram participant G as GitOps participant A as 应用 v2 participant D as 数据库 participant Q as 队列 participant N as 通知系统 G->>A: 部署 v2 A->>D: 写入兼容扩展字段 A->>Q: 发送新格式事件 Q->>N: 触发通知 Note over N: 外部事实已发生 A-->>G: 人工只回滚镜像 G->>A: 再次纠正为 v2

    正确处理不是单独执行 rollout undo，而是：

    1. 先关闭产生新事件的写入 Flag；
    2. 修改 GitOps 事实源为已批准版本；
    3. 确认 v1 能读取扩展 Schema；
    4. 让兼容消费者排空或隔离新格式事件；
    5. 对错误通知执行明确的客户补偿流程，而不是“撤回部署”；
    6. 验证下单、查询、消息积压和通知状态。

    边界与不可逆点¶

    以下边界越过后，直接回滚通常不再是默认动作：

    • destructive schema migration 已删除旧字段或约束；
    • 新版本写入了旧版本无法解析的数据；
    • 新协议已经被不可控客户端采用；
    • 支付、退款、邮件、短信或第三方资源创建已经发生；
    • 多个数据源各自包含独有写入；
    • GitOps、Terraform 或发布控制器仍会重建坏状态。

    安全设计应在发布前建立：

    • 不可变产物和可追溯版本；
    • expand-contract 数据迁移；
    • 新旧 API/消息格式重叠期；
    • 同时控制读写和副作用的 Feature Flag；
    • 幂等键和补偿动作；
    • 与运行时一致的 GitOps/IaC 变更；
    • 以用户路径、数据完整性和 SLO 为核心的恢复验证。

    反例：什么时候不要回滚¶

    场景 错误动作 更安全的方向
    新版本写入不可逆格式 强制启动旧版本 Forward Fix 或数据恢复
    AOF/数据库状态权威不明 多个节点同时恢复写入 冻结写入、保存证据、确定权威源
    已完成第三方支付 回滚代码后重试请求 停止扩散并执行幂等补偿
    GitOps 仍指向坏版本 手工改 Deployment 先修正事实源，再恢复运行时
    小修复可在几分钟内完成 跨系统反向迁移 受控前滚修复

    可执行下一步：发布前恢复合同¶

    每个发布必须在上线前回答：

    1. 哪些组件可以 Rollback，哪些只能 Disable、Recover、Compensate 或 Forward Fix？
    2. 旧版本能否读取新版本可能写入的全部数据和事件？
    3. 哪个动作会越过不可逆边界？
    4. GitOps、配置、数据库和消息的权威事实源分别是什么？
    5. 哪些用户路径、数据抽样和运行信号证明恢复？
    6. 什么条件必须停止尝试并升级为数据或业务事故？

    无法回答时，不应把该发布标记为“可回滚”。
