    JumpServer 堡垒机访问标准¶

    目的¶

    本标准定义如何把 JumpServer 或堡垒机平台作为特权基础设施访问的受控入口。

    核心判断：

    堡垒机不是登录捷径，而是特权操作的审计边界。

    如果工程师通过它访问服务器、数据库、Kubernetes 节点、网络设备或内部控制台，堡垒机必须保留下五个问题的答案：

    • 谁操作了？
    • 碰了哪个具体资产？
    • 使用了什么权限？
    • 留下了什么证据？
    • 压力场景下如何撤销访问？

    一个只集中 SSH 流量，但仍保留直连旁路、共享账号、无人评审录屏和永久临时用户的堡垒机，不是访问控制系统，只是登录页更整齐的绕行通道。

    适用场景¶

    适用于：

    • 工程师需要 SSH、RDP、数据库或 Web 控制台访问私有资产。
    • 希望移除或减少服务器直连。
    • 运维操作需要会话录制。
    • 供应商或临时用户需要受控访问。
    • Break-glass 访问必须存在，但不能摧毁问责。
    • 资产 Owner 需要访问评审和事故调查证据。

    非目标¶

    本标准不批准共享 root 访问、无人管控的直连 SSH、永久供应商访问，也不把堡垒机当成最小权限云 IAM 的替代品。它也不定义 JumpServer 的每个功能，而是定义堡垒机在承载生产资产前必须满足的访问治理契约。

    生产契约¶

    生产资产接入前，先定义：

    契约 必须决策 为什么重要
    资产 Owner 每台服务器、数据库、集群、网络设备或控制台由谁负责？ 没有资产 Owner 的访问审批没有权威。
    身份来源 哪个身份系统是用户与组的权威来源？ 本地遗留用户会削弱离职回收和事故响应。
    访问类型 哪些访问是日常、特权、供应商或应急？ 不同类型需要不同评审频率和证据要求。
    权限边界 每个角色能使用哪些命令、账号、数据库或控制台？ 登录路径不等于受控权限。
    直连策略 哪些 SSH、RDP、数据库或控制台直连已关闭或批准例外？ 开放直连会绕过堡垒机证据链。
    会话证据 什么必须录制、留存、保护并可评审？ 审计边界需要受保护证据，而不只是日志。
    临时访问 谁审批供应商或限时访问，如何强制过期？ 临时访问如果不自动或定期失效，会自然变成永久访问。
    Break-glass 路径 JumpServer 或身份系统不可用时如何应急访问？ 事故需要入口，但 fallback 必须比正常路径更窄。
    撤销 Owner 谁能快速撤销访问、轮转凭据、终止会话？ 撤销慢，访问治理就只是流程文档。
    平台恢复 JumpServer 配置、录屏、密钥和资产映射如何恢复？ 丢失堡垒机也可能丢失证据和访问 inventory。

    推荐标准¶

    把资产当作治理 inventory¶

    每个资产需要：

    • Owner。
    • 环境。
    • 资产类型。
    • 重要性。
    • 允许访问方式。
    • 批准用户组。
    • 特权账号映射。
    • 应急 Owner。
    • 评审周期。
    • 直连例外状态。
    • 退役 Owner。

    Owner 未知的资产不应为了方便暴露到堡垒机。

    关闭或分类直连路径¶

    普通访问都经过堡垒机时，它才可能成为审计边界。

    每类资产都要决定直连访问是否允许：

    路径 默认策略 必需证据
    从工程师网络 SSH/RDP 关闭 Network policy、security group、firewall 或 VPN 规则。
    数据库客户端访问 关闭或明确限定 来源范围、用户、角色、timeout 和 query logging。
    Kubernetes node 访问 除 break-glass 外关闭 Owner、审批、事故链接和事后 reconcile。
    网络设备管理入口 受限 批准的管理网络和应急路径。
    云控制台或内部控制台 身份治理 MFA、角色映射和审计日志留存。

    如果直连路径必须保留，记录 Owner、原因、范围、过期或评审日期、补偿证据和撤销方式。永久且无记录的例外不是控制，而是绕行。

    分离角色与评审职责¶

    最低角色模型：

    • 平台管理员。
    • 审计员。
    • 资产 owner。
    • 操作员。
    • 临时供应商或应急用户。

    管理员不应是唯一能评审自己会话的人。授权访问的人，也不应是唯一判断录制操作是否可接受的人。

    访问必须映射到具名身份。旧系统可能仍有共享特权账号，但堡垒机必须把共享账号的每次使用绑定到具体人、审批上下文和录制会话。

    记录并评审会话¶

    特权会话应记录和录制。如果法律、策略或工具限制导致高风险路径无法完整录制，访问应默认拒绝；除非存在批准的替代证据路径：命令审计、数据库审计日志、事故或变更工单、独立评审人，以及能证明操作符合审批的留存证据。

    录屏与审计日志要作为证据保护。它们应有留存期、访问控制、可用时的完整性保护、备份和删除规则。如果任何平台管理员都能无痕编辑或删除录屏，这份证据就不可靠。

    评审触发：

    • 生产 root 或 sudo 访问。
    • 数据库管理员访问。
    • Kubernetes node 访问。
    • Firewall、route、VPN、DNS 或 load balancer 变更。
    • 应急访问。
    • 供应商访问。
    • 失败登录激增。
    • 维护窗口外访问。
    • 会话录制失败。
    • 直连例外被使用。

    治理临时与供应商访问¶

    临时访问需要：

    • requester
    • 业务或事故原因
    • 批准的资产 Owner
    • 精确资产范围
    • 允许协议和账号
    • 开始与结束时间
    • 会话录制要求
    • 撤销 Owner
    • 使用后的评审证据

    不要把日历提醒当成控制。使用过期、组移除，或在评审缺失/未通过时默认移除访问的定期评审。

    Break-glass 必须比正常访问更窄¶

    Break-glass 是为事故准备的，不是为了方便。它必须定义：

    • 谁能启用
    • 什么条件允许启用
    • 能到达哪些资产
    • 使用哪种凭据或路径
    • 堡垒机不可用时如何记录行为
    • 谁会收到通知
    • 事后如何轮转凭据
    • 恢复后如何 reconcile 变更

    应急路径应该小于正常常驻访问。如果 break-glass 比堡垒机更宽、更安静、存活更久，它就会在压力场景下变成真正的访问模型。

    验证¶

    接入资产前：

    • 资产 Owner、重要性、环境和评审周期已记录。
    • 用户只能访问批准资产和账号。
    • 直连路径已关闭或批准例外。
    • 会话录制有效，审计员能取回录屏。
    • 支持时命令审计有效。
    • 撤销能从堡垒机和底层账号移除访问。
    • 临时访问无需人工清理也会过期。
    • 应急路径已记录、测试，并在使用后 reconcile。
    • 审计日志和录屏经过平台重启与备份恢复测试。

    运营测试应覆盖每一种批准访问方式：

    访问方式 测试
    SSH 或 RDP 通过堡垒机访问成功；直连被拒绝或有批准例外；会话证据可取回。
    数据库访问 批准的数据库角色只能通过批准路径工作；直连客户端访问被拒绝或限定；存在数据库审计证据。
    Kubernetes node 访问 日常 node 访问被拒绝；break-glass 路径需要审批、证据和事后 reconcile。
    网络设备访问 管理路径限制在批准网络或堡垒机路径；会修改配置的会话证据被留存。
    Web 或云控制台 身份、MFA、角色映射和审计日志条目匹配批准访问类型。
    撤销 移除用户或组后，堡垒机访问和适用的底层资产访问都会消失。

    公开文档只用占位主机名和资产标识，不公开真实资产名。

    失败模式¶

    失败模式 后果 预防
    共享特权账号 无法问责。 将访问映射到个人身份。
    直连 SSH 仍开放 绕过堡垒机审计。 限制直连网络路径。
    数据库或控制台直连仍开放 敏感操作发生在录制会话外。 关闭路径或记录例外范围、Owner 和过期时间。
    会话录屏可被无痕编辑或删除 事故复盘时证据不可信。 保护录屏存储并审计管理操作。
    无会话评审 有录制但风险未管理。 由具名评审人定期和触发式评审。
    供应商访问不过期 临时访问变永久。 强制过期和 Owner 审批。
    Break-glass 比正常访问更宽 事故中治理被绕过。 缩小应急范围并事后 reconcile。
    堡垒机不可用 运维失去受控应急访问。 测试 break-glass 和恢复路径。
    资产 owner 未知 访问审批无权威。 维护资产 inventory。
    身份离职回收未打通 离职用户通过本地账号或组继续访问。 对接身份生命周期并运行撤销测试。
    平台备份遗漏录屏或资产映射 恢复了登录，但丢了证据或范围。 备份配置、映射、密钥和证据元数据。

    回滚与撤销¶

    移除访问：

    • 将用户从角色或组移除。
    • 撤销资产授权。
    • 风险紧急时终止活跃会话。
    • 如共享资产凭据曾使用或暴露，轮转凭据。
    • 禁用临时账号。
    • 如果直连例外原因已经消失，移除例外。
    • 对高风险或未授权访问评审历史会话。
    • 删除账号或映射前保留审计证据。

    堡垒机平台故障时，只能带事故记录使用 break-glass。恢复后要 reconcile 变更、轮转应急凭据、挂接证据，并关闭 fallback 路径。

    安全说明¶

    • 特权用户强制 MFA。
    • 限制 JumpServer 自身管理员访问。
    • 保护会话录制和审计日志。
    • 避免保存无人管理的共享凭据。
    • 单独评审 sudo 和 root 访问。
    • 将数据库管理员访问与普通只读访问分开评审。
    • 保持资产到 Owner 的映射最新。
    • 没有额外控制时，不要公网暴露堡垒机管理接口。
    • 给堡垒机平台及其依赖打补丁。
    • 将堡垒机备份视为敏感数据，因为它可能包含凭据、录屏、资产名和访问映射。

    评审周期¶

    在以下情况评审：

    • 人员变化或团队转移
    • 供应商接入或离场
    • 特权访问事故
    • 会话录制失败或审计日志缺口
    • 创建直连例外或例外到期
    • 堡垒机升级
    • 身份提供方变化
    • 资产重要性变化
    • 使用应急访问
    • 备份或恢复测试
