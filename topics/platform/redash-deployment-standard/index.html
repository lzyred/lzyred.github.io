    Redash 治理标准¶

    目的¶

    本标准定义如何部署和运营 Redash 或类似内部 BI 查询面。

    核心判断：

    BI 工具不是仪表盘应用，而是受治理的数据访问面。

    Redash 能让运营和业务数据更容易被查看，也可能把一个过宽凭据扩散成大量非正式读者、缓存结果、导出文件、共享链接，以及压垮生产数据库的查询。只有访问、数据分类、查询影响、共享、留存和恢复被当成同一份契约治理时，它才适合进入多团队使用。

    适用场景¶

    适用于：

    • 内部团队需要 SQL 仪表盘或临时查询。
    • 数据源可以通过只读账号、只读副本、数仓或受控视图暴露。
    • 查询和仪表盘访问可以按用户组、数据源和数据分类治理。
    • 共享、导出和缓存结果留存有明确 Owner。
    • 查询和仪表盘属于运营资产，需要元数据备份和恢复。

    不要让 Redash 以写入、owner、schema 变更、复制或管理员权限直连生产数据库。

    非目标¶

    本标准不定义：

    • 数据仓库架构
    • 分析建模
    • 业务指标口径
    • 仪表盘设计风格
    • 公共 BI 发布
    • 数据治理审批替代流程

    它只覆盖 BI 查询面的平台和安全契约。

    生产前置条件¶

    以下控制不存在时，不要把 Redash 当成已具备生产治理边界：

    • 批准的身份边界，例如 SSO、IdP 集成，或等效的集中访问管理流程
    • 特权用户启用 MFA
    • 基于用户组的开通和撤权
    • 具名 admin，本地 admin 仅用于受控 break-glass
    • 受支持版本或批准的风险例外
    • patch 和 CVE 监控 Owner
    • 升级回滚计划
    • 升级前完成经过恢复验证的元数据备份

    如果这些条件缺失，本页仍可指导 lab 部署，但系统还不适合承载广泛内部数据访问。

    生产契约¶

    Redash 被多个团队使用前，先定义：

    契约 必须决策 为什么重要
    平台 Owner 谁负责 Redash 服务健康、升级、备份和事故？ BI 工具故障可能阻塞运营决策。
    身份边界 哪个 SSO、IdP、用户组或受控访问流程治理 users 和 admins？ 离职或撤权人员必须失去 query、dashboard、export 和 admin 访问。
    维护姿态 哪个版本被批准，CVE 如何处理，升级如何回滚？ 陈旧的 BI 访问面会变成长期数据访问风险。
    数据源 Owner 谁批准每个数据源和凭据？ 平台管理员不应静默变成数据 Owner。
    数据分类 queries、results、dashboards、exports 和 cache 中允许出现什么数据？ 访问决策取决于数据敏感度，不只取决于数据库主机名。
    凭据范围 使用哪个账号，能读什么，必须拒绝什么？ 过宽凭据会把 BI 被攻陷变成数据被攻陷。
    查询影响 哪条数据库路径承载长查询或高成本查询？ 仪表盘不应变成生产负载测试。
    共享模型 谁能发布 dashboard、创建 public link、邀请用户或导出结果？ 共享会绕过原始查询权限模型。
    结果留存 缓存查询结果和导出文件保留多久？ 撤销访问不会自动删除已复制结果。
    审计与评审 哪些变更需要记录和评审？ 敏感访问需要证据，而不是记忆。
    恢复 元数据、凭据和数据源定义如何恢复或撤销？ 丢失 Redash 元数据可能抹掉运营仪表盘和事故证据。

    推荐标准¶

    先控制身份生命周期，再接入数据源¶

    身份是第一道生产边界。大范围接入数据源前，定义：

    • 用户如何开通
    • 用户如何撤权
    • 哪些用户组映射到 query、dashboard、data-source 和 admin 权限
    • 哪些账号只允许 break-glass
    • 特权角色如何强制 MFA
    • 访问评审证据如何保留

    验证必须证明撤权用户失去 query、dashboard、export、sharing 和 admin 路径。只把用户从聊天群或邮件组移除，不等于完成访问撤销。

    把平台按数据访问面维护¶

    Redash 需要明确维护 Owner。记录：

    • approved version
    • patch and CVE monitoring source
    • upgrade owner
    • upgrade test path
    • upgrade 前完成的 metadata backup
    • downgrade 或 rollback path
    • user-facing impact window

    如果项目版本不受支持或难以升级，扩大数据访问前必须记录风险例外和补偿控制。

    将 Redash 与生产写路径分离¶

    生产数据源必须使用以下模式之一：

    • read replica
    • data warehouse
    • curated view layer
    • 窄范围只读 service account
    • 批准的 analytics endpoint

    如果必须查询生产 primary，把它作为例外处理：需要 Owner、timeout、concurrency limit、query review 和 rollback plan。Redash 永远不应成为修改生产数据的路径。

    把每个数据源当成访问授权¶

    每个数据源记录：

    • data-source owner
    • business purpose
    • data class
    • credential storage location
    • read scope and explicit denied scope
    • network access path
    • query timeout and concurrency limit
    • allowed user groups
    • dashboard sharing rule
    • export rule
    • review date

    避免使用无法轮转、无法归属或无法缩小权限的共享凭据。如果账号能读取的数据超过 Redash 用户应看的范围，即使 UI 权限看起来整齐，这个数据源也是过度授权。

    分开治理查询编写和仪表盘发布¶

    Query authors、dashboard publishers、data-source creators 和 admins 是不同权力。

    定义：

    • 谁可以创建数据源
    • 谁可以写或编辑 query
    • 谁可以发布 dashboard
    • 谁可以导出结果
    • 是否禁用 public link
    • 敏感 dashboard 如何评审
    • 谁可以添加用户或用户组
    • 谁可以轮转凭据

    包含敏感或受监管数据的 dashboard 应有具名 Owner 和评审周期。没有 Owner 的 dashboard，是等日历变化触发的数据泄漏。

    控制缓存结果和导出¶

    Redash 权限保护的是 UI 路径，不能完整保护：

    • cached query results
    • downloaded CSV files
    • screenshots
    • embedded dashboards
    • public or secret links
    • copied SQL queries
    • notification payloads

    定义缓存结果和导出的留存策略。用户失去访问权限时，也要评估是否需要清理缓存结果、共享链接或导出文件。

    限制查询影响¶

    广泛开放前设置护栏：

    • query timeout
    • result size limit
    • concurrency limit
    • warehouse or replica preference
    • slow-query owner
    • 可用时启用数据库侧资源护栏
    • 数据源 emergency disable path

    正确验证不是“仪表盘能打开”，而是慢查询或高成本查询不能悄悄破坏其他服务依赖的数据库路径。

    备份并测试 Redash 元数据恢复¶

    Redash 元数据包含 queries、dashboards、users、groups、data-source definitions、schedules、alerts，有时还包含敏感连接上下文。备份必须受保护并经过恢复测试。

    恢复测试应证明：

    • dashboards 和 queries 可以恢复
    • data-source definitions 恢复后不会以明文暴露密钥
    • credentials 可以在恢复后重新挂载或轮转
    • sharing settings 仍受控
    • admin users 仍经过评审

    验证¶

    通过条件：

    • Redash web service、workers、scheduler 和 metadata database 健康。
    • 身份集成、特权角色 MFA、用户组开通已生效，或存在批准的等效控制。
    • 撤权测试证明被撤销用户失去 query、dashboard、export、sharing 和 admin 路径。
    • 元数据备份最新且已测试恢复。
    • Patch/CVE 监控有 Owner，升级回滚已记录。
    • 每个数据源有 Owner、数据分类、读取范围和评审日期。
    • 数据源凭据是只读，或有批准的例外。
    • 通过 BI 凭据执行写入或 schema 变更会被拒绝。
    • 测试查询在预期副本、数仓或受治理路径上按超时成功。
    • 慢查询、大结果或并发查询被限制，或被路由 away from production primary。
    • Public link 默认禁用或需要评审。
    • Dashboard export 和 cached-result retention 已记录。
    • Admin users、data-source creators 和 dashboard publishers 已评审。
    • 凭据轮转路径已测试。

    失败模式¶

    失败模式 后果 预防
    身份生命周期薄弱 离职用户或陈旧用户组继续拥有数据访问。 使用 SSO 或等效控制、MFA、用户组评审和撤权测试。
    Redash 不受支持或未修补 BI 访问面积累已知漏洞。 维护版本姿态、CVE 监控和升级回滚。
    生产凭据过宽 BI 被攻陷会暴露或修改生产数据。 使用只读范围账号并验证拒绝范围。
    UI 权限掩盖数据库过度授权 用户能查询所在组不应看到的数据。 按数据分类缩小凭据和视图。
    Public 或 secret link 扩散 敏感 dashboard 绕过正常访问评审。 禁用或严格评审 public sharing。
    缓存结果比访问权限活得更久 曾经授权的数据继续可见。 定义 cache、link 和 export 清理。
    长查询打到生产 primary 仪表盘使用拖慢面向用户的系统。 使用副本、数仓、timeout 和 concurrency limit。
    元数据备份不可恢复 queries、dashboards、schedules 和访问记录消失。 测试元数据恢复和凭据重新挂载。
    共享 admin 数据源或共享变更无法问责。 使用具名 admin、MFA 和审计评审。
    凭据通过导出或截图泄漏 数据离开受治理访问面。 限制导出并评审敏感 dashboard。

    回滚与恢复¶

    恢复可能包括：

    • 禁用数据源
    • 撤销或轮转凭据
    • 禁用共享链接
    • 可行时删除缓存结果或导出
    • 恢复 metadata database
    • 重新挂载或轮转恢复后的凭据
    • 将用户切回上一 BI 工具
    • 当敏感结果可能扩散时通知数据 Owner

    如果怀疑凭据泄漏，恢复访问前先轮转。如果 dashboard 暴露了敏感数据，删除链接只是第一步；还要评审缓存结果、导出、截图、通知 payload 和下游副本。

    出现以下情况立即升级：

    • data-source credential 可能泄漏
    • dashboard、link、export 或 cached result 暴露敏感数据
    • query 影响 production primary 或共享数据库路径
    • metadata restore 失败
    • 发现未授权 sharing 或 admin activity

    平台 Owner 协调 Redash 服务动作。数据源 Owner 决定是否禁用数据源、凭据范围和重新启用批准。DBA 或 SRE 负责数据库影响响应。当涉及凭据、受监管数据、客户数据或未授权共享时，Security 或 Privacy Owner 参与。

    重新启用被禁用的数据源前，需要收集证据：凭据已轮转或仍有效、访问组已评审、缓存或共享暴露在可行范围内已收敛、查询影响已受控，并且数据源 Owner 已批准重新启用。

    安全说明¶

    • 不公开查询结果、客户数据、内部表名、dashboard 截图、数据源名、数据库主机或凭据示例。
    • 数据源凭据放入批准的密钥存储。
    • 使用最小权限数据库账号。
    • Redash 放在批准的访问控制后；没有额外控制时不要暴露到公网。
    • 特权用户启用 MFA，并受控处理 break-glass admin。
    • Redash 版本、依赖和安全补丁保持评审。
    • 将 public link 和 export 视为数据共享事件评审。
    • 审计 admin、data-source、permission、sharing 和 credential 变更。
    • 不要把 Redash 当成应急数据库控制台。

    评审周期¶

    在以下情况评审 Redash：

    • 新增数据源
    • 数据源 Owner 或数据分类变化
    • 新团队获得查询访问
    • public sharing 或 export 规则变化
    • dashboard 变成运营关键资产
    • 凭据轮转
    • 测试元数据恢复
    • 慢查询影响生产路径
    • 出现安全事故或疑似数据暴露
