    GitLab 运维标准¶

    目的¶

    本标准定义如何把 GitLab 作为源码控制和交付平台来运营。

    核心判断：

    GitLab 不只是仓库服务器，它是生产变更授权中心。

    当 GitLab 控制源码、CI/CD、受保护分支、审批、制品、镜像仓库、token 和基础设施流水线时，GitLab 的薄弱点会变成生产变更的薄弱点。要把它当成变更治理系统，而不是代码偶然存放的地方。

    适用场景¶

    适用于：

    • GitLab 托管生产源码或基础设施代码。
    • GitLab CI/CD 可以部署工作负载或修改基础设施。
    • Runner 可以访问云、Kubernetes、registry、package、signing 或 secret 系统。
    • 多团队使用 GitLab group 做发布治理。
    • GitLab 故障会阻塞应急变更或回滚。

    非目标¶

    本标准不定义 GitLab 每个功能、UI 选项或授权版本，只定义 GitLab 作为生产变更授权中心的运营契约。

    生产契约¶

    GitLab 成为生产变更来源前，先定义：

    契约 必须决策 为什么重要
    平台 Owner 谁负责 GitLab 服务健康、升级、备份和事故？ GitLab 故障会阻塞部署、回滚和审计。
    变更授权 哪些 project、branch、tag 和 pipeline 能影响生产？ 生产修改不应隐藏在普通仓库活动里。
    Group 所有权 每个 group 和 project 谁负责？ 无主 group 会变成未评审的发布授权。
    身份生命周期 哪些 SSO、MFA、group sync、沉默用户、密钥和离职回收控制治理人的访问？ 离职 maintainer、旧 SSH key 或无人管理的 token 仍可能修改生产。
    Protected refs 哪些 branch 和 tag 需要 review、status check 或 approval？ 一个未保护 ref 就能绕过发布治理。
    Runner 信任 哪些 runner 能看到 secret 或部署？ Runner 被攻陷常常比仓库读权限更危险。
    Secret 边界 CI/CD variables、deploy tokens 和 access tokens 存在哪里、如何轮转？ CI 凭据经常就是生产凭据。
    制品与 registry 留存 哪些 build、release、package 和 image 证据需要保留？ 恢复和审计不只依赖 Git history。
    备份与恢复 备份什么、频率如何、如何证明可恢复？ 没有恢复证明的备份不是恢复计划。
    应急发布 GitLab 不可用时生产变更如何发生？ 事故响应不能依赖唯一不可用的授权中心。
    审计评审 哪些事件需要评审？ 变更授权需要证据，而不是记忆。

    推荐标准¶

    把生产 refs 当作发布授权保护¶

    为以下对象定义保护：

    • default branches
    • release branches
    • production tags
    • infrastructure repositories
    • deployment manifests
    • CI/CD templates
    • group-level variables

    对影响生产的 refs 要求适当 review、status check 和 approval。应急绕过必须具名、记录、限时，并在事后 reconcile。

    分离构建信任和部署信任¶

    Runner 访问是最常见的隐藏信任边界。

    Runner 类型 用途 规则
    共享低信任 runner 构建和测试非特权代码。 无生产凭据或 protected variables。
    Project runner 项目专用流水线。 只授予项目范围凭据。
    Protected runner 从受保护 branch 或 tag 部署。 只运行 protected jobs。
    Privileged runner 容器构建、Docker-in-Docker、特殊操作。 隔离、最小化、打补丁并审计。
    生产部署 runner 修改生产或基础设施状态。 专用、受保护，并绑定发布审批。

    不要让不可信 merge request 运行带部署凭据、protected variables、privileged runners 或云/Kubernetes 访问的 job。

    把 secret 和 token 当成生产凭据治理¶

    记录并评审：

    • CI/CD variables
    • deploy tokens
    • project access tokens
    • group access tokens
    • personal access tokens
    • SSH keys 和 deploy keys
    • runner registration tokens
    • runner authentication tokens
    • robot accounts
    • service accounts
    • 支持时的 OAuth 或 active sessions
    • signing keys
    • registry credentials
    • webhooks and outbound integrations

    每个 token 都应有 Owner、scope、过期或评审日期、轮转路径和事故响应 Owner。Masked variables 本身不是安全边界；scope 和 protected job access 更重要。

    把人的访问也作为发布授权治理¶

    人的访问也是生产变更面的一部分。定义：

    • 权威身份提供方，或本地账号例外
    • 特权用户 MFA 范围
    • group sync 或 membership Owner
    • owner 与 maintainer 角色评审
    • 沉默用户检测
    • SSH key 和 deploy key 评审
    • personal access token 过期与 scope 评审
    • 离职回收测试，证明 Git、UI、CI/CD、registry 和 runner 路径上的访问都会消失

    每个影响生产的 group 和 project 都需要当前负责的 Owner。访问例外应有过期日期、撤销 Owner，以及这个例外仍有生产原因的证据。

    备份完整变更授权中心¶

    备份必须覆盖：

    • repositories
    • database
    • GitLab configuration
    • secrets file
    • CI/CD variables 和 protected settings
    • group 和 project membership
    • merge-request 和 approval metadata
    • 如使用，container registry
    • 如需要，package artifacts
    • 发布证据需要的 CI artifacts

    恢复测试不能只证明“UI 能打开”。它要证明 repository 访问、protected branch rules、runner 注册或替换、CI/CD variable 恢复、registry 或 artifact 访问，以及 admin review state。

    GitLab 备份与恢复介质是敏感数据。它们可能包含 repositories、secrets、CI/CD variables、registry references、membership state 和发布证据。备份要加密、限制访问；隔离恢复环境也要按敏感系统处理；如果怀疑备份介质暴露，要轮转暴露的 token 或 key。

    在事故前定义 GitLab 不可用时的行为¶

    生产事故中 GitLab 不可用时，团队需要受控路径处理：

    • emergency rollback
    • emergency config change
    • disabling compromised credentials
    • freezing deployments
    • communicating source-of-truth status
    • reconciling changes back into GitLab

    应急路径应该比正常路径更窄。如果 fallback 比 primary 更宽，GitLab 治理会在压力最高时蒸发。

    最小应急包应包含：

    • incident ID
    • 具名 approver
    • 允许操作的 repository、artifact、manifest 或 infrastructure target
    • 允许动作和禁止动作
    • 被应用 artifact 或 configuration 的 source of truth
    • 限时 credential 或 runner path
    • GitLab 不可用时的 evidence capture method
    • 使用后的 credential rotation
    • GitLab 恢复后的 reconciliation 或 back-merge proof

    验证¶

    定期检查：

    gitlab-rake gitlab:check
gitlab-rake gitlab:doctor:secrets
gitlab-ctl status

    通过条件：

    • Web、Git SSH 或 HTTP、registry、package 和 CI 服务健康。
    • 影响生产的 branch 和 tag 已受保护。
    • 必需 review、status check 和 approval 匹配发布策略。
    • 不可信 merge request 无法访问 protected variables 或 deploy runners。
    • Runner fleet 在线、按信任级别分域，并保持补丁。
    • 生产部署 runner 受保护且可审计。
    • 备份完成，并在隔离环境测试恢复。
    • 恢复测试证明 repositories、protected refs、CI/CD variables、runners、registry 或 artifacts、admin state 可恢复。
    • Admin membership、group owners、deploy tokens 和 project access tokens 已评审。
    • 影响生产的 group 和 project 有负责 Owner，不存在无主 owner 或 maintainer 权限，访问例外有过期时间和撤销 Owner。
    • 离职回收测试证明离职用户失去 Git、UI、CI/CD、registry、token、SSH key 和 runner 间接访问。
    • 应急发布和回滚路径有 Owner 和 reconcile 步骤。
    • Pipelines、runners 或 protected deploy jobs 在疑似被攻陷后，只有重新验证发布授权、Owner 批准和 blast radius 后才能重新启用。

    失败模式¶

    失败模式 后果 预防
    Protected branch 被关闭 未评审代码或 manifests 可以部署。 强制 protected refs 并审计变更。
    Runner 凭据过宽 CI 被攻陷变成生产被攻陷。 分离 runner 信任级别和 secret scope。
    不可信 MR 接触 protected secrets 外部或低信任代码可窃取部署凭据。 阻断不可信 job 访问 protected variables 和 protected runners。
    Token 无 Owner 或过期策略 旧凭据持续拥有生产访问。 跟踪 Owner、scope、评审日期和轮转。
    无主 owner 或 maintainer 角色 离职或未知人员仍可修改生产。 评审 group/project ownership、沉默用户和离职回收证据。
    无恢复测试 最需要 GitLab 时恢复失败。 在隔离环境测试恢复。
    备份介质暴露 Repositories、variables、tokens 和 membership state 可能泄漏。 加密、限制、审计，并在疑似暴露后轮转凭据。
    Artifact 留存无治理 存储写满或发布证据消失。 定义 artifact、registry 和 package retention。
    共享 admin 账号 变更授权修改无法问责。 使用具名 admin、MFA 和 break-glass 流程。
    GitLab-down 路径未定义 事故回滚被阻塞或临场发挥。 定义应急发布和 reconcile 流程。

    回滚与恢复¶

    恢复方式包括：

    • 从测试过的备份恢复 GitLab
    • 禁用被攻陷 runner
    • 轮转 CI/CD variables、deploy tokens 和 access tokens
    • 撤销或禁用 group/project membership、owner 与 maintainer 角色、personal access tokens、SSH keys、deploy keys、service accounts、OAuth sessions、webhooks 和被攻陷 integrations
    • 撤销 runner registration tokens
    • Runner 可能被攻陷时轮转 runner authentication tokens
    • 通过禁用 protected deploy jobs 冻结部署
    • 回退 protected branch 或 tag 变更
    • 从备份或 mirror 恢复 repository
    • 恢复 registry 或 release artifacts
    • 恢复后把应急变更 reconcile 回 GitLab

    如果 runner、token、protected ref 或人的访问路径疑似被攻陷，先停止部署 job，保留审计证据，撤销访问，轮转凭据，然后由 Owner 验证 scope、blast radius、发布授权和 reconcile 证据后再重新启用。

    安全说明¶

    • 特权用户强制 MFA。
    • 可用时使用 SSO 或其他受治理身份来源；记录本地账号例外。
    • 管理员成员保持小范围并定期评审。
    • 评审沉默用户、owner 与 maintainer 角色、SSH keys、deploy keys、personal access tokens 和 service accounts。
    • 分离 build、test 和 deploy 凭据。
    • 保护 deploy tokens、project access tokens、group tokens 和 runner tokens。
    • 评审 webhook targets 和 outbound integrations。
    • 不要把 secret 存入 repository、issue comments、job logs 或 artifacts。
    • 加密并限制 GitLab backups 和 restore media。
    • 保持 GitLab 和 runners 补丁更新。
    • 把 CI job logs 和 artifacts 视为潜在 secret 暴露面。

    评审周期¶

    在以下情况评审：

    • runner 事故
    • 访问或 group owner 变化
    • 身份提供方、MFA、group sync、SSH key、deploy key 或 personal access token 策略变化
    • protected branch 或 release-policy 变化
    • GitLab 升级
    • 备份失败或恢复测试
    • deploy token 或 project access token 轮转
    • registry 或 artifact retention 变化
    • 发布治理变化
    • GitLab 在事故中不可用
