    系统 OS 补丁标准¶

    目的¶

    本标准定义如何为 Linux 操作系统打补丁，避免把例行安全工作变成计划外故障。

    核心判断：

    OS 补丁是以安全为动机的可靠性变更。补丁不是在包安装成功时完成，而是在风险降低且服务路径仍然健康时完成。

    补丁必须有计划、可观测、尽可能可回退、绑定所有权，并在需要重启时验证重启后的生效状态。“到处跑升级命令”不是补丁策略。

    适用场景¶

    适用于：

    • 月度或紧急 Linux 补丁
    • kernel、libc、OpenSSL、OpenSSH、systemd、driver 和容器宿主机更新
    • bastion、CI runner、monitoring、database 和平台主机维护
    • Kubernetes node image 或 host patch wave
    • fleet patch compliance review
    • 补丁无法按时应用时的安全例外评审

    非目标¶

    本标准不定义应用依赖升级、容器镜像重建策略、托管 OS 补丁服务内部机制，或发行版特定加固基准。

    补丁契约¶

    补丁生产主机类别前，必须定义：

    契约 必须决策 为什么重要
    Owner 谁负责补丁审批、服务验证和回滚？ Owner 未知的主机会把补丁变成盲变更。
    Host class 主机角色是什么：stateless app、database、bastion、runner、monitoring、gateway、node？ 不同角色有不同 drain、reboot 和 recovery 行为。
    Failure domain 哪些主机可以一起补丁而不损失服务容量？ 安全的维护窗口也可能因所有副本同时变化而造成故障。
    Urgency 补丁基于暴露面和可利用性属于 routine、expedited 还是 emergency？ 严重度本身不决定时机，可达暴露面才决定操作模式。
    Reboot boundary 哪些更新需要 reboot 或 service restart 才真正生效？ 包已安装，不代表易受攻击 kernel 或 library 已退出。
    Validation path 哪个 smoke test、metric 或 owner check 证明主机安全返回？ “主机起来了”不等于“服务路径可用”。
    Rollback path 回滚是 downgrade、previous kernel boot、snapshot restore、rebuild，还是 service failover？ 有些补丁在压力下是前向变更。
    Exception rule 谁批准延迟补丁，有什么补偿控制？ 永久例外是隐藏风险。

    推荐标准¶

    Inventory 必须能支撑决策¶

    每台主机或节点组需要：

    • owner
    • environment
    • criticality
    • OS distribution 和 version
    • kernel version
    • workload role
    • exposure class
    • reboot tolerance
    • backup、snapshot 或 rebuild path
    • maintenance window
    • monitoring owner
    • patch exception status

    Owner 未知的主机不应静默打补丁，而应作为治理风险升级，因为没人能验证影响或批准停机。

    按可利用性和暴露面分类紧急度¶

    类别 使用条件 操作模式
    Routine 普通安全与稳定性更新，没有紧急暴露。 计划维护窗口。
    Expedited 漏洞可利用、暴露，或影响重要主机类别。 有 owner 在场的批准加速窗口。
    Emergency 正在被利用、严重远程暴露，或业务风险极高。 带明确沟通的事故管理变更。

    紧急补丁仍然需要沟通、回滚思考和变更后验证。速度不能取消对“主机重启失败会影响什么”的理解。

    按故障域补丁，而不是按方便程度¶

    按波次补丁：

    1. 非生产或 canary 主机。
    2. 同角色的一台低风险主机。
    3. 每个 failure domain、availability zone、node pool 或 backend group 的一台主机。
    4. 指标保持健康后再推进剩余 fleet。

    除非工作负载已证明可以承受容量损失，否则不要同时补丁同一服务角色的所有主机。Kubernetes node 应一次 drain 和 uncordon 一个受控单位，除非 node pool 设计允许更大波次。

    重启才是真正的生效边界¶

    Kernel、libc、OpenSSL、OpenSSH、systemd 和 driver 更新可能需要 service restart 或 reboot。包管理器报告成功时，易受攻击代码可能仍在内存中运行。

    重启前：

    • 如适用，先 drain workload
    • 确认容量可承受该主机不可用
    • 有状态服务按专属 runbook 协调
    • 确认 console 或 rebuild access
    • 确认告警静默或维护窗口
    • 可行时记录 previous kernel 或 snapshot

    重启后，必须证明预期 kernel 或服务版本已经生效。

    例外必须是会过期的风险¶

    一个例外必须包含：

    • owner
    • affected hosts
    • missing patch or version
    • reason patch cannot be applied
    • exposure and impact
    • compensating control
    • expiry date
    • next review date

    不要接受“应用可能会坏”作为无限期例外。要么测试应用路径，要么把该主机归类为 unsupported risk。

    操作流程¶

    1. 准备¶

    uname -a
cat /etc/os-release
df -h
systemctl --failed

    确认：

    • host owner 和 service owner 已知
    • 变更前监控健康
    • 必要时存在 backup、snapshot 或 rebuild path
    • 容量可承受一台主机不可用
    • package repository source 已批准
    • reboot requirement 已明确
    • validation command 或 smoke test 已准备
    • 高风险主机的 rollback owner 在场

    2. 应用补丁¶

    使用发行版包管理器和批准的软件源。生产补丁时避免临时下载包。

    Debian 或 Ubuntu 类系统：

    sudo apt update
sudo apt list --upgradable
sudo apt upgrade

    RHEL 类系统：

    sudo dnf check-update
sudo dnf upgrade

    当变更影响重要主机类别时，记录 package set 或 patch batch。

    3. 必要时重启或重启服务¶

    如果需要重启，先完成 drain 或 failover。如果重启服务足够，也要在 service restart 后验证服务路径。

    不要跳过 kernel 或 core library 更新后的 reboot，然后把主机标记为 compliant。

    验证¶

    补丁后：

    uname -a
systemctl --failed
uptime
journalctl -p err -b --no-pager | tail -50

    通过条件：

    • 主机可通过批准的管理路径访问
    • 预期 kernel 或 package version 已经生效
    • 关键服务运行中
    • 如适用，workload 已回到 rotation
    • 监控报告健康
    • 没有异常 boot、driver、disk、time-sync、SSH 或 service error
    • 应用或平台 smoke test 通过
    • 告警已关闭或有解释
    • patch compliance record 已更新

    故障模式¶

    故障模式 结果 预防
    一次补所有节点 包安装成功但服务中断。 按 failure domain 补丁，并在波次之间观察指标。
    跳过重启 易受攻击 kernel 或 library 仍在运行。 跟踪 reboot-required 状态，并证明 active version。
    无 owner 补丁 影响未知，没人有权验证。 维护 inventory，并升级 owner 未知主机。
    软件源漂移 主机收到不同包或未批准版本。 使用批准 mirror，并记录 package set。
    不检查磁盘 包安装中途失败。 补丁前检查可用磁盘、inode 和 package cache 空间。
    把有状态主机当无状态 数据服务失去 quorum 或可用性。 遵循服务专属 drain、failover 和 recovery runbook。
    无回滚边界 补丁失败变成长时间故障。 补丁前定义 previous kernel、snapshot、rebuild 或 failover path。
    永久例外 已知暴露变成常态。 要求过期时间、补偿控制和 review owner。

    回滚与恢复¶

    回滚取决于变更类型：

    • 当仓库仍提供旧版本时，用户态包可能可以 downgrade
    • kernel rollback 可能使用 previous boot entry
    • 对 pet 或有状态主机，snapshot restore 可能合适，但必须理解数据一致性
    • 对 cattle-style 主机，rebuild 可能比手工修复更安全
    • 数据服务需要服务专属恢复和 quorum 检查

    如果补丁导致无法启动，按主机类别使用 console access、previous kernel、snapshot restore 或 image rebuild。如果补丁破坏应用行为，停止后续波次，保留最后健康主机不动，并在 package rollback、application mitigation 或 traffic removal 之间选择。

    安全说明¶

    • 补丁源必须是可信仓库。
    • 不要为了强行打补丁而关闭安全控制。
    • 例外必须记录 owner、reason、expiry 和 compensating control。
    • 紧急补丁凭据必须限时。
    • 维护后验证 patch compliance。
    • 不公开真实 hostname、IP address、package mirror、绑定真实系统的漏洞例外细节、credential 或运维证据。

    复审周期¶

    紧急补丁后、补丁相关事故后、新增主机类别、OS 发行版支持窗口变化、例外反复出现，或补丁波次经常需要手工恢复时，复审本标准。
