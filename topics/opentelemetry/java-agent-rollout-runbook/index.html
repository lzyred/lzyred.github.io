    OpenTelemetry Java Agent 灰度发布 Runbook¶

    使用场景¶

    用于不通过 Operator webhook、而是以 -javaagent 或专用 JAVA_TOOL_OPTIONS 给 JVM 挂载 OpenTelemetry Java agent 的生产变更。适用于 Kubernetes、容器和虚拟机，但执行命令需按平台改写。

    Java agent 在类加载时修改字节码并运行于应用进程内，会改变启动、CPU、内存、GC、网络和数据出口。官方 facts 最后核验于 2026-07-12；本文不保证任意 agent/JVM/framework 组合兼容。每次升级按实际版本的 Supported Libraries 和 release notes 重新验证。

    开始前必须决定¶

    输入 必须记录
    二进制 agent 版本、来源 URL/内部镜像、SHA-256 或制品签名记录
    JVM 供应商、主版本、架构、容器基础镜像、启动命令
    应用 镜像 digest、框架/应用服务器版本、现有 APM/agent
    Canary 独立实例/Deployment、流量入口、样本量、观察窗口
    导出 signals、OTLP 协议/endpoint、TLS、认证、NetworkPolicy/防火墙
    身份 service.name、namespace、environment、version；不得含客户/租户身份
    阈值 独立 p95/p99、错误率、启动、CPU/RSS、GC、重启；Collector queue/failures；成本
    回滚 上一版启动配置、制品、执行人和触发器

    任何字段缺失则停止。

    前置检查¶

    1. 建立真正 canary¶

    建立独立 JVM 实例或独立 Deployment/revision，只接收固定比例、请求头或专用合成流量。不要给稳定 Deployment 全部副本加入 JAVA_TOOL_OPTIONS 后称为 canary。保存独立于 OTel 的入口请求数、p95/p99、错误率、启动时间、CPU/RSS、GC pause 和重启基线。

    2. 审核 agent 供应链¶

    只从 OpenTelemetry 官方 release 或批准的内部制品库取得固定版本。验证 SHA-256/签名/内部 provenance，并把结果与 agent 文件一起归档。不要运行浮动 URL、未经校验的 JAR 或临时下载目录中的文件。

    shasum -a 256 /opt/otel/opentelemetry-javaagent.jar

    官方入门文档说明 JAR 来自 opentelemetry-java-instrumentation releases，并通过 -javaagent 挂载：Java agent getting started。

    3. 盘点 JVM 参数和 agent¶

    记录完整启动命令、JAVA_TOOL_OPTIONS、JDK_JAVA_OPTIONS、JAVA_OPTS、容器 entrypoint、应用服务器脚本和现有 -javaagent。确认是否已有 APM/security/profiler agent，按双方官方兼容说明验证加载顺序；不能仅因 JVM 能启动就认为兼容。

    java -version
ps -ef
env

    在共享主机上，不要把 JAVA_TOOL_OPTIONS 设置为全局登录环境，因为它可能影响批处理、CLI 和其他 JVM。优先在唯一服务的启动命令中添加 -javaagent。

    4. 验证安全出口和 Collector¶

    证明 OTLP receiver 的协议/端口、TLS trust、客户端认证、NetworkPolicy/防火墙与 Collector 配置匹配。不能通过关闭证书验证、打印 token 或把凭据放入命令行历史来测试。确认 Collector 容量和内部指标阈值已建立。

    变更步骤¶

    1. 只启用 traces¶

    首次变更只开启 traces；metrics 和 logs 分开评审。以下仅是变量形态示例，endpoint、证书和阈值必须来自变更单：

    export OTEL_SERVICE_NAME="<service-name>"
export OTEL_RESOURCE_ATTRIBUTES="service.namespace=<namespace>,deployment.environment.name=<environment>,service.version=<version>"
export OTEL_EXPORTER_OTLP_ENDPOINT="https://<collector-endpoint>:4318"
export OTEL_EXPORTER_OTLP_PROTOCOL="http/protobuf"
export OTEL_TRACES_EXPORTER="otlp"
export OTEL_METRICS_EXPORTER="none"
export OTEL_LOGS_EXPORTER="none"

    Java agent 2.0+ 当前默认协议是 http/protobuf，但仍应显式配置以避免版本默认变化；官方配置优先级从 system properties、environment、configuration file 到 SPI 提供值，具体按部署版本核对：Java agent configuration。不要同时在多处定义相反的值。

    2. 在 canary 挂载¶

    优先在 canary 的专用启动命令中添加：

    java -javaagent:/opt/otel/opentelemetry-javaagent.jar <existing-jvm-arguments> -jar <application.jar>

    如果运行平台只能使用 JAVA_TOOL_OPTIONS，确保该环境只属于目标 JVM，并保留原值的经测试合并结果：

    export JAVA_TOOL_OPTIONS="<existing-options> -javaagent:/opt/otel/opentelemetry-javaagent.jar"

    先用与生产相同的镜像/启动脚本完成 dry run 或预生产启动。不要在生产 shell 中猜测参数顺序。

    3. 发起已知请求并验证¶

    1. 确认 canary JVM 启动，agent 日志报告预期版本，无类转换/导出错误。
    2. 比较实际进程参数，证明只有目标 JVM 挂载一个预期 agent，heap/GC/应用参数未丢失。
    3. 发送带唯一非敏感标记的合成请求，记录时间、入口和请求 ID。
    4. 在后端找到完整 trace，确认 service.name、namespace、environment、version 和上下游传播。
    5. 检查 Collector accepted/refused/send_failed/queue；不能只看后端 UI。
    6. 用网关、合成探针或应用原生指标验证独立 SLI、资源和 GC 阈值。
    7. 使用批准的假 token、Cookie、查询参数、用户 ID 和 body 标记做负向测试；Collector 出口及后端必须零匹配。

    停止条件¶

    出现任一项停止流量并回滚：JVM/框架不在验证矩阵；启动或 readiness 超时；类转换异常；独立 SLI、CPU/RSS/GC/Collector/成本阈值越界；非目标 JVM 被挂载；现有 agent 冲突；敏感标记抵达出口；TLS/认证只能通过降级才能连接。

    Agent debug 输出非常详细。只允许在 canary 上短时使用 OTEL_JAVAAGENT_DEBUG=true，先确认日志不会包含敏感上下文，限制采集窗口，排障后立即撤销。官方文档也提示 debug 日志非常冗长：Java agent getting started。

    回滚¶

    1. 将 canary 流量降为零。
    2. 从专用启动命令移除 -javaagent:/opt/otel/opentelemetry-javaagent.jar，或恢复变更前完整 JAVA_TOOL_OPTIONS；不要只设 OTEL_SDK_DISABLED 冒充卸载。
    3. 用上一版不可变配置重新创建/重启 canary。
    4. 证明实际进程参数不再包含 agent，agent 线程/日志和新 telemetry 消失。
    5. 用独立 SLI 和 JVM/GC 指标确认回到基线。
    6. 在问题定位前保留经校验的制品和配置证据，但不要让未批准 agent 继续挂载。
    7. 若数据已泄露，启动数据事件处置；卸载 agent 不能删除后端数据。

    升级¶

    • 类转换、JVM crash、框架冲突：升级给服务/JVM 与可观测性负责人，并附最小非敏感错误和版本矩阵。
    • Collector 队列/发送失败：升级给平台/后端负责人，停止扩大输入。
    • 敏感字段：立即升级数据安全负责人。
    • 未知或不匹配 JAR：按供应链事件处理，不运行该制品。

    完成条件¶

    只有当观察窗口和样本量完成、所有阈值通过、负向测试零匹配、回滚预演成功且证据归档，才能批准一个后续批次。每个服务/JVM/框架组合都需要自己的 canary；一个 Spring Boot 服务的结果不能替另一个应用服务器证明兼容。
