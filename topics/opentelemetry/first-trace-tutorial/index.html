    Observability OpenTelemetry Tutorial

    生成第一条 Trace¶

    本教程让你亲手验证最小 Trace 数据路径：telemetrygen 创建 Span，经 OTLP 发送到本地 Collector，再由 Debug Exporter 和 zPages 提供接收证据。

    这不是生产部署教程。它没有后端持久化、TLS、认证、高可用、容量控制或敏感数据治理；所有端口只绑定到 127.0.0.1。不要把本教程的 Collector 配置、开放端口或调试出口复制到生产环境。

    本文于 2026-07-12 按官方 Collector Quick Start 复核，并固定 Collector 与 telemetrygen 为 0.156.0。若该版本镜像或 Go Module 在你的平台不可用，停止实验并到官方发行页选择同一版本的 Collector 与生成器，不要混用不明版本。

    你将证明什么¶

    完成后，你应能分别提供三段证据：

    1. 生成器报告已生成并发送三条 Trace。
    2. Collector 日志中出现接收到的 Span，且同一 Trace 内能看到 Trace ID、Span ID 和 Parent ID。
    3. 本地 zPages 的 TraceZ 页面能显示 Collector 处理过的 Trace 活动。
    flowchart LR T[telemetrygen\n生成测试 Span] -->|OTLP/gRPC\n127.0.0.1:4317| C[Collector 0.156.0] C --> D[Debug Exporter\n终端日志] C --> Z[zPages\n127.0.0.1:55679] D --> V1{核对 ID 与父子关系} Z --> V2{核对接收活动}

    这三个观察点不能互相替代：生成器成功不证明 Collector 接收；Collector 日志有 Span 不证明真实生产后端可查询；zPages 是本地排障界面，不是监控后端。

    前置条件¶

    在 macOS、Linux 或 WSL 的 Bash 兼容终端准备：

    • 可运行 Linux 容器的 Docker 或兼容容器运行时；
    • 已安装 Go，且 go install 能访问公开的 Go Module；
    • 本机 4317、4318、55679 端口未被占用；
    • 至少能拉取 otel/opentelemetry-collector:0.156.0；
    • 当前目录允许创建一个临时实验目录。

    先检查工具，不要跳过错误：

    docker version
go version

    任一命令失败就停止。不要使用 sudo 绕过 Docker 权限问题；先修复本机容器运行时配置。

    检查端口：

    lsof -nP -iTCP:4317 -sTCP:LISTEN
lsof -nP -iTCP:4318 -sTCP:LISTEN
lsof -nP -iTCP:55679 -sTCP:LISTEN

    这些命令没有输出表示没有进程监听对应 TCP 端口。若有输出，停止并释放端口，或在理解 Collector 与客户端参数后整体改用另一组仅本地绑定的端口。

    建立隔离工作目录¶

    export OTEL_LAB_DIR="$(mktemp -d "${TMPDIR:-/tmp}/otel-first-trace.XXXXXX")"
export OTEL_CONTAINER="otel-first-trace-$(date +%Y%m%d%H%M%S)-$$"
mkdir -p "$OTEL_LAB_DIR/bin"
touch "$OTEL_LAB_DIR/.l2do-otel-lab"
cd "$OTEL_LAB_DIR"
export GOBIN="$PWD/bin"

    确认后续安装只写入该目录：

    printf 'GOBIN=%s\n' "$GOBIN"

    输出应以当前随机实验目录的 /bin 结尾。OTEL_LAB_DIR、OTEL_CONTAINER 和后续命令必须在同一个 Shell 中使用；变量缺失时停止，不要手工猜测路径或容器名。

    安装固定版本的生成器¶

    go install github.com/open-telemetry/opentelemetry-collector-contrib/cmd/telemetrygen@v0.156.0
"$GOBIN/telemetrygen" --help

    第二条命令应显示 telemetrygen 帮助，而不是“找不到文件”。这里固定版本是为了让客户端与本教程使用同一发行线；官方 Quick Start 页面可能随发布更新到新版本，执行时应同时核对页面中的当前镜像版本。

    拉取并确认 Collector 镜像¶

    docker pull otel/opentelemetry-collector:0.156.0
docker image inspect otel/opentelemetry-collector:0.156.0 \
  --format '{{index .RepoDigests 0}}'

    保存第二条命令显示的镜像摘要作为实验记录。摘要取决于镜像仓库和平台，本文不会编造一个固定输出。如果摘要为空或镜像拉取失败，停止，不要临时替换成 latest。

    启动本地 Collector¶

    if docker container inspect "$OTEL_CONTAINER" >/dev/null 2>&1; then
  printf '容器名冲突，停止：%s\n' "$OTEL_CONTAINER" >&2
  exit 1
fi

docker run --detach \
  --name "$OTEL_CONTAINER" \
  --label "l2do.otel-lab=$OTEL_LAB_DIR" \
  -p 127.0.0.1:4317:4317 \
  -p 127.0.0.1:4318:4318 \
  -p 127.0.0.1:55679:55679 \
  otel/opentelemetry-collector:0.156.0

    检查容器状态和启动日志：

    docker ps --filter "name=^/${OTEL_CONTAINER}$"
docker logs "$OTEL_CONTAINER"

    预期容器状态为 Up。日志中不应出现配置解析失败、端口绑定失败或 Collector 立即退出。

    此镜像版本的官方 Quick Start 默认配置暴露：

    • 4317：OTLP/gRPC；
    • 4318：OTLP/HTTP；
    • 55679：zPages。

    这些是 0.156.0 Quick Start 的已复核行为，不是所有发行版和自定义配置的永久默认值（官方步骤）。

    停止条件¶

    出现以下任一情况，不要继续发送数据：

    • 容器不是 Up；
    • Docker 报告端口冲突；
    • 日志显示配置无效或 Receiver/Exporter 启动失败；
    • 端口绑定不是 127.0.0.1；
    • 运行的镜像标签不是 0.156.0。

    执行下方“清理”步骤后，先解决根因。

    生成三条测试 Trace¶

    保持 Collector 运行，执行：

    "$GOBIN/telemetrygen" traces \
  --otlp-endpoint 127.0.0.1:4317 \
  --otlp-insecure \
  --traces 3

    --otlp-insecure 仅适用于这个回环地址上的本地明文实验。它不验证 TLS，不能成为生产参数。

    生成器应报告已生成三条 Trace 并停止 Batch Span Processor。时间戳、文件行号和 ID 会因版本与每次运行而变化，不要把它们与官方示例逐字比较。若命令超时或报告导出失败，先检查 Collector 容器与 4317 监听，不要不断重试制造噪声。

    验证 Collector 确实收到了 Span¶

    将日志保存到实验目录：

    docker logs "$OTEL_CONTAINER" > collector-output.txt 2>&1
grep -E '^Span|(Trace ID|Parent ID|ID|Name|Kind|Start time|End time|Status code)[[:space:]]*:' \
  collector-output.txt

    预期能看到一个或多个 Span # 段落，并包含：

    • 非空的 Trace ID 和 ID；
    • 根 Span 的 Parent ID 为空；
    • 子 Span 的 Parent ID 等于同一 Trace 中另一个 Span 的 ID；
    • Name、Kind、开始与结束时间；
    • 同一组父子 Span 使用相同的 Trace ID。

    具体 ID 是每次生成的随机值。验收的是字段关系，不是某一串示例字符。

    可以先列出本次日志中的 Trace ID：

    grep 'Trace ID' collector-output.txt | awk '{print $NF}' | sort -u

    如果生成器成功但日志没有 Span，数据路径断在客户端导出、网络入口或 Collector Pipeline；此时生成器输出不能作为 Collector 接收证据。

    在 zPages 中交叉验证¶

    浏览器打开：

    http://127.0.0.1:55679/debug/tracez

    选择表中的 Trace 活动，确认页面不是连接失败或空白错误。zPages 展示 Collector 自身处理 Trace 的调试视图，适合核对活动和错误；它不提供长期保留，也不代表某个外部后端已成功接收。

    如果终端日志有 Span 而 TraceZ 无法访问，优先检查 55679 的端口映射和 zPages Extension，不要否定已经由 Debug Exporter 建立的接收证据。

    结果判定¶

    检查 通过证据 失败说明
    客户端生成 telemetrygen 报告生成三条 Trace 生成器、参数或本地网络问题
    Collector 接收与处理 日志存在 Span 及合理的 ID 父子关系 OTLP 入口或 Pipeline 断裂
    调试界面 TraceZ 可访问并显示 Trace 活动 zPages Extension 或端口映射问题

    三项全部通过，才能得出有限结论：测试 Span 已从本地生成器到达并经过这个 Collector。 不能据此推断生产应用埋点、跨服务传播、后端持久化、告警或容量设计已经正确。

    常见失败与定位顺序¶

    connection refused¶

    先运行：

    docker ps --filter "name=^/${OTEL_CONTAINER}$"
docker logs --tail 100 "$OTEL_CONTAINER"

    确认容器存活，再确认生成器指向 127.0.0.1:4317。不要在本教程中把端点改成容器私网 IP。

    Collector 立即退出¶

    查看完整日志。最常见的本地原因是同名容器已存在或端口被占用。本教程使用镜像自带配置；如果你挂载了自定义配置，已经偏离本实验边界，应移除挂载后重做。

    日志很多但找不到 Span¶

    先在生成 Trace 后重新执行 docker logs 保存步骤。不要只查看容器启动前的旧日志。Debug Exporter 用于确认 Collector 接收和处理数据，官方也建议在这类局部排障中使用它（Collector Troubleshooting）。

    清理¶

    无论实验成功或失败，都只删除带有本实验标签的容器和带有本实验标记的随机目录。以下保护检查失败时必须人工检查，不能跳过保护直接执行删除：

    if [ "$(docker inspect "$OTEL_CONTAINER" --format '{{ index .Config.Labels "l2do.otel-lab" }}' 2>/dev/null)" = "$OTEL_LAB_DIR" ]; then
  docker rm --force "$OTEL_CONTAINER"
else
  printf '拒绝删除未确认归属的容器：%s\n' "$OTEL_CONTAINER" >&2
fi

if [ -n "$OTEL_LAB_DIR" ] && [ -f "$OTEL_LAB_DIR/.l2do-otel-lab" ]; then
  cd "${TMPDIR:-/tmp}"
  rm -rf -- "$OTEL_LAB_DIR"
else
  printf '拒绝删除未确认归属的目录：%s\n' "${OTEL_LAB_DIR:-<empty>}" >&2
fi

    确认没有遗留容器：

    docker ps --all --filter "name=^/${OTEL_CONTAINER}$"

    应只显示表头。镜像默认保留在本机缓存；若也要删除固定版本镜像，执行：

    docker image rm otel/opentelemetry-collector:0.156.0

    若其他容器仍引用该镜像，Docker 会拒绝删除；不要使用强制删除影响其他实验。

    从实验走向真实系统¶

    下一步不是把 --otlp-insecure 和 Debug Exporter搬进集群，而是逐层增加真实约束：

    1. 用应用 SDK 或受支持的自动埋点替换 telemetrygen。
    2. 为每个服务定义稳定的 service.name 与 Resource 规则。
    3. 用真实的同步或异步调用验证 Context Propagation。
    4. 为 Collector Receiver 配置 TLS、认证和网络边界。
    5. 增加内存限制、队列、重试、容量与 Collector 自身监控。
    6. 接入测试后端，并分别验证 Collector 发送与后端可查询。

    每增加一层，只改变一个假设并保留前一层证据。这比一次部署完整平台后面对“看不到 Trace”更容易定位。

    官方资料与版本边界¶

    • Collector Quick Start：本实验的官方基础步骤、端口和预期观察点。
    • Collector Configuration：自定义 Pipeline 时的组件与启用规则。
    • Collector Troubleshooting：Debug Exporter 与 zPages 的定位用途。
    • telemetrygen source：本教程固定版本的生成器源码。
