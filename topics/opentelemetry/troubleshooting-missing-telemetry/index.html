    OpenTelemetry 遥测缺失故障排查¶

    触发与目标¶

    用于以下症状：后端找不到服务或已知请求、trace 断裂、指标/日志突然消失、Operator 注解存在但 Pod 没有被注入。目标不是“让 UI 出现一点数据”，而是找到第一处有证据的断点，恢复一条已知请求的端到端链路，同时不扩大数据暴露或应用风险。

    官方事实最后核验于 2026-07-12。运行时默认值和 Operator 行为可能变化，先记录实际版本，再参考兼容性与配置参考。

    先保护业务¶

    如果应用错误率、延迟、启动或资源已经恶化，按自动埋点灰度发布 Runbook回滚，不要为了保住 telemetry 延迟业务恢复。发现真实敏感字段时立即停止相关导出并升级数据事件；agent 回滚不能撤回已导出的数据。

    不要在生产中直接改为 OTEL_TRACES_SAMPLER=always_on、打开全量 debug 日志、关闭 TLS/认证或把 payload 打到控制台。这些“排障捷径”可能造成容量或数据事件。

    事件记录¶

    记录：首次发现时间、受影响信号、服务/namespace/版本、最后已知正常时间、已知请求 ID 与时间窗、Operator/agent/Collector/后端版本、近期变更、独立业务 SLI。没有已知请求时，先发送一条不含真实数据的合成请求；不要用普通流量猜测。

    逐跳定位¶

    1. 后端查询是否正确¶

    用明确时间窗、service.name、环境、版本和已知请求 ID 查询。确认时区、租户/项目、信号类型和采样策略。若只有一部分 span，记录最后一个上游 span 和预期下游服务，问题可能是传播、下游埋点或采样，不是整个 Collector。

    继续条件： 已排除查询条件错误。否则修正查询并结束。

    2. 请求是否真的到达目标实例¶

    使用网关访问日志、应用原生请求计数或合成探针证明请求进入目标 canary/Pod。若请求未到达，修复路由，不要改 OTel。

    kubectl -n <app-namespace> get pod -l <workload-selector> -o wide
kubectl -n <app-namespace> logs deploy/<deployment> -c <app-container> --since=<window>

    日志可能含敏感数据；只获取批准时间窗和必要字段。

    3. Pod 是否被预期地注入¶

    kubectl -n <app-namespace> get deploy <deployment> -o yaml
kubectl -n <app-namespace> describe pod <pod-name>
kubectl -n <app-namespace> get events --sort-by=.metadata.creationTimestamp
kubectl -n <app-namespace> describe otelinst <instrumentation-name>

    检查：注解位于 spec.template.metadata.annotations；语言、资源名和容器名正确；Instrumentation 先于 Pod 存在；Pod 创建后出现预期 init container/sidecar/变量。Operator 官方排障页给出相同部署顺序与检查入口：Auto-instrumentation troubleshooting。

    若 namespace 注入与工作负载注解冲突，以实际 Pod 变更为证据。不要仅凭 Deployment YAML 判断已注入。

    4. 启动配置是否被覆盖¶

    比较镜像声明、ConfigMap、Pod spec 与容器实际环境/进程参数：

    kubectl -n <app-namespace> exec <pod-name> -c <app-container> -- env
kubectl -n <app-namespace> exec <pod-name> -c <app-container> -- ps -ef

    重点检查 JAVA_TOOL_OPTIONS、PYTHONPATH、NODE_OPTIONS。官方文档记录自动注入当前可能覆盖镜像或 ConfigMap 中这些变量：配置源限制。若应用依赖的参数丢失，立即回滚，不要现场拼接未验证的值。

    5. SDK/agent 是否生成信号¶

    用一条已知合成请求检查 agent 的有界诊断输出和应用日志。只在 canary、短时间窗口、批准日志级别下临时增加诊断，结束后恢复。检查：运行时/库是否受支持、埋点是否被禁用、采样是否可能丢弃该请求、传播头是否在代理层被删除、service.name 是否意外变化。

    如果应用手工 SDK 与自动 agent 同时初始化 provider，先核对语言官方兼容方式；不要假定二者可叠加。

    6. 协议、端点和网络是否一致¶

    检查实际生效的 OTEL_EXPORTER_OTLP_PROTOCOL 与 endpoint，不从端口猜协议。OTLP 规范规定 4317/4318 是未配置时的默认端口，端点可以配置为其他端口；http/json 是 SDK 可以实现的可选形式，不是所有运行时都必须支持：OTLP Exporter specification。

    核对 DNS、NetworkPolicy、Service Endpoint、TLS SNI/CA、客户端认证和 Collector receiver。不要通过关闭证书校验来证明网络可达。

    kubectl -n <collector-namespace> get svc,endpoints,networkpolicy
kubectl -n <collector-namespace> logs deploy/<collector-deployment> --since=<window>

    7. Collector 在哪里丢失¶

    按 pipeline 查看 receiver accepted/refused、processor drop、exporter sent/send_failed 和 queue size/capacity。指标名随版本变化，先对照实际 Collector 的 Internal telemetry 文档。

    • receiver 没增长：问题在 agent、网络、认证或 receiver。
    • accepted 增长但 exporter sent 不增长：检查 processor/filter、memory limiter、routing 和 pipeline 绑定。
    • send_failed 或 queue 增长：检查后端认证、限流、网络与容量。
    • queue 满：停止扩大输入；发送队列满仍会丢数据，见 Collector resiliency。

    不要为了“先通”而移除敏感字段过滤器；使用带唯一、非敏感标记的测试请求跟踪 pipeline。

    8. 后端是否接收但不可查询¶

    检查后端接收错误、租户路由、索引延迟、保留、限流和时钟偏差。比较 Collector exporter sent 与后端 accepted。如果 Collector 证明成功发送，而后端无记录，将证据升级给后端负责人，不要反复重启应用。

    症状到动作¶

    证据 最可能断点 动作
    Pod 无注入变更 资源顺序、注解位置/名称、webhook 修正 canary 配置并重新创建 Pod。
    Pod 变更但应用启动异常 版本/架构不兼容或环境覆盖 立即显式 "false" 回滚。
    agent 报连接/TLS/认证错误 endpoint、协议、信任、NetworkPolicy 修正批准的连接配置，不降级安全。
    receiver accepted 为零 agent 到 receiver 路径 逐项验证 DNS、Service、NetworkPolicy、TLS 和认证。
    receiver accepted 增长、exporter sent 不增长 processor/pipeline 审核 filter、routing、memory limiter 和 pipeline 绑定。
    send_failed/queue 增长 后端或 Collector 容量 停止推广，保护队列，升级后端/平台。
    后端有 root span、缺下游 context propagation 或下游埋点 检查代理头部、下游采样和下游 agent。
    trace 存在但服务名错误 resource 配置 修正显式资源属性；不删除旧数据冒充修复。

    停止、回滚与升级¶

    立即回滚：应用独立 SLI 越界、进程参数被覆盖、CrashLoop、无关 Pod 被注入、无法限定诊断范围。回滚时对语言注解设置 "false" 并创建新 Pod；namespace 可能启用注入时，仅删除注解不足。

    立即升级数据安全：真实敏感字段或负向测试标记抵达未批准位置。立即升级平台：webhook 阻断无关 Pod 创建、证书/CA 失效、Collector 队列接近耗尽。升级后端：Collector 有成功发送证据但后端无法查询。

    完成与复盘证据¶

    关闭故障前必须保存：第一处断点、根因、版本矩阵、已知请求的端到端证据、独立 SLI、Collector 指标、修复/回滚 diff 和临时诊断已撤销的证据。若仍不能证明数据去哪了，状态应为“缓解但未解决”，不能以 UI 恢复少量数据结案。
