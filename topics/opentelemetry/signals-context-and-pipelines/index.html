    信号、上下文与 Pipeline¶

    OpenTelemetry 的难点不在于记住四个名词，而在于建立一条完整因果链：系统活动产生信号，Context 把分散活动连接起来，Pipeline 把数据可靠地送到可查询的位置。 三者任意一处断裂，都可能出现“有数据却解释不了故障”。

    本文按截至 2026-07-12 的官方 OpenTelemetry 概念与 Collector 配置文档复核。信号和组件的稳定性会演进，尤其是 Profiles、Events 与各 Collector 组件；生产采用时应核对目标版本。

    四类信息回答不同问题¶

    信息 基本单位 最擅长回答 不擅长单独回答
    Trace Trace / Span 一次请求经过哪里，父子调用和耗时如何分布 整体影响面与长期趋势
    Metric 时间序列中的测量点 多少请求受影响，趋势与分位数是否越界 某一次请求为何失败
    Log 带时间和上下文的事件记录 某个代码路径记录了什么离散事件 跨服务因果关系，除非带有可关联上下文
    Baggage 随 Context 传播的键值 将有限的上下文带到下游埋点逻辑 作为安全的数据仓库或观测记录

    官方信号概览目前列出 Traces、Metrics、Logs 和 Baggage，并将 Profiles、Events 标注为开发或提案阶段（Signals；复核日期 2026-07-12）。这里把 Baggage 与前三者分开理解：它的主要作用是随调用传播上下文，本身并不自动成为一条可查询的 Span、Metric 或 Log。

    Trace：重建一次执行的路径¶

    Trace 是一组共享 Trace ID 的 Span。Span 表示一个有开始和结束的操作，例如接收 HTTP 请求、调用库存服务或执行一条数据库语句。除了时间，Span 通常还携带：

    • 自己的 Span ID 与父 Span ID；
    • 操作名称与 Span Kind；
    • 状态、事件和属性；
    • 产生它的 Resource，例如服务名与版本。

    父子关系表示因果，不只是先后。两个操作在时间上接近，不等于其中一个导致另一个；只有正确传播并提取 Context，后端才能重建调用树。

    Trace 特别适合解释尾延迟和局部失败，但采样意味着“没有看到某条 Trace”不能直接证明请求没有发生。诊断影响面仍应结合指标或其他独立数据源。

    Metric：压缩后的总体行为¶

    Metric 把大量事件聚合成随时间变化的测量。Counter 适合累计数量，Gauge 表示某时刻状态，Histogram 描述值的分布。它们让工程师看到错误率、请求率、队列深度和延迟分布，而不必保存每一次操作的完整细节。

    Metric 的力量来自聚合，风险也来自标签维度。把请求 ID、用户 ID 或任意 URL 放入属性，会把有限的时间序列扩张成高基数集合，增加内存、网络和存储成本。一个能出现在 Span 上的属性，不代表适合成为 Metric 维度。

    Log：离散事件的证据¶

    Log 记录某个时刻发生的事件，既可能是结构化记录，也可能是传统文本。高质量日志应表达事件、结果和必要上下文，而不是把整份请求、凭据或对象转储出来。

    当日志记录包含当前 Trace ID 和 Span ID 时，后端可以从失败 Span 跳到同一次执行的日志。但这种关联依赖：

    1. 日志产生时进程内存在正确的当前 Context；
    2. 日志 SDK 或桥接器把标识加入记录；
    3. 采集与后端保留并索引这些字段。

    少任何一步，都不能因为“接入了 OTel”而假设日志已经与 Trace 关联。

    Baggage：传播的数据，不是放秘密的地方¶

    Baggage 是随 Context 穿过服务边界的键值。下游埋点可以读取它并决定添加什么属性，例如有限的业务流程类别。它不会自动成为 Span 属性，也不应该被当成分布式 Session。

    Baggage 可能经过多个进程甚至外部服务。不得放入密码、令牌、个人信息或无需下游知道的内部标识。官方安全指引明确要求谨慎接收不可信来源的 Context，并避免在 Baggage 中传播凭据、API Key 或个人信息（Context propagation：Security best practices）。

    Context 如何把服务连接起来¶

    Context 是当前执行所携带的关联信息；Propagation 是把它序列化到传输载体，并在另一端反序列化的机制。HTTP 常用 Header 作为载体，消息队列可能使用消息属性。

    sequenceDiagram participant U as Client participant A as Checkout API participant Q as Message Broker participant B as Inventory Worker U->>A: HTTP request + traceparent A->>A: extract Context<br/>create checkout Span A->>Q: publish message<br/>inject updated Context Q-->>B: deliver message + metadata B->>B: extract Context<br/>create reserve-inventory Span Note over A,B: Same Trace ID; each operation has its own Span ID

    发送端执行 inject，把当前 Trace ID、当前 Span ID 和采样标记写入载体；接收端执行 extract，将远端 Context 设为新 Span 的 Parent。OpenTelemetry 默认传播器使用 W3C Trace Context 的 traceparent Header，其结构为：

    <version>-<trace-id>-<parent-id>-<trace-flags>

    这个 Header 是关联信息，不是访问凭据。外部调用边界仍需决定是否接受、清洗或继续传播入站 Context，以防伪造数据污染 Trace。相关机制与安全边界见官方上下文传播文档。

    Context 断裂会是什么样¶

    • 下游创建了新的 Trace ID：通常是没有提取，或传输层没有携带 Header/消息属性。
    • Trace ID 相同但父子关系错误：可能在异步或并发执行中错误复用了当前 Context。
    • HTTP 正常、消息链断裂：HTTP 库被自动埋点，消息客户端或自定义协议没有对应埋点。
    • 日志没有 Trace ID：记录日志时没有活动 Span，或日志桥接未配置。

    不要用时间戳“猜”两段 Trace 是否属于一次请求。时间接近只能形成假设，不能恢复丢失的因果证据。

    Resource 与 Context 不可互换¶

    Resource 描述产生遥测的实体，例如 service.name=inventory、服务版本或 Pod；它在许多记录之间相对稳定。Context 描述当前执行关系，会随每次请求和 Span 改变。

    简单记忆：

    • Resource 回答“谁产生了它”；
    • Context 回答“它属于哪一次执行、哪个父操作”；
    • Attribute 回答“这个操作还有什么特征”。

    把服务名塞进 Baggage、把请求 ID 做成 Resource、把用户 ID 做成 Metric 维度，都会造成错误模型和高昂返工。

    Pipeline 如何处理数据¶

    Collector 配置中定义组件并不等于组件已运行。只有把组件放进 service.pipelines，对应数据流才启用。一个典型 Pipeline 是 Receiver → Processor → Exporter；Processor 可为零个或多个，顺序决定处理顺序。

    Receiver¶

    Receiver 接受 Push 数据或主动抓取数据。它解决的是协议与入口问题，不证明数据已经通过 Processor 或被后端接受。对网络 Receiver，监听地址、TLS、认证和暴露范围都属于安全边界。

    Processor¶

    Processor 在导出前修改、丢弃、聚合或批处理数据。顺序会改变结果：先过滤再批处理与先批处理再过滤的资源行为不同。配置了 Processor 但没有把它列入 Pipeline，不会产生任何效果。

    Exporter¶

    Exporter 把数据发送到后端或其他 Collector。Exporter 成功通常只表示目标端接受了发送，不等于数据已可查询、已被正确索引或满足保留策略。后端侧仍要独立验证。

    Connector 与 Extension¶

    Connector 同时作为一个 Pipeline 的 Exporter 和另一个 Pipeline 的 Receiver，可在信号间生成或路由数据。Extension 提供健康检查、认证等能力，但不直接位于遥测数据路径中。两者也必须在相应的 service 配置中启用。组件结构和启用规则见官方Collector Configuration。

    数据可能在哪里丢失¶

    业务操作
  → 未被埋点
  → SDK 采样丢弃
  → 进程退出前未导出
  → 网络、协议或认证失败
  → Collector 接收或队列拒绝
  → Processor 主动过滤
  → Exporter 重试耗尽
  → 后端拒绝、限流或索引失败

    这条链说明为什么“后端没有 Trace”不是一个根因。排障必须逐段建立证据：应用是否创建 Span、SDK 是否尝试导出、Collector 是否接收、Processor 是否保留、Exporter 是否发送、后端是否可查询。

    调试 Exporter 可以证明数据到达某个 Collector Pipeline，并展示处理后的内容；它不能证明生产后端可用。官方也将其定位为本地或现场排障工具（Collector Troubleshooting）。

    采样改变你能得出的结论¶

    Head sampling 在 Trace 开始附近作决定，成本较低，但尚不知道请求最终是否失败。Tail sampling 要在 Collector 等位置等待更多 Span 后决定，能保留错误或高延迟 Trace，却需要缓存完整度、等待窗口和容量设计。

    无论哪种方式，采样都不是单纯的成本旋钮：它改变证据集合。若入口与下游对采样决定理解不一致，可能形成局部 Trace；若 Collector 在等待尾部决策时过载，最需要保留的异常 Trace 也可能丢失。

    因此应把“采样策略、实际保留率、丢弃位置和适用查询”一起记录，而不是只写一个百分比。

    用三个问题检查设计¶

    能否定位一条执行¶

    从入口请求到下游同步和异步边界，Trace ID 是否连续？每个边界由谁 inject/extract？不可信边界怎样处理 Context？

    能否判断总体影响¶

    哪些指标不依赖 Trace 采样？标签基数上限是什么？错误率与延迟来自应用、代理还是后端派生？

    能否证明数据链路健康¶

    应用、Collector 和后端各有什么独立证据？能否区分“没有业务事件”和“遥测链路丢数”？Collector 的接收、拒绝、发送失败和队列指标由另一条可靠路径监控吗？

    如果这三组问题没有答案，增加更多自动埋点通常只会产生更多无法解释的数据。
