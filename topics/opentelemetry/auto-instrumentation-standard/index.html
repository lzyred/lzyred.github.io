    OpenTelemetry 自动埋点生产标准¶

    目的与适用范围¶

    本标准规定在 Kubernetes 工作负载中通过 OpenTelemetry Operator 注入 Java、Node.js、Python、.NET、Go 或 SDK 配置时的最低生产门禁。它约束的是变更是否可以上线，具体操作见自动埋点灰度发布 Runbook，运行时差异见兼容性与配置参考。

    自动埋点会在 Pod 创建时改变容器环境、启动参数、文件系统内容或 sidecar，并增加 CPU、内存和遥测出站。它必须按生产代码变更管理，不能作为命名空间级“可观测性开关”。Operator 官方文档说明了注入发生在 Pod 创建阶段，并要求先创建 Instrumentation 资源再创建应用 Pod：Injecting Auto-instrumentation。

    本文中的 MUST 表示上线阻断项，SHOULD 表示默认要求，偏离时必须记录理由，MAY 表示可选实现。

    版本与事实边界¶

    • 本文官方事实最后核验于 2026-07-12。
    • 本文不声明任意 Operator、Kubernetes、Collector、运行时和 agent 版本彼此兼容。
    • 每次变更 MUST 记录实际版本、镜像 digest、基础镜像 libc、协议、Collector 发行版与最后一次预生产验证结果；缺一项即不能推广。
    • 升级任一组件后，历史验证证据失效，MUST 重新执行 canary。

    版本记录应至少包含：Kubernetes 小版本、Operator 版本、Instrumentation CRD 版本、自动埋点镜像 digest、应用运行时版本、应用镜像 digest、Collector 版本和后端接收版本。易变限制集中维护在兼容性与配置参考。

    上线规则矩阵¶

    规则 要求 最低证据 未满足时
    OTI-001 MUST 固定并验证全部版本与镜像 digest，不得以“最新版”或浮动标签作为批准依据。 变更单中的版本矩阵、镜像 digest、预生产测试日期。 阻断上线。
    OTI-002 MUST 只对一个独立 canary 工作负载生效。不能给现有 Deployment 的全部 Pod 模板直接加注解后称为 canary。 独立 Deployment/ReplicaSet 名称、固定流量比例或专用合成流量、回滚对象。 阻断上线。
    OTI-003 MUST 在变更前盘点 JAVA_TOOL_OPTIONS、PYTHONPATH、NODE_OPTIONS 及运行时启动参数。 镜像配置、ConfigMap、Pod 环境和实际进程参数的审查记录。 存在冲突则阻断。
    OTI-004 MUST 显式配置 OTLP 协议、端点、TLS 和认证，并用 NetworkPolicy 或等价控制限制遥测出口。 Collector receiver 配置摘要、证书信任链测试、认证测试、允许的源/目的网络规则。 阻断上线。
    OTI-005 MUST 证明敏感字段不会离开批准边界。 包含假令牌、Cookie、查询参数、用户标识和请求体标记的负向测试；后端查询结果为零。 阻断并按数据事件升级。
    OTI-006 MUST 证明 Collector 在当前负载和后端中断预算下有容量及高可用。 副本、反亲和/PDB、资源、队列、重试、容量测量及内部指标仪表板。 阻断上线。
    OTI-007 MUST 审核 admission webhook 的 failurePolicy、timeoutSeconds、CA bundle、证书到期和 Service Endpoint。 webhook 配置快照、证书到期监控、故障演练结果。 阻断上线。
    OTI-008 MUST 使用不依赖新遥测链路的 SLI 判断应用健康。 负载均衡器/网关指标、合成请求、应用原生指标或业务成功率的基线与阈值。 阻断上线。
    OTI-009 MUST 预先定义量化的通过、停止和回滚阈值。 变更单中的观察窗口、样本量、p95/p99、错误率、重启、RSS/CPU、Collector 拒收/队列及成本阈值。 阻断上线。
    OTI-010 MUST 能以显式 "false" 覆盖 namespace 注入，并创建未注入的新 Pod。 回滚补丁、预演输出及干净 Pod 证据。 阻断上线。
    OTI-011 SHOULD 先仅启用 traces；metrics 与 logs 作为独立变更。 信号范围记录。 记录偏离理由和额外容量/数据评审。
    OTI-012 SHOULD 每种语言、每个命名空间使用明确命名的 Instrumentation 资源，并显式指定目标容器。 资源名和容器名注解。 记录为何不会误注入。

    真正的 Canary¶

    Canary 必须拥有独立缩放、独立回滚和可识别流量。以下任一方案可接受：

    • 从稳定 Deployment 复制出 example-service-otel-canary，只运行少量副本，并由网关按固定比例或请求头路由；
    • 使用专用测试入口，仅发送无敏感数据的合成请求；
    • 在支持分批发布的控制器中建立独立 canary revision，并证明只影响选定副本。

    直接修改稳定 Deployment 的 spec.template.metadata.annotations 会触发整个 Deployment 的滚动替换。这是一次全量渐进 rollout，不是独立 canary，不满足 OTI-002。

    量化变更单¶

    阈值不能由本文替所有服务编造。服务负责人必须根据变更前基线填写绝对值或相对变化，并说明样本量与观察窗口。

    输入 示例写法（仅说明格式）
    流量与样本 canary 5% 流量；至少 10,000 请求；观察 30 分钟
    延迟 p95 增幅不超过已批准值；p99 不超过服务 SLO
    正确性 5xx/业务失败率增量不超过已批准值
    资源 CPU、RSS、启动时间、GC 增幅均有上限
    稳定性 无新增重启；readiness 失败数不增长
    Collector queue/capacity、send_failed、refused、export latency 均有上限
    成本 估算和实测 span/s、bytes/s、保留成本不超过预算
    停止条件 任一 MUST 证据缺失、敏感字段命中或独立 SLI 恶化即停止

    采样率必须由入口吞吐、平均 trace 宽度、错误/慢请求保留策略、Collector 容量和后端预算反推。固定写成 5% 或 100% 都不能作为通用标准。

    数据安全门禁¶

    OTI-004 与 OTI-005 必须在首次注入前完成：

    1. 应用到 Collector 使用组织批准的 TLS/认证方式；跨信任边界时不得使用明文 OTLP。
    2. 凭据优先保存在 Collector，而不是复制到每个工作负载。
    3. NetworkPolicy 或等价控制只允许应用访问批准的 Collector，只允许 Collector访问批准的后端。
    4. Collector 的 filter、transform 或属性处理不是“已配置即通过”；必须发送带假敏感标记的请求并在 Collector 出口和后端查询验证为零。
    5. 若发现真实令牌、Cookie、个人数据或租户标识，立即停止导出、保存最小调查证据并按组织的数据事件流程升级；回滚 agent 不能撤回已经导出的数据。

    OpenTelemetry 官方安全指南强调敏感数据识别依赖组织语境，并建议在采集器或 SDK 处删除、哈希或过滤：Handling sensitive data。

    Collector 与 Webhook 门禁¶

    Collector MUST 至少满足：

    • 生产环境无单副本单点；副本数、拓扑分布和 PDB 与可用性目标一致；
    • 根据实测接收率、平均批大小、允许后端中断时长和内存预算配置发送队列、重试与持久化需求；
    • 配置 memory_limiter 或经验证的等价保护，并为容器设置资源请求/限制；
    • 监控 otelcol_receiver_accepted_*、otelcol_receiver_refused_*、otelcol_exporter_sent_*、otelcol_exporter_send_failed_* 以及 exporter queue size/capacity；具体指标名会随 Collector 版本变化，部署时以该版本官方 internal telemetry 文档为准；
    • 用后端中断演练证明队列耗尽前能告警，恢复后不会无限积压。

    官方文档说明发送队列在下游不可用且队列满时仍会丢数据，并建议按队列、内存与接收速率规划：Collector resiliency、Scaling the Collector、Internal telemetry。

    Webhook 是 Pod 创建控制路径的一部分。团队必须明确选择 failurePolicy: Fail 还是 Ignore，并记录安全与可用性权衡；不得假设一种选择普遍正确。Kubernetes 文档说明超时后的行为由 failurePolicy 决定，且 timeoutSeconds 可配置：Dynamic Admission Control。证书到期、CA bundle 不匹配、Service 无 Endpoint 和 webhook 延迟都必须告警。

    独立验证与合规判定¶

    通过必须同时具备四类证据：

    1. 变更证据：只有 canary Pod 被注入，目标容器和 agent 版本正确。
    2. 遥测证据：一条已知合成请求形成预期 trace，资源属性可归属，Collector 无拒收或失败增长。
    3. 独立健康证据：网关、合成监控或应用原生 SLI 满足变更单阈值；不能用正在被验证的 OTel 链路自证健康。
    4. 负向证据：假敏感标记在 Collector 出口和后端均不可查询。

    任何 OTI-001 至 OTI-010 未满足，结论只能是“不合规”。“Pod Ready”“后端出现服务名”或“没有告警”都不能单独构成批准。

    例外机制¶

    MUST 规则的例外至少记录：规则 ID、业务理由、风险负责人、补偿控制、影响范围、开始与失效时间、回滚触发器和批准人。例外必须有到期时间；到期未续批即停止推广。OTI-005（真实敏感数据泄露）不得用普通变更例外豁免。

    回滚边界¶

    当 namespace 可能启用注入时，回滚必须在工作负载 Pod 模板显式设置对应语言注解为 "false"，而不只是删除注解。Operator 官方文档把 "false" 定义为禁止注入的有效值。随后创建新 Pod，并证明其中不存在注入 init container、sidecar、agent 文件和注入环境变量。详见自动埋点灰度发布 Runbook。
