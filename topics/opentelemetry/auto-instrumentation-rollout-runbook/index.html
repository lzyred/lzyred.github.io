    OpenTelemetry 自动埋点灰度发布 Runbook¶

    何时使用¶

    当已通过自动埋点生产标准的设计评审，需要在 Kubernetes 中首次启用或升级 Operator 自动埋点时使用。本 Runbook 不批准 namespace 全量注入，也不替团队决定采样率、性能阈值或数据保留期。

    官方事实最后核验于 2026-07-12。执行前按兼容性与配置参考重新核对部署版本；本文不假定跨版本兼容。

    角色与变更输入¶

    开始前指定服务负责人、执行人、回滚负责人、可观测性负责人和数据事件升级联系人。变更单必须填入：

    • 稳定与 canary Deployment、镜像 digest、运行时和目标容器；
    • Kubernetes、Operator、CRD、agent 镜像、Collector 和后端版本；
    • OTLP 协议、端点、TLS/认证、NetworkPolicy；
    • canary 流量方式、最小样本量、观察窗口；
    • 独立 SLI 及 p95/p99、错误率、重启、启动、CPU/RSS 阈值；
    • Collector accepted/refused/send_failed/queue 和成本阈值；
    • 敏感字段负向测试标记；
    • 回滚补丁和升级联系人。

    任一输入为空就停止。

    前置检查¶

    1. 证明 canary 独立¶

    canary 必须是独立 Deployment/revision，能够独立缩放和删除。确认 Service 或网关能按固定比例、请求头或专用入口区分流量。不要直接修改稳定 Deployment 的 Pod 模板。

    kubectl -n <app-namespace> get deploy <stable-deployment> <canary-deployment> -o wide
kubectl -n <app-namespace> get svc,httproute,ingress

    2. 盘点环境覆盖¶

    Operator 官方故障排查记录了一个当前已知问题：镜像或 ConfigMap 中的 JAVA_TOOL_OPTIONS、PYTHONPATH、NODE_OPTIONS 可能被自动注入覆盖。官方说明

    kubectl -n <app-namespace> get deploy <canary-deployment> -o yaml
kubectl -n <app-namespace> get configmap <runtime-config> -o yaml
kubectl -n <app-namespace> exec deploy/<canary-deployment> -c <app-container> -- env
kubectl -n <app-namespace> exec deploy/<canary-deployment> -c <app-container> -- ps -ef

    如果这些变量承载应用必需参数、无法确认合并结果，停止并在预生产镜像中解决；不要在生产现场试错。

    3. 检查 Operator webhook¶

    kubectl -n <operator-namespace> get deploy,pod,svc,endpoints
kubectl get mutatingwebhookconfiguration -o yaml
kubectl -n <operator-namespace> logs deploy/<operator-deployment> --since=15m

    确认 failurePolicy 和 timeoutSeconds 与变更单一致，CA bundle 非空，证书未临期，Service 有 Endpoint，日志没有 TLS、超时或注入错误。Kubernetes 说明 webhook 超时后是忽略还是拒绝请求由 failurePolicy 决定：Dynamic Admission Control。

    4. 检查 Collector 边界¶

    kubectl -n <collector-namespace> get deploy,statefulset,pod,svc,pdb,networkpolicy
kubectl -n <collector-namespace> get secret <collector-client-identity> -o name
kubectl -n <collector-namespace> logs deploy/<collector-deployment> --since=15m

    证明接收端协议/端口、TLS 信任和认证匹配；Collector 有批准的副本与容量；内部指标无拒收、导出失败或队列逼近阈值。不能用打印 Secret 内容的方式验证认证。

    5. 保存基线¶

    从独立于 OTel 的来源保存观察窗口内的请求数、p95/p99、业务错误率、Pod readiness/restarts、CPU/RSS 和启动时间。同时保存 Collector 接收率、拒收率、发送失败和 queue/capacity。没有基线就停止。

    执行¶

    1. 先创建或更新 Instrumentation¶

    以下是仅用于 Java canary、仍需按环境替换并评审的示意资源，不是可直接复制的完整生产配置。它把协议显式设为 http/protobuf，因此示例端点使用 Collector 实际提供的 OTLP/HTTP Receiver。不得把这段配置原样换一个语言注解后复用：Node.js 等运行时必须根据已验证版本显式选择协议，并使用与 Collector 实际 Receiver 匹配的端点。

    apiVersion: opentelemetry.io/v1alpha1
kind: Instrumentation
metadata:
  name: java-canary
  namespace: <app-namespace>
spec:
  exporter:
    endpoint: https://<collector-service>:4318
  propagators: [tracecontext, baggage]
  sampler:
    type: parentbased_traceidratio
    argument: "<approved-ratio>"
  java:
    env:
      - name: OTEL_EXPORTER_OTLP_PROTOCOL
        value: http/protobuf
  # 按已批准方案配置证书信任、认证和其它安全参数。

    先应用 Instrumentation，再创建 canary Pod。Operator 官方文档明确指出顺序错误会导致注入失败：Auto-instrumentation troubleshooting。

    kubectl apply --server-side --dry-run=server -f <instrumentation-manifest>
kubectl apply -f <instrumentation-manifest>
kubectl -n <app-namespace> describe otelinst java-canary

    2. 只给 canary 添加注解¶

    先对补丁做 server-side dry run。注解必须在 spec.template.metadata.annotations，并显式选择资源和容器。

    spec:
  template:
    metadata:
      annotations:
        instrumentation.opentelemetry.io/inject-java: "java-canary"
        instrumentation.opentelemetry.io/container-names: "<app-container>"

    对 Python、Node.js、.NET 或 Go，必须先依据 Reference 和部署版本创建各自的 Instrumentation，显式写出协议与实际端点，再使用对应注解；不要继承上面的 Java 配置。应用经评审的补丁后：

    kubectl -n <app-namespace> rollout status deploy/<canary-deployment> --timeout=<approved-timeout>
kubectl -n <app-namespace> get events --sort-by=.metadata.creationTimestamp
kubectl -n <app-namespace> describe pod -l <canary-selector>

    3. 逐层验证¶

    1. Pod 变更：只有 canary 新 Pod 被创建；目标容器收到正确 agent；稳定 Pod 未变化。
    2. 进程健康：readiness、启动时间、重启、日志和实际进程参数满足阈值。
    3. 已知请求：发送不含真实数据且带唯一测试标记的合成请求，记录时间和请求 ID。
    4. Trace：在预期窗口内找到完整 trace，service.name、环境、版本和传播关系正确。
    5. Collector：accepted 增长与请求量合理，refused/send_failed/queue 未越界。
    6. 独立 SLI：网关/合成/应用原生 SLI 满足阈值。
    7. 负向测试：在测试请求中加入预先批准的假 token、Cookie、查询参数和 body 标记；Collector 出口与后端均为零匹配。

    决策与回滚图¶

    flowchart TD A[前置证据齐全] -->|否| S[停止，不创建注入 Pod] A -->|是| C[创建独立 canary] C --> M{Pod 变更符合预期?} M -->|否| R[显式 false 并替换 Pod] M -->|是| D{敏感字段负向测试为零?} D -->|否| I[停止导出并升级数据事件] D -->|是| H{独立 SLI 与 Collector 阈值通过?} H -->|否| R H -->|是| O[完成观察窗口] O --> P{负责人批准下一批?} P -->|否| K[保持 canary，不扩大] P -->|是| N[一次只推进一个已定义批次]

    立即停止条件¶

    出现任一项立即停止流量扩大：

    • 稳定工作负载被意外注入，或 webhook 影响无关 Pod 创建；
    • 发现真实敏感字段，或负向测试标记出现在出口/后端；
    • 任一独立 SLI、资源或 Collector 阈值越界；
    • canary CrashLoop、启动/readiness 超时或进程参数被覆盖；
    • webhook 证书、CA、Endpoint 或 failurePolicy 行为与记录不符；
    • 无法区分 canary 与稳定流量，或样本量不足。

    回滚¶

    如果 namespace 可能存在注入，不要只删除注解。在 canary Pod 模板显式设置对应语言注解为 "false"；Operator 官方定义该值为不注入：Injecting Auto-instrumentation。

    spec:
  template:
    metadata:
      annotations:
        instrumentation.opentelemetry.io/inject-java: "false"
    1. 先将 canary 流量降为零。
    2. 应用显式 "false" 补丁并 rollout canary。
    3. 确认新 Pod 不含注入 init container/sidecar、agent 文件和注入变量。
    4. 用独立 SLI 确认服务恢复基线。
    5. 保留 Instrumentation 和 Operator，直到所有依赖工作负载均清理；不要先拆平台组件。
    6. 若有真实数据泄露，另行处置后端数据；回滚不能撤回已导出数据。

    升级¶

    以下情况升级给平台/可观测性负责人：Collector 接收成功但后端缺数、队列持续增长、证书或 webhook 失效、agent 与运行时冲突。敏感数据命中升级给数据安全负责人；特权 Go sidecar 未获批准升级给安全负责人。记录命令、时间、资源版本和非敏感错误摘要，不复制凭据或真实 payload。

    完成条件¶

    只有当完整观察窗口和样本量满足、所有 MUST 证据归档、回滚预演有效、稳定服务无变化，才可建议下一批。下一批仍是新的受控变更，不能因为 canary 一次通过就批准整个 namespace。
