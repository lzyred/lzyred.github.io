    OpenTelemetry 兼容性与配置参考¶

    使用边界¶

    本页是查询入口，不是“已认证版本清单”。官方事实最后核验于 2026-07-12，但 OpenTelemetry Operator、各语言 agent、Deno 和 Collector 独立发布。部署者必须用实际版本的官方 release notes、镜像 digest 和预生产测试建立自己的兼容性证据，不能从本页推导任意组合都受支持。

    每次升级重新填写：

    组件 必填证据
    Kubernetes 小版本、节点架构、容器运行时
    Operator 版本、Helm chart/manifest 版本、feature gates
    CRD Instrumentation apiVersion 与已安装 schema
    自动埋点 语言、镜像 tag 和 digest、agent/runtime 版本
    应用镜像 digest、CPU 架构、glibc/musl、启动参数
    Collector 发行版/版本、启用组件、receiver/exporter 配置摘要
    后端 OTLP 接收版本、认证/TLS、限流与保留
    验证 环境、日期、已知请求、性能/数据安全结果

    任一项变化都应视为新组合。latest、只记录 chart 版本或“测试环境能启动”都不是兼容证据。

    OTLP 协议与端口¶

    传输 未配置端口时的默认值 常见 endpoint 形态 注意
    OTLP/gRPC 4317 https://collector.example:4317 Collector 端必须启用 gRPC receiver。
    OTLP/HTTP 4318 https://collector.example:4318 信号通常使用 /v1/traces、/v1/metrics、/v1/logs 路径；SDK 对 base URL/信号 URL 的拼接遵循实现和规范。

    4317 和 4318 是规范在端口未配置时的默认值，不是固定端口。Collector 可监听其他端口，Service 也可重映射；必须检查实际 endpoint 和 receiver，不能仅凭端口猜协议。OTLP Exporter specification

    规范定义 grpc、http/protobuf，并允许 SDK 可选支持 http/json。因此：

    • 不要假设所有语言 agent 都支持 http/json；
    • 不要依赖某语言当前默认协议；显式设置并进行端到端测试；
    • endpoint 的 http:///https:// 描述安全传输，不表示“这是 OTLP/HTTP 而不是 gRPC”；
    • 生产跨信任边界使用批准的 TLS 与认证，不用关闭校验证明连通。

    Operator 注入入口¶

    注解放在 spec.template.metadata.annotations。值可以是当前 namespace 默认资源的 "true"、资源名、namespace/name 或禁止注入的 "false"。namespace 级注解会让其中工作负载 opt in，因此回滚时工作负载应显式设 "false"。Operator auto-instrumentation

    运行时 注入注解 主要边界
    Java instrumentation.opentelemetry.io/inject-java 检查 JAVA_TOOL_OPTIONS 覆盖和 JVM/库兼容。
    Node.js instrumentation.opentelemetry.io/inject-nodejs 检查 NODE_OPTIONS 覆盖。
    Python instrumentation.opentelemetry.io/inject-python 检查 PYTHONPATH 与 glibc/musl。
    .NET instrumentation.opentelemetry.io/inject-dotnet 检查基础镜像、架构和 runtime identifier。
    Go instrumentation.opentelemetry.io/inject-go eBPF sidecar、提升权限、目标可执行文件、单容器限制。
    Deno/SDK-only instrumentation.opentelemetry.io/inject-sdk 只注入 SDK 配置；不会给普通应用安装完整语言 SDK。

    多容器 Pod 必须显式指定目标。单一语言可用 instrumentation.opentelemetry.io/container-names；启用 Operator 的 multi-instrumentation feature gate 后，可用 java-container-names、nodejs-container-names、python-container-names、dotnet-container-names、go-container-names 和 sdk-container-names。一个容器不能同时注入多种语言；Go 不支持多容器 Pod。以部署版本的 Operator 文档为准。

    运行时兼容边界¶

    Java¶

    • Operator 注入可能覆盖镜像或 ConfigMap 提供的 JAVA_TOOL_OPTIONS，必须比较实际进程参数。官方故障排查
    • 手工 Java agent 通过 -javaagent:<path> 或专用 JAVA_TOOL_OPTIONS 挂载；agent 是进程内代码，版本和 JAR checksum/digest 必须固定。
    • Java agent 2.0+ 当前默认 OTLP 协议是 http/protobuf，与 SDK autoconfigure 的历史默认不能混为一谈；显式配置更安全。Java agent configuration
    • 支持的 JVM、库和框架随 agent 版本变化，按该版本的 Supported Libraries 页面验证，不用“Java 8+”替代框架兼容测试。

    Python 与 musl¶

    • Operator 注入可能覆盖镜像或 ConfigMap 提供的 PYTHONPATH。官方故障排查
    • glibc 是默认平台；Operator 文档说明从 v0.113.0 起可用下列注解选择 musl：
    instrumentation.opentelemetry.io/otel-python-platform: "musl"
    • 这只说明 Operator 提供平台选择，不证明应用的 Python 版本、C 扩展、架构和 agent 镜像兼容；仍需在相同基础镜像中验证。

    Node.js¶

    • Operator 注入可能覆盖镜像或 ConfigMap 提供的 NODE_OPTIONS。官方故障排查
    • 预加载器、APM agent、ESM loader 或自定义 --require 可能争用启动选项；比较实际 process.execArgv 与环境，不假定自动合并。

    .NET¶

    • 记录架构、发行版、glibc/musl 和 runtime identifier；同为 .NET 不代表自动埋点二进制可以跨基础镜像复用。
    • 在匹配生产镜像的预生产环境验证 profiler 加载、启动时间和本机依赖；失败时回滚，不在生产替换未知二进制。

    Go¶

    • 必须启用对应 Operator feature gate；具体名称和默认状态按部署版本核对。
    • Go 自动埋点使用 eBPF sidecar，官方示例要求 privileged: true、runAsUser: 0，因此必须单独安全批准。
    • 必须通过 OTEL_GO_AUTO_TARGET_EXE 或 instrumentation.opentelemetry.io/otel-go-auto-target-exe 提供目标可执行路径。
    • 官方文档当前声明不支持多容器 Pod。一个 Pod 包含 service mesh、日志 sidecar 等也属于多容器，应改用手工埋点或重新设计，而不是绕过检查。Operator Go auto-instrumentation

    Deno¶

    • Operator 使用 inject-sdk，并在 Instrumentation 中设置 OTEL_DENO=true；当前 Deno 官方文档也以该变量开启内置 OpenTelemetry，并默认向 localhost:4318 使用 http/protobuf，所以生产必须覆盖 endpoint。Deno OpenTelemetry
    • Deno 的能力和启动标志随运行时快速变化。部分 Operator 版本文档仍要求 --unstable-otel，而当前 Deno 文档的 quick start 已不再使用它。应以实际 Deno 版本的官方文档和启动测试为准，不能把任一写法跨版本复制。
    • Deno 可能同时导出 console logs；在启用前明确 OTEL_DENO_CONSOLE 行为并执行敏感日志负向测试。

    SDK-only 注入¶

    inject-sdk 注入通用配置和资源属性，不会把缺失的语言 SDK、自动埋点库或业务 span 注入普通应用。应用必须已经包含并初始化兼容 SDK；否则“环境变量存在但没有遥测”是预期结果而非 Collector 故障。

    配置来源与优先级¶

    通用环境变量规范说明实现可以提供这些环境变量，但并非每个实现都必须实现所有变量；语言 agent 还可能有自己的优先级。Environment Variable Specification

    必须显式检查：

    • OTEL_SERVICE_NAME（与 OTEL_RESOURCE_ATTRIBUTES 中 service.name 冲突时通常优先）；
    • OTEL_EXPORTER_OTLP_PROTOCOL、通用 endpoint 和各信号 endpoint；
    • traces/metrics/logs exporter 与 sampler；
    • propagators；
    • agent/SDK 的配置文件变量及其是否覆盖普通环境变量；
    • Operator 注入后的实际容器环境，而不是只看 CR。

    不要在资源属性中放客户、租户、邮箱、令牌、完整 URL 查询串或请求级标识。

    最小验证命令¶

    kubectl version
kubectl -n <operator-namespace> get deploy <operator-deployment> -o yaml
kubectl get crd instrumentations.opentelemetry.io -o yaml
kubectl -n <app-namespace> describe otelinst <name>
kubectl -n <app-namespace> get pod <pod> -o yaml
kubectl -n <app-namespace> exec <pod> -c <container> -- env
kubectl -n <app-namespace> exec <pod> -c <container> -- ps -ef

    这些命令只能证明实际配置的一部分。最终兼容证据还必须包含一条已知请求、独立应用 SLI、Collector 内部指标和敏感字段负向测试。
