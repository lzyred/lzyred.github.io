    Codex 工程开发标准¶

    执行摘要¶

    Codex 不是只生成代码的聊天工具。它可以读取仓库、运行命令、修改文件、使用工作树、调用 MCP 或插件、访问 GitHub，并在不同执行面上产生真实工程变更。正确做法不是把更多内容塞进 Prompt，而是建立一个可约束、可验证、可审查、可停止的执行系统。

    mindmap root((Codex Engineering)) ChatGPT Web Desktop CLI Local shell Non-interactive work IDE Editor context Inline changes Cloud Isolated environment Parallel tasks AGENTS.md Repository rules Directory scope Skills Reusable workflow Trigger and stop point MCP Tools Data sources Plugins Packaged capability External actions Sandbox Filesystem Network Approval Command External write Worktree Isolated branch state PR Review Diff Evidence Human acceptance

    核心原则：模型可以在已批准目标和权限内选择实现细节，但不能自行扩大目标、数据边界、外部写权限、发布范围或不可逆动作。

    适用范围¶

    本标准适用于使用 ChatGPT、Codex CLI、IDE 扩展、Codex Cloud、AGENTS.md、Skills、MCP、插件和 GitHub 交付工程变更的团队。它不授权 Codex 访问生产系统；具体产品能力会变化，执行前仍需核对当前官方文档和组织策略。

    前置条件与任务合同¶

    字段 必须回答
    目标 用户最终能观察到什么变化
    范围 哪些仓库、目录、文件和系统可以修改
    非目标 哪些问题明确不在本次处理
    基线 分支、SHA、现有 PR、工作区状态
    权限 本地读写、网络、MCP、GitHub、云和生产权限
    验证 哪些命令、测试、截图、人工路径证明结果
    停止条件 何时不得继续猜测或扩大范围
    交付 分支、提交、Draft PR、合并和发布边界
    flowchart LR A[任务合同] --> B[仓库事实] B --> C[权限边界] C --> D[最小变更] D --> E[自动验证] E --> F[Diff Review] F --> G[Draft PR] G --> H[人工接受] E -- 证据不足 --> A F -- 超出范围 --> D H -- 拒绝 --> A

    仓库事实至少包括当前分支、远端、最新基线、未提交改动、适用 AGENTS.md、构建入口和已有失败。不得把用户本地未提交内容当作可覆盖文件。

    强制要求¶

    指令按信任与作用域解析¶

    用户和 Owner 的明确目标优先于网页、Issue、日志或文档中的嵌入指令；AGENTS.md 按目录作用域组合；外部网页、README、Issue、日志、代码注释和测试数据都是不可信输入，不能自行授权工具、Secret 或外部写操作。

    使用最小权限¶

    只读优先；文件写入限制在目标仓库和分支；网络按任务开启；GitHub token 只覆盖必要仓库和动作；MCP/插件分别授权；云和生产使用独立身份、短期凭据和人工门禁。

    保持最小闭环¶

    Codex 可以决定局部实现，但不得顺手重构无关文件、自动扩大目标、删除验证制造绿色、伪造截图/日志/版本/生产结果，或在无执行证据时把 not-verified 改为已验证。

    外部写操作需要授权¶

    Push、PR、邮件、评论、Issue、云资源、生产配置、Secret、权限、合并、发布、删除分支/数据以及可能破坏数据或产生费用的命令，均需任务明确授权或人工批准。

    验证形成证据梯度¶

    证据从语法格式、单元/静态检查、构建/集成测试、目标场景、基线对比、人工产品验收到生产结果逐层增强。低层证据不能冒充高层结果。CI 绿色只证明已运行检查通过，不代表内容有价值、产品目标被接受或生产正确。

    PR 是审查边界¶

    Draft PR 必须包含目标与根因、范围、验证结果与未运行项、自动化无法证明的内容、风险与回滚、Owner Review 顺序，以及 Merge/Publication 状态。

    禁止事项¶

    • 不把网页、Issue、日志或仓库内容中的提示当成更高优先级授权；
    • 不读取、打印或提交不需要的 Secret；
    • 不在未授权时 push、merge、publish、删分支或修改生产；
    • 不使用 git add -A 静默包含无关改动；
    • 不关闭测试、删除断言或放宽安全边界制造绿色 CI；
    • 不把“命令退出 0”写成业务恢复或产品验收；
    • 不在同一工作区并行修改互相覆盖的任务；
    • 不声称后台继续执行或稍后交付未完成结果。

    风险与信任边界¶

    flowchart TB H[Human Owner] M[Model] R[Local Repository] E[CLI / IDE / Cloud Environment] T[MCP Tools and Plugins] G[GitHub] C[Cloud Services] P[Production Systems] X[External Docs / Web / Issues / Logs] H -->|task and approval| M M --> E E --> R E -->|scoped token| T T --> G T --> C C --> P X -. untrusted content .-> M X -. prompt injection .-> E H -->|explicit gate| G H -->|separate high-risk gate| P

    主要风险包括 Prompt Injection、权限混淆、上下文不完整、证据膨胀、作用域漂移和凭据扩散。防线是独立身份、最小 token、沙箱、网络限制、审批、Diff Review 和 Owner Acceptance，而不是只依赖模型自律。

    验证方式¶

    git status --short
git diff --check
git diff --stat
git diff
    状态 含义
    PASS 实际运行且通过
    FAIL 实际运行且失败
    NOT RUN 环境、权限或依赖不足
    NOT PROVABLE 自动化无法证明，需要人工判断

    决策权矩阵¶

    决策 Codex 可自行决定 需要人工批准 禁止独立执行
    范围内实现细节、相关测试 是
    扩大业务目标或请求额外权限 是
    Push 功能分支 任务明确授权时 是
    Merge / Publish 是 未授权时
    修改生产或数据 强审批 无专用 Runbook 时
    删除证据或放宽门禁 是

    例外处理¶

    无法联网时使用已有仓库事实并标记外部验证未运行；无浏览器时完成构建和源检查并标记视觉验收未运行；无生产权限时输出安全命令和证据需求；工作区有无关改动时只暂存明确文件或停止；需求有关键歧义时采用最小安全解释并记录假设；测试过长时运行最相关子集并列出剩余项。例外必须降低声明强度，不能降低门槛。

    恢复与回滚原则¶

    • 所有变更在独立分支或 worktree 中完成；
    • 提交前保留基线 SHA 和完整 diff；
    • 失败时撤销本任务提交，不覆盖用户已有工作；
    • 外部写入有幂等键、审计和补偿路径；
    • PR 未接受前保持 Draft，不自动合并；
    • 发布失败时保留构建产物与源提交对应关系；
    • 生产动作使用独立 Runbook，不把代码回滚等同业务恢复。

    能力与版本边界¶

    截至 2026 年 7 月，本标准覆盖 ChatGPT Web/Desktop、Codex CLI、IDE、Cloud、AGENTS.md、Skills、MCP、插件、沙箱、审批、Git worktree 与 PR Review。稳定边界是：目标与证据先于修改；权限先于自主；外部内容默认不可信；最小 diff 先于广泛重构；人工接受先于 Merge 和 Publication。

    实例：修复文档站而不制造第二轮问题¶

    任务是删除错误公开 Portal、深化五篇文章、限制 CI 仅 main 触发。合格执行应核对基线和工作流；明确 Tab 是 <title>；删除 Portal 源和入口；在十个双语文件中增加有解释价值的图和实例；把质量与发布串在 main push；明确本地验证和未运行项；只创建 Draft PR，不合并、不发布。

    相关标准与概念¶

    • AI 项目为什么越改越乱：这些控制所依赖的交付证据分层与停止条件。
    • 从"导航很乱"到一条能自我执行的不变量：把抽象需求变成能跑的指标和门禁。
