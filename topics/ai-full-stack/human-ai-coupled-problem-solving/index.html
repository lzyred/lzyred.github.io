    从"导航很乱"到一条能自我执行的不变量：人机意图对齐的入口歧义度系统¶

    当人只能抽象表达意图、又不会审查每个细节时，对齐不来自更长的需求文档，也不来自一次性测量，而来自一套系统：把意图编码成指标，为全站建立基线，再装一道拦住新漂移的门禁。

    先用三行读完结论¶

    判断什么 结论
    难在哪 需求抽象且融合（"四个导航入口重叠，优化一下"），而人不会审查 74 页
    为什么"演示"不够 测一页一次证明不了任何东西能存活；意图必须变成 CI 持续执行的不变量
    这套系统 grader（测量）+ 策略（canonical + 至多一个横切）+ 门禁（执行，带已采纳的债务基线）
    是证据不是口号 今天 74 页中有 13 页超标；门禁采纳它们并通过，且以 exit 1 拦住了一次注入的回归

    问题：抽象意图的鸿沟——并已在全站量化¶

    维护者看着顶部导航——任务、Runbook、AI 工程、领域——说：

    这四个入口交叉重叠，看着很乱，优化一下。

    这句话真实，但抽象、融合、未经审查。原样甩给 AI，会得到范围爆炸或猜歪意图，而 mkdocs build --strict 通过只证明机械完整，不证明意图达成。问题是：

    一个抽象表达、又不会审查每个细节的人，如何让 AI 产出对齐真实意图——不是一次，而是永久？

    你无法审查全部，也无法在每次未来改动后重新审查。你需要一个指标和一道门禁。

    案例盒① —— 全站诊断，且已量化。 grader（tools/nav-entry-ambiguity-grader.mjs）模拟浏览型读者，对每个内容页统计有几个不同顶层入口给它一条策展门。跑遍全站：

    $ node tools/nav-entry-ambiguity-grader.mjs dist --all
...
site=zh  content_pages=74  ambiguity_histogram={"1":38,"2":23,"3":11,"4":2}
    入口歧义度 页数 解读
    1 38 只有 canonical 门——可预测
    2 23 canonical + 一个横切——健康
    3 11 过度呈现
    4 2 被每个入口都争——最差

    74 页中有 13 页（18%）超过"canonical + 一个横切"的预算，且这 13 页里有 8 页是 runbook（含两个歧义度 4 的页）。重叠不是感觉，而是一个有主因的分布：Runbook 控制台（按类型）、Tasks 门户（按目标）、Home 精选，都在重复呈现那些本已住在 Topics 下的 runbook。

    核心定义：意图 → 指标 → 可执行的不变量¶

    抽象下的意图对齐是指：把一个人只能抽象表达的意图，收敛成一个程序能计算的可观察指标，再把这个指标绑定成一条门禁执行的不变量——于是数字不会漂移，人也永远不必重读全站。

    意图"读者能预测该点哪扇门"变成 入口歧义度（每页几扇门），策略是 canonical + 至多一个有意横切（预算 = 2），不变量是一道 CI 门禁。

    系统有三部分：

    部分 工具 职责
    测量 nav-entry-ambiguity-grader.mjs 把抱怨变成一个全站的数字
    策略 预算 2 + 每类页型的有意横切 定义"对齐"是什么
    执行 verify-nav-ambiguity.mjs + config/nav-ambiguity-baseline.json 拦住新违规；显式采纳现有债务

    这就是用 LLM 做东西的基本功——把意图变成 eval，再变成门禁。 门禁有总结所不具备的性质：它无法被甜言蜜语糊弄，也不会遗忘。（对本文早先的一次评审曾接受了 AI 自信的"7 分"，直到证据把它逼到 3.5；门禁一碰就会让自信的散文失败。）

    工作机制：在真实站点上跑通被执行的闭环¶

    flowchart TD A[抽象意图：<br/>导航重叠，优化它] --> B[编码：入口歧义度 grader] B --> C[全站基线：<br/>74 页中 13 页超标] C --> D[策略：canonical + 一个横切<br/>预算 = 2] D --> E[把现有债务采纳为<br/>可追踪基线] E --> F[CI 门禁：<br/>verify-nav-ambiguity.mjs] F --> G{新页超标<br/>且不在基线？} G -- 是 --> H[FAIL exit 1：<br/>点名页面及其门] G -- 否 --> I[PASS：意图持续被执行] H --> J[给它一个 canonical 门<br/>+ 一个横切，或说明理由] J --> F

    第一、二步 —— 人命名意图；AI 诚实地编码¶

    读者任务（到达 Redis AOF runbook、到达 OpenTelemetry 是什么、到达 Codex 实践、不知道技术时找到恢复资产）代表"可用导航"。grader 模拟浏览型读者：只跟随每页正文里的策展链接，故意忽略全局侧栏／标签页（它们到处摊开整棵树）。歧义度是预测成本的结构代理，不是用户实测。

    第三步 —— 为全站建立基线，并定位每个杠杆¶

    上面的分布就是基线。每扇门都可溯源：Home 精选（docs/*/index.md）、Tasks 叶子列表（docs/*/tasks/index.md）、Runbook 控制台（docs/*/runbooks/index.md）、canonical 的 Topics 卡。runbook 同时背着三个横切，所以它们占了大头。

    第四步 —— 每种页型的策略¶

    页型 canonical 有意横切 预算
    runbook Topics Runbook 控制台（事故快捷门） 2
    standard / reference Topics Tasks（与目标相关时） 2
    concept / case Topics 默认无 ≤2

    规则是统一的——canonical 加至多一个有意横切——而 Home 精选是编辑行为，不是结构性的门。

    第五步 —— 先证明改动为真，再把它执行成不变量¶

    案例盒② —— 真实前后对比，然后闭环。 对最差的页面，一份废弃副本去掉了 Home 精选和 Tasks 叶子，用真实管线重建（198 页，--strict），再重新评分：

    BEFORE (真实 dist)     redis-aof-recovery-runbook   ambiguity=4  [Home,Runbooks,Tasks,Topics]
AFTER  (重建的临时副本) redis-aof-recovery-runbook   ambiguity=2  [Runbooks,Topics]

    但一次性修复不是对齐。门禁把当前 13 个违规采纳为可追踪债务（config/nav-ambiguity-baseline.json），从而今天就能启用，并拦住任何新增。它确实闭环的证据——在一次重建的站点里把某页在 Home 上过度呈现，已提交的门禁抓住了它：

    $ node tools/verify-nav-ambiguity.mjs <引入回归的 dist>
new_violations: [ { page: /topics/opentelemetry/what-is-opentelemetry/, ambiguity: 3 } ]
FAIL: 1 个内容页超过预算 2 且不在基线中   → exit 1

$ node tools/verify-nav-ambiguity.mjs dist    # 真实站点
PASS: 除追踪基线外无新违规                   → exit 0

    门禁已接进 make verify（site-verify），因此这条不变量现在每次构建都会运行。

    诚实说明，不是脚注。 (1) "after"是临时实验；上线整合是另一个需 owner 批准的任务。(2) 门禁当前检查 canonical（zh）站点；en 站结构对称，为它加门是一个已记录的下一步。(3) grader 本身曾因路径解析 bug 报出一个虚假安心的"没变化"，只因 sanity check 与它矛盾才被抓住——你不去审视的 eval，会像 AI 总结一样顺滑地撒谎。

    第六步 —— 人靠读门禁验证对齐，而不是读全站¶

    维护者不再审查 74 页，而是读一条门禁结果：现有债务已追踪、零新违规。意图已经变成不变量。

    边界¶

    适用于：需求抽象或融合、人无法审查每个细节、意图能收敛成可计算代理、且自动化否则只能证明机械完整。

    不适用于：已有现成 runbook 的单点故障、本质主观且抗拒任何诚实代理的意图、需要立即升级的安全事件、或没有人愿意认领目标的情况。

    指标是代理，所以 Goodhart 定律成立：把歧义度压到 0 会抹掉有用的可发现性。预算是 2 而非 0，人仍要判断这个代理是不是对的代理、以及能否往基线里加行。

    反例：什么时候不该建门禁¶

    移动端某个菜单标签被截断。

    这是视觉缺陷，改 CSS，别写 eval。同样，歧义度为 2 是健康的。系统的职责是守住那些读者无法预测的 3 和 4，而不是把每页压到 1。敌人是不可预测的歧义，不是歧义本身。

    可执行下一步¶

    1. 为你的站点建立基线： 
      node tools/nav-entry-ambiguity-grader.mjs dist --all
    2. 定预算和每类页型的横切策略；
    3. 显式采纳现有债务，并对新工作执行： 
      node tools/verify-nav-ambiguity.mjs dist --update-baseline   # 一次
node tools/verify-nav-ambiguity.mjs dist                     # 进 CI，拦住新漂移
    4. 通过整合真实页面把基线往下压——这是另一个需 owner 批准的任务；
    5. 把门禁扩展到 en 站，并在有用时加一个基于视觉的可读性 grader。

    本文交付的是系统及其门禁，不是首页重设计。

    结构化模型¶

    抽象意图
→ 编码成 grader（测量）
→ 为全站建立基线
→ 定策略（canonical + 一个横切）
→ 采纳债务，装门禁（执行）
→ 人读门禁，而非读全站
= 即使人从不读完每个细节、也永不静默回归的对齐

    抽象需求本身不是问题，未被执行的需求才是。门禁不让人审查更多——它让人真正检查的那一小步，在未来每一次构建上都继续决定结果。
