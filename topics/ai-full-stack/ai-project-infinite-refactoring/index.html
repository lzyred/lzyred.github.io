    AI 项目为什么越改越乱：一次“合并成功、交付失败”的真实复盘¶

    AI 项目最危险的失败，不是模型不会写代码，而是仓库持续变化、PR 不断合并，用户问题却始终没有被关闭。

    先用 3 分钟读完结论¶

    你需要记住什么 结论
    无限重构是什么 完成标准持续漂移，导致每轮修改都无法形成可关闭的交付闭环
    为什么 AI 更容易触发它 模型发现和生成问题的速度远高于人类审核与决策速度
    根因在哪里 产品、决策、验证、发布四个系统同时失效
    “完成”有几层 源码、规则验证、构建、发布、用户结果五层，不能互相替代
    如何停止 唯一结果 → 变更边界 → 最小修改 → 五层证据 → 用户验收 → 冻结版本

    全文只需要最终带走两个模型：

    1. 四个失效系统解释为什么项目越改越乱；
    2. 六步交付闭环决定下一次任务应该如何停止。

    事故现场：PR 合并了，错误页面仍然在线¶

    2026 年 7 月，一次 DevOps 文档站维护任务要求完成四件事：

    • 删除一个不应公开存在的版本化入口；
    • 重写五组内容过浅的中英文文章；
    • 增加真正帮助理解的 Mermaid 图；
    • 把 GitHub Actions 改成只在 main push 后执行质量检查与发布。

    最终 PR 修改了 24 个文件，新增 1747 行，删除 2406 行，并成功合并到 main。

    从源码仓库看，一切像是完成了：

    • PR #39 已合并；
    • 合并提交 190bf8e 已存在；
    • 源码中删除了 /topics/devops-answers/ 对应页面；
    • 发布工作流声明使用 rsync --delete 清理旧文件。

    但用户打开线上站点后，仍然看到本应被删除的旧入口。页面还出现严重布局异常：卡片被压缩成狭窄竖条，中英文逐字换行。

    已确认事实、推断与未知项¶

    类型 内容
    已确认事实 源码 PR 已合并到 main
    已确认事实 线上旧路由仍返回旧页面
    已确认事实 当时发布仓库最新提交仍来自旧源提交 73102bb...
    已确认事实 AI 曾在未确认发布和线上结果前报告合并完成
    合理推断 完整交付链没有被验证和关闭
    未确认项 工作流当时究竟在哪一步未执行、失败或未完成发布

    这里必须保持边界：不能因为线上仍是旧版本，就编造具体 CI 根因。

    这次事故真正证明的是：

    源码完成、PR 合并和用户交付，是三个不同状态。

    sequenceDiagram participant O as 项目所有者 participant A as AI participant S as 源仓库 participant P as 发布仓库 participant W as 线上站点 O->>A: 删除错误入口并完成发布 A->>S: 修改 24 个文件并创建 PR S-->>A: PR merged A-->>O: 报告合并完成 O->>W: 打开目标路由 W-->>O: 仍返回旧页面 P-->>O: 最新发布提交仍映射旧源提交 Note over A,W: 缺失了构建、发布、线上与用户验收证据

    这为什么不是一次普通发布故障¶

    如果只把它定义为“发布没成功”，下一步通常会是：

    • 再改工作流；
    • 再增加一个检查脚本；
    • 再开一个 PR；
    • 再写一份完成报告。

    但这会掩盖更深的问题：团队为什么会在没有发布证据和线上验收时，把任务视为完成？

    核心定义：什么是 AI 项目的无限重构¶

    无限重构不等于代码被多次修改，也不等于大规模重构一定错误。

    正常重构具有明确边界：

    问题已确认
→ 影响范围已确认
→ 修改边界已确认
→ 验收证据已确认
→ 通过后停止

    无限重构则是：

    项目的完成标准持续漂移，导致每轮修改都不能形成可关闭的交付闭环。

    flowchart LR A[目标描述模糊] --> B[AI 自行补全需求] B --> C[任务范围扩大] C --> D[变更超过人工审核能力] D --> E[依赖 AI 总结和自动检查] E --> F[合并被视为完成] F --> G[线上或用户验收失败] G --> H[重新定义方案] H --> B I[没有停止条件] -.维持循环.-> B J[没有稳定产品真相] -.维持循环.-> H

    这个循环最危险的地方在于：每一步都可能局部正确。

    • AI 发现的问题可能真的存在；
    • 新增的测试可能确实有用；
    • 重写后的文章可能确实更完整；
    • 工作流修改也可能符合工程规范。

    但局部正确不会自动合成为整体正确。项目最终仍可能没有解决用户最初的问题。

    工作机制：四个系统如何把项目推入无限重构¶

    总模型：四个失效系统¶

    mindmap root((AI 无限重构)) 产品系统失效 没有唯一用户结果 没有不做清单 页面数量代替真实价值 技术完整代替产品闭环 决策系统失效 AI 可以自行扩大范围 新会话重新定义项目 所有者只看最终总结 没有不可推翻的项目事实 验证系统失效 测试只验证形式 构建成功等于质量成功 PR 合并等于任务完成 缺少用户任务验收 发布系统失效 源仓库与发布仓库分离 工作流状态没有确认 线上结果没有抽样检查 旧文件和缓存没有证据

    1. 产品系统失效：目标是一种愿望，而不是结果¶

    下面这种目标无法交付：

    全面优化 AI 项目的内容、UI、架构、CI 和用户体验。

    它没有说明：

    • 谁会使用；
    • 用户最终观察到什么变化；
    • 哪些内容明确不做；
    • 达到什么状态后停止。

    目标不可验收时，AI 会持续补全。“更专业”“更丰富”“更高级”都没有自然终点。

    2. 决策系统失效：AI 同时定义标准、实施和验收¶

    很多项目实际上采用了这条链：

    AI 定义标准
→ AI 实施
→ AI 编写测试
→ AI 运行测试
→ AI 根据自己的规则宣布完成

    这是一套内部一致的自证系统，却不一定对应用户真实目标。

    模型可以决定局部实现，但不应该自行决定：

    • 是否扩大产品范围；
    • 是否修改全局架构；
    • 是否接受发布风险；
    • 内容是否值得公开；
    • 用户是否已经验收。

    3. 验证系统失效：自动化证明规则，不证明规则正确¶

    自动化擅长检查：

    • 文件存在；
    • 语法正确；
    • 元数据一致；
    • Mermaid 数量达到要求；
    • 页面没有水平滚动；
    • 构建命令退出码为 0。

    但它很难判断：

    • 文章是否值得读；
    • 图是否真正帮助理解；
    • 内容是否兑现标题；
    • 用户是否能完成任务；
    • 这个页面是否应该存在。
    flowchart TD A[自动化验证] --> B[语法] A --> C[格式] A --> D[构建] A --> E[基础布局] A --> F[规则一致性] G[人工验收] --> H[内容价值] G --> I[理解难度] G --> J[真实问题解决] G --> K[结构性思维] G --> L[标题兑现] B --> M[工程合格] C --> M D --> M E --> M F --> M H --> N[产品合格] I --> N J --> N K --> N L --> N

    4. 发布系统失效：项目没有区分五层交付真相¶

    “完成”至少有五层：

    flowchart TD A[1. 源码完成] --> B[2. 规则验证完成] B --> C[3. 构建产物完成] C --> D[4. 发布完成] D --> E[5. 用户目标完成] A -.不能证明.-> C B -.不能证明.-> D C -.不能证明.-> E D -.仍不能完全证明.-> E
    层级 能证明什么 不能证明什么
    源码 文件、Commit、PR 已发生变化 项目可运行
    规则验证 已编码的规则通过 规则本身正确
    构建 生成了目标产物 产物已发布
    发布 线上文件已更新 用户体验和内容正确
    用户结果 目标任务被真正关闭 后续不需要维护

    本文案例完成了源码层，却没有拿到发布层和用户结果层证据。

    一个局部问题如何膨胀成 24 个文件¶

    AI 特别擅长发现关联问题，但不天然承担优先级责任。

    flowchart LR A[原始问题<br/>移动端页面过窄] --> B[直接修复<br/>容器与 Grid] A --> C[AI 发现关联问题] C --> D[字体不统一] C --> E[导航不合理] C --> F[文章内容太浅] C --> G[缺少治理状态] C --> H[CI 设计不合理] D --> I[设计系统重构] E --> J[信息架构重构] F --> K[批量内容重写] G --> L[新增入口与标签] H --> M[工作流重构] I --> N[变更规模失控] J --> N K --> N L --> N M --> N

    这些问题可能都是真的，但关键判断是：

    真实存在的问题，不等于必须进入当前 PR。

    决策债务：AI 项目最容易积累的隐性成本¶

    技术债务意味着代码难维护；决策债务意味着仓库中存在大量已经实施、但没人真正理解或认可的决定。

    例如：

    • 为什么新增这个入口；
    • 为什么修改导航；
    • 为什么这个元数据代表可信度；
    • 为什么一个测试能够代表内容质量；
    • 为什么要同时重写 UI、内容和 CI。
    flowchart LR A[AI 快速生成] --> B[变更文件增加] B --> C[隐藏决策增加] C --> D[人工审核不足] D --> E[依赖 AI 总结] E --> F[未经理解的决定被合并] F --> G[未来再次返工] G --> H[决策债务继续增加]

    当团队不知道旧设计为什么存在时，局部修改会显得危险，整体重构反而显得简单。于是下一轮更大的重构开始了。

    边界：不是所有大改动都叫无限重构¶

    下面这些情况可能合理地要求大范围重构：

    • 同一架构根因持续制造同类生产缺陷；
    • 一个局部修改必然破坏多个模块；
    • 关键逻辑无法测试；
    • 数据模型无法支持已经确认的需求；
    • 安全边界不明确；
    • 发布和回滚无法保证。

    真正的判断标准不是文件数量，而是：

    1. 不重构会阻止哪个已确认目标；
    2. 是否存在更小修复；
    3. 是否有迁移、验证和回滚方案；
    4. 完成条件是否稳定。
    flowchart TD A[提出重构] --> B{不重构会阻止哪个已确认目标} B -- 无法明确回答 --> C[停止重构] B -- 可以明确回答 --> D{是否存在更小修复} D -- 有 --> E[采用最小修复] D -- 没有 --> F{是否有迁移、验证与回滚方案} F -- 没有 --> G[暂缓重构] F -- 有 --> H[批准受控重构]

    反例：一次合理的大型重构应该是什么样¶

    假设一个单体部署系统存在以下已确认问题：

    • 每次修改部署状态都会同时改动 API、数据库和前端；
    • 失败状态无法持久化；
    • 审计记录无法还原；
    • 三次生产事故都来自同一状态模型。

    此时把状态机独立成明确领域模型，可能会修改几十个文件，但它不属于无限重构，因为：

    • 根因有多次事故证据；
    • 目标是稳定状态流转，而不是“让架构更优雅”；
    • 迁移和回滚边界已定义；
    • 验收可以通过真实发布任务证明；
    • 达标后版本会冻结。

    相反，下面这些理由不足以启动重构：

    • 新模型推荐了另一套架构；
    • 页面还可以更高级；
    • 目录看起来不够完整；
    • 未来也许会需要；
    • AI 顺手发现了更多问题。

    可执行下一步：六步反无限重构闭环¶

    无限重构不能靠“请不要过度设计”解决。项目需要一个不可绕过的停止系统。

    flowchart LR A[1. 唯一结果] --> B[2. 变更边界] B --> C[3. 最小修改] C --> D[4. 五层证据] D --> E[5. 用户验收] E --> F[6. 冻结版本] D -->|明确失败| C C -->|发现非阻塞问题| G[进入 Backlog] G --> H[独立任务重新排期]

    第一步：定义唯一结果¶

    错误目标：

    全面提升 AI 项目的内容、UI、架构、CI 和体验。

    正确目标：

    删除错误公开入口，并证明线上目标路由不再返回旧页面。

    一个任务可以包含多个技术动作，但只能服务一个用户结果。

    第二步：建立变更边界和预算¶

    change_budget:
  max_files: 6
  allowed_paths:
    - docs/zh/topics/ai-full-stack/
    - docs/en/topics/ai-full-stack/
    - mkdocs.yml
  forbidden_paths:
    - docs/zh/index.md
    - docs/en/index.md
    - docs/assets/stylesheets/global.css
  allow_new_dependencies: false
  allow_unrelated_refactoring: false

    预算不是为了机械限制文件数量，而是迫使 AI 在扩大任务前解释：

    为什么一个删除页面的任务需要修改 24 个文件？

    第三步：只做最小必要修改¶

    每发现一个新问题，都先问：

    flowchart TD A[发现新问题] --> B{是否阻止当前唯一结果} B -- 是 --> C[进入本次修改] B -- 否 --> D[记录 Backlog] C --> E{是否存在更小修复} E -- 是 --> F[采用最小修复] E -- 否 --> G[申请扩大范围]

    第四步：提交五层证据¶

    层级 必须提交的证据
    源码 Commit、diff、文件列表
    验证 测试结果、静态检查
    构建 构建产物、目标路由
    发布 发布仓库提交、来源 SHA
    用户 线上响应、桌面截图、移动端截图、真实任务结果

    禁止用下面这些话替代证据：

    • 理论上会删除；
    • 应该已经发布；
    • 测试应该覆盖；
    • 工作流会自动处理。

    第五步：从用户任务验收¶

    错误验收：

    页面构建成功。
测试全部通过。

    正确验收：

    访问旧路由时，不再返回旧入口页面；
导航、搜索索引和 sitemap 中不存在旧入口；
桌面端与移动端均无异常布局；
目标文章兑现标题承诺并能完成读者任务。

    第六步：验收后冻结¶

    stateDiagram-v2 [*] --> Defined Defined --> Implementing: 范围已批准 Implementing --> Verifying: 最小修改完成 Verifying --> Fixing: 明确失败 Fixing --> Verifying Verifying --> Publishing: 自动验证通过 Publishing --> OnlineChecking: 发布完成 OnlineChecking --> Accepted: 用户任务通过 OnlineChecking --> Fixing: 线上失败 Accepted --> Frozen Frozen --> [*] Implementing --> Backlog: 发现非阻塞问题 Backlog --> [*]

    Frozen 的含义是：

    • 不再因为“还可以更好”继续改；
    • 新问题进入独立任务；
    • 新任务重新定义唯一结果和预算；
    • 不允许在旧任务里无限追加。

    十分钟诊断：你的项目是否已经进入无限重构¶

    每项回答“是”得 1 分。

    1. 当前版本无法用一句话说明唯一结果；
    2. 最近三个 PR 同时修改多个不相关模块；
    3. AI 经常因为发现新问题而扩大当前任务；
    4. 项目没有明确的不做清单；
    5. 同一个 AI 同时负责设计、实现、测试和验收；
    6. 人类主要依赖 AI 总结，没有理解主要变更；
    7. 新会话经常重新定义产品方向或目录结构；
    8. 只确认 PR 或测试状态，没有验证线上结果；
    9. 无法演示一个完整真实用户任务；
    10. 不知道达到什么状态后必须停止修改。
    得分 判断
    0–2 正常开发波动
    3–4 已出现范围失控风险
    5–7 正在发生无限重构
    8–10 应冻结新增功能，先重建交付闭环

    回到本文案例：正确完成链应该是什么¶

    sequenceDiagram participant O as 项目所有者 participant A as AI participant S as 源仓库 participant C as CI participant P as 发布仓库 participant W as 线上站点 O->>A: 确认唯一结果与变更边界 A->>S: 提交最小必要修改 A->>S: 创建 PR 并提交证据 O->>S: 审查后批准合并 S->>C: main push C->>C: 测试、构建与视觉验证 C->>P: 发布构建产物 P-->>A: 返回发布提交与源 SHA A->>W: 检查旧路由、新页面和设备布局 A-->>O: 提交完整五层证据 O->>O: 接受或只退回明确失败项

    这套闭环的价值不在于增加流程，而在于防止低层状态冒充高层完成。

    最终结构性思维¶

    整篇文章可以收敛为两条公式。

    根因模型¶

    产品系统失效
+ 决策系统失效
+ 验证系统失效
+ 发布系统失效
= 无限重构

    解决模型¶

    唯一结果
→ 变更边界
→ 最小修改
→ 五层证据
→ 用户验收
→ 冻结版本

    AI 项目陷入无限重构，通常不是因为模型能力太弱，而是因为模型生成能力增长得太快，项目的决策、验证与停止机制没有同步升级。

    “代码已修改”“测试通过”“PR 已合并”都只能证明取得了进展。真正的交付必须同时满足：

    源码正确
+ 规则通过
+ 构建成功
+ 发布完成
+ 用户目标实现

    少任何一层，都不能宣布完成。

    无限重构的终点也不是完美，而是：

    问题已经解决，证据已经成立，用户已经验收，现在停止修改。
