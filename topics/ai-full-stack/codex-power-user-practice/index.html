    从会用 Codex 到驾驭工程代理¶

    两个工程师使用同一个 Codex、同一个仓库和同一个模型，结果仍可能相差一个数量级。第一个人不断补充提示词：先让它写代码，发现方向不对，再解释目录，再补测试，再提醒不要改某个文件。第二个人先让 Codex读取仓库契约、确认边界和验收方式，再允许它修改；改完以后，要求它用测试、差异、渲染结果和人工判断证明完成。

    差距不在“谁更会和 AI 聊天”。真正的差距是：一个人在索取答案，另一个人在运行一套受约束、可验证的工程系统。

    资深 Codex 用户并不是能写出最长提示词的人，而是能稳定回答下面五个问题的人：

    1. Codex 此刻应该知道什么，不应该被什么噪声干扰？
    2. 哪些判断可以交给模型，哪些必须由人保留？
    3. 它可以使用哪些工具、修改哪些状态、何时必须停下来申请权限？
    4. 什么证据足以证明工作完成，而不是“看起来完成”？
    5. 这次纠正应该留在当前任务，还是沉淀成测试、AGENTS.md、skill 或工具？

    这篇文章给出一条由浅入深的实践路线。它不承诺“让 AI 替你完成所有工程工作”，而是解释如何把 Codex 从一个偶尔有用的代码生成器，变成一个可审查的工程执行者。

    适用范围与版本边界¶

    本文基于截至 2026-07-12 的 OpenAI 官方 Codex 文档和本地工程实践。Codex 的界面、模型名称、账户权限和功能可用性会变化；AGENTS.md、skills、MCP、沙箱、审批、代码审查、subagent 和 worktree 的具体入口应以当前官方文档为准。

    本文关注的是相对稳定的方法：任务契约、最小权限、工具边界、验证证据和人类复核。即使产品界面变化，这些原则仍然成立。

    问题：把工程代理当成聊天框¶

    初学者最常见的失败不是提示词太短，而是把所有责任都塞给一句自然语言：

    帮我把这个项目优化好，继续处理直到完成。

    这句话没有定义“优化”、没有约束改动范围、没有说明验证方法，也没有规定何时停止。模型只能从仓库中的局部信号猜测目标。它可能完成大量工作，却让用户面对更难审查的差异、更高的维护成本和一组无法证明的结论。

    工程代理与聊天模型的根本区别在于，它会读取文件、执行命令、修改状态并调用外部工具。输入不再只是影响回答风格；输入决定了一段真实执行过程的控制面。模糊任务因此不是“少说了几句话”，而是缺少变更管理。

    核心模型：证据驱动的代理闭环¶

    成熟工作流不是“提示—回答”，而是六个连续状态：

    flowchart LR A[观察<br/>仓库、约束、现状] --> B[定义<br/>结果、边界、验收] B --> C[执行<br/>最小可审查改动] C --> D[验证<br/>测试、构建、运行证据] D --> E[审查<br/>风险、判断、用户体验] E --> F[沉淀<br/>测试、规则、skill、工具] F --> A D -- 证据不足 --> B E -- 结果不接受 --> B

    其中最重要的转变是：验证不再是生成后的附加步骤，而是生成过程的支配条件。 在开始修改以前，就要知道最终需要交付哪些证据。测试命令、截图、性能指标、数据对账或人工确认，会反过来决定实现方式。

    第一阶段：把愿望改写成任务契约¶

    一个可执行任务至少包含五部分：

    部分 必须回答的问题 示例
    结果 用户最终要看到什么变化？ 首页显示按更新时间排序的文档列表
    范围 哪些文件、系统或用户路径在范围内？ 首页、样式、构建时索引生成
    禁区 明确不允许改变什么？ 不改文章正文、URL 和发布机制
    验收 用什么事实判断成功？ 每页 12 篇、前后翻页、双语一致、移动端无溢出
    交付边界 Codex 做到哪里停下？ 在临时分支完成验证，等待人工 review，不合并

    好的提示并不需要很长，但必须让结果可证伪。例如：

    目标：把首页改为最近更新文档列表。
范围：首页中英文、构建时索引、分页样式和前端分页逻辑。
不改：文章正文、主题路径、发布 workflow。
排序：以 Git 中最后一次内容修改为准，纯路径迁移不算更新。
验收：每页 12 篇；支持页码、上一页、下一页和直接 URL；
      390px 无横向滚动；中英文顺序一致；严格构建通过。
交付：保留在临时分支，提供渲染证据，等待人工确认。

    这不是提示词技巧，而是一个小型变更单。它让 Codex 可以自主选择实现细节，同时无法把“自主”解释成无限扩大范围。

    第二阶段：先让它建立事实，再允许它行动¶

    资深用户会把“诊断”和“修改”分开。第一轮先要求 Codex回答：

    • 当前分支、工作区和基础版本是什么？
    • 哪些仓库指令适用？
    • 目标功能现在由哪些文件和生成器负责？
    • 哪些事实已经验证，哪些只是推断？
    • 最小改动面是什么？
    • 哪些行为需要网络、外部写入或提升权限？

    这一步的价值不是让模型写一份漂亮计划，而是暴露错误前提。很多返工并非代码能力不足，而是模型在错误的基础分支上工作、漏读嵌套规则、把生成文件当源文件编辑，或把构建成功误认为发布成功。

    计划也不应该成为仪式。只有当任务存在多个依赖步骤、风险边界或不可逆状态时，计划才值得维护；一个单文件文字修正不需要十步计划。计划的作用是让状态和决策可见，不是制造管理感。

    第三阶段：把纠正放到正确的持久层¶

    一次对话中的纠正很容易消失。资深用户要判断它应当住在哪里。

    载体 适合保存什么 不适合保存什么
    当前提示 一次性目标、当前范围、临时判断 每个任务都应遵守的规则
    AGENTS.md 仓库惯例、构建命令、审查要求、目录边界 某次任务的详细实施方案
    .codex/config.toml 可信仓库的沙箱、模型、MCP、hook 等设置 文章写作规范或临时目标
    skill 可跨任务复用的完整工作流、参考资料和脚本 只对一个仓库成立的事实
    MCP 或 connector 外部系统中的实时数据和受控动作 可以直接放在仓库里的静态规则
    测试或验证器 必须机械保持的不变量 “内容是否有洞察”这类编辑判断

    官方文档建议让 AGENTS.md 保持精简，把反复出现的构建、测试、审查和目录规则放在最接近适用代码的位置。真正有效的反馈闭环不是“以后记得”，而是：

    1. 第一次失败，修正当前任务；
    2. 第二次出现，判断是否为稳定模式；
    3. 能机械判断的，写成测试或验证器；
    4. 需要行为约束的，写进最近的 AGENTS.md；
    5. 跨仓库重复的完整流程，提炼为 skill；
    6. 需要实时外部数据或动作时，才接入 MCP。

    不要把每次不满意都追加成规则。规则越多，上下文越拥挤，冲突越难发现。应保留能改变行为、可以验证、会重复发生的少数约束。

    第四阶段：从“让它写代码”升级为能力编排¶

    Codex 的能力来自不同层，它们不是同一个东西的不同名称。

    • AGENTS.md 规定这个仓库如何工作。
    • Skill 封装一个可重复的任务流程，可附带脚本、参考资料和模板。
    • MCP 把 Codex连接到外部工具和上下文，例如文档、浏览器、设计系统或内部平台。
    • Plugin 用于分发一组 skills、connectors 和相关能力。
    • Subagent 把真正独立的工作交给不同上下文并行完成。
    • Hook 或验证器 在生命周期节点机械执行约束。

    “工具越多，能力越强”是错误的。每多一个工具，就多一个选择分支、权限面和失败模式。成熟配置追求的是最小充分能力集：

    1. 本地文件已经包含事实，就不要先接外部系统；
    2. 官方文档可以回答，就不要让模型凭记忆猜当前产品行为；
    3. 只读查询足够，就不要授权写入；
    4. 一次性步骤不要急着做成 skill；
    5. 无法独立验收的子任务不要拆给 subagent。

    Skill 的价值也不在于塞入更多说明，而在于让触发条件、步骤、参考资料和验证方式变得稳定。官方文档指出，Codex会先看到 skill 的名称和描述，只有匹配后才加载完整 SKILL.md；因此，一个模糊的 description 会让再好的 skill 也无法稳定触发。

    第五阶段：让验证支配生成¶

    “代码已经写完”不是完成。资深用户要求一个证据阶梯：

    层级 能证明什么 不能证明什么
    静态检查 语法、格式、类型或链接符合规则 功能在真实环境中可用
    自动化测试 已编码行为没有回归 未编码的用户需求正确
    构建与运行 产物可以生成并启动 关键交互和视觉体验达标
    浏览器或端到端验证 用户路径真实可执行 业务判断和长期维护合理
    人工 review 范围、技术判断、风险和体验可接受 未来永远不会变化

    以一个分页首页为例，最低证据不应只有“构建通过”，而应包括：

    • 记录总数、每页数量和最后一页边界；
    • ?page=2 直接访问与浏览器前进后退；
    • “上一页”“下一页”的禁用状态；
    • 第一条记录确实是最近内容修改；
    • 中英文条目数量和顺序一致；
    • 桌面、移动端、浅色和暗色渲染；
    • 点击文档链接后实际返回目标页面；
    • 控制台没有新增错误；
    • 仓库完整质量门禁通过。

    验证失败时，要求 Codex先解释失败属于实现、环境、测试还是权限边界，再修复。不要允许它通过放宽测试、关闭严格模式或吞掉错误来换取绿色结果。

    第六阶段：分离作者、验证者和批准者¶

    同一个模型完成自检很有价值，但它不是独立审查。作者知道自己想实现什么，会自然地沿着自己的假设寻找证据。

    Codex 的 /review 可以针对未提交差异、分支、提交或自定义范围生成按优先级排序的发现，并且默认不修改工作区。成熟流程可以分成三层：

    1. 作者验证：运行测试、构建和目标路径检查；
    2. 独立代理审查：只读检查差异，寻找回归、遗漏和错误假设；
    3. 人类批准：判断内容、风险、用户体验和公开边界是否可接受。

    关键不是一定要使用 PR，而是批准记录不能伪装成作者自己的结论。对高风险变更，审查者还应重新运行关键证据，而不是只读作者的总结。

    第七阶段：只在任务真正独立时并行¶

    Subagent 和 worktree 可以显著加速大型任务，但并行不是高级感的装饰。

    适合并行的任务通常满足三个条件：

    • 输入边界清晰，例如一个代理审计可访问性，另一个审计构建流程；
    • 输出可以独立判断，例如各自返回发现清单或修改不同文件；
    • 不需要频繁共享中间状态。

    不适合并行的情况包括：两个代理同时修改同一页面、一个代理的架构决定会改变另一个代理的全部前提、或最终只能通过“相信摘要”来合并结果。

    官方文档说明，subagent 会消耗额外 token，但能把探索日志和中间噪声隔离在独立上下文中；worktree 则让多个 Git 任务拥有独立文件副本，避免直接干扰当前工作区。正确模式是：

    • 主任务保留目标、约束和最终判断；
    • 子任务承担可独立完成的调查、实现或复核；
    • 每个子任务返回证据和未决问题，而不只是结论；
    • 共享文件修改使用不同 worktree 或串行协调；
    • 主任务重新验证组合后的整体结果。

    第八阶段：把代理权限当成生产权限¶

    Codex 能执行 shell、访问网络、修改文件和调用外部工具，因此权限设计不能依赖“它应该不会做错”。

    沙箱和审批是两个不同控制：沙箱规定技术上能做什么，审批策略规定越过边界时是否必须停下来问人。资深用户不会为了减少提示而关闭所有边界，而会把常规操作放进可自动执行的安全区域：

    • 默认只写当前工作区；
    • 网络默认关闭，需要下载依赖或查外部状态时按范围开启；
    • 只读诊断不获得写权限；
    • 删除、发布、发送消息、修改权限和生产操作需要动作时确认；
    • 密钥只进入需要它的最短阶段，不写进提示、日志或仓库；
    • 为重复且安全的命令申请窄前缀，而不是授权任意 shell。

    高自主并不等于高权限。真正可持续的自主来自边界清晰：代理知道安全区内可以连续行动，越界时知道必须停下。

    完整实战：把“改一个首页”变成可审查交付¶

    假设任务是把一个双语静态文档站的首页改为最近更新列表，并提升视觉质量。

    1. 观察¶

    先读取仓库契约、主题配置、首页源文件、自动生成脚本和发布 workflow。确认站点是 MkDocs 静态构建，不引入数据库或后端服务。

    2. 固定语义¶

    定义“最近更新”为 Git 中最后一次实际内容修改，而不是文件系统 mtime；纯路径迁移不应把所有文章变成同一天。这个定义比 CSS 更重要，因为它决定首页是否可信。

    3. 固定边界¶

    只修改首页生成、分页脚本和视觉样式；不改文章正文、URL、发布仓库和主题分类。所有工作留在临时分支。

    4. 选择最小架构¶

    构建时读取 Git 历史并生成双语首页；浏览器端只负责分页。这样不需要数据库、API、管理后台，也不要求作者维护 updated 字段。

    5. 实现可退化交互¶

    无 JavaScript 时页面仍保留完整链接列表；启用 JavaScript 后每页显示固定数量，URL 使用查询参数保存页码，浏览器前进后退可以恢复状态。

    6. 视觉对照¶

    不要凭“高级感”猜字体。直接检查参考站的 computed style，记录字体栈、字号、行高、颜色和列表密度，再为本站建立跨平台回退与无障碍对比度。

    7. 验证¶

    运行生成器一致性检查、静态安全检查和严格构建；在中文桌面、英文桌面、中文移动、英文移动和暗色桌面上验证；点击分页、前后翻页和文章链接；检查控制台。

    8. 交付¶

    报告改动文件、未改内容、排序语义、浏览器实测数字和验证命令。停在未提交分支，让维护者看到真实页面后再批准。

    这个流程的本质不是“Codex 帮忙写了 CSS 和 JavaScript”。它把需求定义、信息来源、架构约束、实现、运行验证和批准边界连成了一条证据链。代码只是其中一段。

    常见失败模式¶

    失败模式 为什么失败 纠偏方式
    不断追加提示词 上下文变长，但目标和验收仍不清楚 重写任务契约，而不是继续补句子
    把全部规则放进 AGENTS.md 每次任务都背负无关上下文 只保存稳定、重复、能改变行为的规则
    看到重复步骤就做 skill 尚未理解流程就固化错误模式 至少复用并修正数次，再抽象工作流
    给 Codex 所有权限 错误动作的影响面与便利一起扩大 使用最小权限和动作时审批
    大量启用 subagent 产生冲突、重复探索和摘要损失 只拆真正独立、可验收的子任务
    CI 绿色就自动合并 自动化不能证明内容判断和体验 保留渲染 review 与人类批准
    让 Codex 自己宣布成功 作者验证自己的假设 要求外部证据、独立 review 和用户确认
    直接编辑生成文件 下次同步会覆盖修改 找到事实源和生成器再改

    四轮成长练习¶

    这不是能力评分，而是一组逐步增加控制面的练习。

    第一轮：每个任务都有验收证据¶

    连续选择五个小任务。每次开始前写清结果、范围、禁区和验证命令；结束时拒绝接受只有“已完成”的总结，必须看到差异和证据。

    第二轮：建立仓库反馈闭环¶

    记录 Codex 重复犯的错误。能机械判断的写成测试，稳定的仓库约束写进最近的 AGENTS.md。一周后删掉没有改变行为的规则。

    第三轮：抽象一个真实工作流¶

    选择一个已经手工成功多次的流程，例如双语文档发布、依赖事故审计或前端视觉验收。把触发条件、输入、步骤、验证和参考资料做成 skill；用一个相似但不相同的任务测试它是否真正可复用。

    第四轮：运行一次多角色交付¶

    让一个任务负责实现，一个独立任务只读 review；只有存在真正独立调查时才启用 subagent；必要时使用 worktree 隔离修改。最终由主任务重新运行整体验证，再由人决定是否接受。

    边界：不能交给 Codex 的责任¶

    Codex 可以扩大工程师的观察和执行能力，但不能替代责任主体。以下判断必须有人明确拥有：

    • 需求是否值得做，以及机会成本是否合理；
    • 生产数据、客户信息和凭据是否可以进入某个工具；
    • 不可逆操作的业务后果是否可接受；
    • 技术结论是否足以支撑公开发布或生产变更；
    • 自动化遗漏了哪些用户、组织和长期维护风险；
    • 最终是否提交、合并、部署、发送或删除。

    当这些责任没有人拥有时，更强的模型只会更快地产生未经承担的决策。

    可操作结论¶

    资深 Codex 实践可以压缩成十条：

    1. 给结果、边界和验收，不给模糊愿望。
    2. 先建立事实，再授权修改。
    3. 区分已验证事实、合理推断和未知项。
    4. 让验证条件在实现之前出现。
    5. 把重复纠正沉淀到正确载体，而不是无限增长提示词。
    6. 使用最小充分工具集和最小权限。
    7. 只并行真正独立、可以分别验收的工作。
    8. 用独立 review 对抗作者偏差。
    9. 自动化负责机械完整性，人负责判断、风险和接受。
    10. 把每次成功工作流变成下一次更短、更安全的起点。

    当你做到这些，Codex 不再是一个需要反复哄回正确方向的聊天对象。它成为一名在明确契约内工作、能够提交证据、会在权限边界停下、最终仍由人负责的工程代理。
