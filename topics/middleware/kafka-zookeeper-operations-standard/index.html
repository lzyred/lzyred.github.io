    Kafka 与 ZooKeeper 运维标准¶

    目的¶

    本标准定义如何安全运行 Kafka——包括现场仍存在的 ZooKeeper 后端集群——以及如何迁移或升级集群，避免丢失已确认记录或意外重放消费者工作。

    本标准的核心判断：

    Kafka 的持久化不是“副本因子 3”，而是副本因子、min.insync.replicas、生产者确认、leader election 策略，以及变更期间写入边界的组合。

    适用场景¶

    • 运维 KRaft 或存量 ZooKeeper 模式的自管 Kafka 集群。
    • 需要在事故复盘中站得住脚的持久化、监控和迁移实践。
    • 计划增加 broker、滚动升级、元数据模式迁移或跨集群迁移。

    非目标¶

    • Kafka Streams、ksqlDB、connector 等流处理应用设计。
    • 托管流服务内部机制。
    • 完整 ZooKeeper 到 KRaft 迁移流程。
    • Active-active 冲突解决设计。

    版本边界¶

    • Kafka 4.x 使用 KRaft 元数据模式。
    • ZooKeeper 后端集群必须按准确 Kafka 版本使用受支持流程。
    • metadata、feature level 或存储格式推进后，不要假设 broker 二进制可直接降级。

    数据与控制拓扑¶

    flowchart LR Producers[生产者] --> Source[(源 Kafka 集群)] Source --> SourceConsumers[源消费者] Source --> Mirror[有方向的复制流] Mirror --> Target[(目标 Kafka 集群)] Target --> TargetConsumers[目标消费者] Metadata[元数据模式：KRaft 或 ZooKeeper] --> Source Metadata --> Target Catalog[Topic、责任人、ACL 与重要性目录] --> Source Catalog --> Target Observe[Quorum、URP、磁盘、复制延迟、消费延迟] --> Source Observe --> Target Observe --> Mirror Change[变更控制器] --> Producers Change --> SourceConsumers Change --> TargetConsumers

    该拓扑将数据面与控制面分离。生产者、broker、复制流和消费者负责移动记录；资产目录、身份、可观测性和变更控制负责判断这种移动是否安全。

    跨集群切换流程¶

    flowchart TD A[盘点 Topic、分区、ACL、Group 与 Offset] --> B[启动已批准的单向复制流] B --> C[校验命名、记录、延迟与 Offset 同步] C --> D{目标是否满足声明的契约？} D -- 否 --> E[停止并修正拓扑或重放策略] E --> C D -- 是 --> F[在记录位置冻结源写入] F --> G[等待目标追到最终源位置] G --> H[转换并校验消费者 Offset] H --> I[在一个受控窗口内迁移生产者和消费者] I --> J[确认只有目标接受新写入] J --> K[持续观察，并保留源直到回滚窗口关闭]

    产品落位¶

    产品能力 管理对象 必须提供的行为
    资产管理 集群、broker、元数据模式、topic、分区、责任人、重要性 变更前展示完整拓扑和故障域。
    Topic 与 Schema 治理 Topic 配置、留存、分区数、命名、兼容性 阻止无治理拓扑漂移和不安全目标命名。
    身份与访问 SASL 身份、ACL、TLS 信任、复制凭据 分离生产者、消费者、运维和复制权限。
    可观测性 离线分区、URP、法定人数、磁盘、复制延迟、消费延迟 证明集群健康和切换就绪状态。
    变更管理 最终源位置、生产者冻结、Offset 映射、回滚窗口 将流量迁移组织为一次有边界的操作。
    证据中心 Topic 清单、Offset、延迟图、校验结果、Reviewer 保留切换和回切决策依据。

    规划检查清单¶

    领域 必须回答的问题
    Broker 多少 broker、什么存储、跨多少故障域？
    元数据 KRaft 还是存量 ZooKeeper？源和目标版本是否支持？
    Topic 分区数、副本因子、留存、清理策略和目标命名？
    持久化 acks、min.insync.replicas、副本因子和 leader election 是否一致？
    消费者 哪些 group 迁移、offset 如何转换、允许何种重复或重放？
    写入边界 如何冻结源生产者，最后一个有效源位置是什么？
    回切 是否有反向复制、明确双写契约或对账流程？
    安全 SASL、ACL、TLS 和复制凭据是否受控？
    可观测性 是否采集离线分区、欠副本分区、法定人数、复制延迟、消费延迟和磁盘？

    推荐标准¶

    持久化是一组联合契约¶

    对不能丢失已确认记录的 topic：

    • 平台具备独立故障域时，副本因子至少为 3。
    • min.insync.replicas = 2，用于容忍一个 broker 故障。
    • 生产者使用 acks=all。
    • 客户端与业务契约允许时启用幂等生产。

    副本因子 3 配 acks=1，leader 在副本追平前故障时仍可能丢失已确认数据。更强持久化的代价是延迟，以及 ISR 低于要求时写入被拒绝。消费者仍可能读取现有数据，因此把整个 topic 描述为“只读”并不准确。

    部署¶

    1. 保持 broker 或 node 身份稳定。
    2. 按写入量、留存、副本和恢复余量规划持久存储。
    3. 将 broker 与 KRaft controller 或 ZooKeeper 法定成员分散到故障域，并使用奇数成员。
    4. 显式设置每个 topic 的留存；无界留存最终会变成磁盘故障。

    真正重要的监控¶

    • 离线分区和欠副本分区。
    • KRaft 或 ZooKeeper 法定人数健康。
    • 每个 group 的消费延迟。
    • 迁移期间的跨集群复制延迟和 checkpoint 延迟。
    • broker 磁盘、JVM heap 和 GC。

    迁移 / 升级¶

    把变更作为有明确写入边界、验收标准和回滚负责人的受控操作。

    Broker 滚动升级¶

    1. 确认官方支持的源到目标兼容路径。
    2. 冻结不兼容配置变更。
    3. 每次只升级一个 broker。
    4. 等待 broker 重新加入，并让欠副本分区恢复为 0。
    5. 在理解回滚边界前，不推进不可逆的 metadata 或 feature level。

    跨集群迁移¶

    MirrorMaker 2 的复制流是有方向的。双向复制需要显式启用两条流，并验证 topic 命名、offset 转换、重复处理和 loop prevention 策略。

    1. 记录 topic、分区数、副本因子、配置、ACL、消费者组和源 offset。
    2. 建立经过批准的复制流和安全连接。
    3. 校验目标 topic 命名、记录、分区映射、配置、复制延迟和消费者 offset 同步。
    4. 定义最后一个源写入位置，并冻结迁移 topic 的写入。
    5. 等待目标复制到该最终位置。
    6. 按重放契约暂停消费者，转换并验证目标 offset，然后在同一个受控边界内迁移生产者和消费者。
    7. 确认只有预期集群接受迁移 topic 的新写入。
    8. 在验收完成且回滚窗口关闭前，保留源数据和复制能力。

    不存在通用的“先生产者”或“先消费者”切换顺序：

    • 先启动目标消费者，可以先消化镜像积压。
    • 先启动目标生产者，可以形成有界的目标积压。
    • 当源写冻结点和目标消费者位置经过证明时，两种方式都可能成立。
    • 两个集群同时接受不受控写入且没有合并契约时，两种方式都不安全。

    校验¶

    kafka-topics.sh --bootstrap-server <broker>:9092 --describe --under-replicated-partitions
kafka-topics.sh --bootstrap-server <broker>:9092 --describe --topic example-topic
kafka-consumer-groups.sh --bootstrap-server <broker>:9092 --all-groups --describe

    确认：

    • 没有离线分区，也没有无法解释的欠副本分区。
    • 关键 topic 的副本因子与 min.insync.replicas 符合要求。
    • 生产者确认和幂等设置符合持久化要求。
    • 消费延迟有界且在收敛。
    • 跨集群复制达到约定切换位置。
    • 目标消费者 offset 对应预期记录。
    • 切换后只有预期集群接受写入。

    已知失败模式¶

    失败模式 症状 可能原因 正确处置
    已确认记录丢失 broker 故障后数据消失 acks、ISR 或 leader election 设置过弱 对齐生产者与 broker 持久化设置，并测试 broker 故障。
    写入被拒绝 生产者收到副本不足错误 ISR 低于 min.insync.replicas 恢复副本健康，不要下意识降低持久化要求。
    消费者落后 延迟持续增长 消费者慢、rebalance、分区倾斜或 broker 压力 先排查工作负载和 broker 容量，再决定扩分区。
    Offset 转换错误 迁移后重复或跳过处理 目标 offset 未与实际记录核对 停止切换，按真实记录和重放策略重新映射。
    写入分叉 两个集群各有独有记录 两边同时写入且无冲突处理 冻结一侧、保留两边历史并完成对账。
    磁盘撑满 broker 不可用或集群不稳 留存或容量规划失败 恢复容量，再修正留存和告警。
    滚动升级风险 broker 之间 URP 未恢复为 0 过早升级下一个 broker 停止发布，恢复 ISR 后再继续。
    复制 loop 或 topic 冲突 重复记录或远端 topic 命名异常 复制方向和命名策略冲突 修正流拓扑，并先用隔离 topic 验证。

    回滚与回切¶

    目标开始写入前，回滚通常是让生产者继续留在源端、干净停止迁移流，并证明消费者状态没有推进到目标独有记录之后。

    目标开始写入后，安全回切必须具备以下之一：

    • 已验证的目标到源反向复制；
    • 具备顺序、幂等和冲突处理能力的双写设计；
    • 对目标独有记录执行受控导出、重放和对账。

    如果这些契约都不存在，直接回源会产生缺失记录或分叉历史。应停止写入、保留两个集群、对账记录与消费者位置，再执行第二次受控切换。接受已知缺口属于明确的数据损失决策，需要业务授权，不是普通回滚步骤。

    Broker 升级失败时，应按当前 metadata 和 feature level 使用受支持恢复路径。不可逆级别推进后，不要假设安装旧二进制即可安全恢复。

    安全注记¶

    • 启用 SASL 认证和 ACL 授权，不运行开放 broker。
    • 需要保密或身份验证时，为客户端、broker 间和跨集群流量启用 TLS。
    • 不公开真实 broker 地址、业务敏感 topic 名或凭据。
    • 限制 topic、ACL 和复制流的管理权限。
    • 通过批准的密钥管理路径保护 MirrorMaker 与 connector 凭据。

    评审周期¶

    • 负责人：平台/消息团队。
    • Topic 丢失容忍度变化时重新确认持久化设置。
    • 每次升级或迁移重新确认版本、元数据模式和迁移拓扑。
    • 大版本升级后重测 broker 故障、滚动升级、offset 转换和回切行为。
