    EKS 与 Kubernetes 升级标准¶

    执行摘要¶

    一次 Kubernetes 小版本升级不是一个升级。它是三个必须按顺序耦合的升级——控制面、然后插件、然后节点——外加一次必须在它们任何一个之前运行的兼容性检查。 控制面的版本号最显眼，但真正影响应用的故障，通常暴露在插件偏斜、目标版本已移除的 API、准入或转换 webhook、PDB、节点迁移这些耦合处。

    事故来自那些耦合。控制面动了之后还停在旧版本的某个插件，会产出看起来像控制面问题、其实不是的 DNS、网络或存储抖动。一个仍在调用新版本已移除 API 的工作负载，会在它的控制器重启的那一刻崩掉——而那可能是几小时之后，远在升级被宣布"完成"之后。一个阻塞全部驱逐的 PodDisruptionBudget，会让节点 drain 卡在半途，一部分工作负载在新版本、一部分在旧版本。

    最后一个完全可逆的时间点，是控制面升级开始之前。升级前的兼容性检查，会在旧版本仍然承载流量时，找出被移除的 API、不兼容的 CRD、失效的 webhook 和版本偏斜的插件。跳过它，你就会在 drain 期间发现同样的事实——那时半个集群已经迁移，控制面也已经无法回退。这份标准，主要讲的就是把发现提前，并让节点、插件、应用这些仍可回退的路径保持开放，直到新版本在真实流量下证明了自己。

    适用场景¶

    在以下情况使用本标准：

    • 跨小版本升级 EKS、自管 Kubernetes 或兼容平台。
    • 作为平台升级的一部分替换工作节点组。
    • 升级核心插件（CNI、CoreDNS、kube-proxy、CSI 驱动、Ingress 控制器、cert-manager、metrics-server）。

    这是升级标准，它假设你已有一个运行中的集群。不要把它当作：

    • 集群创建指南——见 Kubernetes 集群供给标准。
    • 应用发布流程——见部署回滚 Runbook。
    • 某个云厂商精确控制台/CLI 步骤的替代——在本序列内部使用云厂商记录的升级路径。
    • 跳过小版本的办法——一次只升一个小版本，除非平台明确支持更大跨度。

    假设¶

    本标准假设：

    • 集群当前处于受支持版本，只升级到下一个小版本。
    • 工作负载以无状态为主，状态放在托管数据服务里，或放在你能验证 CSI 驱动兼容性的存储里。
    • 你有能力新增一个节点组，而不是原地变更节点——正是这一点让节点回滚变成"停止 drain"而不是"把一个节点降级回去"。
    • 集群状态和密钥有加密的备份路径（平台的流程，或自管集群的 etcd 备份）。
    • 升级后有一个观察窗口，期间旧节点组、兼容的插件版本和应用回滚路径保持可用。托管控制面本身应按单向变更处理。

    如果你的集群落后好几个小版本，这就不是一次升级——而是一连串有计划的升级，每一次都端到端跑完，因为 Kubernetes 只支持单小版本跳跃，且每一步都有偏斜规则。

    按比例缩放¶

    下面的完整序列，是按一个承载真实流量的生产集群校准的。让投入匹配风险：

    • 一个可以压缩流程的集群应同时满足：没有客户流量、没有关键 StatefulSet/PVC 依赖、停机或重建 RTO 已被接受、集群状态可以从代码重建。可以缩短观察窗口或用更大的批次 drain，但前提是已经完成 API 兼容性检查，并且旧节点组保留到冒烟测试和指标通过之后。
    • 一个无状态集群可以更激进地 drain，因为节点迁移的回滚是"停止 drain、uncordon 或扩容旧节点组、把工作负载重新调度回去"——但只在旧节点组还存在的时候。一旦你下线它，回滚就变成在新版本上重新发布应用，而不是回到那个已知良好的节点版本。在你把"无状态"当成安全网之前，让旧节点组挺过观察窗口。当存在有状态工作负载、且一次糟糕的节点迁移无法靠重新调度撤销时，分批、验证过的 drain 才最重要。
    • 一个有状态或有严格 SLA 的集群需要完整模型：小批 drain、批次间验证、观察窗口、演练过的恢复。这里不要因为小集群路径允许就跳过分批 drain。

    永远不会被缩放掉的那个检查，是 API 兼容性的升级前预检。其余一切都随一场宕机的代价而伸缩。

    升级原则¶

    1. 一次只升一个小版本，除非平台明确支持更大跳跃。
    2. 把插件当作集群的一部分，而非可选附加——控制面与插件之间的版本偏斜是首要的故障来源。
    3. 严格按序升级：控制面 → 插件 → 节点。
    4. 优先新节点组 + drain，而非原地变更节点，让节点回滚是"停止 drain、uncordon 或扩容旧节点组"，而不是"把一个节点降级回去"。
    5. 在宣布完成前，验证 Kubernetes 健康和应用行为两者。
    6. 提前定义回滚触发条件，并在观察窗口内保持回滚路径开放。

    升级前评估¶

    捕获基线并运行兼容性检查。这是你预防宕机的地方——趁一切还在旧版本、可逆的时候。

    kubectl version
kubectl get nodes -o wide
kubectl get deploy,statefulset,daemonset -A
kubectl get ingress,svc,pvc,storageclass -A
kubectl get pdb -A                 # 检查 ALLOWED DISRUPTIONS——为 0 的 PDB 会卡住 drain
kubectl top nodes; kubectl top pods -A

    同时现在就从监控里捕获一份基线错误率、延迟和 SLO burn rate，趁集群还在旧版本上健康。下面的回滚触发条件必须是预先批准的数字阈值或 SLO 偏移，而不是事故中途临时定义的口径。

    区域 检查
    目标版本已移除的 API 找出仍在使用目标版本已移除 API 的工作负载（如 pluto、kubent）。在清单、Helm chart、GitOps 仓库和 CI 模板都停止输出这些 API 之前，阻断升级。
    CRD 与转换 webhook 确认 CRD 版本、stored versions 和转换 webhook 与目标版本兼容。失效的转换 webhook 可能让已有对象无法读取，或阻塞控制器 reconcile。
    准入 webhook 在控制面升级前验证 mutating/validating webhook、证书有效期、timeout policy、failure policy 和 API 兼容性。失效的 webhook 可能在升级后阻塞 Pod 创建。
    聚合 API 与 APIService 检查 APIService 健康度和扩展 API 兼容性。聚合 API 失效时，核心 Kubernetes 看起来健康，控制器或 CLI 却可能失败。
    Operator 与控制器 确认 operator 版本、controller-runtime 兼容性、leader election 和 RBAC 与目标 Kubernetes 版本匹配。控制器常常在升级数小时后才暴露目标版本已移除 API 的问题。
    插件 CNI、CoreDNS、kube-proxy、CSI 驱动、Ingress 控制器、cert-manager、metrics-server 与目标版本的兼容性。
    工作负载 PodDisruptionBudget、探针、资源请求、优先级类、安全上下文。
    存储 CSI 驱动兼容性、PVC 绑定、快照/备份支持。
    网络 Ingress class 行为、负载均衡器注解、NetworkPolicy。
    访问 IAM/RBAC、服务账号，以及一条可用的 break-glass 路径。

    一个阻塞全部驱逐的 PDB、一个目标版本已移除的 API，或一个不兼容的 webhook/控制器，都会让升级卡在半途——现在就找出它们，而不是在 drain 期间。

    标准升级序列¶

    1. 冻结并通告¶

    为该窗口暂停无关的部署和基础设施变更。沟通影响、回滚标准和负责人。

    2. 备份集群状态¶

    kubectl get all -A -o yaml > cluster-workloads.snapshot.yaml
kubectl get configmap,ingress,svc,pvc,storageclass -A -o yaml > cluster-config.snapshot.yaml

    不要把 Secret 值导出成明文。通过平台的加密流程备份密钥；对自管集群，用推荐的加密方法备份 etcd（见 etcd 标准）。

    3. 升级控制面¶

    通过云商/集群工具升级，并等待云商上报的升级完成状态（例如托管集群回到 ACTIVE），而不只是 /readyz 通过——一次托管控制面升级可能在 /readyz 上报健康、而云商仍在滚动它。随后先验证 API server discovery、准入/转换 webhook、聚合 API 和核心控制器，再触碰插件或节点。

    kubectl get --raw='/readyz?verbose'
kubectl get --raw='/apis' >/dev/null
kubectl get apiservice
kubectl get nodes

    4. 把核心插件升到目标兼容版本¶

    kubectl get pods -n kube-system
kubectl rollout status deployment/coredns -n kube-system
kubectl get daemonset -n kube-system

    在迁移工作负载之前做这一步——以新版本加入的节点需要兼容的 CNI/kube-proxy。这是最常被跳过的一步，也是那些"控制面升级搞坏了 DNS"报告背后、其实是插件偏斜的那一步。

    5. 新增一个节点组，逐步 drain 旧的¶

    在目标版本上创建一个新节点组，然后小批迁移工作负载：

    kubectl cordon <old-node>
kubectl drain <old-node> --ignore-daemonsets --delete-emptydir-data --timeout=10m

    每一批之后验证应用健康再继续。这就是可逆模型：如果某一批行为异常，就停下并保留旧的（仍然健康的）节点。

    6. 验证，然后下线旧节点¶

    观察窗口之后，缩容并移除旧节点组。保留快照和记录。

    验证¶

    kubectl get pods -A
kubectl get events -A --sort-by='.lastTimestamp'
kubectl get ingress,pvc -A

    应用验证必须包括：HTTP 健康、一个核心 API/登录冒烟测试、后台作业处理、数据库连通性、外部 webhook/回调，以及告警/看板仍然工作。只有在这些通过、且指标在观察窗口内稳定之后，才宣布成功——而不是在新节点组 Ready 时。

    已知失败模式¶

    失败模式 症状 可能原因 正确响应
    目标版本已移除的 API 升级后控制器/CI 失败，有时几小时后才发生 工作负载、chart、CRD、webhook 或控制器仍在使用目标版本已移除的 API 升级前用 pluto/kubent 找出；迁移清单和控制器依赖。
    Webhook/APIService 不兼容 Pod 创建、控制器 reconcile 或 CLI discovery 失败 准入 webhook、转换 webhook 或聚合 API 不兼容或不可用 在节点迁移前验证 webhook 证书、failure policy、timeout 和 APIService 健康度。
    插件版本偏斜 控制面升级后 DNS/网络/存储不稳 插件未升到匹配版本 在迁移节点前升级插件。
    Drain 卡住 kubectl drain 在迁移途中挂起 PDB 阻塞驱逐 / 单副本 修 PDB/副本数；绝不盲目 --force。
    Pod 无法调度 节点迁移后 Pending 新节点组上的污点/标签/容量不匹配 匹配标签/污点/规格；检查资源请求。
    存储无法挂载 StatefulSet Pod 卡住 CSI 驱动不兼容 在升级前评估中验证 CSI 兼容性；回滚节点。
    控制面不可逆 出问题后无法降级 托管升级是单向的 通过节点/插件/应用回滚 + 备份恢复（见下）。

    回滚¶

    托管控制面升级通常不直接可逆——所以回滚是围绕那些仍然可逆的层设计的。这恰恰是为什么原则 4（新节点组 + drain）重要：即使控制面不可逆，它也让节点迁移保持可逆。

    1. 立即停止节点迁移；在旧节点组仍受支持时 uncordon 或扩容旧节点组。
    2. 在确认安全可行时，把插件回滚到兼容版本。
    3. 回滚应用部署（kubectl rollout undo）。
    4. 如果数据受影响，从备份恢复有状态组件。

    开始前定义回滚触发条件：

    • 一个核心业务冒烟测试失败。
    • 预先批准的 SLO 触发条件在约定时长内被击穿，例如 5xx 率 > <threshold>、p95/p99 延迟 > <threshold>，或 burn-rate delta > <threshold> 持续超过 <duration>。
    • 关键 Pod 无法调度。
    • 存储挂载错误打到有状态工作负载。
    • Ingress/负载均衡器行为破坏了外部流量。

    升级后复盘¶

    记录源/目标版本、前后的插件版本、节点组变更、目标版本已移除的 API、webhook/APIService 发现、事故与缓解、后续工作。

    安全说明¶

    • 备份时绝不把 Secret 值导出成明文；用加密/密钥管理器备份。
    • 在下线旧路径前，验证 RBAC、IAM 和 break-glass 访问在新版本上仍然工作。
    • 不要发布真实集群名、账号 ID、端点或节点名。

    复审周期¶

    • 负责人：平台/集群团队。
    • 每次升级前重新运行已移除 API、webhook/APIService、控制器和插件兼容性检查。
    • 每个升级窗口前重新确认回滚触发条件和备份/恢复路径。
    • 跟踪 Kubernetes 版本的支持窗口，让升级是有计划的，而不是被生命周期终止逼着做的。
