    kubeadm 安装标准¶

    目的¶

    本标准定义什么时候 kubeadm 适合用于构建 Kubernetes 集群，以及初始化第一个控制面节点前必须先决定什么。

    核心判断：

    kubeadm 给你集群 bootstrap，也把控制面运营负担交给你。

    kubeadm 集群不是“按钮更少的托管服务”，而是自管控制面。团队要负责 API 可用性、etcd 恢复、证书生命周期、主机补丁、插件兼容、节点替换、升级顺序、审计姿态和重建决策。

    适用场景¶

    适用于：

    • 无法使用托管 Kubernetes，或托管 Kubernetes 不适合、被基础设施约束阻断
    • 团队需要显式拥有控制面主机、PKI、网络或 etcd 拓扑
    • 环境存在网络、合规、离线、硬件或本地性要求
    • 运维者能演练备份、恢复、证书轮换、升级和重建
    • 平台 ownership 足够稳定，能承担集群生命周期

    如果团队不能可靠拥有 etcd 恢复、证书轮换、控制面补丁、节点替换和事故恢复，优先使用托管 Kubernetes。

    非目标¶

    本标准不是命令记录，也不批准发布真实 kubeadm join token、certificate key、kubeconfig、内部 endpoint、节点名或生产 CIDR。它补充更完整的 Kubernetes 集群供给标准。

    生产契约¶

    运行 kubeadm 前，先定义：

    契约 必须决策 为什么重要
    API endpoint 客户端和节点使用哪个稳定 endpoint？ API 地址变化会破坏 kubelet、自动化和人工操作。
    控制面拓扑 单节点、stacked HA，还是 external etcd？ 拓扑决定故障和恢复行为。
    etcd 恢复 Snapshot 如何创建、保护、恢复和测试？ 只有 etcd 恢复真实可用，集群状态才可恢复。
    PKI 归属 谁拥有 CA 材料、证书过期和轮换？ 证书故障可能变成集群级中断。
    CIDR 选择 Pod 与 Service CIDR 最终是什么？ 错误 CIDR 通常是重建决策，不是可轻松调参项。
    CNI 选择 先安装哪个 CNI 和网络策略模型？ Pod 网络假设从集群初始化开始。
    插件顺序 CNI、DNS、CSI、Ingress、Metrics 和可观测性的哪些版本被批准？ 未定版本插件会在工作负载到来前就产生漂移。
    主机生命周期 控制面和工作节点如何打补丁、替换？ kubeadm 不替你管理操作系统。
    访问与审计 谁能管理集群，动作如何记录？ 自管集群需要显式问责。
    重建边界 哪些错误触发重建而不是修补？ 有些 bootstrap 选择重建比长期背负更便宜。

    推荐标准¶

    初始化前决定拓扑¶

    不要先初始化集群，再补生产叙事。先决定：

    • 单控制面或高可用控制面
    • API endpoint 的负载均衡或虚拟 IP Owner
    • stacked etcd 或 external etcd
    • 控制面主机放置与反亲和
    • 节点操作系统基线
    • 支持的 Kubernetes minor version
    • 升级与回滚顺序

    单控制面集群可用于实验、边缘场景或明确低关键性环境。没有书面例外和恢复预期时，不应称为生产。

    冻结不可轻易变更的 bootstrap 输入¶

    将以下内容作为设计决策：

    • Pod CIDR
    • Service CIDR
    • DNS domain
    • API endpoint name
    • CA ownership
    • etcd topology
    • CNI family

    工作负载到来后再修改它们，往往比重建一个年轻集群更危险。

    将插件安装为平台契约¶

    最小顺序：

    1. OS 与容器运行时基线。
    2. kubeadm、kubelet、kubectl 版本对齐。
    3. 使用批准 endpoint 和 CIDR 初始化控制面。
    4. 安装并验证 CNI。
    5. 从 pod 内验证 CoreDNS。
    6. 如果自动扩缩或资源可见性依赖它，安装 metrics server。
    7. 有状态工作负载前安装 CSI。
    8. 发布服务前安装 Ingress。
    9. 应用接入前安装可观测性和日志采集。
    10. 关键工作负载前完成备份与恢复验证。

    CNI、DNS、存储、Ingress、可观测性和 etcd 恢复验证前，不应接入应用。

    将 etcd 当作集群状态边界¶

    必需 etcd 控制：

    • snapshot 计划
    • snapshot 保留
    • snapshot 加密与访问控制
    • 隔离环境恢复演练
    • 恢复目标和预期停机记录
    • snapshot 失败与磁盘压力告警
    • 事故中 operator 能访问恢复材料

    如果团队不能在测试环境恢复 etcd，集群就不是生产就绪。

    在证书变成故障前跟踪它¶

    记录：

    • CA Owner
    • 证书过期时间
    • 续期和轮换流程
    • kubelet client certificate 行为
    • 控制面证书轮换窗口
    • CA 材料备份和访问规则

    证书材料应作为集群 root authority 保护。不要存入公开仓库、工单、聊天或示例。

    验证¶

    安装后验证运行模型，而不只是节点 Ready：

    kubectl get nodes -o wide
kubectl get pods -A
kubectl cluster-info
kubectl get --raw='/readyz?verbose'

    通过条件：

    • 所有控制面组件通过 API readiness endpoint 报告 ready。
    • 节点 Ready，且版本偏移在支持范围内。
    • CNI 支持 Pod-to-Pod 和 Pod-to-Service 连通。
    • 从 pod 内能解析 service name。
    • Service 路由正确。
    • 如果自动扩缩或资源可见性依赖 metrics server，它能工作。
    • 如果计划有状态工作负载，StorageClass 和 PVC provisioning 能工作。
    • 如果发布服务，Ingress 路径能工作。
    • 可观测性和日志采集能看到控制面与工作负载信号。
    • 已执行 etcd snapshot 和恢复演练。
    • 证书过期与轮换路径已记录。
    • 控制面维护测试中 API endpoint 稳定，或有记录的例外。

    失败模式¶

    失败模式 后果 预防
    Init 后临时拼 API endpoint 节点和客户端依赖不稳定地址。 Init 前决定 endpoint 与归属。
    Pod 或 Service CIDR 错误 工作负载到来后出现网络冲突。 Bootstrap 前冻结 CIDR；年轻集群错误时重建。
    无 etcd 恢复测试 集群状态丢失不可恢复。 工作负载接入前在隔离环境恢复 snapshot。
    忽略证书生命周期 证书过期导致集群异常。 跟踪、告警并演练轮换。
    Init 后才选 CNI Pod 网络假设漂移。 应用接入前选择 CNI 和策略模型。
    插件未定版本 升级路径变成猜测。 维护插件版本矩阵。
    应用先于可观测性接入 第一次故障没有证据。 工作负载前安装可观测性和日志。
    主机补丁未定义 控制面安全或重启工作临场发挥。 定义主机补丁和替换路径。

    重建或修复边界¶

    优先重建：

    • Pod 或 Service CIDR 错误。
    • 控制面 endpoint 设计错误。
    • etcd 拓扑不适合生产。
    • CA 归属不清或已受损。
    • Bootstrap 配置无记录。
    • 集群年轻且没有关键工作负载。

    优先原地修复：

    • 插件配置错误但数据面可恢复
    • 工作节点需要替换
    • 版本偏移可在支持范围内纠正
    • Ingress、storage 或 observability 需要重新配置

    已有关键工作负载时，重建会变成迁移。必须显式定义 cutover、rollback、data movement 和 workload validation。

    安全说明¶

    • 保护 kubeconfig、CA 材料、service-account signing keys 和 etcd snapshots。
    • 限制控制面主机访问。
    • 尽量使用具名管理员身份和最小权限。
    • 可行时保留 audit logs。
    • 加密并限制 etcd backups。
    • 不公开真实 cluster endpoints、join tokens、certificate keys、kubeconfigs、node names 或生产 CIDR。
    • 公开示例只使用占位符。

    评审周期¶

    在以下情况前评审：

    • 新 kubeadm 集群建设
    • Kubernetes minor upgrade
    • 控制面拓扑变化
    • 证书轮换
    • etcd 恢复演练
    • CNI、CSI、Ingress 或 CoreDNS 升级
    • 主机 OS 补丁波次
    • 生产工作负载接入
