    NFS PVC 容量管理¶

    目的¶

    本标准定义如何管理由 NFS 或类似共享文件存储支撑的 Kubernetes PVC 容量。

    核心判断是：

    NFS-backed PVC 容量，本质是披着 Kubernetes 元数据外衣的共享文件系统问题。

    PVC request 只告诉 Kubernetes 这个工作负载申请了多少容量。它不一定告诉你后端文件系统真实剩余多少、还有多少 PVC 共享同一个后端，也不一定说明后端是否能强制每个 claim 的容量限制。容量管理必须把两层信息连接起来。

    适用场景¶

    在以下情况使用本标准：

    • Kubernetes PVC 来自 NFS 服务器、托管文件服务或共享文件系统。
    • 多个 PVC 映射到同一个后端 export 下的不同目录。
    • PVC 使用率告警与后端文件系统使用率不一致。
    • 应用团队使用共享文件、上传、模型产物、报表或批处理输出。
    • 平台需要容量所有权和告警阈值标准。

    当 NFS 被作为便利的 ReadWriteMany 方案时，本标准尤其重要。便利不等于可以放弃容量隔离、监控和清理规则。

    非目标¶

    本标准不覆盖：

    • 从零设计高可用 NFS。
    • 数据库存储调优。
    • 用 NFS 替代对象存储保存大型不可变产物。
    • 共享文件的完整备份架构。
    • 厂商特定文件服务操作。

    双层容量模型¶

    把 NFS PVC 容量作为两个相关但不同的层来管理：

    层 能看到什么 不能证明什么
    Kubernetes PVC 元数据 Namespace、claim 名、申请容量、StorageClass、PV、挂载关系。 真实后端剩余空间、共享使用量、文件系统压力。
    NFS 后端文件系统 实际总量、已用、可用、inode 使用、目录增长。 Kubernetes 所有者、命名空间、工作负载意图、数据重要性。

    任何单层都不够。

    只看 Kubernetes，可能错过多个 claim 共同写满后端文件系统。只看 NFS，可能知道某个目录很大，却不知道它属于哪个 namespace、工作负载和团队。

    必需 Inventory¶

    维护每个 PVC 到后端现实的映射：

    字段 示例
    Namespace payments-prod
    PVC uploads-data
    PV pvc-abc123
    StorageClass shared-rwx
    Backend export nfs-shared-01:/exports/k8s
    Backend path /exports/k8s/payments-prod/uploads-data
    Workload payments-api
    Owner 平台或应用团队别名
    Data class 上传、缓存、报表、产物、日志、未知
    Retention 30 天、负责人定义或无
    Backup 必须、不需要或未知
    Expansion model 后端池扩容、目录 quota 或迁移

    如果平台无法把某个目录完整映射到负责人，就不得删除该目录下内容。

    标准阈值¶

    使用两组阈值：后端文件系统阈值，以及每 PVC 或每目录增长阈值。

    信号 Warning Critical 动作
    后端文件系统使用率 75% 90% 评审增长，计划扩容或清理批准数据。
    后端 inode 使用率 70% 85% 找出小文件来源和负责人。
    单 PVC 目录增长 24 小时增长 20% 24 小时增长 40% 识别写入者和数据类别。
    预计写满 7 天 72 小时 准备容量动作。
    未知负责人数据 任何持续增长 任何 critical 压力 升级；不要盲删。

    按后端规模和业务风险调节阈值。大型文件系统通常需要比百分比告警更早的预测告警。

    监控标准¶

    收集 Kubernetes 元数据¶

    收集：

    • PVC 名。
    • Namespace。
    • PV 名。
    • StorageClass。
    • 申请容量。
    • Volume mode。
    • Access mode。
    • 使用该 claim 的工作负载。

    Kubernetes 指标不总是暴露后端路径。如果 provisioner 把路径写在 PV attributes 或 annotations 中，应把它纳入 inventory。

    收集后端文件系统使用量¶

    从 NFS 服务器、托管文件服务指标或批准的观测主机收集：

    • 文件系统总量、已用、可用和使用率。
    • inode 总量、已用、可用和使用率。
    • 每个 export 使用量。
    • 可行时的每目录使用量。
    • 增长速率。
    • 按大小和文件数排序的 top 目录。

    对于 Linux NFS 服务器，平台侧采集可能使用：

    df -h /exports/k8s
df -i /exports/k8s
du -xh --max-depth=2 /exports/k8s 2>/dev/null | sort -h

    在繁忙文件服务器的高峰期，不要在不了解影响的情况下运行昂贵的递归扫描。

    连接两层信息¶

    容量仪表盘和告警应显示：

    • 后端文件系统压力。
    • Top PVC 目录。
    • Namespace 和 owner。
    • Workload 和数据类别。
    • 增长趋势。
    • 最近评审日期。

    真正有用的问题不是“哪块盘满了”，而是“哪个工作负载、哪类数据正在消耗共享容量，什么动作是安全的？”

    容量响应模型¶

    1. 行动前先分类数据¶

    使用以下分类：

    数据类别 默认动作
    缓存 通过批准的应用方法清理或重建。
    临时导出 应用保留策略，并把长期产物迁到其他位置。
    用户上传 未经负责人批准和备份确认，不删除。
    应用状态 当作业务数据；扩容或迁移。
    日志 迁移到日志管道或对象存储，按保留期清理。
    未知 升级；不做破坏性操作。

    2. 选择容量动作¶

    情况 动作
    后端容量充足，但单个 PVC 目录增长。 与负责人协作；执行保留策略、迁移数据或调整 claim 预期。
    多个 claim 共享一个池导致后端写满。 扩容后端、拆分租户或迁移高增长 claim。
    inode 先于字节耗尽。 找到小文件工作负载，归档或压缩，并改变应用行为。
    NFS 在负载下性能下降。 降低扫描压力、拆分工作负载或迁移到更合适后端。
    PVC request 与实际使用不匹配。 更新 inventory 和告警；不要只依赖申请容量。

    3. 谨慎扩容¶

    对 NFS，扩容可能意味着增加后端文件系统或调整 quota，而不是 patch 某个 PVC。

    扩容前确认：

    • 后端池余量或云厂商 quota。
    • 备份或快照姿态。
    • 扩容是否影响所有租户。
    • 扩容后的告警阈值。
    • 新容量和预期 forecast 已记录。

    4. NFS 抽象错误时迁移¶

    NFS 经常被误用于本该放在其他地方的数据。

    以下情况考虑迁移：

    • 大型不可变产物更适合对象存储。
    • 数据库被放在共享文件存储上且缺少强理由。
    • 单个租户主导了共享文件系统容量。
    • 应用性能依赖 NFS 无法保证的文件系统语义。
    • 清理依赖高风险手工删除。

    清理安全¶

    绝不能因为某个目录很大就直接从 NFS 删除。

    最低删除要求：

    • 负责人已识别。
    • 数据类别已识别。
    • 保留策略或明确审批已确认。
    • 备份或非关键分类已确认。
    • 精确路径已评审。
    • 命令范围限定在目标目录。
    • 结果已验证。

    优先使用应用级删除，而不是平台 shell 删除。应用通常更知道哪些文件被索引、被引用或可以安全移除。

    验证¶

    本标准落地的条件：

    检查 通过条件
    Inventory 每个 NFS-backed PVC 都映射到 namespace、workload、owner、backend export 和 path。
    指标 后端字节、后端 inode 和 PVC 元数据能一起看到。
    告警 后端压力和每目录快速增长都能被发现。
    所有权 未知 owner 目录被作为风险跟踪。
    响应 清理、扩容、迁移和升级路径已记录。
    安全 未经 owner 和数据分类，不做破坏性清理。

    常见失败模式¶

    失败模式 后果 预防
    只监控 PVC 申请容量 后端写满而 PVC 看似正常。 监控后端文件系统使用率。
    只监控 NFS 服务器 大目录没有清晰 Kubernetes owner。 维护 PVC 到路径的 inventory。
    共享池没有租户隔离 单个工作负载消耗所有人的容量。 使用 quota、独立 export 或独立 StorageClass。
    手工删除未知文件 数据丢失或应用损坏。 清理前要求 owner 和数据类别。
    高峰期递归扫描 调查时进一步拉高 NFS 延迟。 安排扫描窗口或使用轻量指标。
    没有 inode 告警 大量小文件在字节未满前破坏写入。 对 inode 使用率和文件数增长告警。

    安全说明¶

    • 不公开真实 export 路径、服务器名、IP 或客户目录名。
    • 限制 NFS 服务器 shell 访问。
    • 避免对共享应用文件赋予过宽读取权限。
    • 用户上传和报表都按潜在敏感数据处理。
    • 公开文档中的清理命令必须是通用占位示例。

    评审周期¶

    在以下时点评审本标准：

    • 接入新的 NFS-backed StorageClass 前。
    • 每次 NFS 容量或性能事故后。
    • 某个租户成为 top 容量消费者时。
    • 显著增加后端容量前。
    • 应用数据保留策略变化时。
