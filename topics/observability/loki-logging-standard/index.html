    Loki 日志标准¶

    目的¶

    本标准定义如何为 Kubernetes 和云原生工作负载运营 Loki 兼容日志体系。

    核心判断：

    日志是事故证据。只有证据能被找到、可信、留存并安全分享时，Loki 才有价值。

    部署 collector 并在 Grafana 里看到日志行还不够。真正的运行契约是标签设计、采集范围、留存、访问、隐私过滤、查询成本，以及真实故障中的验证。

    适用场景¶

    适用于：

    • 部署 Loki、Promtail、Grafana Agent、Alloy 或其他 Loki 兼容 collector
    • 增加 Kubernetes 日志采集
    • 修改标签、留存或 tenant 边界
    • 从 ELK/EFK 类日志系统迁移
    • 向多个团队开放日志
    • 排查嘈杂、昂贵或难查询的日志

    非目标¶

    本标准不要求每个环境都必须使用 Loki，也不定义每个 Helm value。它定义日志被当成生产证据前必须存在的运行契约。

    日志契约¶

    日志成为共享生产面之前，先定义：

    契约 必须决策 为什么重要
    采集范围 哪些 namespace、workload 和 node path 被采集或排除？ 意外采集可能泄露敏感日志或浪费存储。
    标签模型 哪些 label 允许，哪些禁止？ Loki 成本和可查询性主要由 label cardinality 决定。
    Tenant 与访问模型 哪些团队能读哪些日志？ 日志经常包含运行上下文和用户上下文。
    留存 每类日志保留多久？ 留存同时是成本、合规和事故调查决策。
    隐私过滤 Secret 和个人数据在哪里 mask 或 drop？ 一旦落库，敏感日志会扩散进查询、截图和工单。
    查询预算 哪些查询限制保护平台？ 昂贵查询可能把调试变成日志平台故障。
    事故证据 已知故障时必须能看到哪些日志信号？ 事故中失效的日志流水线不能当证据。
    Owner 谁调优标签、留存、collector 健康和访问？ 无主日志平台会变成垃圾倾倒场。

    标签设计¶

    推荐低基数标签：

    cluster
namespace
pod
container
app
environment
team

    避免以下高基数标签：

    request_id
trace_id
order_id
exception_fingerprint

    这些字段只有在非敏感、受留存和访问策略覆盖，并且调查确实需要时，才可以作为结构化字段存储。

    敏感值或个人数据不能只是从 label 挪到日志正文。存储前应 mask、drop、按批准方案 tokenize，或取得 data Owner 明确批准：

    user_id
email
session_id
full_url
authorization_header
request_body

    有用的 Loki label 应缩小搜索范围，不应该为每个请求、用户或敏感值创建一条 stream。

    采集流程¶

    Kubernetes 场景：

    1. 以 DaemonSet 或批准的节点级 agent 部署日志 collector。
    2. 从节点日志路径采集容器日志。
    3. 附加 Kubernetes metadata。
    4. 丢弃禁止采集的 namespace、label 和字段。
    5. 在可行时，存储前 mask 或 drop 敏感字段。
    6. 通过批准的 endpoint 发送日志到 Loki。
    7. 在 Grafana 或批准的日志 UI 中验证查询。

    不要把 kubectl logs 当成生产日志系统。它适合本地排查，但不提供留存、访问控制、历史搜索或跨工作负载调查。

    部署模型¶

    部署模型要有意选择。

    模型 适用场景 注意点
    Single binary / monolithic 小型或单租户环境。 本地存储压力和有限扩展边界。
    Simple scalable 中等规模，需要读写分离。 对象存储、gateway、query frontend 和 retention 配置。
    Microservices 高容量或多租户环境。 运维成本、组件健康和升级复杂度。

    非 Helm 的 systemd 变体见 Loki 二进制部署与告警标准。

    推荐标准¶

    固定版本并拥有 values¶

    Helm chart、镜像版本和 values 文件应明确并纳入版本控制。不要跟踪 latest。values 文件应体现 retention、storage、resource limits、auth mode、service exposure 和 collector relabeling rules。

    把 Grafana 保持为受治理访问路径¶

    如果已有平台 Grafana，优先复用。为了方便再装一个 Grafana，会分裂访问控制、仪表盘和事故工作流。

    Grafana data source 应使用稳定 service name 或批准的内部 endpoint。kubectl port-forward 是调试便利，不是生产访问模式。

    显式定义留存与删除¶

    定义：

    • 默认留存
    • 关键服务延长留存
    • 排除的 namespace
    • 日志容量预算
    • 敏感误采集的删除流程
    • 谁能批准留存例外

    从不复查的留存，要么过早删除事故证据，要么让日志存储变成无限成本。

    保护查询性能¶

    设置查询限制并规范查询形态：

    • 从较窄时间范围开始
    • 先用低基数 label 过滤
    • 避免跨所有 namespace 做无边界文本扫描
    • 定义查询长度、结果大小和并发限制
    • 监控查询延迟和失败查询

    可观测平台不应该因为一次事故查询扫描全世界而不可用。

    验证¶

    通过条件：

    • 当前模式下 Loki write、read、compactor 或 retention 组件健康。
    • Collector 在预期节点运行并发送日志。
    • 已知 workload 打出一条日志，并以预期 label 出现。
    • 敏感测试字段按策略被 drop 或 mask。
    • 查询近期 error 能在预期时间内返回。
    • Retention 已配置且可观测。
    • 团队访问规则能阻止无关团队读取受限日志。
    • Collector 或 Loki 故障能在 Loki 之外触发平台告警。
    • 受控应用故障中，能用约定 label 找到日志。

    验证查询：

    {namespace="example-namespace"}
{app="example-service"} |= "error"

    失败模式¶

    失败模式 后果 预防
    request ID 被当成 label Stream 数爆炸，查询变慢。 非敏感高基数值放进结构化字段，不做 label。
    敏感值从 label 挪到正文 隐私风险仍存在，只是不那么显眼。 存储前 mask、drop、tokenize，或取得 data Owner 批准。
    无留存 Owner 成本增长或证据意外消失。 按服务等级定义并评审留存。
    敏感日志原样入库 Secret 或个人数据扩散到工单和截图。 在应用或 collector 层 mask/drop。
    Collector 健康未监控 事故最需要日志时日志消失。 在 Loki 之外告警 collector 和 Loki ingestion 健康。
    重复 Grafana 团队看到不同事实。 可用时复用受治理 Grafana。
    Port-forward 变成访问模型 生产调查依赖开发机隧道。 使用稳定内部 service 或 gateway。
    标签过少 日志存了但找不到。 要求 namespace、app、environment 和 owner 标签。

    安全说明¶

    • 不要记录 credentials、tokens、session IDs、authorization headers、private keys 或完整 request bodies。
    • 尽量在应用或 collector 层 mask personal data。
    • 按团队、环境和数据等级限制日志访问。
    • 将导出的日志文件和截图视为敏感工件。
    • Collector 配置如果包含内部路径、tenant、endpoint 或 credentials，不要放进公开示例。

    评审周期¶

    在以下情况评审：

    • 日志量突变
    • 事故中团队找不到日志
    • 查询延迟或失败增加
    • 新的监管数据或个人数据可能进入日志
    • 留存或存储成本变化
    • 发生日志迁移
    • 计划升级 collector、gateway 或 Loki
