    SigNoz 可观测性标准¶

    目的¶

    本标准定义如何把 SigNoz 或类似 OpenTelemetry 兼容后端，放进生产可观测体系。

    核心判断：

    SigNoz 本身不是可观测性。它是查询遥测数据的地方。只有当采集、标签、存储、访问、告警、Owner 和失败验证都被设计成运行契约时，它才成为生产可观测能力的一部分。

    统一展示 traces、metrics 和 logs 很有用。但如果团队把 UI 能打开当成“故障时一定能定位”的证据，它也会很危险。

    适用场景¶

    适用于：

    • 服务已经输出 OpenTelemetry traces、metrics 或 logs。
    • 团队需要以 trace 为中心分析服务延迟、错误和依赖关系。
    • 平台希望在 OpenTelemetry Collector 后面提供一个受控遥测入口。
    • 组织接受运行 ClickHouse 支撑的可观测栈所需的运维成本。
    • SigNoz 与既有告警和日志标准配合使用，而不是意外替代它们。

    SigNoz 最适合回答“这条请求路径为什么变慢或失败”。如果把它作为唯一基础设施告警源、长周期日志归档或合规审计证据，就要非常谨慎。

    非目标¶

    本标准不：

    • 教 OpenTelemetry 埋点。埋点见下方 OpenTelemetry 与 Java 相关文档。
    • 替代基础设施和 SLO 告警的 Prometheus 标准。
    • 在组织需要专门日志留存和查询控制时替代 Loki 或其他日志平台。
    • 批准把敏感请求体、headers、tokens、用户标识或客户数据写入 traces 或 logs。
    • 把 demo 部署视为生产可用。

    决策模型¶

    部署前先决定角色。角色决定运行契约。

    角色 可接受用途 必需边界
    实验工具 学习 OpenTelemetry、验证埋点、查看样例服务 traces。 不接生产流量，不接敏感数据，存储可丢弃。
    团队分析后端 调试选定服务，并把 traces 与 metrics 或 logs 关联。 Collector 边界、访问控制、留存、Owner 和回滚路径。
    生产可观测服务 支撑真实服务的事故定位。 存储生命周期、备份或重建决策、告警路由、SLO 影响检查、容量规划和失败演练。

    如果团队说不清自己正在部署哪一种角色，就不要把它接入生产路径。

    生产契约¶

    允许 SigNoz 接收生产遥测前，先定义这些契约：

    契约 必须决策 为什么重要
    遥测所有权 谁负责 collector、backend、dashboard、alert 和 retention？ 事故中“UI 挂了”必须有人负责。
    Collector 边界 应用发到 OpenTelemetry Collector，还是直接发到 SigNoz？ 过滤、批处理、重试、内存限制和导出路由应在 collector 控制。
    属性策略 哪些 resource attributes 必须有？哪些属性禁止？ 坏标签让搜索失效；危险属性会泄漏数据。
    存储生命周期 ClickHouse 的留存、容量、备份和重建模型是什么？ trace 后端会悄悄变成大型有状态数据库。
    访问模型 谁能看 traces、logs、dashboards 和 alerts？ 遥测可能包含生产元数据和接近用户的上下文。
    告警模型 哪些告警归 SigNoz，哪些留在 Prometheus 或其他路径？ 工具重叠会制造告警空洞和重复噪音。
    自监控 SigNoz、ClickHouse 和 collector 健康如何通过 SigNoz 之外的路径告警？ 可观测后端不能只靠自己报告自己的失败。
    采样与基数 谁负责采样？哪些 trace 必须保留？哪些基数禁止？ 把故障采没，或被标签打爆的后端，不能支撑事故定位。
    失败验证 什么真实测试能证明应用到 collector 到后端到人的路径？ Pod 绿色只证明进程在跑。
    回滚边界 应用如何停止导出到该后端，而不夹带无关发布？ 可观测变更必须可逆，且不能破坏应用。

    推荐标准¶

    把 OpenTelemetry Collector 放在前面¶

    应用应将遥测发送到 OpenTelemetry Collector 服务，再由 collector 导出到 SigNoz。

    这样平台有一个受控位置来：

    • 默认丢弃或脱敏敏感属性。
    • 只有经过数据分类评审后，才允许对低风险伪匿名标识做哈希；绝不能把 token、authorization header、cookie、请求体、完整 URL 或 secret 的哈希当作安全替代。
    • 设置内存限制、队列、重试和批处理。
    • 规范化 service.name、namespace、environment、team、version 等属性。
    • 在迁移或回滚时把同一份遥测路由到其他后端。
    • 不改应用清单就暂停或限流导出。

    不要让应用 agent 指向 localhost、开发机、kubectl port-forward 或无人负责的端点。生产端点必须是平台拥有的稳定服务名和端口。

    把 ClickHouse 当作生产状态¶

    SigNoz 通常依赖 ClickHouse 保存遥测数据。ClickHouse 里不一定有业务记录，但它保存事故证据、性能历史和敏感元数据。

    生产部署需要：

    • 固定 chart 和 image 版本。
    • 持久卷具备已知扩容行为和回收策略。
    • traces、metrics、logs 的留存策略。
    • ClickHouse、collector、query service、frontend 和告警组件的资源请求与限制。
    • 备份或重建决策。如果团队选择不备份遥测存储，这必须是明确接受数据损失的决策。
    • 磁盘、写入速率、dropped spans、查询延迟和队列压力告警。

    不安全的模式，是让遥测存储一直增长，直到可观测系统本身变成下一场事故。

    区分三类信号的职责¶

    SigNoz 可以展示 traces、metrics 和 logs，但这些信号承担的工作不同。

    信号 强项 护栏
    Traces 请求路径、延迟拆解、依赖错误、服务拓扑。 采样、安全属性、一致的服务名。
    Metrics 服务级延迟、错误率、吞吐和后端健康。 SLO 与基础设施告警 Owner 必须明确。
    Logs 选定失败上下文与 trace 关联。 不要把 trace 后端日志当作无人管理的长期日志归档。

    不要因为 UI 支持，就在所有地方开启所有信号。每种信号都会带来写入成本、数据风险和运维责任。

    先让属性可搜索，再增加流量¶

    SigNoz 的第一份生产价值，不在于 chart 是否装好，而在于事故中能否按关键维度过滤遥测。

    必须具备安全 resource attributes：

    • service.name
    • service.namespace
    • deployment.environment
    • service.version
    • k8s.namespace.name 或等价平台 namespace
    • 组织使用时的 team 或 Owner 标签

    避免危险或高基数属性：

    • 用户 ID、邮箱、token、authorization header、cookie
    • 请求体、完整 URL、原始 SQL、客户名
    • 订单 ID、请求 ID、会话 ID 作为索引维度
    • 公开示例中的内部主机名或私有拓扑细节

    坏属性不是外观问题。它决定真实事故能否被快速收窄，也决定遥测是否适合保留。

    先治理采样和基数，再 rollout¶

    采样不只是成本设置。它决定事故证据是否会留下来。

    生产流量进入 SigNoz 前，先定义：

    • 采样 Owner 和批准路径。
    • 普通流量的默认采样策略。
    • 错误 trace、慢 trace 和显式诊断流量的保留规则。
    • 单次变更允许接入的服务数或流量上限。
    • resource attributes 和 span attributes 的基数预算。
    • 合成慢请求和合成错误请求不会被采样丢弃的验证。

    不要等 ClickHouse 或 collector 已经有压力后才调采样。那时系统可能已经丢掉解释压力来源的 trace。

    Kubernetes 部署标准¶

    部署前：

    • 确认角色：实验工具、团队分析后端，还是生产可观测服务。
    • 确认 OpenTelemetry Collector 部署位置，以及 receiver/exporter 端口。
    • 确认默认 StorageClass 支持所需扩容和回收行为。
    • 固定 Helm chart 与 image 版本。
    • 创建独立 namespace。
    • 在暴露 frontend 前决定 ingress、认证、TLS 和网络访问。
    • 评审 ClickHouse、collector、Alertmanager 或告警服务、frontend、query service 的资源请求和限制。
    • 生产流量进入前决定留存和容量告警。

    部署只有在真实埋点工作负载能通过预期 collector 路径发送遥测，并且真实操作员能通过预期访问路径找到它时，才算完成。

    验证¶

    分层验证，并在第一层不清楚时停止。

    层级 证据 停止条件
    后端健康 SigNoz frontend、query service、ClickHouse、collector 和告警组件 ready。 任一核心组件不健康或反复重启。
    存储 ClickHouse 表初始化，PVC 绑定，retention 已配置，磁盘告警存在。 存储无法扩容、留存缺失，或磁盘压力无人负责。
    Collector 路径 测试工作负载通过 OTLP 发到 collector，collector 导出到 SigNoz，无队列饱和或丢弃。 导出错误、队列压力、memory limiter drop 或协议不匹配。
    属性质量 测试 traces 展示安全的 service.name、environment、namespace、version 和 owner 属性。 服务显示 unknown、泄漏危险属性，或标签噪音大到无法搜索。
    采样与基数 合成慢请求和错误请求可查询，高基数属性在大规模 rollout 前被阻断。 测试故障被采样丢弃，或标签制造无界序列和查询压力。
    访问与告警 真实操作员通过批准路径访问 UI，测试告警进入预期渠道。 访问依赖 port-forward、共享 admin，或告警路由无人负责。
    自监控 SigNoz、ClickHouse 和 collector 健康、drop、队列压力、磁盘压力、查询失败通过不依赖 SigNoz 的路径告警。 SigNoz 的失败只能在 SigNoz 自己里面被发现。
    事故可用性 一个合成慢请求/错误请求能从服务定位到 trace，并关联相关 logs 或 metrics。 UI 有数据，但无法按 service、environment、version 或 owner 定位失败。

    Pod ready 不够。最低有用证明，是一个已知失败信号能从 workload 到 collector 到 backend 再到人。

    回滚与退出¶

    回滚取决于 SigNoz 在遥测路径中的位置。

    场景 更安全的回滚
    应用发到 collector，collector 导出到 SigNoz 删除或禁用 collector 中的 SigNoz exporter；应用埋点保持不变。
    应用直接发到 SigNoz 修改应用 endpoint 配置并重新部署受影响工作负载，影响面更大。
    SigNoz 不健康，但遥测仍需保留到其他后端 将 collector 导出路由到替代后端，并暂停 SigNoz 写入。
    永久移除 SigNoz 先停止 exporter，确认无依赖工作负载后卸载 release。只有明确批准后才删除 PVC。

    不要把删除 PVC 当作随手回滚的一部分。PVC 可能包含事故证据和受留存策略控制的遥测。

    如果该栈只是实验工具，在确认没有生产 collector 导出到它后，可以删除。

    失败模式¶

    失败模式 现象 正确处理
    UI 能打开但无法定位 有 traces，但无法按 service、version、environment 或 owner 过滤。 先修 resource attributes，再接入更多服务。
    绕过 collector 应用直接发到后端，没有过滤或路由控制。 把导出路径移到 OpenTelemetry Collector 后面。
    ClickHouse 磁盘压力 写入变慢、查询失败或 Pod 重启。 降低写入量，检查 retention，扩容存储，评审采样和基数。
    敏感遥测泄漏 Traces 或 logs 包含 token、headers、请求体、用户数据或客户上下文。 停止问题导出，在 collector 和应用层过滤，并评审留存。
    Demo 访问变成生产访问 操作员依赖 kubectl port-forward 或共享 admin。 生产使用前建立受控 ingress、认证和授权。
    告警脑裂 Prometheus 和 SigNoz 对类似症状重复告警，或都不负责。 定义告警所有权，把每个症状路由到一个响应路径。
    自监控依赖 SigNoz backend、ClickHouse 或 collector 故障只能在故障栈内部看见。 通过外部路径告警 SigNoz 健康、ingestion drops、队列压力、磁盘压力和查询失败。
    采样隐藏事故 合成慢请求或失败请求在 traces 中找不到。 保留错误/慢诊断流量，降低 rollout 范围，并在接入更多服务前评审采样。
    rollout 后后端过载 Dropped spans、collector 队列压力、ClickHouse 写入率高。 降低采样，暂停 rollout，验证 collector 内存和队列限制。
    Logs 被当成免费附属品 成本和留存无 Owner 地增长。 除非 SigNoz 日志留存和访问有 Owner，否则日志留在批准的日志平台。

    安全说明¶

    • 把遥测视为生产数据导出。
    • 限制 UI 访问，避免共享 admin 账号。
    • 后端凭据放在 collector 或密钥管理中，不放应用清单。
    • 过滤请求体、authorization headers、cookies、tokens、用户标识和完整 URL。
    • frontend 和 collector 路径使用 TLS 与网络控制。
    • 限制谁能修改 collector pipeline、retention 和 alert route。
    • 不公开真实服务名、namespace、域名、用户标识、dashboard 截图或查询结果。

    评审周期¶

    在以下情况评审：

    • 从实验角色迁到团队或生产角色前
    • 启用新的信号类型，例如 logs 或 metrics
    • 接入高流量服务时
    • collector drop、存储压力或告警路由失败后
    • OpenTelemetry Collector、SigNoz、ClickHouse 或 Kubernetes 升级期间
    • 留存、访问策略或数据分类变化时
