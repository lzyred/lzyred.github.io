    Docker 主机运维标准¶

    目的¶

    本标准定义如何运营承载单机 workload、build job、内部工具或过渡服务的 Docker 主机。

    核心判断：

    Docker 主机是一个故障域。如果它的磁盘、daemon、bridge、socket 或 volume 无人管理，容器只会让故障更容易重启，不会让故障更容易理解。

    Docker 适用于有边界的 workload。它不应变成隐藏编排平台、无人负责的生产状态，或绕过调度、镜像仓库、备份和访问控制的捷径。

    适用场景¶

    适用于：

    • 单机 Docker workload
    • 开发或集成主机
    • 轻量内部工具
    • CI/build machine
    • Kubernetes 或托管平台前的过渡服务
    • 有明确 owner 和恢复路径的 host-managed container stack

    当 workload 需要多节点调度、自动故障转移、水平扩展、强租户隔离或正式发布治理时，优先使用 Kubernetes、托管调度器或托管服务。

    非目标¶

    本标准不批准在一台 Docker 主机上运行关键多服务生产平台，也不替代 registry governance、image build standard、Compose stack standard 或 Kubernetes operations。

    主机契约¶

    Docker 主机承载共享或近生产 workload 前，必须定义：

    契约 必须决策 为什么重要
    Owner 谁负责 daemon config、cleanup、patching、incident 和退役？ 无 owner 的 Docker 主机会变成被遗忘的生产依赖。
    Workload class 是 build runner、internal tool、transient app、stateful service，还是 migration bridge？ 清理、重启和备份规则不同。
    Data boundary 哪些 volume 或 bind mount 保存业务数据？哪些可丢弃？ docker volume prune 可以删除状态。
    Image source 哪个 registry 和 tag/digest 被批准？ 回滚依赖可恢复镜像。
    Log policy 容器日志如何限量和采集？ JSON logs 可以写满主机文件系统。
    Network boundary 使用哪些 bridge CIDR、端口和主机路由？ Docker network 可能与 VPN、云或 Kubernetes 地址段冲突。
    Socket boundary 谁能访问 Docker socket 或 daemon API？ Docker socket 访问几乎等同于主机级控制。
    Recovery path 恢复是 restart、image rollback、volume restore、host rebuild，还是迁移出去？ 重启容器不能恢复丢失状态。

    推荐标准¶

    Workload 到来前先配置 daemon¶

    长期容器运行前，先准备经过评审的 daemon 配置：

    {
  "log-driver": "json-file",
  "log-opts": {
    "max-size": "100m",
    "max-file": "3"
  },
  "storage-driver": "overlay2"
}

    应用并验证：

    sudo systemctl daemon-reload
sudo systemctl enable --now docker
docker info

    Docker data root 应放在容量足够的存储上，能承载 images、layers、logs、build cache 和 volumes。如果 /var/lib/docker 在 root filesystem 上，Docker 磁盘压力会变成主机压力。

    Volume 和 bind mount 是数据契约¶

    运行或清理 workload 前，记录：

    • named volumes
    • bind mount paths
    • 每个路径是业务数据、cache，还是可丢弃数据
    • backup owner
    • restore command 或 rebuild path
    • retention expectation

    不要把所有 Docker 数据都当成 cache。部分数据可能是应用状态的唯一副本。

    磁盘压力出现前先限制日志¶

    第一个长期 workload 启动前就应限制容器日志。即使已有集中日志，也要限制本地日志增长。

    最低控制：

    • daemon config 或 per-container log rotation
    • Docker data root 磁盘使用告警
    • cleanup owner
    • 如果本地日志被清理，证明应用日志可从其他位置获得

    按风险顺序执行受控清理¶

    当 build cache、旧镜像、停止容器或未使用网络占用磁盘时，使用清理。

    前置条件：

    • 已查看 docker system df
    • 已识别关键容器
    • 已检查 stopped container 是否包含 writable layer 临时数据、故障排查证据，或未在其他位置保留的日志
    • 已识别 stateful volume，除非批准否则排除
    • 当前 rollback images 已确认在本地或 registry 可用
    • 已检查是否处于活跃发布或回滚窗口

    满足前置条件后，按从低到高风险顺序清理：

    docker container prune -f
docker network prune -f
docker image prune -f
docker builder prune -f

    高风险清理：

    # 高风险：删除所有未被运行中容器使用的镜像。
# 使用前确认 rollback images 可从 registry 获取。
docker image prune -a

    除非已知 volume owner 和数据生命周期，否则避免 volume cleanup：

    # 高风险：仅在明确批准且确认 volume 可丢弃时使用。
docker volume prune

    验证：

    • 回收了预期磁盘空间
    • 运行中容器仍然健康
    • rollback images 仍在本地或 registry 可用
    • 磁盘告警恢复
    • 没有未批准删除 stateful volume

    删除状态前先恢复 Docker 网络¶

    Docker 网络问题包括容器间名称解析失败、bridge CIDR 冲突、IP range 变化后服务不可达，或默认 bridge 损坏。

    低风险检查：

    docker network ls
docker network inspect bridge
ip addr
iptables -S

    低风险恢复：

    1. 停止受影响的非关键容器。
    2. 只删除未使用 Docker network。
    3. 重启 Docker。
    4. 通过已评审 Compose 或部署文件重建容器。

    只有当 bridge 损坏、与主机路由冲突，或使用错误 CIDR 时，才执行完整 bridge reset。

    前置条件：

    • 维护窗口已批准
    • 已识别运行中容器
    • 已备份 /etc/docker/daemon.json
    • 新 bridge CIDR 已与 host、VPN、Kubernetes 和 cloud range 比对
    • 远程访问不依赖即将重置的 Docker bridge

    流程：

    1. 干净停止 Docker。
    2. 确认没有关键容器被期望继续运行。
    3. 使用批准 bridge CIDR 和日志轮转设置更新 daemon.json。
    4. 在 Docker 仍处于停止状态时，只有当 stale docker0 仍存在并阻塞批准 CIDR，才删除它：
    sudo ip link set docker0 down
sudo ip link delete docker0
    1. 启动 Docker，让它从配置重建默认 bridge。
    2. 验证 bridge 地址、容器网络、NAT 和服务可达性。

    不要把删除 /var/lib/docker 当成常规修复步骤。数据删除是单独动作，需要自己的备份和批准关口。

    保护 Docker socket¶

    访问 /var/run/docker.sock 几乎等于控制主机。挂载 Docker socket 的容器可以启动特权容器、通过 mount 读取主机文件，并改变运行中 workload。

    必需控制：

    • 普通应用容器不挂载 Docker socket，除非 host-control 角色明确
    • 使用 Docker socket 的 CI runner 与不可信 job 隔离
    • daemon TCP API 禁用或严格保护
    • Docker group membership 视为特权访问
    • 人员或 runner 变化后复核 socket access

    验证¶

    docker version
docker info
docker run --rm <approved-registry>/<project>/<smoke-test-image>@<digest>
docker system df

    通过条件：

    • Docker daemon 从已评审配置启动
    • storage driver 和 data root 符合主机契约
    • log limit 已生效
    • disk usage 可见且有告警
    • 运行中容器健康
    • volume 和 bind mount 归属已记录
    • Docker socket 访问仅限批准 operator 或 workload
    • network CIDR 不与 host、VPN、Kubernetes 或 cloud range 冲突

    故障模式¶

    故障模式 结果 预防
    Docker data root 写满 root filesystem 主机不稳定或故障。 将 data root 放到规划存储，并对使用率告警。
    日志无限增长 Container JSON logs 写满磁盘。 workload 运行前配置 log rotation。
    把 volume 当 cleanup 删除 Stateful data loss。 盘点 volume owner，并排除 stateful volume。
    rollback image 被删除 事故中无法快速回滚。 cleanup 前确认 previous image digest 在本地或 registry 存在。
    Docker bridge 与网络地址段冲突 容器或主机路由异常。 预留并验证 bridge CIDR。
    Docker socket 暴露 容器或用户获得主机级控制。 限制 socket 和 Docker group access。
    修复时删除 /var/lib/docker image、volume 和 container state 丢失。 数据删除必须作为单独批准恢复动作。
    未评审 Compose 运行关键服务 隐藏生产平台出现。 要求 stack owner、state、exposure、backup 和 rollback contract。

    回滚与恢复¶

    恢复可能是：

    • restart failed container
    • restore previous daemon config
    • roll back to previous image digest
    • restore named volume from backup
    • restore Docker bridge configuration
    • rebuild host and redeploy workloads
    • migrate workload to scheduler or managed platform

    如果 stateful volume data 丢失，image rollback 不够。必须先恢复数据，或确认 workload 可丢弃。

    Daemon 配置回滚：恢复上一版 /etc/docker/daemon.json，reload systemd，restart Docker，并验证 workload。镜像回滚使用上一批准 digest，不使用 mutable tag。

    安全说明¶

    • 从可信仓库安装 Docker。
    • 保持 Docker 和 host OS 补丁更新。
    • 限制 Docker group membership。
    • 避免 privileged container，除非有记录清楚的 host integration reason。
    • 普通应用容器不要挂载 Docker socket。
    • Registry credential 不进入 Compose file 或 shell history。
    • 不公开真实 hostname、registry URL、image name、volume path、private IP range、credential 或生产命令输出。

    复审周期¶

    发生磁盘压力事故、daemon 或 network 变更、cleanup 失误、socket access 变化、host patching、runner onboarding，或单机 workload 变得长期存在/业务关键前，复审 Docker 主机。
