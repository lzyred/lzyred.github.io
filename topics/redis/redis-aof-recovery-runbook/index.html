    Redis Reliability Runbook

    Redis AOF 检查与恢复 Runbook¶

    禁止在生产数据目录、唯一副本或证据副本上运行 redis-check-aof --fix。 本文的默认路径是寻找更完整的恢复源，而不是修复 AOF。

    flowchart TD A[Redis 无法加载 AOF] --> B{存在健康副本吗} B -- 有 --> C[验证复制位置与数据完整性] C --> D[从健康副本重建] B -- 无 --> E{存在满足 RPO 的合格备份吗} E -- 有 --> F[恢复备份并验证] E -- 无 --> G[只读检查 AOF] G --> H{仅最后尾部可有限截断吗} H -- 否 --> I[停止并升级数据恢复事件] H -- 是 --> J[批准预计损失] J --> K[仅修复 repair_dir] K --> L[隔离加载与业务验证]
    mindmap root((Redis AOF 恢复模型)) 优先恢复源 健康副本 合格备份 AOF 候选 证据保全 source 只读 evidence checksum 日志与配置 有限修复 recovery 只读检查 repair 唯一可修改 估算 RPO 隔离边界 独立目录 独立端口 禁用复制与 Cluster 无生产网络 接受标准 key 与 TTL 业务样本 唯一写入权威 独立切换计划

    触发条件¶

    只在 Redis 因 AOF、manifest 或格式错误无法启动，节点故障后需要判断 AOF 候选，单文件尾部疑似截断，Redis 7/8 multi-part AOF 关系异常，或 rewrite 中断导致权威文件集不明时进入本 Runbook。

    Redis Cloud、Redis Software/Enterprise、Operator 托管底层不可访问、Redis Cluster 全集群灾难恢复、供应商专有加密/模块/存储层，或仍有健康 primary/replica 与合格备份时，应使用对应平台恢复流程。

    风险等级¶

    这是高风险数据恢复。错误动作可能截断有效写入、把较旧副本误判为权威、修改唯一证据、让隔离实例加入原拓扑、因版本/模块/ACL/TLS 不匹配误读 AOF，或同时开放多个写入源。

    任何 --fix、候选提升和生产切换都必须由数据 Owner 批准。verification_status: not-verified 说明本文未在用户真实环境执行。

    所需权限¶

    需要原配置、日志和数据目录只读访问；查询副本、备份、复制状态和业务数据的权限；受控 root/sudo 或平台权限；至少容纳三份数据集的空间；与生产网络隔离的恢复环境；数据 Owner 对 RPO/修复/切换的书面批准；以及与来源相同 major/minor 的 Redis 工具。

    前置检查¶

    先确认角色、流量和更完整候选¶

    redis-cli -h <host> -p <port> ROLE
redis-cli -h <host> -p <port> INFO replication
redis-cli -h <host> -p <port> INFO persistence
redis-cli -h <host> -p <port> CONFIG GET dir
redis-cli -h <host> -p <port> CONFIG GET appendonly
redis-cli -h <host> -p <port> CONFIG GET appendfilename
redis-cli -h <host> -p <port> CONFIG GET appenddirname

    必须回答角色、是否仍有客户端写入、自动拉起机制、哪个候选更完整、是否有已演练备份、允许的最大 RPO。仍承载权威写入时不得停止；停止方式必须匹配真实部署。

    记录版本、配置和文件事实¶

    redis-server --version
redis-check-aof --version 2>/dev/null || true
find <source-dir> -maxdepth 3 -type f -printf '%TY-%Tm-%Td %TH:%TM:%TS %s %p\n' | sort
df -h <source-dir>

    记录 redis_conf、source_dir、appendfilename、appenddirname、部署方式和角色。模块、ACL、TLS、加密、压缩层或文件系统无法复现时停止。

    建立四层目录和校验基线¶

    flowchart LR S[source_dir<br/>原始现场] -->|只读复制| E[evidence_dir<br/>不可变证据] E -->|完整复制| R[recovery_dir<br/>只读检查候选] R -->|批准后复制| P[repair_dir<br/>唯一允许 --fix] R --> I[隔离加载未修复候选] P --> I2[隔离加载修复候选]
    case_id="redis-aof-$(date -u +%Y%m%dT%H%M%SZ)"
work_root="/var/tmp/$case_id"
evidence_dir="$work_root/evidence"
recovery_dir="$work_root/recovery"
repair_dir="$work_root/repair"
sudo install -d -m 0700 "$evidence_dir" "$recovery_dir" "$repair_dir"
sudo cp -a --reflink=auto "<source-dir>/." "$evidence_dir/"
sudo cp -a --reflink=auto "$evidence_dir/." "$recovery_dir/"
sudo find "$evidence_dir" -type f -print0 | sort -z | xargs -0 -r sha256sum | sudo tee "$work_root/evidence.sha256" >/dev/null
sudo find "$recovery_dir" -type f -print0 | sort -z | xargs -0 -r sha256sum | sudo tee "$work_root/recovery-before.sha256" >/dev/null
    目录 允许修改 用途
    source_dir 否 原生产现场
    evidence_dir 否 不可变证据与 checksum
    recovery_dir 否 只读检查与未修复加载
    repair_dir 是 唯一允许 --fix 的副本

    操作步骤¶

    识别 Redis 6.2 与 Redis 7/8 文件结构¶

    flowchart TB A[Redis 6.2] --> A1[appendonly.aof<br/>single file] B[Redis 7 / 8] --> B1[appendonlydir] B1 --> B2[appendonly.aof.manifest] B1 --> B3[base AOF or RDB file] B1 --> B4[incremental AOF files] B2 --> B3 B2 --> B4

    以真实配置和 manifest 为准。Redis 7/8 必须确认 manifest 引用的 base 和 incremental 文件存在、顺序合理且未混合不同 rewrite 周期。缺失或中间损坏时不要手工拼接。

    使用匹配版本工具只读检查¶

    redis-check-aof "$recovery_dir/<appendfilename>" 2>&1 | tee "$work_root/redis-check-aof-readonly.log"
redis-check-aof "$recovery_dir/<appenddirname>/<appendfilename>.manifest" 2>&1 | tee "$work_root/redis-check-aof-readonly.log"

    文件有效则直接隔离加载 recovery；仅最后命令或最后 incremental 尾部截断时才评估损失；中间损坏、manifest/文件缺失、版本无法识别或多个候选各自有独有写入时立即停止。

    估算修复损失¶

    有效 offset 与文件大小只能估算字节截断量，不能单独证明业务时间范围。必须结合 Redis 日志、复制 offset/backlog、应用写入审计、备份时间和业务流水；mtime 仅是辅助。记录待截断字节、影响文件、可能丢失命令、时间范围、批准人和最大 RPO。无法说明边界时不得 --fix。

    仅在批准后修复 repair 副本¶

    sudo rm -rf "$repair_dir"
sudo install -d -m 0700 "$repair_dir"
sudo cp -a --reflink=auto "$recovery_dir/." "$repair_dir/"
sudo find "$repair_dir" -type f -print0 | sort -z | xargs -0 -r sha256sum | sudo tee "$work_root/repair-before.sha256" >/dev/null
redis-check-aof --fix "$repair_dir/<appendfilename>" 2>&1 | tee "$work_root/redis-check-aof-fix.log"
redis-check-aof --fix "$repair_dir/<appenddirname>/<appendfilename>.manifest" 2>&1 | tee "$work_root/redis-check-aof-fix.log"
sudo find "$repair_dir" -type f -print0 | sort -z | xargs -0 -r sha256sum | sudo tee "$work_root/repair-after.sha256" >/dev/null
diff -u "$work_root/repair-before.sha256" "$work_root/repair-after.sha256" | tee "$work_root/repair-files.diff" || true

    不得对同一 repair 副本反复尝试不同修复。失败时丢弃并从 recovery 重新生成。

    在隔离边界内启动候选¶

    flowchart LR P[原生产拓扑] -. 禁止连接 .-> I[隔离恢复实例] I --> D[独立数据目录] I --> PORT[独立端口 16379] I --> NET[无生产网络出口] I --> REP[replication disabled] I --> CL[cluster disabled] V[验证人员] -->|localhost or controlled tunnel| I I --> LOG[独立日志与 PID]

    复制原配置以保留模块、ACL 和编码行为，再覆盖 bind 127.0.0.1、独立端口、独立目录、禁用 replicaof/slaveof、cluster-enabled no、独立日志和 PID。bind 不是完整隔离，仍应使用断网主机、容器或 network namespace。

    standalone 可以独立加载；primary 必须先确认无更完整 replica；replica 需比较 offset；Cluster 的 slot、节点配置和多分片一致性不在本文范围。模块缺失时应复现原模块，不能删除未知命令来强行启动。

    每一步预期结果¶

    阶段 预期结果
    候选判断 健康副本、备份和 AOF 完整性/RPO 已比较
    现场保全 source 未修改，evidence checksum 已保存
    只读检查 recovery 未修改，工具和入口明确
    修复审批 截断和业务损失边界已批准
    有限修复 只有 repair 发生预期变化
    隔离加载 独立目录/端口，无复制、Cluster 或生产连接
    数据验证 key、TTL、类型、数量级和业务样本可接受
    切换准备 有独立变更、备份、监控和回退计划

    停止条件¶

    无法区分四层目录、无法判断候选权威、工具版本不匹配、manifest/base/incremental 缺失、损坏不在尾部、损失超过 RPO 或无法评估、依赖无法复现、隔离实例尝试连接原拓扑、关键数据解析失败或需要合并多个独有写入源时，立即停止并升级。

    成功验证¶

    redis-cli -p 16379 PING
redis-cli -p 16379 ROLE
redis-cli -p 16379 INFO persistence
redis-cli -p 16379 INFO replication
redis-cli -p 16379 INFO keyspace
redis-cli -p 16379 DBSIZE
redis-cli -p 16379 CONFIG GET dir
redis-cli -p 16379 CONFIG GET appendonly
redis-cli -p 16379 CONFIG GET appendfilename
redis-cli -p 16379 CONFIG GET appenddirname
redis-cli -p 16379 CONFIG GET cluster-enabled

    确认无生产 upstream/downstream、目录和 AOF 入口正确、日志无持续错误、关键 key/TTL/类型和业务样本可读、数量级与权威来源一致、恢复点满足 RPO、应用能解析、生产切换另有计划且只有一个写入源。

    失败处理¶

    工具无法识别则核对版本/manifest/入口；修复损失过大则废弃 repair 并改用副本/RDB/备份；隔离启动失败则保留日志并从 recovery 重建；模块缺失则复现模块；抽样失败不得接入业务；多个独有写入源进入数据合并与业务裁决。

    回滚与升级路径¶

    本文的回滚是丢弃 repair/recovery 候选并回到不可变 evidence，不是修改生产原目录。生产切换必须保持旧节点和证据冻结，以候选创建新实例，保证唯一写入权威，逐步恢复客户端并监控；失败时重新隔离候选并回到健康副本或已验证备份。保留日志、manifest、文件清单、checksum、审批和预计损失。
