    Redis 生产运维标准¶

    Redis 默认使用异步复制。副本、AOF、Sentinel、Cluster 和 WAIT 都不能自动保证零数据损失。生产设计必须先确定业务可以接受的数据丢失量和恢复时间，再选择配置。

    适用范围¶

    本标准适用于自管 Redis 6.2 及以上版本，包括单实例、主从加 Sentinel，以及 Redis Cluster。

    它不替代云厂商托管 Redis 的产品约束，也不把 Redis 定义为所有业务的权威数据库。缓存、会话、队列、排行榜和关键业务状态需要分别评估。

    前提¶

    上线前必须记录：

    • 数据角色：缓存、可重建状态或权威数据；
    • 数据权威源和重建方式；
    • RPO、RTO 与允许的降级方式；
    • 峰值数据量、写入速率和增长预估；
    • Owner、值班责任和升级联系人；
    • 使用版本、部署模式和持久化目录。

    如果这些信息未知，不应通过“先部署再观察”补齐。

    强制要求¶

    内存与淘汰¶

    • 设置 maxmemory，为复制缓冲、客户端缓冲、fork 和操作系统保留空间；
    • 按数据角色选择 maxmemory-policy，权威数据不得依赖不可控淘汰；
    • 监控 used_memory、used_memory_rss、碎片率、evicted keys 和 blocked clients；
    • 容器 limit 不能等于 maxmemory，否则 Redis 可能在达到自身阈值前被 OOM Kill。

    持久化与数据保护¶

    • 缓存可以关闭持久化，但必须证明数据能够重建；
    • 需要恢复能力时，明确使用 RDB、AOF 或组合模式，并测试实际恢复；
    • Redis 7+ 的多段 AOF 必须连同 manifest、base 和 incremental 文件作为一个文件集管理；
    • 备份必须复制到独立故障域，并定期执行恢复验证。

    高可用与切换¶

    • Sentinel 和 Cluster 解决自动选主或分片可用性，不等于同步复制；
    • 切换前检查复制状态、延迟、积压和持久化健康；
    • 故障切换后必须确认客户端连接、写入目标和旧主隔离；
    • 回切属于新的数据迁移，必须重新确认数据方向、写入冻结和验证步骤。

    安全¶

    • 只监听受控网络接口，并限制安全组或防火墙来源；
    • 使用 ACL 分离应用、运维和监控权限；
    • 禁止在文档、命令历史或流水线日志中暴露凭据；
    • 对跨主机或不可信网络通信启用 TLS；
    • 生产变更必须经过身份审计和最小权限控制。

    监控与变更¶

    至少监控：可用性、命令延迟、内存、连接数、复制状态、持久化状态、主从角色变化、拒绝连接和慢查询。

    升级、配置修改、扩容、切换和数据迁移必须有：基线、影响范围、验证信号、停止条件、回滚或恢复方案。

    验证¶

    上线或重大变更后，至少完成：

    redis-cli INFO server
redis-cli INFO memory
redis-cli INFO persistence
redis-cli INFO replication
redis-cli ACL WHOAMI

    验证结果必须证明：

    • 应用连接到预期实例和角色；
    • 内存余量满足 fork 与负载峰值；
    • 持久化无错误，最近保存或 AOF 重写状态正常；
    • 副本延迟与链路状态在约定范围内；
    • 备份可以在隔离环境恢复；
    • 告警能够触发并在状态恢复后关闭。

    例外¶

    允许例外的情况必须记录业务原因、风险、补偿控制、Owner 和到期时间。

    例如，纯缓存实例可以不启用持久化，但必须有稳定的回源能力、限流保护和缓存重建方案。临时测试环境可以降低高可用要求，但不能使用生产凭据或生产数据。

    回滚¶

    配置变更应保留旧配置和启动参数。版本升级应保留上一制品以及兼容性验证结果。

    涉及数据方向变化时，不得把“重新指回旧主”当作简单回滚。必须先冻结写入、确认权威数据、评估差异，再决定回切、补偿或从备份恢复。

    专项操作见：Redis AOF 检查与恢复 Runbook。
