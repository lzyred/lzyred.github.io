    站点体验升级记录¶

    本文记录 DevOps 博客站点的体验升级方向、取舍边界和落地顺序。它不是面向普通读者的专家文章，而是维护者在后续调整主题、导航、博客能力和 AI 助手入口时使用的决策记录。

    背景¶

    当前站点已经完成了两个基础能力：

    • 使用 MkDocs Material 作为文档站点框架。
    • 使用 mkdocs-static-i18n 将中文作为默认站点根路径，将英文发布到 /en/，并保留旧 /zh/ 路径的跳转兼容。

    这解决了“能发布”和“能中英文切换”的问题，但还没有解决“看起来像一个资深 DevOps 专家的长期站点”的问题。后续升级需要围绕可信度、可读性、导航效率和内容生产秩序展开，而不是只做视觉装饰。

    设计目标¶

    站点体验升级应服务于四个目标：

    1. 可信：首页、导航和文章结构要让读者快速判断作者确实做过生产级 DevOps 工作，而不是只整理工具命令。
    2. 耐读：长文、标准、复盘、案例都要在桌面和移动端保持稳定阅读体验。
    3. 可维护：主题、语言、导航、博客能力和发布流程要能被后续维护者理解和回滚。
    4. 适合问答助手复用：内容结构应保留清晰标题、稳定 URL、明确上下文和可检索知识块。

    已采用的基础方案¶

    国际化¶

    当前采用插件级国际化，而不是手写语言切换链接。

    • 中文源文件位于 docs/zh/。
    • 英文源文件位于 docs/en/。
    • 中文构建到站点根路径。
    • 英文构建到 /en/。
    • 旧 /zh/ 路径保留跳转，避免外部链接失效。

    维护原则：

    • 新增正式内容时优先同步中英文版本。
    • 如果英文暂时无法达到同等质量，可以先不发布该英文页面，而不是发布低质量直译。
    • 不再在页面顶部手写 [English] 或 [中文] 链接，语言切换交给插件生成。

    内容结构¶

    当前站点按专家博客结构组织：

    • Start Here
    • Blog
    • 专家方法论
    • 工程体系
    • 标准与手册
    • 技术知识库
    • 站点与维护

    维护原则：

    • 新的深度文章优先进入专家方法论和工程体系，不要堆进技术知识库。
    • 技术知识库用于承载已清洗的工具、标准和操作资料，不承担站点主叙事。
    • 面向读者的文章和面向维护者的规则要分开，避免公开站点像内部说明书。

    推荐升级路线¶

    第一阶段：主题与导航质感¶

    目标是用低风险配置提升第一眼体验。

    建议加入：

    • 明暗主题切换，支持系统偏好。
    • 更克制的品牌色，不使用大面积花哨渐变。
    • 顶部导航标签，突出主要栏目。
    • 返回顶部、目录跟随、搜索高亮和搜索建议。
    • 轻量 logo 或图标，保持技术站点气质。

    判断标准：

    • 首页不像模板站。
    • 长文阅读不刺眼。
    • 移动端目录和导航不挤压正文。
    • 改动主要集中在 mkdocs.yml 和少量 CSS，便于回滚。

    第二阶段：博客化能力¶

    目标是让站点不只是文档库，也像一个持续更新的专家博客。

    建议评估：

    • 博客插件。
    • 文章日期。
    • 标签和分类。
    • 归档页。
    • RSS。
    • 作者说明。

    取舍：

    • 如果站点还没有稳定的文章发布节奏，不急着启用完整博客系统。
    • 标签数量必须克制，避免出现大量只有一篇文章的标签。
    • 归档和 RSS 只有在持续发布后才有明显价值。

    第三阶段：外部传播能力¶

    目标是让文章被分享时更专业。

    建议评估：

    • 社交分享卡片。
    • Open Graph 元信息。
    • 站点 favicon。
    • 自定义文章摘要。
    • 轻量访问分析。

    取舍：

    • 不应为了追踪数据牺牲加载速度或隐私。
    • 社交卡片应服务于深度文章传播，不必覆盖每一篇低价值归档文档。

    第四阶段：问答 AI 助手入口¶

    目标是为后续 DevOps 问答助手预留自然入口，而不是把站点变成聊天应用。

    建议方式：

    • 在导航保留 Assistant 入口或 Start Here 中放置入口。
    • 助手回答必须引用站内文档小节。
    • 助手不能绕过公开知识库直接回答疑似敏感、内部、凭证类问题。
    • 对无依据回答要明确说明缺少证据，而不是编造。

    取舍：

    • 早期可以先做“文档检索 + 引用回答”，不急着做复杂多轮自动化。
    • 助手入口应辅助阅读，不应抢走深度文章的主位置。

    不建议现在做的事¶

    • 大规模自定义主题，尤其是复杂暗色 UI、动效和重写布局。
    • 在内容质量不足时先追求评论系统、会员系统或复杂分析。
    • 为每篇旧文档都强行补标签。
    • 把内部维护规则暴露成主导航里的读者内容。
    • 为了“高级感”加入大量装饰图、渐变背景或视觉噪音。

    这些改动容易让站点看起来更复杂，但不会提高专家可信度。

    内容质量与视觉质量的关系¶

    一个资深 DevOps 博客的“高级感”主要来自三件事：

    • 文章能解释生产环境里的约束、失败模式和取舍。
    • 案例能保留匿名化后的规模、现象、决策压力和复盘结论。
    • 站点结构能帮助读者从问题进入，而不是从工具分类进入。

    视觉升级只能放大内容的可信度，不能替代内容本身。如果 Deep Dives、Case Studies 和 Decision Reviews 没有真实密度，再漂亮的主题也只会让站点更像模板。

    后续变更记录要求¶

    每次站点体验相关改动都应记录：

    • 改动日期。
    • 改动范围。
    • 变更原因。
    • 影响的文件。
    • 验证方式。
    • 可回滚路径。

    示例：

    Date: 2026-06-27
Scope: i18n
Reason: Replace manual language links with plugin-managed bilingual routing.
Files: mkdocs.yml, docs/zh, docs/en, tools/create-zh-redirect-aliases.mjs
Verification: make build, make verifyDist, online HTTP status checks
Rollback: Restore previous mkdocs.yml and remove i18n plugin config

    当前维护优先级¶

    当前优先级是补齐正式内容结构：博客入口、作者可信度、文章卡片、正式导航和栏目首页。评论系统、统计分析和动态社交卡片在没有配置边界前不进入默认发布路径。

    2026-06-27 执行记录¶

    范围¶

    本次按前三阶段执行：

    • 阶段一：主题与导航质感。
    • 阶段二：博客化能力。
    • 阶段三：外部传播能力。

    已实施¶

    • 在 mkdocs.yml 中启用明暗主题切换、系统偏好、顶部导航、固定导航标签、路径导航、页脚上一篇/下一篇、目录跟随、搜索高亮和搜索建议。
    • 正文字体改为系统字体，避免页面运行时依赖 Google Fonts。
    • 使用 mkdocs-material[imaging] 支持官方社交卡片图像生成能力，并采用本地静态品牌卡片布局，避免构建过程依赖 Google Fonts 下载。
    • 启用 Material blog 插件，新增中英文 Blog 入口、作者配置和第一篇站点体验升级记录。
    • 启用 Material tags 插件，新增中英文 Tags 页面。
    • 使用构建后脚本 tools/create-rss-feeds.mjs 为中英文博客文章输出 RSS。
    • 新增轻量品牌图标和少量 CSS，控制链接、标题、引用块、表格和 admonition 的阅读质感。

    取舍¶

    • 没有重写 Material 主题，避免未来升级和回滚困难。
    • 没有引入评论系统、访问分析或会员能力，因为当前阶段更需要内容可信度和发布稳定性。
    • Blog 只放一篇真实变更记录，不用低质量文章填充归档。
    • RSS 没有继续使用 mkdocs-rss-plugin，因为它与当前 mkdocs-static-i18n 组合在严格模式下会重复触发 date_from_meta.default_time warning；改为构建后脚本生成，保留 RSS 能力并保证 --strict 可通过。
    • 社交卡片先使用统一品牌卡片，而不是每页动态标题卡片；这样可以先保证构建稳定，后续如需动态标题再引入可提交到仓库的开源字体。

    验证要求¶

    每次修改后必须执行：

    make build MKDOCS=/private/tmp/devops-kb-mkdocs-venv/bin/mkdocs
make verifyDist

    发布后至少检查：

    • 首页。
    • 中英文 Blog 首页。
    • 中英文 Tags 页面。
    • 中英文第一篇博客文章。
    • RSS 文件。

    2026-06-27 博客优化补齐记录¶

    现状复盘¶

    上一轮已经完成主题切换、导航 tabs、路径导航、品牌色、favicon、Blog 插件、Tags、RSS、作者配置和 Social Cards，但仍存在三个会让站点显得半成品的问题：

    • /blog/ 首页只有标题，没有文章卡片、栏目说明和订阅入口。
    • 作者信息过于简略，不能支撑“资深 DevOps 专家博客”的可信度。
    • 导航结构把大纲、维护文档和旧知识库平铺在同一级，读者很难判断主阅读路径。

    本轮补齐¶

    • Blog 首页改为正式入口，展示推荐文章、RSS、标签入口、栏目定位和发布标准。
    • 作者信息补充为“生产架构、交付工程、可靠性、平台工程和内容安全维护者”，并指向 GitHub。
    • 增加文章卡片、状态卡片和维护卡片 CSS，避免 Blog 和维护页看起来像空白 Markdown。
    • 导航重组为：首页、开始阅读、Blog、专家方法论、工程体系、标准与手册、技术知识库、站点与维护。
    • 栏目首页去掉临时待办式表述，改为正式栏目定位、阅读路径和质量标准。

    暂不实现¶

    • 评论系统暂不启用。评论需要先确认 GitHub Discussions、Giscus category、公开互动边界和内容审核责任；在没有这些配置前上线评论入口会制造坏体验。
    • 统计分析暂不启用。统计需要先确定隐私策略和工具选择，例如 Plausible、Umami 或 GitHub Pages 外部日志；当前不加入任何会收集访问者数据的脚本。
    • Social Cards 暂时继续使用统一品牌图，不做每篇动态标题排版。动态标题卡需要字体、中文换行和构建稳定性验证，适合作为后续独立任务。

    完成标准¶

    • Blog 首页不再是空壳，能直接看到文章入口和订阅方式。
    • 导航不再使用旧归档命名作为主结构，而是把旧内容归入正式的技术知识库。
    • 维护文档说明哪些功能已实现、哪些被延后，以及延后的原因。
