    自评与成长路径¶

    大多数 DevOps 自评都让你在几个维度上给自己打分，然后把数字加起来。下面你会看到这样一张矩阵，因为它是个有用的起步快照。但它产出的那个数字，是这一页上最不可靠的东西，而相信它，正是工程师误判自己水平最常见的方式。

    陷阱在这里。评分网格奖励广度——当你在很多维度上都有那么一点能力时，它涨得最快。但真正把一个中级工程师和一个资深工程师区分开的不是广度。是你有没有在至少一个维度上深到——深到知道它危险的默认选项在哪，并能在压力下拒绝它们。分数看不见这个，因为"拒绝"不长得像更高的数字，它长得像同一个数字，但背后多了判断。

    分数为什么误导¶

    设想两个工程师。

    第一个，六个维度全是 2 分：总分 12。他能部署集群、跑流水线、看仪表盘、配权限、维护主机、打模板。平常的日子里一切正常。按网格，他稳稳的中级。

    第二个，五个维度 1 分，一个维度 3 分——比如变更交付：总分 8。他几乎处处更弱。但在交付上，他亲手发过那个坏变更，看着它把一个线上服务搞垮，并在身体里学会了一件事："流水线成功了"和"服务是健康的"是两个不同的论断。现在他拒绝在没有演练过回滚的情况下发布任何东西，而且他能精确讲出上一次一条绿色的流水线是怎么骗了他的。

    网格说第一个工程师强 50%。生产说正相反。第二个工程师跨过了第一个没跨过的东西：他在一个地方触到了底，而这一次经历重组了他对待每一次变更的方式。第一个工程师从未在任何地方触过底——这恰恰是为什么，在生产替他找到一个底的那天，他是危险的。没有任何一点真实深度的广度不是中级工程师，它是一个初级工程师，只是未经检验的自信摊得更广。

    这就是为什么分数是你输入的快照，而不是你水平的度量。水平活在网格从不发问的那个问题里：你在哪里深到了敢说"不"？

    "说不"到底是什么意思¶

    每个维度都有一套诱人、看起来合理、却在压力下错误的默认选项，而能力大体上就是识别并拒绝它们的累积。给几个这种动作的例子，让它具体而不是口号：

    • 在交付上：拒绝"回滚我们以后再补"，因为你经历过那次事故——"以后"从未到来，第一个坏变更就成了那场宕机。
    • 在可观测性上：拒绝"我们有监控，所以我们盖住了"，因为你看着一条迁移过来的告警变哑，而一个安静的系统和一个健康的系统长得一模一样。
    • 在平台工作上：拒绝"我们把这个集中起来，团队自然会采用"，因为你见过一个平台变成工单队列，让交付更慢、却看起来像进步。
    • 在容量与成本上：拒绝"它在演示里能跑"，因为你撞到过那个节点级的限制——它让一个工作负载调度得好好的，然后在真实流量下卡住。

    注意这些的共同点。没有一个是你学会的工具。每一个都是你不再信任的默认选项，因为现实曾为它向你收过一次费。这才是真实能力的质地，而它对一张 0–4 的网格是不可见的。网格度量你能做什么；而资深，越来越由你学会了拒绝什么来定义。

    用这张矩阵，然后越过它去看¶

    给自己打分吧——它是你广度的一份公平初稿，而在职业早期，广度恰恰是对的，原因却容易被忽略：你还看不出未来会把你逼深进哪个维度。过早收窄的工程师，把自己的整个水平押在一列上，然后一次角色变动、一次事故、或一次重组，把重量挪到了他从未碰过的那一列。又广又浅，是正确的对冲——直到你接触得足够多，知道该把深度花在哪。

    每个维度按 0–4 打分：0 你无法判断它；1 你能照文档操作；2 你能独立交付日常工作；3 你能设计方案并处理异常；4 你能定义标准并用数据改进它。

    维度 0-1 分 2 分 3-4 分
    Linux / 网络 / 存储 只会常用命令 能排查常见系统问题 能设计容量、隔离、恢复与安全基线
    容器 / Kubernetes / 云 能部署示例 能维护日常集群与服务 能设计生产高可用、升级与恢复策略
    CI/CD / 制品 / 环境 能跑一条流水线 能维护发布流程 能设计安全、可回退、可审计的交付体系
    可观测性 / 事故处理 能看日志和仪表盘 能诊断常见事故 能设计告警分级、SLO 与复盘回路
    安全 / 合规 / 成本 知道基本概念 能跑扫描、配权限 能把治理嵌入平台与交付流程
    平台 / 标准化 / 影响力 只服务自己 能打包脚本与模板 能服务多个团队并量化平台价值

    现在，按它真正预测生产行为的方式去读这个结果——那几乎和"加总"相反：

    • 你的下限比总分更重要。 在一个你负责的维度上得 0，就是一个潜伏的事故，无论其他维度有多高。先补下限，再抬上限。
    • 你最高的那一个维度比平均值更重要。 一个真实的 3——一个你深到能拒绝坏默认的地方——是比六个 2 更强的资深证据。如果你的最高分是 2，你还没有一个被检验过的地方；那个，而不是你的总分，才是要去攻的。
    • 形状告诉你下一步。 全低：建广度，跟着基础走。全 2 的平坦：停止铺摊，靠端到端地扛住一件事、穿过一次失败，把一个维度推到真正的 3。带一个 3 和几个 1 的尖刺：你有了第一处深度；现在去抬高你这处深度所依赖的下限。

    成长路径是一连串形状¶

    这一行的职业生涯，不是沿一架梯子向上爬，而是你网格形状的一种可预测的变化——而知道下一个形状，就是成长路径的全部。

    你从又平又低开始——什么都会一点，没有一处深。正确的动作不是把平均值平均地抬高，而是靠端到端地扛住一件事、穿过一次失败，直到你拒绝过一个坏默认并被证明是对的，把一个维度推到真正的 3。那第一处尖峰，就是从初级到中级的跳跃：你不再是一个操作工具的人，而成了一个有地方可以说"不"的人。

    接着形状变得尖锐——一两处真实的尖峰，几处低谷。从中级到资深的动作是反直觉的：它主要关于下限，不是新的尖峰。你在一个维度上的深度，只有当它所倚靠的那些维度不是 0 时才是安全的——一个出色的交付设计，会栽在一个你无法恢复的基础上；一个强大的平台，会塌在你从未建过的可观测性上。资深工程师花数年抬高尖峰之下的那些下限，好让尖峰能承受生产的重量。

    架构师与专家的形状最后才到，而且性质不同：网格不再关于你自己。问题变成了你能否抬高别人的下限、并把你的尖峰复制到各个团队——把你挣来的那些"拒绝"，编码进一条别人无需被烧过一次也能照走的路。这个高度上的风险，和初学者的正好相反：离开一线太远，远到你只能讲原则——那是它自己的一种下限跌到 0。

    所以这条路是：先变广，挣到一处尖峰，花数年补它之下的下限，然后学会把你的尖峰和下限装进别人身上。每个阶段，工具都是同一张网格、同一种读法——先看下限，再看最高尖峰，永远不看总和。

    诚实的证据¶

    当你想知道自己真实的级别时，别数你的工具，也别信你的总分。问你改变了什么：

    • 发布变得更安全，是因为你设计的某个体系，还是只是更快了？
    • 恢复变得更快，是因为你搭建并演练过的某条路径，还是你只是走运？
    • 某一类事故变得更少，是因为你做了结构性修复，还是它只是暂时没再发生？
    • 一个团队变得不那么依赖某一个人的记忆了，还是那个人变成了你？

    每一个背后带着机制的"是"，都比网格上的任何一分更值钱。这张矩阵的意义，是找出你哪里一个这样的"是"都没有，然后去挣一个回来。

    知道自己的档位后，用内容成熟度地图找出接下来读什么，并用 DevOps 能力拓扑 看你正在加深的那个维度如何与周围的维度相连。
