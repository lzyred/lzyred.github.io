    DevOps Deep Article Quality Standard¶

    Purpose¶

    This standard defines how to decide whether a source file, note, diagram, runbook, script, or case from the Trash review workspace is valuable enough to become a public DevOps knowledge base article.

    It is used for three goals:

    • Select high-value source material for the public DevOps knowledge base.
    • Rewrite internal or source-specific material into safe, reusable engineering guidance.
    • Prepare retrieval-friendly documents for a future DevOps Q&A assistant.

    Core Principle¶

    A deep DevOps article should not only explain what to run. It should explain the operational problem, the constraints, the design choice, the trade-offs, the implementation path, the validation method, and the rollback strategy.

    If a document only contains commands, raw configuration, screenshots, credentials, logs, or environment-specific notes, it is not a deep article by itself. It may still be useful as source material, but it must be rewritten, merged, or excluded.

    Publication Gate¶

    No source material may become public content unless it passes all publication gates.

    Gate Requirement Fail Result
    Public safety No secrets, keys, tokens, passwords, private certificates, internal domains, private IPs, customer names, personal data, private repositories, raw SQL dumps, production configs, or unreleased business details. Exclude or quarantine.
    Anonymization Project, customer, team, user, region, asset, domain, repository, account, and business names are replaced with generic placeholders. Rewrite before publication.
    Reusable value The material teaches a repeatable DevOps pattern, method, checklist, standard, or decision model. Merge into FAQ or exclude.
    Engineering context The article explains scenario, constraints, assumptions, and target state. Downgrade unless context can be reconstructed safely.
    Operational closure The article includes validation, rollback, risk, and maintenance guidance when operational change is involved. Rewrite before publication.
    Retrieval quality Headings are stable, sections are small, and each section answers one operational question. Restructure before publication.

    Deep Article Characteristics¶

    1. Problem-First¶

    The article begins with the operational problem, not the tool.

    Good:

    • A cluster upgrade needs to reduce workload disruption.
    • A CI/CD process needs safer rollback and auditability.
    • A logging platform needs predictable retention and query performance.
    • A cloud network migration needs isolation, route control, and validation.

    Weak:

    • How to install a tool without explaining why it matters.
    • A list of commands copied from a shell history.
    • A diagram with no explanation of constraints or target state.

    2. Clear Scope And Boundaries¶

    The article states when the guidance applies and when it does not.

    Examples:

    • Applies to production Kubernetes clusters with multiple node groups.
    • Applies to single-host Docker workloads, not production orchestration.
    • Applies to log aggregation patterns, not application-level logging design.
    • Applies to cloud migration planning, not raw customer network replication.

    3. Architecture Or Process Judgment¶

    The article explains why a method is selected.

    Examples:

    • Why use GitOps instead of ad hoc deployment scripts.
    • Why separate build, release, and deploy permissions.
    • Why prefer managed add-ons for EKS control plane integrations.
    • Why keep stateful data outside disposable compute nodes.

    4. Trade-Offs¶

    The article names the cost of the recommendation.

    Common DevOps trade-offs:

    • Availability vs cost.
    • Standardization vs flexibility.
    • Automation vs manual control.
    • Managed services vs self-managed control.
    • Security isolation vs operational complexity.
    • Fast delivery vs release governance.

    5. Implementation Path¶

    The article provides a safe sequence:

    1. Preparation.
    2. Baseline capture.
    3. Change plan.
    4. Execution.
    5. Validation.
    6. Rollback.
    7. Handoff and maintenance.

    6. Validation And Rollback¶

    The article explains how to prove success and how to recover.

    Validation should include:

    • What command, metric, log, dashboard, or user journey to check.
    • Expected healthy output or behavior.
    • Failure signals.
    • Post-change observation window.

    Rollback should include:

    • What can be reverted.
    • What cannot be safely reverted.
    • What data or state must be preserved.
    • Who approves rollback in a production-like environment.

    7. Security Boundary¶

    The article treats security as part of the engineering design.

    Security sections should cover relevant points:

    • Identity and access control.
    • Secret storage and rotation.
    • Network exposure and ingress.
    • CI/CD supply-chain risk.
    • Log and artifact leakage.
    • Backup and restore access.
    • Least privilege for automation.

    8. Reusable Model¶

    The article should produce a reusable model, not just a one-off story.

    Examples of reusable models:

    • Kubernetes upgrade readiness model.
    • Deployment rollback decision model.
    • Observability onboarding model.
    • Cloud network migration validation model.
    • Secrets handling model for CI/CD.
    • Public documentation sanitization model.

    Article Type Standards¶

    Standards¶

    Use when the content defines a repeatable operating baseline.

    Required sections:

    • Purpose.
    • When To Use.
    • Prerequisites.
    • Standard Procedure.
    • Validation.
    • Rollback or Recovery.
    • Security Notes.
    • FAQ.
    • Related Documents.

    Examples:

    • Docker Host Operations Standard.
    • Kubernetes Cluster Provisioning Standard.
    • Terraform IaC Standard.

    Runbooks¶

    Use when the content guides incident response or recurring operational action.

    Required sections:

    • Symptom.
    • Impact.
    • Triage.
    • Decision Tree.
    • Remediation.
    • Validation.
    • Escalation.
    • Rollback.
    • Prevention.

    Examples:

    • PVC Usage Alert Runbook.
    • TLS Certificate Expiry Runbook.
    • Deployment Rollback Runbook.

    Migration Cases¶

    Use when the content comes from a project or architecture change and can be anonymized.

    Required sections:

    • Anonymous Scenario.
    • Source And Target State.
    • Constraints.
    • Migration Strategy.
    • Phases.
    • Validation.
    • Rollback.
    • Risks.
    • Lessons Learned.
    • Reusable Checklist.

    The case must be rewritten as a general standard approach. Do not preserve customer-specific topology, business workflow, domains, internal service names, cost tables, or private architecture details.

    Troubleshooting Articles¶

    Use when the source explains a failure mode or recurring diagnostic pattern.

    Required sections:

    • Symptom.
    • Scope.
    • Likely Causes.
    • Diagnostic Commands.
    • Evidence Interpretation.
    • Fix Options.
    • Validation.
    • Prevention.

    Automation Articles¶

    Use when the source contains scripts or repeatable automation.

    Required sections:

    • Problem Automated.
    • Inputs.
    • Outputs.
    • Required Permissions.
    • Failure Modes.
    • Dry Run Or Safe Mode.
    • Logging.
    • Rollback Or Manual Recovery.
    • Security Notes.

    Never publish raw scripts that contain credentials, internal paths, production hostnames, account IDs, or private operational logic.

    Scoring Matrix¶

    Score each candidate from 0 to 5.

    Dimension 0 3 5
    Problem value No clear problem. Useful but narrow operational problem. Common high-value DevOps problem with clear impact.
    Context completeness No scenario or assumptions. Some environment and constraints. Clear scope, constraints, target state, and non-goals.
    Engineering depth Commands only. Some reasoning and steps. Architecture/process judgment with trade-offs.
    Reusability One-off record. Can become a checklist or FAQ. Can become a standard, runbook, or model.
    Validation and rollback Missing. Basic validation only. Clear validation, failure signals, rollback, and maintenance.
    Security posture Risk unknown. Can be sanitized with effort. Cleanly anonymized with explicit safety guidance.
    Retrieval quality Hard to chunk or cite. Some useful headings. Stable headings, focused sections, direct Q&A value.
    Freshness Obsolete or version-locked. Needs version notes. Current or written as version-aware guidance.

    Rating Decision¶

    Rating Rule Action
    A Score >= 32, security risk <= 2, and reusable value >= 4. Rewrite as a standalone public article.
    B Score 22-31, security risk <= 3, and at least one strong reusable section. Merge into an existing standard, runbook, FAQ, or checklist.
    C Score 12-21 or context is incomplete but some experience is useful. Keep as internal review note or extract a small sanitized snippet.
    D Score < 12, security risk >= 4, or mostly private/business/runtime material. Exclude, quarantine, archive, or delete according to cleanup policy.

    Security risk is judged separately:

    Security Risk Meaning
    0 Public-safe generic material.
    1 Minor identifiers that are easy to remove.
    2 Some environment details requiring careful rewrite.
    3 Heavy anonymization required; publish only if value is high.
    4 Contains sensitive or private operational content; do not publish raw.
    5 Contains credentials, keys, secrets, production data, private customer data, or personal data; quarantine or destroy.

    Public Rewrite Rules¶

    Replace Specific Identifiers¶

    Use placeholders:

    <cluster-name>
<namespace>
<service-name>
<region>
<account-id>
<vpc-id>
<subnet-id>
<example.com>
<private-registry>
<repository-url>
<customer-name>
<team-name>

    Rewrite Raw Commands¶

    Commands must be safe examples, not production history.

    Bad:

    kubectl config use-context real-prod-cluster
kubectl delete pod real-customer-payment-7d9d

    Good:

    kubectl config use-context <cluster-name>
kubectl -n <namespace> rollout status deployment/<deployment-name>

    Rewrite Architecture Cases¶

    Do not publish original architecture diagrams, customer routes, service names, or network topology. Convert them into a generalized pattern:

    • Source state.
    • Target state.
    • Migration constraints.
    • Reusable design decision.
    • Validation checklist.

    Retrieval-Friendly Writing Rules¶

    For Q&A assistant readiness:

    • Use descriptive headings.
    • Keep each section focused on one question or procedure.
    • Prefer short paragraphs and explicit lists.
    • Include definitions for ambiguous operational terms.
    • Avoid hidden assumptions.
    • Add validation and rollback sections even when short.
    • Link related documents using stable relative links.

    Each article should answer likely questions such as:

    • When should I use this?
    • What are the prerequisites?
    • What is the safest implementation order?
    • How do I validate success?
    • How do I roll back?
    • What are the security risks?
    • What should I monitor after the change?

    Minimum Publication Checklist¶

    Before a document is published:

    • It has a clear title and stable path.
    • It is linked from mkdocs.yml.
    • It contains no raw secrets, production data, or private identifiers.
    • It has public-safe placeholders.
    • It includes validation and rollback where relevant.
    • It includes security notes where relevant.
    • It builds successfully with MkDocs.
    • It is suitable for both human reading and retrieval-based Q&A.
