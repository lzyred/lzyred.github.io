    Terraform IaC 标准¶

    目的¶

    本标准定义如何使用 Terraform，使基础设施变更可复现、可评审、可恢复，并从设计上消除 Terraform 两个最危险的失败模式（state 文件损坏/丢失，以及一次本应替换却销毁了不该销毁之物的 apply）。

    本标准的核心判断：

    Terraform 的 state 文件就是你基础设施的可信源，而 terraform plan 是你唯一能提前看到 apply 真正会做什么的地方。保护好 state，并且绝不应用你没读过的 plan。

    适用场景¶

    在以下情况使用本标准：

    • 用 Terraform/OpenTofu 供给云上或自建基础设施。
    • 在多人都能执行 apply 的团队中工作。
    • 需要可审计、含回滚意识的基础设施变更流程。

    非目标¶

    本标准不涵盖：

    • 某个特定 provider 的模块设计模式细节。
    • 选择云厂商或设计目标架构（那是架构决策）。
    • CI 系统细节；本标准定义任何流水线都必须强制的 Terraform 纪律。

    原则¶

    1. state 远端存储、加密、启用锁。
    2. 每次 apply 都锁定 state，使两名操作者无法竞争。
    3. 密钥不放入 .tf，并尽量不进入 state。
    4. 用模块与固定版本实现可重复的基础设施。
    5. 应用前读每一个 plan；共享环境的 apply 设评审门。

    推荐标准¶

    远端、加锁、加密的 state¶

    笔记本上的本地 state 是单点故障，且无法安全共享。使用带锁和加密的远端后端。

    terraform {
  required_version = ">= 1.6"
  required_providers {
    example = {
      source  = "example/example"
      version = "~> 5.0"
    }
  }
  backend "s3" {
    bucket         = "example-tf-state"     # 占位；切勿公开真实名称
    key            = "env/prod/terraform.tfstate"
    region         = "example-region"
    dynamodb_table = "example-tf-locks"     # state 锁
    encrypt        = true
  }
}
    • 在 state 存储上开启版本化，以便恢复到先前 state。
    • 每个环境使用独立的 state key；切勿让 prod 与非 prod 共用一份 state。
    • 在团队中锁表/锁机制不是可选项——没有它，并发 apply 会损坏 state。

    固定版本¶

    同时固定 required_version 与 provider 版本。未固定的 provider 意味着一个无关变更可能悄悄改变 plan。提交 .terraform.lock.hcl 锁文件。

    plan、评审、apply¶

    terraform fmt -check
terraform validate
terraform plan -out=plan.tfout      # 保存确切的 plan
# 评审 plan.tfout 输出，然后应用正是这个 plan：
terraform apply plan.tfout

    保存 plan 并应用那个文件，可保证 apply 只做你评审过的事——而非几分钟后世界变成的样子。plan 输出中最重要的符号：

    • + 创建、~ 就地更新——通常安全。
    • -/+ 销毁并重建——这里正是承载数据的资源会丢失之处。对数据库、卷或任何有状态资源，看到 -/+ 必须先停下确认再应用。

    密钥不入 .tf 与 state¶

    • 通过环境变量或密钥管理系统/vault provider 传递密钥，而非 .tf 中的字面量。
    • 记住：即便你安全注入，许多资源仍会把敏感值写入 state——因此 state 文件必须像密钥一样加密并做访问控制。
    • 将敏感输出标记为 sensitive = true；切勿提交含密钥的 *.tfstate 或 *.tfvars。

    校验¶

    terraform validate          # 配置内部一致
terraform plan              # apply 后无意外变更（plan 干净 = 无漂移）
terraform state list        # 预期资源被纳管

    确认：

    • apply 后的 plan 为干净（空）——表示真实 state 与期望一致。
    • 没有对有状态资源发生非预期的 -/+ 销毁重建。
    • state 远端存储，且锁已获取并释放。

    已知失败模式¶

    失败模式 症状 可能原因 正确处置
    state 丢失或损坏 apply 报错；资源在 Terraform 中"消失" 本地 state、无版本化或手工编辑 从 state 存储的版本化恢复；切勿手改 state。
    并发 apply state 冲突/覆盖 无锁后端 启用锁表；串行化 apply。
    非预期销毁 资源被删除/重建 不可变属性变更触发强制替换 读 plan 中的 -/+；用 create_before_destroy 或 prevent_destroy。
    漂移 plan 显示无人做过的变更 控制台带外改动 协调：import 或回退手改；强制变更走 Terraform。
    密钥进入 state/仓库 凭据泄漏 .tf 中字面量密钥或提交了 state 轮换密钥；从历史清除；加密 state。
    provider 升级意外 代码未变 plan 却变 provider 版本未固定 固定版本；提交锁文件。

    销毁与回滚¶

    Terraform 没有通用的"撤销"——回滚是向前应用一个先前的已知良好配置，外加防止误销毁。

    1. 用 lifecycle { prevent_destroy = true } 保护有状态资源，使一个失误的 plan 无法删除它们。
    2. 回滚变更：检出先前配置版本并 plan/apply——要评审 plan，因为反转一个变更本身也可能触发替换。
    3. 若 state 受损，先从后端版本化恢复先前 state 版本，再应用任何变更。
    4. 用 terraform plan -destroy 预览拆除；切勿在没有明确批准和保存 plan 的情况下对共享环境运行 terraform destroy。

    安全注记¶

    • 切勿公开真实账号 ID、资源 ID、后端 bucket 名、区域、state 文件或 provider 凭据。
    • 把 state 文件当作含密钥之物：静态加密、限制读取、纳入审计。
    • Terraform 运行者使用最小权限凭据；不要随意用宽泛的管理员密钥跑生产 apply。
    • 审查模块来源；固定到可信版本/提交，而非浮动引用。

    评审周期¶

    • 负责人：平台/基础设施团队。
    • 定期重新固定并复核 provider/模块版本。
    • 每季度确认 state 存储的版本化、加密与锁仍然就位。
    • 架构变化时，重新评审有状态资源的 prevent_destroy 覆盖。
