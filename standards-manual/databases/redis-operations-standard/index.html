    Redis 运维标准¶

    目的¶

    本标准定义如何在生产环境安全地运行 Redis，以及如何在不丢数据、不丢可用性的前提下变更一个运行中的 Redis。

    本标准的核心判断：

    Redis 作为"可丢弃的缓存"和作为"状态存储"时，行为完全不同。在选择持久化、淘汰策略和故障切换之前，先确定每个实例属于哪一种——大多数 Redis 事故都源于把一种当成了另一种。

    适用场景¶

    在以下情况使用本标准：

    • 自管 Redis 用作缓存、会话存储、限流、队列或分布式锁。
    • 需要为某个负载在 RDB、AOF 或两者之间做出选择。
    • 计划 Redis 版本升级、迁移主机或调整拓扑。
    • 需要一套在事故复盘时站得住脚的内存压力与故障切换策略。

    非目标¶

    本标准不涵盖：

    • 应用数据模型设计（键命名、TTL 策略、热点 key 规避）。
    • 由云厂商负责持久化与故障切换的托管 Redis 服务，应改用厂商的持久化与切换契约。
    • 把 Redis 当作无法重建数据的主数据库——这是一个需要单独评审的架构决策。
    • 核心服务行为之外的模块运维（search、time series、JSON 等）。

    先确定角色¶

    角色 丢失容忍度 持久化 淘汰策略 故障切换
    纯缓存 数据可重建 RDB 或不持久化 allkeys-lru / allkeys-lfu 可选；客户端容忍冷缓存
    状态存储（会话、队列、锁） 丢失会导致用户可见故障 AOF（常配合 RDB） noeviction 必须，且需副本 + 自动切换

    逐实例写下角色，后续所有配置都由它推导。一个被悄悄改成 noeviction 的缓存会在内存压力下开始返回写错误；一个被留在 allkeys-lru 的状态存储会悄悄淘汰你真正需要的数据。

    推荐标准¶

    主机调优¶

    Redis 对少数几个主机设置很敏感，要显式设置并纳入主机基线。

    # 透明大页（THP）会在 fork/save 时引发延迟抖动。
echo never | sudo tee /sys/kernel/mm/transparent_hugepage/enabled

# 让内存压力下的后台保存 fork 能够成功。
echo 'vm.overcommit_memory = 1' | sudo tee /etc/sysctl.d/99-redis.conf
sudo sysctl --system

    将文件描述符上限提高到覆盖 maxclients 并留出余量：

    redis soft nofile 65535
redis hard nofile 65535

    实例基线¶

    从一份最小、显式的配置开始。仅绑定私有网卡，保持 protected-mode 开启，并设置与角色匹配的内存上限和淘汰策略。

    bind 127.0.0.1 <private-interface>
protected-mode yes
port 6379
dir /var/lib/redis

# 角色：状态存储示例
appendonly yes
appendfsync everysec
maxmemory 4gb
maxmemory-policy noeviction

    requirepass 及任何 ACL 用户均通过密钥管理系统注入。切勿提交真实密码，也切勿公开真实绑定地址或主机名。

    从角色出发选择持久化，而非沿用默认¶

    模式 恢复行为 成本 适用
    仅 RDB 时间点快照；丢失上次保存后的写入 廉价、重启快 缓存，或可接受分钟级丢失的状态
    仅 AOF 重放命令日志；everysec 下最多丢约 1 秒 文件更大、需定期重写 需要低丢失的状态存储
    AOF + RDB RDB 快速重启，AOF 保证持久 磁盘/IO 最高 同时需要快重启和低丢失的状态存储

    appendfsync always 持久但慢；仅在确实无法容忍 everysec 窗口时使用，并先做压测。

    把故障切换当作显式设计，而非事后补救¶

    单个 Redis 实例对依赖它的一切都是单点故障。对状态存储：

    • 在独立主机/可用区运行至少一个副本。
    • 使用 Redis Sentinel 或 Redis Cluster 实现自动切换；不要在事故中依赖手动提升。
    • 确认客户端库会在切换时刷新拓扑。许多故障出在客户端：服务端切换正常，客户端却仍在连旧主。
    • 在依赖它之前，先在预发环境带负载测试切换。

    迁移流程¶

    适用于迁移主机、版本升级或拓扑调整。把它当作有回滚负责人在场的变更。

    1. 记录基线：角色、INFO persistence、INFO replication、键数量、内存以及应用错误率。
    2. 确认源与目标 Redis 版本兼容（AOF/RDB 格式与复制兼容性）。
    3. 尽量让目标从源进行实时复制（REPLICAOF）追平，而不是拷贝陈旧的 dump。
    4. 若改用文件拷贝：停写或进入维护窗口，生成 AOF/RDB，通过安全通道拷贝，并校验摘要：
    sha256sum appendonly.aof
    1. 对 Redis Cluster，逐个 shard/副本组迁移，每步后确认槽位归属。
    2. 将客户端切到目标。
    3. 在下线源之前完成校验（见下）。
    4. 在约定的观察窗口通过前，保持源实例完好。

    校验¶

    redis-cli -h <redis-host> -p 6379 INFO persistence
redis-cli -h <redis-host> -p 6379 INFO replication
redis-cli -h <redis-host> -p 6379 CLUSTER INFO     # 仅 cluster

    确认：

    • INFO persistence 显示预期的 aof_enabled，上次保存/重写成功，且无 rdb_last_bgsave_status:err。
    • INFO replication 显示副本已连接、master_link_status:up，复制偏移延迟有界。
    • 对 cluster，cluster_state:ok 且已分配预期的槽位数。
    • 键数量与内存与基线一致；应用错误率无变化。

    已知失败模式¶

    失败模式 症状 可能原因 正确处置
    负载下写报错 OOM command not allowed maxmemory 达上限且为 noeviction 扩容、分片；仅当角色允许淘汰时才改策略。
    静默丢数据 键意外消失 对状态存储启用了淘汰 状态存储设 noeviction；按工作集规划内存。
    保存时延迟抖动 周期性卡顿 THP 开启，或内存压力下 fork 关闭 THP；设 overcommit_memory=1；留出保存余量。
    切换后读到陈旧数据 客户端仍连旧主 客户端未刷新拓扑 使用感知 Sentinel/Cluster 的客户端并验证切换行为。
    重启丢失近期写入 重启后数据偏旧 状态存储仅用 RDB 状态存储启用 AOF。
    AOF 无法加载 Redis 拒绝启动 AOF 损坏/截断 在副本上用 redis-check-aof --fix 修复；必要时从备份恢复。

    回滚¶

    1. 切换当天不要删除源实例及其数据，保留到校验通过。
    2. 回滚迁移：将客户端指回源，确认 INFO replication 与键数量。
    3. 回滚配置变更：恢复旧配置文件，在窗口内通过服务管理器重启。
    4. 任何应急访问后轮换 requirepass/ACL 凭据。

    安全注记¶

    • 切勿将 Redis 暴露到公网。通过安全组、防火墙或网络策略限制到已知客户端。
    • 强制认证（requirepass 或 ACL 用户）；凭据存于密钥管理系统。
    • 当负载要求传输保密时启用 TLS。
    • 通过 ACL 为应用侧用户禁用或重命名危险命令（FLUSHALL、CONFIG、KEYS）。
    • 迁移、迁主机或应急访问后轮换凭据。

    评审周期¶

    • 负责人：依赖该实例的服务所属团队。
    • 每次重大变更时重新确认各实例的角色记录。
    • 工作集增长时复核淘汰策略与 maxmemory。
    • 每次 Redis 小版本升级、每次客户端库升级后重测切换。
