    AWS EKS 生产级高可用集群设计与交付标准¶

    English version

    结论先行¶

    在 AWS 上搭建一个生产级 EKS 高可用集群，重点不是把节点数量调大，也不是把 Kubernetes 控制面创建出来。真正的难点是把控制面、节点、网络、身份权限、入口流量、存储、发布、观测、升级和故障恢复放进同一个可验证的交付模型里。

    一个可信的 EKS 高可用方案至少要做到：

    • 集群跨多个 Availability Zone 运行。
    • 工作负载可以在单个 AZ 故障时继续调度。
    • 节点容量可以自动恢复和扩缩。
    • Pod 权限不依赖节点级长期凭证。
    • Ingress、DNS、证书、日志、指标、告警都有明确责任边界。
    • 每个关键变更都有验证方法和回滚路径。

    如果一个方案只写 eksctl create cluster，它只能证明集群能创建，不能证明它能承载生产。

    适用场景¶

    本文适用于以下场景：

    • 新建面向生产环境的 Amazon EKS 集群。
    • 将已有虚拟机或自建 Kubernetes 迁移到 EKS。
    • 为现有 EKS 集群补齐高可用、权限、安全、监控和发布标准。
    • 为团队制定可复用的 EKS 交付检查清单。

    本文不适用于：

    • 单人实验环境。
    • 短期 PoC。
    • 对 Kubernetes 运行机制完全不了解的入门教程。
    • 要求跨 Region 零停机容灾的金融级多活系统。跨 Region 是另一个设计问题，不能用普通多 AZ 集群替代。

    基本假设¶

    本文采用以下假设：

    • 集群运行在一个 AWS Region 内，至少使用 3 个 AZ。
    • 工作负载以无状态服务为主，有少量状态服务通过托管数据库、对象存储或持久卷承载。
    • 团队使用 GitOps 或 CI/CD 进行变更交付。
    • 集群需要公开或半公开入口流量。
    • 集群需要满足最小权限、可审计和可回滚要求。

    如果你的业务只有几个低风险服务，可以先降低复杂度：两类节点组、一个入口控制器、基础指标和日志即可。不要为了“看起来生产级”一开始就堆满所有组件。

    核心原则：AWS 托管 EKS 控制面，但业务高可用仍然取决于你的 VPC、子网、节点组、调度策略、Pod 副本、PDB、存储、入口和发布方式。

    关键设计决策¶

    1. 多 AZ 是底线，不是亮点¶

    生产集群至少使用 3 个 AZ。只使用 2 个 AZ 时，很多故障仍然能扛住，但容量恢复、调度弹性和维护窗口会更紧。只使用 1 个 AZ 的 EKS 不能称为高可用生产集群。

    需要检查：

    • VPC 是否覆盖至少 3 个 AZ。
    • 每个 AZ 是否有私有子网。
    • 节点组是否跨 AZ。
    • 应用副本是否通过 topology spread 或反亲和规则分散。
    • 存储类是否理解 AZ 绑定特性。

    常见失败模式：集群使用了 3 个 AZ，但节点组只在 1 个 AZ 中有容量。页面上看起来是多 AZ，实际调度时仍然单点。

    2. 节点容量要分层¶

    不要把所有 Pod 都跑在同一类节点上。最低限度应分成：

    • system pool：CoreDNS、Ingress Controller、日志采集、监控组件。
    • application pool：业务服务。
    • batch or spot pool：可中断任务、异步任务、低优先级任务。

    分层的目的不是复杂化，而是防止业务突增把系统组件挤掉。

    推荐原则：

    • 系统组件使用稳定容量。
    • 业务服务根据资源画像选择实例族。
    • 可中断任务可以使用 Spot，但必须能重试。
    • 关键服务配置 PDB，避免维护时被一次性驱逐。

    3. Pod 权限不要继承节点权限¶

    生产集群不应让 Pod 依赖节点 IAM Role 获得 AWS 权限。应使用 EKS Pod Identity 或 IRSA，为每类工作负载绑定最小权限 IAM Role。

    判断标准：

    • 一个 Pod 只拿到它需要的 AWS API 权限。
    • Secret 管理、对象存储、消息队列、镜像仓库访问分开授权。
    • 权限变更可审计。
    • 删除应用时可以同步删除对应权限。

    常见失败模式：为了省事，把 S3、ECR、CloudWatch、Secrets Manager 权限都挂到节点角色上。短期能跑，长期会让任意被攻破的 Pod 拥有过大横向移动能力。

    4. Ingress 不是流量入口的全部¶

    一个生产入口链路至少要明确：

    • DNS 由谁管理。
    • TLS 证书在哪里签发和续期。
    • 负载均衡器是公网还是内网。
    • WAF 是否需要。
    • 健康检查路径是否稳定。
    • 灰度或回滚如何切流。

    如果只创建 Ingress YAML，没有回答这些问题，入口层还没有设计完成。

    5. 状态数据优先用托管服务¶

    EKS 可以运行状态服务，但生产系统不要默认把数据库、消息队列、搜索引擎都塞进集群。优先考虑 RDS、ElastiCache、MSK、OpenSearch、S3 等托管服务。

    在集群内运行状态服务之前，需要回答：

    • 数据备份在哪里。
    • 跨 AZ 恢复如何做。
    • 升级失败如何回滚。
    • 节点丢失时数据如何恢复。
    • 团队是否真的有运维该组件的能力。

    交付顺序¶

    阶段 1：网络和账号边界¶

    交付内容：

    • VPC、子网、路由表、NAT Gateway 或替代方案。
    • 集群 IAM Role。
    • 节点 IAM Role。
    • 管理员和 CI/CD 访问边界。
    • 标签规范。

    验证：

    aws sts get-caller-identity
aws ec2 describe-subnets --filters "Name=vpc-id,Values=<vpc-id>"
aws eks describe-cluster --name <cluster-name>

    风险点：NAT Gateway 成本容易被低估；私有集群访问路径容易被忘记；过宽的管理员权限会在后续阶段变成默认做法。

    阶段 2：创建 EKS 控制面¶

    交付内容：

    • EKS 集群。
    • Kubernetes 版本选择。
    • API Server endpoint 访问策略。
    • 日志开关。
    • 加密配置。

    验证：

    aws eks update-kubeconfig --name <cluster-name> --region <region>
kubectl cluster-info
kubectl get nodes
kubectl get ns

    注意：刚创建完控制面时，kubectl get nodes 可能为空。控制面健康不等于节点已经加入。

    阶段 3：节点容量和系统组件¶

    交付内容：

    • Managed Node Groups 或 Karpenter。
    • CoreDNS、kube-proxy、VPC CNI 版本确认。
    • 系统组件和业务组件的调度隔离。
    • Cluster Autoscaler 或 Karpenter 扩缩容策略。

    验证：

    kubectl get nodes -o wide
kubectl -n kube-system get pods -o wide
kubectl describe node <node-name>
kubectl top nodes

    失败模式：节点 Ready，但 CNI IP 不足，业务 Pod 卡在 Pending。此时问题不在 Deployment，而在子网地址容量或 CNI 配置。

    阶段 4：身份、权限和密钥¶

    交付内容：

    • Pod Identity 或 IRSA。
    • 最小权限 IAM Policy。
    • Secret 管理策略。
    • CI/CD 部署权限。

    验证：

    kubectl -n <namespace> describe serviceaccount <service-account>
kubectl -n <namespace> exec deploy/<deployment-name> -- aws sts get-caller-identity

    判断标准：业务 Pod 看到的身份应该是它自己的工作负载角色，而不是节点角色。

    阶段 5：入口、证书和发布路径¶

    交付内容：

    • AWS Load Balancer Controller。
    • IngressClass。
    • DNS 记录。
    • TLS 证书。
    • 健康检查。
    • 灰度或蓝绿发布策略。

    验证：

    kubectl -n kube-system get deploy aws-load-balancer-controller
kubectl -n <namespace> get ingress
kubectl -n <namespace> describe ingress <ingress-name>
curl -I https://<service-domain>

    失败模式：Ingress 创建成功，但 Load Balancer 没有被创建。通常是控制器权限、子网标签或 Ingress 注解错误。

    阶段 6：可观测性和告警¶

    交付内容：

    • 节点指标。
    • Pod 指标。
    • 应用日志。
    • Kubernetes Events。
    • API Server 或审计日志。
    • 关键告警。

    最低告警集合：

    • 节点 NotReady。
    • Pod CrashLoopBackOff。
    • Deployment 副本不足。
    • PVC 容量高水位。
    • Ingress 5xx 升高。
    • HPA 长时间无法扩容。
    • CoreDNS 错误率升高。

    验证：

    kubectl get events -A --sort-by=.lastTimestamp
kubectl top pods -A
kubectl -n <monitoring-namespace> get pods

    回滚策略¶

    EKS 集群回滚不能只靠 kubectl rollout undo。需要分层处理。

    层级 回滚方式
    应用发布 回滚镜像、回滚 Helm release、回滚 GitOps commit
    Ingress 切回旧规则、旧证书或旧负载均衡器
    节点 回滚节点组 Launch Template 或创建旧版本节点组
    权限 回滚 IAM Policy 或 ServiceAccount 绑定
    集群插件 回滚到兼容版本或临时扩容旧节点
    Kubernetes 版本 通常不能原地降级，应通过新集群迁移恢复

    最容易被忽略的是 Kubernetes 版本升级。升级前必须确认插件、节点 AMI、控制器和关键 CRD 的兼容性。不要把生产集群升级当成普通包升级。

    生产验收清单¶

    上线前至少检查：

    • 集群跨 3 个 AZ。
    • 私有子网容量足够。
    • 节点组跨 AZ 且有自动恢复能力。
    • 系统组件和业务组件有容量隔离。
    • CoreDNS 有足够副本并分散调度。
    • 关键应用有多副本、PDB、资源请求和限制。
    • 入口链路有 DNS、TLS、健康检查和回滚方案。
    • Pod 使用最小权限身份。
    • 密钥没有写入镜像、ConfigMap 或 Git。
    • 日志、指标、事件和告警可用。
    • CI/CD 权限可审计。
    • 备份和恢复流程被演练过。
    • 升级和故障演练有记录。

    常见错误¶

    错误 1：把 EKS 托管控制面等同于业务高可用¶

    AWS 托管控制面降低了控制面运维成本，但业务 Pod、节点容量、Ingress、存储、发布和监控仍然由你负责。业务高可用不是购买出来的，是设计和验证出来的。

    错误 2：只看节点数量，不看调度分布¶

    6 个节点如果都在一个 AZ，遇到 AZ 故障仍然会整体受影响。副本数和节点数必须结合拓扑分布来看。

    错误 3：先迁移状态服务¶

    迁移时应先选择无状态服务验证镜像、发布、Ingress、日志、指标和回滚链路。状态服务最后处理，因为它的问题不是能不能调度，而是数据一致性、恢复时间和恢复点。

    错误 4：把权限问题留到最后¶

    权限如果最后处理，通常会变成临时授权和长期例外。生产集群应从第一天就设计 Pod 身份和部署权限边界。

    问答助手知识块¶

    问：EKS 高可用是不是只要使用 Managed Node Group 就够了？

    不是。Managed Node Group 解决节点生命周期的一部分问题，但高可用还依赖多 AZ 子网、调度分散、Pod 副本、PDB、Ingress、存储、权限、监控和回滚。

    问：生产 EKS 集群建议几个 AZ？

    通常建议至少 3 个 AZ。2 个 AZ 可以用于较低风险场景，但容量恢复和维护弹性较弱。1 个 AZ 不应称为生产高可用。

    问：Pod 访问 AWS 服务应该怎么授权？

    使用 EKS Pod Identity 或 IRSA，为工作负载绑定最小权限 IAM Role。不要让业务 Pod 继承节点角色上的宽权限。

    问：状态服务是否应该部署在 EKS 中？

    默认优先使用 AWS 托管服务。只有当团队具备备份、恢复、升级、监控和故障处理能力时，才考虑在集群内运行状态服务。
