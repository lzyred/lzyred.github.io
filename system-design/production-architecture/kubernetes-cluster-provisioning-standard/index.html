    Kubernetes 集群供给标准¶

    目的¶

    本标准定义如何供给一个可以承载生产工作负载的 Kubernetes 集群。

    最重要的判断是：

    集群供给不是创建控制面的那条命令，而是按顺序构建一张平台依赖图。

    kubectl get nodes 返回 Ready，只能说明节点加入了集群，不能说明集群已经可生产使用。只有当工作负载身份、网络边界、节点池、存储、Ingress、可观测性、发布控制、备份预期和所有权规则都被明确并验证后，集群才算真正准备好。

    本标准适用于 EKS、CCE、ACK、GKE、AKS 等托管集群，也适用于 kubeadm 自建集群。实现细节不同，但就绪契约相同。

    适用场景¶

    在以下情况使用本标准：

    • 建设新的生产或预发 Kubernetes 集群。
    • 在重大网络、身份或运行时变更后重建集群。
    • 为多个团队创建可复用的集群画像。
    • 评审现有集群是否成熟到可以承载业务工作负载。
    • 把实验性集群转换为平台团队正式拥有的服务。

    本标准不是本地开发集群的快速清单。开发集群可以跳过很多生产控制，但不能把开发集群误认为生产参考架构。

    非目标¶

    本标准不覆盖：

    • 某个云厂商控制台的逐步点击流程。
    • 完整 Kubernetes 安全加固。
    • 集群内应用架构设计。
    • 服务网格设计。
    • 超出供给就绪边界之外的灾难恢复设计。

    供给判断¶

    生产集群在部署第一个业务工作负载前，必须回答七个问题：

    问题 必须明确的决策
    谁拥有平台？ 平台负责人、升级路径、变更窗口和生命周期策略。
    流量能流向哪里？ VPC 或 VNet、CIDR、Service CIDR、Pod CIDR、DNS、出站、私有端点和入口暴露方式。
    谁可以执行操作？ 人员访问、工作负载身份、服务账号、应急访问和审计日志。
    工作负载运行在哪里？ 节点池边界、污点、标签、自动伸缩、运行时、镜像和维护策略。
    状态数据如何保存？ StorageClass、CSI 驱动、扩容、备份、快照、回收策略和拓扑约束。
    用户如何进入？ IngressClass、负载均衡模型、TLS 归属、DNS 归属和证书续期。
    故障如何被看见和回退？ 指标、日志、事件、告警、发布控制、回滚流程和恢复边界。

    如果任何答案缺失，集群可能已经存在，但还不是生产就绪。

    决策输入¶

    供给前先固化这些输入：

    领域 必须输入
    Kubernetes 版本 目标版本、版本偏移支持、插件兼容性、升级窗口。
    网络 VPC 或 VNet、CIDR 规划、Pod/Service 网段、DNS、出站 NAT、私有端点要求。
    身份 集群管理员模型、用户组映射、工作负载身份机制、应急账号。
    节点池 CPU 与内存画像、按量或竞价策略、GPU 或特殊硬件、污点、标签。
    存储 默认 StorageClass、有状态负载需求、备份与恢复预期、扩容策略。
    Ingress 公网或内网暴露、控制器、TLS 签发者、DNS 归属、WAF 或网关边界。
    可观测性 指标、日志、事件、链路、告警路由、仪表盘归属。
    交付 GitOps 或流水线归属、命名空间 onboarding、回滚路径、发布策略。
    治理 命名、标签、成本归集、资源配额、镜像策略、平台评审周期。

    供给记录不需要很长，但必须明确。依赖默认值的集群，会在第一次事故中变成运营惊喜。

    参考供给顺序¶

    1. 定义平台边界¶

    先决定这个集群允许拥有什么。不要因为安装时顺手配置了 DNS、Secret、证书、Ingress、CI/CD、数据库和监控，就让集群意外成为这些能力的所有者。

    至少明确：

    • 生产、预发、开发或共享平台范围。
    • 允许承载的工作负载类型。
    • 网络暴露模型。
    • 团队接入流程。
    • 平台插件所有权。
    • 事故升级路径。

    2. 冻结版本与插件矩阵¶

    选择 Kubernetes 版本时必须同时选择要运行的插件版本。不要先选集群版本，再事后发现插件不兼容。

    矩阵必须包含：

    • CNI 插件及版本。
    • CoreDNS。
    • kube-proxy 或替代组件。
    • CSI 驱动。
    • Ingress 控制器。
    • metrics-server。
    • Cluster Autoscaler 或 Karpenter 类组件。
    • 日志与指标 agent。
    • 如使用策略控制器，也要纳入矩阵。

    最低可接受产物是一张表：哪些版本已批准、哪些版本被阻断、需要什么升级顺序。

    3. 先验证网络，再创建节点¶

    网络比集群更难改。创建节点池前先验证网络。

    必查项：

    • Pod 与 Service CIDR 不与 VPC/VNet、对等网络、VPN 网段或本地数据中心网段冲突。
    • DNS 可以解析云 API、内部服务和私有端点。
    • 节点可以访问必要的镜像仓库和控制面端点。
    • 出站策略是刻意设计的，不是偶然全开放。
    • 如有需要，入口子网与内网负载均衡子网分离。

    如果网络规划不清楚，就停止。重建一个年轻集群，比长期忍受网段冲突便宜得多。

    4. 应用接入前配置身份与访问¶

    人员访问和工作负载访问必须在应用团队进入前设好。

    必须决策：

    • 哪些用户组拥有 cluster-admin。
    • 哪些用户组拥有命名空间级管理员权限。
    • 服务账号如何获得云权限。
    • 应急访问如何授予、记录和回收。
    • 哪些审计日志需要保留。

    避免把永久 cluster-admin 绑定到个人用户。使用用户组、限时应急访问和有记录的定期评审。

    5. 把节点池当作工作负载契约¶

    节点池不只是容量，而是工作负载放置契约。

    至少定义：

    • 通用无状态节点池。
    • 如有需要，系统或平台插件节点池。
    • 如有要求，有状态或存储敏感节点池。
    • 竞价或可抢占节点池只能给可容忍中断的工作负载。
    • GPU 或特殊硬件节点池必须通过污点与标签隔离。

    每个节点池都应有标签、必要的污点、自动伸缩边界、镜像生命周期规则和补丁或替换策略。

    6. 在工作负载接入前安装存储¶

    应用部署前先安装 CSI 驱动和 StorageClass。没有经过测试的默认存储路径，集群就不适合承载有状态工作负载。

    最小验证：

    • PVC 可以动态创建。
    • Pod 可以挂载并写入文件系统。
    • 扩容行为已知。
    • 回收策略是刻意选择的。
    • 备份或快照预期已记录。
    • zone 或拓扑约束对调度器可见。

    存储细节见 Kubernetes 存储与 Ingress 标准。

    7. 发布服务前安装 Ingress 与 TLS¶

    Ingress 必须是平台托管能力，不能让每个团队自行 improvisation。

    验证：

    • IngressClass 归属。
    • 公网或内网暴露方式。
    • TLS 证书签发者与续期责任。
    • DNS 归属。
    • 请求超时、请求体大小和 WebSocket 预期。
    • 访问日志和指标可用。
    • 如需要，明确 WAF 或网关边界。

    8. 业务流量进入前安装可观测性¶

    可观测性是上线依赖，不是事故后补装项。

    最小信号：

    • 节点健康和资源饱和。
    • Pod 重启、调度失败、拉镜像失败。
    • 如平台暴露，API Server 健康。
    • Ingress 请求量、延迟和错误率。
    • 存在有状态工作负载时的 PVC 使用率与扩容风险。
    • 用于排障的事件采集。
    • 告警路由到责任团队。

    9. 加入交付与回滚控制¶

    第一个部署机制先要能回滚，再追求速度。

    必须控制：

    • 命名空间 onboarding 模式。
    • 发布负责人。
    • 镜像来源预期。
    • 滚动发布策略。
    • 回滚命令或 GitOps revert 路径。
    • 发布健康检查。
    • 变更记录。

    操作回滚流程见 部署回滚 Runbook。

    托管集群画像¶

    对托管 Kubernetes 服务，优先使用云厂商托管控制面，避免自管那些云厂商能更稳定运营的组件。

    必须决策：

    • 控制面端点是私有还是公开。
    • 控制面访问白名单。
    • 托管节点组还是自管节点组。
    • IAM 或工作负载身份集成。
    • 云厂商支持的 CSI 和负载均衡控制器。
    • 云厂商版本支持窗口。
    • 平台团队与云厂商之间的升级责任边界。

    不要把“托管”理解成“生产就绪”。云厂商只运营平台的一部分；你的团队仍然拥有工作负载访问、插件、交付、可观测性、数据保护和事故响应。

    kubeadm 自建画像¶

    对 kubeadm 集群，团队必须显式拥有控制面。

    额外要求：

    • 高可用 API Server 入口。
    • etcd 备份与恢复测试。
    • 证书轮换流程。
    • 控制面节点补丁流程。
    • CNI 安装与升级策略。
    • kubelet 与容器运行时生命周期。
    • 灾难恢复 Runbook。

    如果这些能力无法配人并测试，除非有硬约束阻止，否则应使用托管集群。

    应用接入就绪门禁¶

    接收业务工作负载前，执行以下门禁：

    门禁 通过条件
    节点 目标节点池全部 Ready，标签、污点符合设计，自动伸缩边界正确。
    系统 Pod CoreDNS、CNI、CSI、Ingress、metrics-server、可观测性 agent 健康。
    存储 测试 PVC 可以创建、挂载、写入、按支持情况扩容，并符合回收策略。
    Ingress 测试服务能通过预期公网或内网路径访问，TLS 有效。
    身份 工作负载能通过预期云身份访问资源，不依赖静态密钥。
    可观测性 测试故障能产生可搜索的日志、指标、事件和告警。
    交付 测试部署能通过批准机制前进和回滚。
    成本与所有权 命名空间、标签、配额和升级路径可见。

    每个门禁都有负责人和结果后，集群才可以开放 onboarding。

    验证¶

    从干净的运维工作站或 CI runner，以正常平台权限执行验证：

    kubectl get nodes -o wide
kubectl get pods -A
kubectl get storageclass
kubectl get ingressclass
kubectl top nodes
kubectl auth can-i create deployments --namespace <test-namespace>
kubectl auth can-i '*' '*' --as <non-admin-user>

    然后创建并删除一个小型验证工作负载：

    • 带 readiness probe 的无状态 Deployment。
    • 使用 PVC 的 Pod。
    • 内部 Service。
    • 带 TLS 的 Ingress 路由。
    • 一次故意失败的发布，以及随后的回滚。

    把结果记录在集群就绪文档中。截图不如可重复命令和预期输出可靠。

    常见失败模式¶

    失败模式 为什么危险 预防方式
    CIDR 冲突 对等、VPN 和服务路由后续失败。 创建集群前验证全部地址范围。
    插件版本不匹配 CNI、CSI 或 Ingress 在升级后损坏。 冻结并评审版本矩阵。
    未测试 StorageClass 有状态工作负载阻塞或丢数据。 执行 PVC 创建、挂载、写入、扩容和回收测试。
    先做 Ingress 后定 TLS 归属 服务公开了，但无人负责证书续期。 暴露前明确 DNS 与证书所有权。
    应用先于可观测性上线 第一次事故没有有用证据。 把可观测性作为上线依赖。
    个人永久 admin 权限 审计和离职回收变弱。 使用用户组、应急访问和定期评审。
    节点自动伸缩无边界 成本暴涨或调度不稳定。 设置最小、最大、标签、污点、配额和告警。

    回滚与重建边界¶

    有些供给错误应该通过重建集群解决，而不是持续打补丁。

    优先重建的情况：

    • Pod 或 Service CIDR 与其他网络冲突。
    • 集群创建在错误 VPC、区域或账号边界。
    • 身份集成围绕长期静态凭据建立。
    • 插件所有权不清晰，没人能安全升级。
    • 集群已经接入业务但缺少基线可观测性，且迁移成本仍低。

    优先原地修正的情况：

    • 节点池需要扩容、缩容或替换。
    • 插件配置错误，但版本兼容。
    • StorageClass 或 IngressClass 需要在迁移计划后更换默认值。
    • RBAC 需要通过可审计访问评审收紧。

    问题发现得越早，就越应认真考虑重建。

    安全说明¶

    • 不要在公开文档中保存 kubeconfig、token、证书或云密钥。
    • 不要公开私有端点地址、客户域名或内部 CIDR。
    • 使用 <pod-cidr> 或 <service-cidr> 这类符号占位符，不复制生产网段。
    • 优先使用工作负载身份，不把静态云凭据挂进 Pod。
    • 在授予团队访问前启用审计日志。
    • 每次团队接入和事故之后，评审 cluster-admin 绑定。

    评审周期¶

    在以下时点评审本标准：

    • 每次新建集群前。
    • 每次 Kubernetes minor 版本升级前。
    • 每次网络、身份、存储或 Ingress 事故后。
    • 新工作负载类型需要新节点池或存储画像时。
