    Kubernetes 存储与 Ingress 标准¶

    目的¶

    本标准定义在业务工作负载依赖 Kubernetes 存储和 Ingress 之前，如何设计和验证这两项能力。

    核心判断是：

    存储和 Ingress 不是插件，而是平台与每个应用团队之间的生产契约。

    存储决定状态能否保留、扩容、迁移和恢复。Ingress 决定用户如何访问服务、TLS 如何续期、流量如何记录，以及集群的外部边界从哪里开始。如果把它们当作默认安装组件，生产风险会一直不可见，直到第一次故障才爆出来。

    适用场景¶

    在以下情况使用本标准：

    • 供给新的 Kubernetes 集群。
    • 引入或变更 CSI 驱动、StorageClass、IngressClass 或 Ingress 控制器。
    • 接入有状态工作负载。
    • 通过公网或内网 Ingress 发布服务。
    • 在集群成为共享生产平台前进行评审。

    非目标¶

    本标准不覆盖：

    • 完整数据库备份架构。
    • 服务网格流量策略。
    • 云负载均衡成本优化细节。
    • 应用层数据模型设计。
    • 某个 Ingress 控制器的所有功能。

    设计原则¶

    生产平台应暴露少量命名清晰的选择，而不是一堆隐式默认值。

    应用团队应该能回答：

    • 普通持久化数据应使用哪个 StorageClass？
    • 哪个 StorageClass 支持扩容？
    • 如有，哪个存储画像支持 ReadWriteMany？
    • 谁拥有备份和恢复？
    • 哪个 IngressClass 用于公网流量？
    • 哪个 IngressClass 用于内网流量？
    • 谁拥有 DNS 和 TLS 续期？
    • 默认请求限制和超时是多少？

    如果这些答案必须翻安装历史才能找到，平台契约就不够清晰。

    存储决策¶

    显式定义存储画像¶

    至少定义这些存储画像：

    画像 典型用途 访问模式 扩容 备份预期 说明
    standard-rwo 普通应用持久化 ReadWriteOnce 支持 应用或平台策略 只有对大多数应用安全时才设为默认。
    fast-rwo 延迟敏感状态 ReadWriteOnce 支持 必须 仅在性能需求被证明后使用。
    shared-rwx 共享文件、上传、模型产物 ReadWriteMany 平台定义 必须 通常是 NFS 或托管文件存储；必须监控真实文件系统使用率。
    ephemeral 可丢弃缓存或临时状态 视实现而定 不保证 无 不得保存业务关键数据。

    在创建、挂载、扩容、删除和失败行为都被验证前，不要把某个 StorageClass 设为默认。

    把回收策略作为刻意决策¶

    Delete 很方便，但 PVC 被删除时可能直接销毁数据。Retain 能保护数据，但会制造清理责任。

    使用以下规则：

    • 只有可丢弃或易重建数据才使用 Delete。
    • 业务数据默认使用 Retain，除非备份和恢复契约另有说明。
    • 记录谁可以删除保留卷，以及如何确认数据已经不再需要。

    把扩容当作 Runbook，而不是勾选项¶

    allowVolumeExpansion: true 只能说明 Kubernetes 接受扩容请求，不证明底层文件系统、CSI 驱动、配额和应用都能安全承受这个变化。

    批准某个画像支持扩容前，验证：

    • CSI 驱动支持在线或离线扩容。
    • 文件系统能看到新的容量。
    • 应用能容忍扩容路径。
    • 扩容后监控仍跟踪真实文件系统。
    • 回滚预期明确。缩小卷通常不是安全回滚。

    把备份责任与存储供给分开¶

    创建 PV 不等于保护其中的数据。

    每个存储画像都要记录：

    • 是否需要备份。
    • 快照是崩溃一致还是应用一致。
    • 谁拥有恢复测试。
    • 预期恢复耗时。
    • 哪些工作负载不得使用该画像。

    CSI 验证¶

    有状态工作负载接入前，验证 CSI：

    kubectl get csidrivers
kubectl get storageclass
kubectl get pods -n kube-system
kubectl get events -A --sort-by=.lastTimestamp

    然后运行真实 PVC 测试：

    apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: storage-validation
spec:
  accessModes:
    - ReadWriteOnce
  storageClassName: standard-rwo
  resources:
    requests:
      storage: 1Gi
---
apiVersion: v1
kind: Pod
metadata:
  name: storage-validation
spec:
  restartPolicy: Never
  containers:
    - name: writer
      image: busybox:1.36
      command: ["sh", "-c", "date > /data/proof && cat /data/proof"]
      volumeMounts:
        - name: data
          mountPath: /data
  volumes:
    - name: data
      persistentVolumeClaim:
        claimName: storage-validation

    通过条件：

    • PVC 变为 Bound。
    • Pod 调度到预期节点。
    • 文件写入成功。
    • Events 中没有 CSI 错误。
    • 删除测试资源后，表现符合预期回收行为。

    Ingress 决策¶

    把 IngressClass 定义为暴露契约¶

    IngressClass 名称应表达暴露范围和所有权。

    推荐模式：

    IngressClass 暴露范围 典型所有者 必须控制
    public-web 面向互联网 平台或边缘团队 TLS、DNS、访问日志、如需要则 WAF 或网关评审。
    private-web 内部网络 平台团队 内网负载均衡、私有 DNS、访问日志。
    partner-web 受限外部访问 平台与安全团队 白名单、TLS、审计、显式审批。

    避免在没有默认规则的情况下运行多个 Ingress 控制器。应用不应因为漏了一个注解就暴露到错误控制器。

    把 TLS 当作生命周期来拥有¶

    TLS 不是证书首次签发后就结束。

    必须记录：

    • 证书签发者。
    • 续期机制。
    • DNS 验证所有者。
    • 过期告警。
    • Secret 所在命名空间与访问策略。
    • 应急替换流程。

    如果 TLS 续期是手工流程，服务必须有过期告警和明确负责人。

    标准化 Ingress 默认值¶

    设置并记录：

    • 请求超时。
    • 请求体大小。
    • WebSocket 支持。
    • HTTP 到 HTTPS 跳转。
    • 访问日志格式。
    • 真实客户端 IP 处理方式。
    • 健康检查路径。
    • 如使用，限流或 WAF 边界。

    这些值会影响应用行为，不能隐藏在 Ingress 控制器 chart 中而没有平台文档。

    Ingress 验证¶

    暴露真实流量前，先使用小型验证服务：

    apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: ingress-validation
  annotations:
    cert-manager.io/cluster-issuer: letsencrypt-production
spec:
  ingressClassName: public-web
  tls:
    - hosts:
        - example.devops.test
      secretName: ingress-validation-tls
  rules:
    - host: example.devops.test
      http:
        paths:
          - path: /
            pathType: Prefix
            backend:
              service:
                name: ingress-validation
                port:
                  number: 80

    通过条件：

    • Ingress 被分配给预期控制器。
    • 负载均衡或网关地址出现。
    • DNS 解析到预期目标。
    • TLS 证书有效并可续期。
    • 访问日志和请求指标出现。
    • 404、5xx 和慢请求能在预期仪表盘或查询路径里看到。

    验证清单¶

    领域 检查
    StorageClass 默认值是刻意设置、已记录、已验证。
    CSI 控制器和节点插件在各节点池健康。
    PVC 创建、挂载、写入、删除和扩容行为已知。
    回收 数据保留行为匹配工作负载重要性。
    备份 恢复负责人和恢复测试预期已记录。
    IngressClass 公网与内网暴露路径清晰分离。
    TLS 签发者、续期、过期告警和应急替换明确。
    日志与指标 存储和 Ingress 故障可观测。
    所有权 平台责任和应用责任分离。

    常见失败模式¶

    失败模式 症状 预防方式
    缺少默认 StorageClass PVC 一直 Pending。 定义安全默认值，或要求显式设置 storageClassName。
    访问模式错误 Pod 无法挂载卷，或多写工作负载造成数据损坏。 将工作负载映射到批准的画像。
    CSI IAM 或身份失败 PVC 无法创建，Events 显示云权限错误。 接入前验证工作负载身份。
    回收策略不清 数据过早删除或永久保留。 记录回收策略和清理负责人。
    把 NFS 当作隐形容量池 PVC 看似正常，后端文件系统却被写满。 同时监控真实文件系统使用率和 PVC 元数据。
    IngressClass 错误 服务不可达或意外公网暴露。 使用显式 ingressClassName，必要时加准入检查。
    TLS 续期缺失 证书过期后服务失败。 添加过期告警和明确续期负责人。
    控制器默认值隐藏 大文件上传、WebSocket 或慢请求异常失败。 发布默认超时和请求体大小策略。

    回滚与变更边界¶

    存储和 Ingress 变更的回滚属性不同。

    通常可逆：

    • 新增 StorageClass。
    • 新增 IngressClass。
    • 修改验证 Ingress 的注解。
    • 在 canary 后替换 Ingress 控制器配置。

    高风险或无法安全回滚：

    • 在已有工作负载依赖后改变默认 StorageClass。
    • 没有迁移计划就改变回收策略。
    • 缩小存储。
    • 没有备份与恢复证明就迁移有状态工作负载到新存储后端。
    • 没有规划 DNS、TLS 和客户端缓存就替换公网入口。

    如果回滚路径是“从备份恢复”，这就不是普通配置回滚，而是数据恢复操作。

    安全说明¶

    • 不要在示例中公开真实主机名、证书细节、私有 IP 或客户域名。
    • 限制 TLS Secret 访问。
    • 避免把过宽云凭据挂载给 CSI 组件。
    • 存储驱动使用最小权限工作负载身份。
    • DNS 委派前评审公网 Ingress 暴露。
    • 保留访问日志用于事故复盘，但避免记录敏感请求体。

    评审周期¶

    在以下时点评审本标准：

    • 新集群接收工作负载前。
    • 变更 CSI 驱动、StorageClass、Ingress 控制器或 IngressClass 前。
    • 任何数据丢失、存储饱和、证书过期或公网暴露事故后。
    • 接入新的有状态工作负载类别前。
