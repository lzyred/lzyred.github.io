    Containers Delivery Migration

    匿名应用容器化迁移案例¶

    本案例中所有标识均已通用化。未保留任何原始服务名、主机名、域名、镜像、仓库或凭据。

    这篇案例真正讨论什么¶

    这次迁移里最有价值的经验，不是“写一个 Dockerfile，然后把服务搬进去”。真正的经验是：旧主机本身早已是应用的一部分。

    它提供了本地目录、shell profile、软件包版本、证书、cron 任务、启动时序、日志路径，以及少量没有写下来的运维记忆。团队无法让部署变得可重复，直到这些主机行为被拆成一个显式的运行时契约。

    这改变了迁移顺序。镜像构建不是第一个严肃里程碑。第一个严肃里程碑是证明：旧主机过去偷偷替应用做的每件事，在新模型里都有明确归属。

    决策框架¶

    迁移只有在三件事持续成立时才允许进入生产流量：

    决策问题 生产流量前必须有的答案
    容器重启时哪些状态会消失？ 每个可写路径都已外部化、可确定重建，或被明确声明为可丢弃。
    旧进程为什么能启动成功？ 端口、依赖、启动顺序、配置文件、证书和后台任务都已记录并验证。
    回滚边界在哪里？ 原主机部署保持在线，直到容器化服务证明读、写、任务、日志、告警都等价。

    这让团队把“容器化”当成运行时契约提取，而不是打包工作。

    起始状态¶

    • 长期存活的主机经过多次手工补丁和配置调整。
    • 部署脚本只记录了部分运行行为。
    • 一些应用文件写在本地磁盘上，并被默认为持久。
    • 少量配置存在于主机文件或 shell 启动文件中，而不是源代码管理下的配置。
    • 定时任务和维护任务运行在主进程旁边，没有进入发布流水线。
    • 服务之间的启动顺序靠习惯和时序维持，不靠就绪检查声明。

    约束¶

    • 迁移期间服务必须保持可用。
    • 凭据不能烤进镜像，也不能以明文文件搬运。
    • 新部署必须支持渐进式流量切换。
    • 旧主机在新运行时契约被证明前，必须仍能作为回滚路径。
    • 同一窗口内不允许夹带无关应用变更。

    迁移契约¶

    每个服务迁移前，团队先填一张小契约，而不是先动 Dockerfile：

    契约域 必须回答的问题 证据示例
    进程 什么命令启动服务？启动前必须存在什么？ Entrypoint、就绪探针、依赖检查。
    配置 哪些值随环境变化？它们存在哪里？ Config map 或环境变量清单；镜像层中没有密钥。
    密钥 运行时需要哪些凭据？如何轮换？ 密钥管理引用与吊销步骤。
    状态 哪些路径可写？哪些必须跨重启保留？ 卷映射、对象存储路径，或显式可丢弃缓存。
    任务 哪些 cron 或旁路任务属于产品行为？ 任务计划、Owner、重试规则和告警。
    可观测性 迁移后如何看见服务？ 日志、指标、链路、仪表盘和告警路由。
    回滚 什么条件让回到主机是安全的？ 主机部署仍与同一数据权威兼容。

    只有这张表有了明确 Owner，镜像构建才真正有意义。

    执行顺序¶

    1. 盘点运行中主机的进程列表、开放端口、挂载路径、软件包版本、证书、定时任务和启动文件。
    2. 将每个可写路径分类为持久数据、缓存、生成产物或意外状态。
    3. 把配置移入显式环境/config 对象，把凭据移入密钥管理系统。
    4. 用可重复输入构建固定镜像；不使用浮动基础镜像标签，不在启动时临时安装包。
    5. 增加启动、存活和就绪检查，让它们表达真实依赖就绪，而不只是进程存在。
    6. 把旁路任务重建为有调度、重试、Owner 和告警的工作负载。
    7. 在预发环境使用接近生产的配置，并注入依赖失败。
    8. 在旧主机仍作为写入权威时，切一小部分生产流量。
    9. 只有读、写、任务、日志和告警都通过等价验证后，才扩大流量。

    验证证据¶

    团队没有把“容器能启动”当成证据。验收更接近下面这张表：

    检查项 为什么重要 通过条件
    读路径等价 新运行时必须能访问旧主机能访问的每个依赖。 主机和容器执行同一组功能读测试均通过。
    写路径持久性 容器重启不能删除产品状态。 重启测试保留必需文件，或能重建可丢弃文件。
    启动门控 并行调度会暴露隐藏的时序假设。 依赖不可用时服务不会进入 ready 状态。
    任务等价 cron 式行为也是生产行为，即便不在 Web 进程里。 定时任务只执行一次、按规则重试、失败有告警。
    日志与指标路径 看不见的“可用”不能进入生产。 日志、指标和关键告警以新标签出现。
    回滚演练 没执行过一次的回滚计划只是愿望。 流量能回到主机部署，且不需要数据对账。

    暴露出来的失败模式¶

    本地状态其实是产品状态¶

    一个组件把文件写到看似临时的路径。旧主机让这个路径多年跨重启存活，所以团队不再意识到它的存在。容器重调度后，这个路径立刻消失。

    修复不是“把所有目录都挂卷”。团队把该路径拆成持久产品数据、生成缓存和意外残留。只有持久部分迁入受管存储，其余要么可重建，要么可丢弃。

    启动顺序是一种隐藏 API¶

    旧栈以大家熟悉的顺序启动。容器平台会并行调度。一个过去“通常会在依赖之后启动”的服务开始 crash loop。

    长期修复不是 sleep 循环，而是就绪门控和依赖检查。服务可以启动，但在它依赖的行为出现前，不能接收流量。

    Cron 在发布叙事之外¶

    几个小维护任务不属于应用发布产物。它们不在核心请求路径上，因此容易被漏掉；但它们一旦缺席，用户迟早会感受到。

    这些任务被作为一等工作负载迁移：有重试、有 Owner、有告警。这一刻团队才停止把 Web 进程误认为完整服务。

    配置像一次记忆泄漏¶

    有些设置存在，只是因为过去某次事故中有人手工调过主机。它们并不戏剧化，也不邪门，只是没有进入任何持久记录。

    团队通过比较主机与容器的运行行为、环境变量、配置文件和启动日志找到这些差异。任何解释不了的设置都被视为发布阻断项，而不是无害差异。

    可逆边界¶

    只有在旧主机部署仍与同一数据权威兼容时，回滚才容易。因此第一阶段生产迁移被刻意压小：

    • 流量可以在主机和容器之间移动。
    • 写入仍进入两个版本都理解的数据路径。
    • 没有 schema、存储布局或凭据轮换让主机版本失效。
    • 观察窗口结束前，定时任务和告警必须在真实条件下运行过。

    一旦某个步骤会让主机版本无法服务同一份数据，该步骤就被移出第一次切流，作为独立变更处理。

    结果¶

    服务迁移完成，未发生持续停机。关键不是最终镜像，而是那份运行时契约：它让后续服务更快迁移，也更容易审核。

    这次案例也改变了团队对“可以容器化”的定义：可以，不是指应用能被塞进镜像；而是指它能解释自己的配置、状态、任务、依赖和可观测性，不再借用一台长期主机上的隐性事实。

    可复用审查问题¶

    • 旧主机有哪些部分正在被当作不可见基础设施？
    • 如果容器在另一台节点上重启，什么会丢？
    • 哪些后台任务对产品正确性是必需的？
    • 哪些依赖失败应该让服务保持 unready，而不是 crash loop？
    • 流量能否回到旧主机且不需要数据对账？
    • 哪些设置只是因为某个人曾经手工修过事故才存在？
