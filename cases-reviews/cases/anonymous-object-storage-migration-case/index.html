    匿名对象存储迁移案例¶

    本案例中所有标识均已通用化。未保留任何原始 bucket 名、域名、区域、客户数据、访问密钥、前缀或应用名。

    这篇案例真正讨论什么¶

    这次迁移看起来是数据复制问题，实质上是权威迁移问题。

    写切换前，源 bucket 仍是事实来源。失败时通常只要把读重新指回源即可。写切换后，新对象和更新开始落到目标。自那一刻起，回滚不再是改路由，而是对账。

    这个边界塑造了整个计划。团队把所有危险的不确定性都前移到写切换之前，并让写切换成为最后一个小步骤，而不是大复制任务结束后的庆祝动作。

    决策框架¶

    决策问题 为什么重要 必须回答
    每个阶段谁是事实来源？ 没有权威模型，“回滚”这两个字会变得含糊。 写切换前源为权威；目标只有在明确批准后才成为权威。
    如何证明正确性？ 对象数量和总字节数可能一致，但具体对象、元数据或访问行为仍然错误。 Manifest、增量账本、抽样校验和、元数据检查、应用读取。
    写入迁移后会改变什么？ 写切换把恢复方式从重指向变成对账。 目标接收写入前，已有回滚和正向对账计划。
    读切换期间如何保证新写入可见？ 读指向目标、写仍指向源时，如果忽略复制延迟，新写入可能对读不可见。 新写入路径已被栅栏化、支持 read-through/fallback，或明确不属于读后写一致性要求，直到最终增量关闭。

    这次案例变安全的关键，是团队停止问“同步跑完了吗”，开始问“哪个存储当前有权接收事实”。

    起始状态¶

    • 应用持续在源厂商对象存储上读写。
    • 对象前缀承载应用语义。
    • 部分对象的元数据和访问行为对应用有意义，不只是字节内容有意义。
    • 没有足够长的维护窗口支持全量冻结、复制、再切换。
    • 长期迁移凭据不可接受。

    约束¶

    • 应用必须保持可用。
    • 数据不一致必须被限制且可观察。
    • 迁移凭据必须是临时、可吊销的。
    • 源与目标布局差异必须在首次全量复制前审查。
    • 源存储必须在写切换观察期后才退出回滚路径。

    迁移模型¶

    团队用了三本账。这是最重要的设计选择。

    账本 作用 防止的失败
    盘点 Manifest 声明哪些 bucket、前缀、对象类别、元数据和访问规则在范围内。 复制了错误形状的数据，直到很晚才发现前缀缺失。
    同步账本 记录全量复制完成情况、增量窗口、重试、差异和未解决对象。 把“命令结束”误当成“迁移正确”。
    应用验证账本 记录真实应用读取、抽样写入、元数据行为和权限检查。 字节迁过去了，但应用仍然无法使用。

    这些账不是流程装饰。它们是判断回滚会不会丢数据的唯一依据。

    执行顺序¶

    1. 从源 bucket、前缀、对象数量、字节总量、元数据要求、生命周期策略和访问规则生成盘点 Manifest。
    2. 创建目标 bucket、策略、生命周期行为、加密设置和临时迁移凭据。
    3. 应用继续从源读写时，执行初次全量同步。
    4. 将差异和重试失败记录到同步账本，而不是藏在工具日志里。
    5. 对全量复制期间变更的对象执行增量同步，并追踪 create、update、delete、overwrite、tombstone 或版本语义。
    6. 验证数量和大小后，按前缀、对象大小、对象年龄、存储类别进行分层抽样校验和。
    7. 在不迁移写入权威的情况下，对目标执行应用读取测试。
    8. 只有在新写入可见性已通过足够快的复制、read-through fallback、写栅栏，或明确说明该路径没有读后写要求后，才把读切到目标。
    9. 对最终增量窗口执行写冻结、写栅栏或严格顺序控制。
    10. 未解决差异被接受或清零后，才把写切到目标。
    11. 源在回滚窗口内保持只读；迁移关闭后吊销临时凭据。

    验证证据¶

    证据 为什么数量和大小不够
    前缀级 Manifest 对比 全局数量可能掩盖某个前缀缺失或错放。
    分层抽样校验和 总字节数可以一致，但部分对象内容不同。按前缀、大小和年龄抽样比单纯随机点查更有力。
    元数据对比 Content-Type、缓存头、加密和 ACL 行为可能在字节一致时仍破坏应用。
    变更语义 删除、覆盖、tombstone、对象版本和列表一致性决定目标代表当前事实，还是只代表已复制历史。
    应用读取 SDK 行为、签名 URL、Range Read、重定向和权限路径无法由存储 CLI 输出证明。
    最终增量账本 真正危险的是迁移期间持续变化的对象，不是已经干净复制的冷数据。
    写切换观察 新写入必须能从目标读到、被下游任务看见，且不能意外落回源路径。

    暴露出来的失败模式¶

    数量和大小匹配，但抽样失败¶

    第一次聚合对比看起来干净。一次校验和抽样发现了少量对象不匹配。问题小到不会改变全局总量，却足以证明团队不能只相信聚合数字。

    这些对象被重新同步，抽样策略也被扩大。团队保留了差异记录，而不是把它从记忆里擦掉，因为这份记录后来解释了为什么可以进入写切换。

    前缀映射是一种产品契约¶

    一个前缀假设与目标布局准备方式不一致。如果全量同步后才发现，成本会是大规模重复制；如果写切换后才发现，成本还会包括对账和应用混乱。

    盘点 Manifest 把前缀映射变成了复制前决策。这正是发现错误的正确位置。

    元数据也是数据的一部分¶

    部分应用行为依赖元数据和访问语义，而不只依赖对象字节。即使字节完全一致，也可能破坏响应、缓存行为或客户端下载。

    团队在写切换前加入应用读取测试，捕捉到了存储级 diff 看不到的问题。

    凭据必须有结束日期¶

    临时迁移访问太方便，很容易意外变成永久访问。团队把凭据吊销作为迁移关闭的一部分，而不是“以后清理”。

    可逆边界¶

    迁移有两种完全不同的回滚状态：

    阶段 权威 回滚成本
    读切换前 源接收读和写。 无用户侧回滚；目标可以重建。
    读切换后、写切换前 源仍接收写；目标服务读。 将读重新指向源，并调查目标行为。
    写切换后 目标接收新写入、更新、删除和覆盖。 回滚需要反向同步或对账；只改路由不够，源只读本身也不能保留目标侧变更。

    因此最终写切换被当作最小、最受控的步骤。能提前证明的，都必须在它之前证明。

    结果¶

    迁移完成，无长时间停机，未检测到数据丢失。更重要的是，团队知道每个阶段为什么可以继续。数量、大小、校验和、元数据检查、应用读取和增量对账，回答的是不同问题。

    长期经验是：对象存储迁移应围绕权威和证据设计。复制工具移动字节；迁移计划移动对事实的责任。

    可复用审查问题¶

    • 每个阶段哪个存储是权威？
    • 哪些前缀或对象类别如果晚发现会非常昂贵？
    • 元数据、ACL、生命周期规则、加密设置和签名 URL 行为是否属于正确性？
    • 全量复制期间的 create、update、delete、overwrite、tombstone 和对象版本如何被追踪？
    • 读切换期间，什么保证一个新写入对象仍能被读到？
    • 什么测试能证明应用可以从目标读取，而不只是目标上有字节？
    • 如果写切换后已有新写入落到目标，失败时究竟怎么处理？
    • 临时迁移凭据什么时候吊销？
