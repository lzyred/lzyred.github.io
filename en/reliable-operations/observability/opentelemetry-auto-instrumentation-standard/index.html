    Kubernetes Observability OpenTelemetry Reliability

    OpenTelemetry Auto-Instrumentation Standard¶

    Purpose¶

    This standard defines how to enable OpenTelemetry auto-instrumentation on Kubernetes workloads without treating it as a blind namespace-wide toggle.

    OpenTelemetry auto-instrumentation is implemented through the OpenTelemetry Operator and Kubernetes admission mutation. It can add init containers, agents, environment variables, and runtime configuration to new pods. That means it changes application startup, memory use, network egress, and the telemetry data leaving the cluster.

    The core judgment of this standard:

    Auto-instrumentation is a production change, not an observability toggle. Roll it out one workload at a time, with validation and rollback, the same way you would ship any change that mutates a running pod.

    When to Use¶

    Use this standard when:

    • You run the OpenTelemetry Operator, or plan to install it, on Kubernetes.
    • You want traces, and optionally metrics or logs, from services that cannot be manually instrumented yet.
    • You need consistent runtime agent configuration across multiple services.
    • You want a safe bridge from "no traces" to "manual instrumentation for critical paths".

    This is most useful for platform, SRE, and DevOps teams that own observability enablement but do not own every application codebase.

    Non-Goals¶

    This standard does not cover:

    • Manual SDK instrumentation inside application code.
    • Designing the trace backend itself. See SigNoz Observability Standard for backend selection.
    • Full sampling strategy design beyond the rollout-safe baseline below.
    • Non-Kubernetes agents for virtual machines or bare metal.
    • Vendor-specific OpenTelemetry distributions.

    Prerequisites¶

    Before enabling injection, confirm:

    • A reachable OTLP-compatible collector or backend exists.
    • The OpenTelemetry Operator and its admission webhook are installed and healthy.
    • The target namespace has an Instrumentation resource before the target application pod is created.
    • The first target workload is non-critical and has a known rollback owner.
    • The workload has a latency, error-rate, restart, and resource baseline from before injection.
    • The runtime language is known: Java, Node.js, Python, .NET, Go, Deno, or SDK-only injection.

    This standard focuses on application runtime instrumentation. Apache HTTPD and NGINX module injection need a separate webserver-module review because their compatibility and configuration risks are different from application agents.

    Do not start with a high-traffic, latency-sensitive, or incident-prone service. Auto-instrumentation is safer as a staged rollout than as a platform-wide campaign.

    Recommended Standard¶

    Use a collector as the default boundary¶

    Send application telemetry to an in-cluster OpenTelemetry Collector unless there is a deliberate exception.

    The collector gives the platform team one place to control:

    • Backend credentials.
    • Retries and queue limits.
    • Memory limiting.
    • Attribute normalization.
    • Filtering of unsafe span, metric, and log attributes.
    • Export routing to one or more backends.

    Do not point workloads at localhost, a kubectl port-forward, a node IP, or a developer laptop endpoint. A production Instrumentation resource must reference a stable service name.

    Example endpoint pattern:

    http://<otel-collector-service>:4318

    Use the fully qualified service name when the collector is in a different namespace from the application.

    Set the OTLP protocol explicitly; match the collector port¶

    Do not rely on a per-language default OTLP protocol. The default protocol for each auto-instrumentation runtime is version-dependent and has changed across both the language SDKs and the Operator. Treating "Node.js uses gRPC, everything else uses HTTP" as a fixed rule will eventually point a workload at a receiver port the collector is not listening on, and traces will silently disappear.

    The reliable model is a three-part contract you set deliberately, not infer:

    1. Protocol — set OTEL_EXPORTER_OTLP_PROTOCOL explicitly in the runtime section of the Instrumentation resource (grpc, http/protobuf, or http/json).
    2. Endpoint port — must match the chosen protocol:
    OTLP protocol Collector receiver port Endpoint example
    grpc 4317 http://<otel-collector-service>:4317
    http/protobuf or http/json 4318 http://<otel-collector-service>:4318
    1. Collector receiver — the collector must expose the matching OTLP receiver (gRPC on 4317, HTTP on 4318) for the port you chose.

    The http:// scheme on the endpoint is correct for both protocols here; it is the value the agent reads from OTEL_EXPORTER_OTLP_ENDPOINT. What must agree is protocol ↔ port ↔ collector receiver, not the scheme.

    Runtime-specific notes that still matter:

    • Python: older Operator versions may require OTEL_EXPORTER_OTLP_PROTOCOL=http/protobuf explicitly; verify during upgrade.
    • .NET: check the runtime identifier for musl-based images.
    • Go: feature-gated, needs OTEL_GO_AUTO_TARGET_EXE, uses an eBPF sidecar, and does not support multi-container pods (see the Go security exception below).
    • Deno: support may require runtime flags; do not treat it as a stable default without testing.
    • SDK-only: protocol and endpoint come from the SDK configuration you inject; match the collector receiver to whatever you set.

    Whenever the Operator version is upgraded, re-confirm the injected protocol against the receiver you run. The protocol is a value you set, not an architecture boundary you inherit.

    Keep injection opt-in¶

    Use per-workload pod template annotations. Do not use broad namespace-wide injection as the default operating model.

    For a Deployment, the annotation belongs under:

    spec.template.metadata.annotations

    A metadata annotation on the Deployment object itself does not mutate pods.

    Language-specific opt-in annotations:

    Runtime Injection annotation
    Java instrumentation.opentelemetry.io/inject-java: "true"
    Node.js instrumentation.opentelemetry.io/inject-nodejs: "true"
    Python instrumentation.opentelemetry.io/inject-python: "true"
    .NET instrumentation.opentelemetry.io/inject-dotnet: "true"
    Go instrumentation.opentelemetry.io/inject-go: "true"
    Deno or SDK-only instrumentation.opentelemetry.io/inject-sdk: "true"

    Use the Instrumentation resource name instead of "true" when the namespace contains more than one Instrumentation object:

    instrumentation.opentelemetry.io/inject-java: "java-standard"

    Target the container explicitly¶

    Pods with more than one container need explicit targeting.

    For a single language instrumentation, use:

    instrumentation.opentelemetry.io/container-names: "example-app"

    For multiple languages in the same pod, the Operator requires the multi-instrumentation feature flag and language-specific container name annotations:

    instrumentation.opentelemetry.io/inject-java: "true"
instrumentation.opentelemetry.io/java-container-names: "example-api"
instrumentation.opentelemetry.io/inject-python: "true"
instrumentation.opentelemetry.io/python-container-names: "example-worker"

    Do not let the webhook guess the target container in multi-container pods. It creates a common failure mode: the pod is healthy, the annotation exists, but the service you care about produces no useful telemetry.

    Keep the first Instrumentation resource minimal¶

    Use one Instrumentation resource per runtime and namespace until the operating model is stable.

    Example Java baseline:

    apiVersion: opentelemetry.io/v1alpha1
kind: Instrumentation
metadata:
  name: java-standard
  namespace: example-namespace
spec:
  exporter:
    endpoint: http://<otel-collector-service>:4318
  propagators:
    - tracecontext
    - baggage
  sampler:
    type: parentbased_traceidratio
    argument: "0.05"
  java:
    env:
      # Set the protocol explicitly so it matches the 4318 HTTP receiver above.
      - name: OTEL_EXPORTER_OTLP_PROTOCOL
        value: http/protobuf

    This example intentionally starts with a low sampling ratio. A sampling value of "1" traces all traffic and is unsafe as a default for high-throughput services.

    The protocol is set in the runtime-specific section (spec.java.env here) rather than left to the SDK default, so the 4318 endpoint, the http/protobuf protocol, and the collector's HTTP receiver all agree. Use spec.nodejs.env, spec.python.env, and so on for other runtimes.

    Add runtime-specific environment overrides only when they express a deliberate decision, such as disabling a noisy library or setting a compatibility flag. Do not paste a large vendor example into production without reducing it to the behavior you actually want.

    Treat Go as a security exception¶

    Go auto-instrumentation is different from Java, Python, Node.js, and .NET. It uses eBPF and requires an injected sidecar with elevated privileges. It also requires the target executable path through OTEL_GO_AUTO_TARGET_EXE or the matching annotation.

    Do not enable Go auto-instrumentation unless all of these are true:

    • The target pod has only the Go container you intend to instrument.
    • The executable path is known and stable.
    • The security owner accepts the privileged sidecar requirement.
    • A rollback owner is present during the rollout window.

    For many Go services, manual instrumentation is the safer long-term standard.

    Rollout Procedure¶

    1. Select one non-critical workload and one runtime.
    2. Record the baseline: request latency, error rate, pod restart count, CPU, memory, log volume, and current trace volume if any.
    3. Create or update the runtime-specific Instrumentation resource.
    4. If the target is Go, obtain explicit security approval for the privileged sidecar before continuing.
    5. Validate the operator, webhook, collector service, and Instrumentation resource before touching the workload.
    6. Add the injection annotation to the workload pod template.
    7. Let the normal deployment mechanism create new pods, then watch rollout status.
    8. Validate the pod mutation, telemetry path, application health, and backend load.
    9. Wait through an agreed observation window before annotating another workload.

    Example workload annotation:

    apiVersion: apps/v1
kind: Deployment
metadata:
  name: example-service
  namespace: example-namespace
spec:
  template:
    metadata:
      annotations:
        instrumentation.opentelemetry.io/inject-java: "java-standard"
        instrumentation.opentelemetry.io/container-names: "example-app"

    The Instrumentation resource must exist before the application pod is created. If the application pod already exists, the webhook cannot retroactively mutate it. A new pod must be created through a rollout.

    Validation¶

    Validate in layers and stop at the first failed layer.

    Layer 1: Operator and webhook¶

    kubectl -n opentelemetry-operator-system get pods
kubectl get mutatingwebhookconfigurations
kubectl -n opentelemetry-operator-system logs -l app.kubernetes.io/name=opentelemetry-operator --container manager --since=10m

    Expected result:

    • Operator pods are Ready.
    • The mutating webhook configuration exists.
    • Operator logs do not show injection, certificate, or admission errors for the target namespace.

    Layer 2: Collector reachability¶

    kubectl -n observability get svc otel-collector
kubectl -n observability get pods -l app.kubernetes.io/name=otel-collector
kubectl -n observability logs deploy/otel-collector --since=10m

    Expected result:

    • The collector service exposes the protocol and port required by the target runtime.
    • Collector pods are Ready.
    • Collector logs do not show receiver bind failures, exporter authentication errors, queue saturation, or memory limiter drops during the test window.

    Layer 3: Instrumentation resource¶

    kubectl -n example-namespace describe otelinst java-standard

    Expected result:

    • The endpoint points to the collector service, not localhost.
    • Propagators and sampler are explicit.
    • The runtime-specific section exists when runtime overrides are required.
    • Events are empty or explain a non-blocking condition.

    Layer 4: Workload mutation¶

    kubectl -n example-namespace rollout status deploy/example-service
kubectl -n example-namespace get events --sort-by=.metadata.creationTimestamp
kubectl -n example-namespace describe pod -l app.kubernetes.io/name=example-service

    Expected result:

    • New pods were created after the annotation was applied.
    • Events show the auto-instrumentation init container or injected agent behavior when applicable.
    • The target container, not a sidecar or unrelated init container, received the instrumentation.
    • Pods are Ready without abnormal startup delay.

    Layer 5: Telemetry and application health¶

    Validate both observability and service behavior:

    • The backend receives spans from example-service within the agreed test window.
    • Trace service name, namespace, environment, and version attributes are usable for filtering.
    • Request latency, error rate, restart count, and memory remain within the rollout threshold.
    • Log volume does not spike because of agent debug output.
    • Collector queue, dropped spans, and backend ingestion latency remain below alert thresholds.

    If application health regresses, roll back. Do not "leave it enabled and watch" unless the service owner explicitly accepts the risk.

    Rollback¶

    Injection happens when pods are created, so rollback must create clean replacement pods.

    1. Remove the injection annotation from the workload pod template.
    2. Remove runtime-specific container-name annotations from the same pod template.
    3. Roll out the workload through the normal deployment mechanism.
    4. Confirm new pods no longer contain the auto-instrumentation init container, injected agent, or injected OpenTelemetry environment variables.
    5. Confirm application latency, error rate, restart count, and memory return to the pre-injection baseline.
    6. Keep the Instrumentation resource until no annotated workload depends on it.

    Only remove the Operator or Instrumentation resource after annotated workloads have been cleaned up. Removing the platform component first can turn later pod creation into an admission or startup failure.

    Known Failure Modes¶

    Failure mode Symptom Likely cause Correct response
    Annotation on the wrong object Deployment has an annotation, but new pods are unchanged Annotation placed under metadata.annotations instead of spec.template.metadata.annotations Move annotation to pod template and create new pods.
    Resource created too late Annotation exists, but injection did not happen Instrumentation was created after the pod was created Create the Instrumentation first, then roll the workload.
    Collector unreachable Agent logs connection failures and backend has no spans Wrong service name, namespace, port, protocol, or endpoint defaulted to localhost Fix spec.exporter.endpoint; validate collector receiver ports.
    Wrong runtime annotation Pod mutates incorrectly or not at all Runtime annotation does not match application language Replace with the correct language annotation.
    Wrong container targeted Pod is healthy, but no useful spans appear Multi-container pod without explicit container targeting Add container-names or language-specific container name annotation.
    Backend overload Backend latency, dropped spans, or storage spike after rollout Sampling too high or too many workloads enabled at once Lower sampling, pause rollout, and validate collector queue limits.
    Startup regression Pods take longer to become Ready or crash on startup Agent image/runtime incompatibility or resource pressure Roll back and test the runtime image/version in staging.
    Sensitive attribute leakage Traces contain tokens, request bodies, headers, user data, or query strings Default instrumentation captured fields that should have been filtered Disable the offending instrumentation or filter at collector and application layers.
    Go multi-container failure Go workload does not instrument as expected Go auto-instrumentation does not support multi-container pods Do not use Go auto-instrumentation for that pod shape; use manual instrumentation or split workload.

    Security Notes¶

    • Treat telemetry egress as a data export path. Review destination, authentication, TLS, and retention before broad rollout.
    • Do not capture request bodies, full headers, query strings, authorization headers, cookies, or user identifiers by default.
    • Pin or approve auto-instrumentation images. An injected agent runs inside every annotated pod.
    • Do not enable Go auto-instrumentation without a security exception for its eBPF and privileged sidecar model.
    • Keep backend credentials in the collector, not in each application workload, whenever possible.
    • Limit who can modify Instrumentation resources and workload injection annotations.
    • Apply resource limits to the collector and monitor queue saturation so a backend outage does not turn into node pressure.
    • Use safe resource attributes such as service name, namespace, environment, team, and version; avoid customer, tenant, email, or request-level identifiers.

    Review Cycle¶

    • Owner: platform or observability team.
    • Review on every OpenTelemetry Operator upgrade.
    • Review on every Kubernetes minor upgrade.
    • Review when enabling a new runtime language.
    • Review when trace volume, backend cost, or collector drops exceed the agreed threshold.
    • Revalidate sampling whenever a high-traffic service is added.
