    NFS PVC Capacity Management¶

    Purpose¶

    This document defines a standard approach for monitoring and managing Kubernetes PersistentVolumeClaim capacity when NFS or NFS-like shared storage is used.

    Problem¶

    Kubernetes can show requested PVC capacity, but it does not always provide actual filesystem usage for NFS-backed paths. If usage is not monitored at the storage path level, applications can fail unexpectedly when the underlying shared filesystem fills up.

    Standard Approach¶

    Use a two-layer model:

    1. Kubernetes metadata: namespace, PVC name, PV, StorageClass, requested size.
    2. Storage filesystem usage: actual path usage from the NFS server or mounted host.

    The monitoring system should join both views and alert when usage crosses defined thresholds.

    Recommended Thresholds¶

    Level Usage Response
    Warning 75% Review growth rate and owner.
    High 85% Plan cleanup, expansion, or migration.
    Critical 90% Take immediate action and notify service owner.
    Emergency 95% Stop nonessential writes and execute expansion or cleanup.

    Required Metadata¶

    Track these fields for each PVC:

    cluster
namespace
pvc_name
pv_name
storage_class
requested_capacity
storage_path
actual_used
actual_available
usage_percent
owner
service
last_checked_at

    Standard Procedure¶

    1. Identify NFS StorageClasses¶

    kubectl get storageclass
kubectl describe storageclass <storage-class-name>

    Record provisioner, server, path pattern, reclaim policy, and expansion support.

    2. List PVCs Using the StorageClass¶

    kubectl get pvc -A -o wide
kubectl get pv

    Map each PVC to its PV and storage path.

    3. Measure Filesystem Usage¶

    On the storage server or mounted observation host:

    df -h <storage-root>
du -sh <storage-root>/<namespace>/<pvc-name>

    Use a consistent path convention. If dynamic provisioning creates hashed paths, maintain a PV-to-path mapping.

    4. Alert on Usage¶

    Alert messages should include:

    Cluster: example-cluster
Namespace: example-namespace
PVC: example-pvc
StorageClass: example-nfs
Usage: 87%
Threshold: 85%
Recommended Action: review growth and expand or clean up

    Do not include credentials, real internal domains, or sensitive mount paths in public alert examples.

    5. Remediate¶

    Common remediation options:

    • Remove expired application files.
    • Archive old logs or attachments.
    • Expand the backing NFS volume.
    • Migrate a high-growth workload to a dedicated volume.
    • Add retention policies at application or storage level.

    Cleanup Safety¶

    Never delete PVC data using a generic script unless all of these are true:

    • The owner approved the cleanup.
    • The target namespace and PVC are explicitly selected.
    • A backup or snapshot exists if data is business-critical.
    • The script has a dry-run mode.
    • The cleanup rule uses age, path, and file type filters.
    • The action logs every deleted object.

    Validation¶

    After remediation:

    df -h <storage-root>
du -sh <storage-root>/<namespace>/<pvc-name>
kubectl get pvc -n <namespace>
kubectl describe pvc <pvc-name> -n <namespace>

    Verify the affected application can still read and write expected data.

    FAQ¶

    Why not rely only on kubectl get pvc?¶

    PVC capacity shows requested storage, not always real usage. NFS-backed storage requires filesystem-level observation.

    Should cleanup be automatic?¶

    Automatic cleanup is acceptable only for clearly disposable data such as temporary files. Business data should require owner approval or a policy-controlled retention system.
