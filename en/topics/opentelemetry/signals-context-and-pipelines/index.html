    Signals, Context, and Pipelines¶

    The hard part of OpenTelemetry is not memorizing four nouns. It is building one causal model: system activity produces signals, Context connects distributed activity, and a Pipeline moves the data to a queryable destination. A break in any one layer can leave you with data that still cannot explain an incident.

    This page was checked against official OpenTelemetry concept and Collector configuration documentation on 2026-07-12. Signal and component maturity evolves, particularly for Profiles, Events, and Collector components; verify the target version before production adoption.

    Different information answers different questions¶

    Information Basic unit Best at answering Poor at answering alone
    Trace Trace / Span Where one request went, its parent-child calls, and where time was spent Overall impact and long-term trend
    Metric Measurement in a time series How many requests were affected and whether trends or percentiles crossed a limit Why one particular request failed
    Log Timestamped contextual event What a code path recorded at a particular moment Cross-service causality unless correlation context is present
    Baggage Key-value data propagated with Context Carrying a small amount of context to downstream instrumentation Acting as a secure data store or an observation record

    The official signal overview currently lists Traces, Metrics, Logs, and Baggage, while Profiles and Events are in development or proposal stages (Signals, checked 2026-07-12). It is still useful to distinguish Baggage: its primary function is propagation, and it does not automatically become a queryable Span, Metric, or Log.

    Traces reconstruct one execution¶

    A Trace is a collection of Spans that share a Trace ID. A Span represents an operation with a beginning and end: receiving an HTTP request, calling inventory, or executing a database statement. In addition to time, it normally carries:

    • its Span ID and parent Span ID;
    • operation name and Span Kind;
    • status, events, and attributes;
    • the Resource that produced it, such as service name and version.

    Parent-child relationships express causality, not just temporal proximity. Two operations happening at similar times do not prove that one caused the other. Only correct Context propagation and extraction lets a backend reconstruct the call tree.

    Traces are strong evidence for tail latency and localized failure, but sampling means that “I cannot see this trace” does not prove the request never occurred. Use metrics or another independent data source to establish impact.

    Metrics describe aggregated behavior¶

    Metrics aggregate many events into measurements over time. Counters suit cumulative quantities, gauges represent current state, and histograms describe distributions. They expose request rate, error rate, queue depth, and latency distributions without retaining every operation in full detail.

    Aggregation is also where cardinality becomes dangerous. A request ID, user ID, or unconstrained URL in dimensions expands a bounded metric into a large series set, increasing memory, transport, and storage cost. An attribute that is acceptable on a span is not automatically safe as a metric dimension.

    Logs record discrete evidence¶

    A Log records an event at a moment in time, either as a structured record or traditional text. Good logs describe the event, outcome, and necessary context; they do not dump entire requests, credentials, or arbitrary objects.

    When a log record contains the current Trace ID and Span ID, a backend can navigate from a failed span to logs from the same execution. That correlation requires all three conditions:

    1. the correct current Context exists when the log is produced;
    2. the logging SDK or bridge adds identifiers to the record;
    3. the collection path and backend preserve and index the fields.

    Missing any one step means that adopting OTel did not automatically correlate logs and traces.

    Baggage propagates data; it does not store secrets¶

    Baggage is key-value data carried with Context across service boundaries. Downstream instrumentation can read it and decide which attributes to add, for example a bounded workflow category. Baggage does not automatically become a span attribute, and it should not be treated as a distributed session.

    It can cross many processes and even external services. Never place passwords, tokens, personal data, or internal identifiers that downstream systems do not need in Baggage. Official security guidance calls for caution with context from untrusted sources and warns against credentials, API keys, or PII in Baggage (Context propagation security guidance).

    How Context connects services¶

    Context is correlation information attached to the current execution. Propagation serializes it into a transport carrier and deserializes it at the other side. HTTP commonly uses headers; a message broker can use message properties.

    sequenceDiagram participant U as Client participant A as Checkout API participant Q as Message Broker participant B as Inventory Worker U->>A: HTTP request + traceparent A->>A: extract Context<br/>create checkout Span A->>Q: publish message<br/>inject updated Context Q-->>B: deliver message + metadata B->>B: extract Context<br/>create reserve-inventory Span Note over A,B: Same Trace ID; each operation has its own Span ID

    The sender injects the current Trace ID, current Span ID, and trace flags into the carrier. The receiver extracts the remote Context and uses it as the parent of a new span. OpenTelemetry's default propagator uses the W3C Trace Context traceparent header:

    <version>-<trace-id>-<parent-id>-<trace-flags>

    That header is correlation information, not an access credential. At external boundaries, decide whether to accept, sanitize, or continue propagating inbound context so forged data cannot pollute traces. The official Context propagation page describes the mechanism and its security boundary.

    What broken Context looks like¶

    • The downstream service starts a new Trace ID: it probably did not extract context, or the transport omitted the header or message metadata.
    • The Trace ID matches but parentage is wrong: asynchronous or concurrent code may have reused the wrong current Context.
    • HTTP calls connect but messaging breaks: the HTTP library was instrumented while the messaging client or custom protocol was not.
    • Logs lack a Trace ID: no span was active when logging, or the logging bridge is not configured.

    Do not use timestamps to “guess” that two partial traces are one request. Temporal proximity creates a hypothesis; it does not restore causal evidence.

    Resource and Context are not interchangeable¶

    A Resource describes the entity producing telemetry, such as service.name=inventory, a release, or a Pod. It is relatively stable across many records. Context describes the current execution relationship and changes with each request and span.

    A compact model is:

    • Resource answers who produced it;
    • Context answers which execution and parent operation it belongs to;
    • an Attribute answers what else characterizes this operation.

    Putting a service name in Baggage, making a request ID a Resource, or making a user ID a metric dimension creates an incorrect model and expensive rework.

    How a Pipeline handles data¶

    Defining a component in Collector configuration does not run it. Only components listed in service.pipelines are active. A common Pipeline is Receiver → Processor → Exporter. There can be zero or more Processors, and order changes behavior.

    Receiver¶

    A Receiver accepts pushed data or actively collects it. It solves a protocol and entry problem; it does not prove data passed through Processors or was accepted by the backend. For network receivers, bind address, TLS, authentication, and exposure are security boundaries.

    Processor¶

    A Processor modifies, drops, aggregates, or batches data before export. Order changes the result and resource behavior. A configured Processor that is not referenced by a Pipeline has no effect.

    Exporter¶

    An Exporter sends data to a backend or another Collector. Successful sending usually means the destination accepted a request; it does not prove the data is queryable, correctly indexed, or retained. Verify the backend separately.

    Connector and Extension¶

    A Connector is an Exporter in one Pipeline and a Receiver in another, enabling cross-pipeline routing or signal generation. An Extension provides capabilities such as health checks and authentication without directly processing telemetry. Both must be enabled in the relevant service configuration. See Collector Configuration for component structure and activation rules.

    Where data can disappear¶

    Business operation
  → never instrumented
  → dropped by SDK sampling
  → process exits before export
  → network, protocol, or authentication failure
  → Collector receive or queue refusal
  → intentionally filtered by a Processor
  → Exporter retries exhausted
  → backend rejection, throttling, or indexing failure

    That chain is why “there is no trace in the backend” is not a root cause. Troubleshooting must establish evidence at each segment: the application created a span, the SDK attempted export, the Collector received it, Processors retained it, the Exporter sent it, and the backend made it queryable.

    A Debug Exporter proves data reached and passed through one Collector Pipeline. It does not prove a production backend is healthy. The project positions it as a local and live troubleshooting tool (Collector Troubleshooting).

    Sampling changes the conclusions you can draw¶

    Head sampling decides near the beginning of a trace. It costs less but cannot yet know whether the request will fail. Tail sampling waits for more spans in a Collector or similar component, allowing policies to retain errors or slow traces at the cost of buffering, completeness windows, and capacity.

    Sampling is therefore not merely a cost dial; it changes the evidence set. If entry and downstream services disagree on the sampling decision, partial traces can result. If a tail-sampling Collector overloads while waiting, the exceptional traces you most want may be lost.

    Record sampling policy, effective retention, drop location, and valid query interpretation together—not just a percentage.

    Test a design with three questions¶

    Can you locate one execution?¶

    Is the Trace ID continuous from ingress through synchronous and asynchronous boundaries? Who injects and extracts at each boundary? How is context handled at an untrusted edge?

    Can you measure overall impact?¶

    Which metrics are independent of trace sampling? What are the cardinality limits? Do error rate and latency come from the application, a proxy, or backend-derived data?

    Can you prove the telemetry path is healthy?¶

    What independent evidence exists at the application, Collector, and backend? Can you distinguish “no business event occurred” from “the telemetry path lost data”? Are Collector receive, refusal, send-failure, and queue metrics monitored through an independent path?

    If these questions have no answer, more auto-instrumentation usually creates more unexplained data.
