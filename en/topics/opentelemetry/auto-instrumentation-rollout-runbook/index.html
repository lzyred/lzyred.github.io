    OpenTelemetry Auto-Instrumentation Canary Rollout Runbook¶

    When to use this runbook¶

    Use this runbook after the design has passed the Auto-Instrumentation Production Standard, for a first enablement or an Operator auto-instrumentation upgrade in Kubernetes. It does not approve namespace-wide injection and does not choose sampling, performance, or retention limits for the service.

    Official facts were last verified on 2026-07-12. Recheck deployed versions against the Compatibility and Configuration Reference; this runbook makes no cross-version compatibility claim.

    Roles and change inputs¶

    Name the service owner, operator, rollback owner, observability owner, and data-incident contact. The change record must contain:

    • stable and canary Deployments, image digests, runtime, and target container;
    • Kubernetes, Operator, CRD, agent image, Collector, and backend versions;
    • OTLP protocol, endpoint, TLS/authentication, and NetworkPolicy;
    • canary traffic method, minimum sample, and observation window;
    • independent SLI plus p95/p99, error, restart, startup, CPU/RSS limits;
    • Collector accepted/refused/send_failed/queue and cost limits;
    • fake marker for the sensitive-data negative test;
    • rollback patch and escalation contacts.

    Stop if any input is empty.

    Preconditions¶

    1. Prove canary independence¶

    The canary must be an independent Deployment or revision that can be scaled and deleted separately. Confirm that the Service or gateway can distinguish traffic by fixed percentage, request header, or dedicated path. Do not edit the stable Deployment pod template.

    kubectl -n <app-namespace> get deploy <stable-deployment> <canary-deployment> -o wide
kubectl -n <app-namespace> get svc,httproute,ingress

    2. Inventory configuration overrides¶

    The Operator troubleshooting guide documents a current known issue: injection can override JAVA_TOOL_OPTIONS, PYTHONPATH, and NODE_OPTIONS when they come from an image or ConfigMap. Official guidance

    kubectl -n <app-namespace> get deploy <canary-deployment> -o yaml
kubectl -n <app-namespace> get configmap <runtime-config> -o yaml
kubectl -n <app-namespace> exec deploy/<canary-deployment> -c <app-container> -- env
kubectl -n <app-namespace> exec deploy/<canary-deployment> -c <app-container> -- ps -ef

    If these variables carry required application flags or the merged result is unknown, stop and solve it in a pre-production image. Do not experiment in production.

    3. Inspect the Operator webhook¶

    kubectl -n <operator-namespace> get deploy,pod,svc,endpoints
kubectl get mutatingwebhookconfiguration -o yaml
kubectl -n <operator-namespace> logs deploy/<operator-deployment> --since=15m

    Confirm that failurePolicy and timeoutSeconds match the change record, the CA bundle is nonempty, the certificate is not near expiry, the Service has endpoints, and logs have no TLS, timeout, or injection errors. Kubernetes explains that failurePolicy controls whether a timed-out webhook is ignored or rejects the request: Dynamic Admission Control.

    4. Inspect the Collector boundary¶

    kubectl -n <collector-namespace> get deploy,statefulset,pod,svc,pdb,networkpolicy
kubectl -n <collector-namespace> get secret <collector-client-identity> -o name
kubectl -n <collector-namespace> logs deploy/<collector-deployment> --since=15m

    Prove that receiver protocol/port, TLS trust, and authentication match; Collector replicas and capacity are approved; internal telemetry has no refusals, export failures, or queue nearing its limit. Never print Secret content merely to test authentication.

    5. Save the baseline¶

    From a source independent of OTel, save request count, p95/p99, business error rate, pod readiness/restarts, CPU/RSS, and startup time for the agreed window. Also save Collector receive rate, refusal rate, send failure, and queue/capacity. Stop without a baseline.

    Procedure¶

    1. Create or update Instrumentation first¶

    The following is an illustrative Java-canary resource that still requires environment-specific review, not a complete production manifest. It explicitly selects http/protobuf, so the example endpoint targets the Collector's actual OTLP/HTTP receiver. Do not reuse this block by merely swapping the language annotation: Node.js and other runtimes require an explicit protocol selected for the verified version and an endpoint matching the Collector's actual receiver.

    apiVersion: opentelemetry.io/v1alpha1
kind: Instrumentation
metadata:
  name: java-canary
  namespace: <app-namespace>
spec:
  exporter:
    endpoint: https://<collector-service>:4318
  propagators: [tracecontext, baggage]
  sampler:
    type: parentbased_traceidratio
    argument: "<approved-ratio>"
  java:
    env:
      - name: OTEL_EXPORTER_OTLP_PROTOCOL
        value: http/protobuf
  # Configure certificate trust, authentication, and other approved controls.

    Apply Instrumentation before creating canary pods. The Operator documentation states that reversing this order prevents injection: Auto-instrumentation troubleshooting.

    kubectl apply --server-side --dry-run=server -f <instrumentation-manifest>
kubectl apply -f <instrumentation-manifest>
kubectl -n <app-namespace> describe otelinst java-canary

    2. Annotate only the canary¶

    Server-side dry-run the patch first. The annotation belongs at spec.template.metadata.annotations; select the resource and container explicitly.

    spec:
  template:
    metadata:
      annotations:
        instrumentation.opentelemetry.io/inject-java: "java-canary"
        instrumentation.opentelemetry.io/container-names: "<app-container>"

    For Python, Node.js, .NET, or Go, first create a runtime-specific Instrumentation from the verified version and the Reference, with protocol and actual endpoint stated explicitly; do not inherit the Java block above. Then use the matching annotation. Apply the reviewed patch, then:

    kubectl -n <app-namespace> rollout status deploy/<canary-deployment> --timeout=<approved-timeout>
kubectl -n <app-namespace> get events --sort-by=.metadata.creationTimestamp
kubectl -n <app-namespace> describe pod -l <canary-selector>

    3. Validate in layers¶

    1. Pod mutation: only new canary pods are injected; target container and agent are correct; stable pods are unchanged.
    2. Process health: readiness, startup, restarts, logs, and actual process arguments meet limits.
    3. Known request: send a non-sensitive synthetic request with a unique test marker; record time and request ID.
    4. Trace: find the full trace in the expected window; verify service, environment, version, and propagation.
    5. Collector: accepted volume is plausible; refused/send_failed/queue remain within limits.
    6. Independent SLI: gateway, synthetic, or native application SLI meets its threshold.
    7. Negative test: include approved fake token, cookie, query, and body markers; prove zero matches at Collector egress and backend.

    Decision and rollback flow¶

    flowchart TD A[Preconditions complete] -->|No| S[Stop before injected pods] A -->|Yes| C[Create independent canary] C --> M{Expected pod mutation?} M -->|No| R[Set explicit false and replace pod] M -->|Yes| D{Zero sensitive-marker matches?} D -->|No| I[Stop export and escalate data event] D -->|Yes| H{Independent SLI and Collector limits pass?} H -->|No| R H -->|Yes| O[Complete observation window] O --> P{Owner approves next batch?} P -->|No| K[Keep canary; do not expand] P -->|Yes| N[Advance one defined batch]

    Immediate stop conditions¶

    Stop traffic expansion on any of these:

    • a stable workload is injected or the webhook affects unrelated pod creation;
    • real sensitive data is found or a negative-test marker appears at egress/backend;
    • any independent SLI, resource, or Collector limit is crossed;
    • canary CrashLoop, startup/readiness timeout, or overwritten process arguments;
    • webhook certificate, CA, endpoint, or failurePolicy behavior differs from the record;
    • canary and stable traffic cannot be distinguished, or sample size is insufficient.

    Rollback¶

    If namespace injection might exist, do not only remove the annotation. Explicitly set the runtime annotation to "false" on the canary pod template. The Operator defines this as the no-injection value: Injecting Auto-instrumentation.

    spec:
  template:
    metadata:
      annotations:
        instrumentation.opentelemetry.io/inject-java: "false"
    1. Reduce canary traffic to zero.
    2. Apply the explicit-"false" patch and roll the canary.
    3. Prove replacement pods have no injected init container/sidecar, agent files, or injected environment.
    4. Use the independent SLI to confirm return to baseline.
    5. Keep Instrumentation and the Operator until every dependent workload is clean; do not dismantle platform components first.
    6. Handle exported real data separately; agent rollback cannot retract it.

    Escalation¶

    Escalate successful receipt but missing backend data, sustained queue growth, certificate/webhook failure, or agent/runtime conflicts to platform/observability owners. Escalate sensitive-data matches to the data-security owner and an unapproved privileged Go sidecar to the security owner. Record commands, timestamps, resource versions, and non-sensitive error summaries—not credentials or real payloads.

    Completion¶

    Recommend the next batch only after the full observation window and sample pass, all MUST evidence is archived, rollback rehearsal succeeds, and stable service behavior is unchanged. The next batch is another controlled change; one successful canary never approves an entire namespace.
