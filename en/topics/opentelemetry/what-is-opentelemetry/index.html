    What Is OpenTelemetry?¶

    OpenTelemetry (OTel) is an open set of specifications, APIs, SDKs, semantic conventions, and tools for producing, correlating, transporting, and processing telemetry. It does not primarily answer “which dashboard should we use?” It addresses a layer closer to the application: how data from different languages, frameworks, and runtime environments can leave a process with consistent meaning and remain correlatable, governable, and portable before it reaches a backend.

    This page was checked against the official OpenTelemetry documentation on 2026-07-12. Collector and component maturity changes over time. Before deployment, verify the selected distribution's component inventory and stability; do not treat version status on this page as a permanent promise.

    Start with the problem¶

    Suppose a checkout request passes through a gateway, an order service, an inventory service, and a database. The customer sees only “checkout timed out,” while an engineer needs to answer three different questions:

    • What happened? Which request failed, and at which call did the failure occur?
    • How large is the impact? Did error rate, latency percentiles, or affected request volume exceed the service objective?
    • Why did it happen? Does a timeout log belong to the same execution as a slow span, a particular release, and a particular Kubernetes Pod?

    Collecting logs, metrics, or traces separately does not automatically answer those questions. The hard problems are consistency and causality: do services use the same service and attribute semantics, does context survive each call boundary, and can the collection path process, filter, and route data without putting backend-specific code in every application?

    OpenTelemetry provides a common language for that infrastructure. It does not decide whether the business is healthy, but it lets application activity be recorded and related in a form that backends can understand.

    See its place in one diagram¶

    flowchart LR subgraph Runtime[Application runtime] Code[Business code] Lib[Frameworks and libraries] API[OpenTelemetry API] SDK[SDK / auto-instrumentation agent] Code --> API Lib --> SDK API --> SDK end subgraph Collection[Collection and governance] Receiver[Receiver\naccept] Processor[Processor\nbatch, filter, transform] Exporter[Exporter\nsend] Receiver --> Processor --> Exporter end subgraph Backend[Observability backends] TraceDB[Trace backend] MetricsDB[Metrics backend] LogDB[Log backend] end SDK -->|OTLP or another protocol| Receiver Exporter --> TraceDB Exporter --> MetricsDB Exporter --> LogDB

    The diagram contains three boundaries that must remain distinct:

    1. Instrumentation boundary: APIs, SDKs, auto-instrumentation, and library instrumentation decide what data is produced.
    2. Collection boundary: the Collector decides what is received, processed, and sent where.
    3. Storage and analysis boundary: backends decide how data is indexed, queried, alerted on, and displayed.

    OpenTelemetry provides open interfaces and implementations for the first two boundaries, but it is not a complete storage, query, or alerting product. The project defines the Collector as a vendor-neutral implementation for receiving, processing, and exporting telemetry, not as an observability backend (Collector overview).

    What makes up OpenTelemetry¶

    Specification, API, and SDK¶

    The specification defines behavior and data models that should remain consistent across languages, including the API, SDK, OTLP, and semantic conventions. It answers what an implementation is supposed to mean.

    The API is a lightweight interface that application and library code can depend on. A library can create a span or record a measurement without forcing its user to choose a particular SDK or backend. With an API but no configured SDK provider, calls generally do not become exportable telemetry.

    The SDK implements sampling, processing, resource detection, and export. It is where runtime policy enters the application process: which traces are sampled, when spans are batched, and which OTLP endpoint receives the data.

    Separating API from SDK is an architectural choice. Application code states that a meaningful operation occurred; the deployment environment controls how to record and export it. The official component documentation defines these boundaries (OpenTelemetry Components).

    Instrumentation methods¶

    The common instrumentation sources are complementary:

    Method What it can see Main advantage Main boundary
    Manual instrumentation Business stages, results, and important attributes Closest to real business meaning Code must be changed, reviewed, and maintained
    Library instrumentation Common HTTP, database, and messaging calls Reusable and maintained with a framework Does not know business intent
    Zero-code / auto-instrumentation Supported framework and library calls in a runtime Fast initial coverage Constrained by language, version, and startup method
    Collector receivers Host, cluster, or existing protocol data Not all data must originate in an SDK Cannot recreate business meaning never expressed by the application

    Auto-instrumentation and manual instrumentation are not alternatives. A reliable progression is to establish a call skeleton with auto-instrumentation, then add manual spans and attributes for important operations such as payment authorization, inventory reservation, and access decisions.

    Resources and semantic conventions¶

    A span named only GET is almost impossible to assign. A Resource describes who produced telemetry, such as service.name, service version, Pod, and namespace. Semantic conventions define consistent names for HTTP, database, and messaging attributes.

    These are prerequisites for cross-team queries, not decorative metadata. Service-name drift splits one service into separate time series. User IDs in dimensions create cardinality and privacy risk. Inconsistent error attributes let alerts and traces report conflicting outcomes.

    OTLP¶

    OTLP is OpenTelemetry's telemetry transport protocol. It provides a common data model and transport for traces, metrics, and logs, commonly over gRPC or HTTP. OTLP reduces coupling between application and Collector, and between Collector and backend, but it is not a reliability guarantee. Timeouts, queues, retries, TLS, authentication, and backend capacity still require explicit design.

    Collector¶

    The Collector is a separately deployable telemetry data plane. A Pipeline normally contains a Receiver, zero or more Processors, and an Exporter. A component is active only when referenced by service.pipelines. The Collector also supports Connectors that join pipelines and Extensions that provide non-data-path capabilities such as health checks and authentication (Collector configuration).

    Its value is more than forwarding. It centralizes credentials, batching, sensitive-data filtering, attribute normalization, and multi-backend routing instead of repeating those concerns in every process. Centralization also creates a failure domain: insufficient capacity, an incorrect filter, or a blocked destination can affect many services at once.

    Kubernetes Operator¶

    The OpenTelemetry Operator can manage Collector custom resources and mutate supported workloads to inject auto-instrumentation. It is a control-plane component: it determines how a Pod is changed. The SDKs, agents, and Collectors that produce and transport telemetry are the data plane.

    Consequently, a healthy Operator does not prove traces reach a backend, and a Ready Collector does not prove a workload was correctly injected. Verify the two paths independently. The official documentation scopes the Operator to managing Collectors and workload auto-instrumentation (Kubernetes Operator).

    How a trace is formed¶

    A call does not become one trace merely because every service has an agent. The path must continuously satisfy these conditions:

    1. The entry service extracts upstream context or creates a new root span.
    2. The current operation creates a child span and injects context into the downstream call.
    3. The downstream service extracts that context and uses the received span as the parent of its span.
    4. Each SDK exports spans; a Collector receives, processes, and sends them onward.
    5. The backend reconstructs the call tree from the shared Trace ID.

    If one protocol boundary fails to propagate context, the backend sees two unrelated traces. The data may still exist, but causality has been lost. Signals, Context, and Pipelines examines this process in detail.

    What it solves—and what it does not¶

    Engineering problems it can solve¶

    • Give polyglot applications a common telemetry model and propagation protocol.
    • Reduce direct coupling between applications and a particular backend.
    • Reuse instrumentation for common library calls across services.
    • Centralize batching, filtering, routing, and attribute governance.
    • Connect signals with Trace ID, Span ID, and Resource attributes.

    Problems it does not solve automatically¶

    • It does not define service objectives: having data does not define health.
    • It does not guarantee completeness: sampling, queue overflow, network failures, and configuration errors lose data.
    • It does not invent business semantics: auto-instrumentation cannot know that inventory reservation matters more than an ordinary SQL call.
    • It does not remove cost: more attributes, higher sampling, and duplicate export consume resources and storage.
    • It does not automatically protect sensitive data: attributes, log bodies, and Baggage can carry data that must not leave a boundary.
    • It does not replace a backend: queries, retention, alerts, and visualization still belong to a backend.

    Misconceptions that cause rework¶

    “Installing an agent completes observability”¶

    An agent observes only supported runtime and library calls. Asynchronous task boundaries, internal business stages, custom protocols, and unsupported frameworks can remain blind spots. Acceptance should ask whether predefined questions can be answered, not whether injection files exist in a Pod.

    “The Collector is a stateless pipe, so it needs no capacity design”¶

    A Collector can batch, queue, and retry in memory or storage. Even without long-term retention, it has a memory limit, a CPU saturation point, and downstream backpressure. Production design must observe received, refused, failed-send, and queue metrics and define behavior during backend failure.

    “Signals correlate automatically when attributes have the same text”¶

    Matching strings do not necessarily identify the same entity. Reliable correlation needs stable Resource rules, correctly propagated Trace/Span context, controlled attribute cardinality, and backend support for the correlation.

    “Vendor-neutral means zero migration cost”¶

    OTLP and semantic conventions reduce coupling at data production, but query languages, alert models, dashboards, and vendor-specific processors can still lock a system in. Portability comes from explicit boundaries and migration exercises, not from a protocol name.

    When adoption is appropriate¶

    OpenTelemetry is usually a good foundation when an organization has inconsistent polyglot instrumentation, applications directly connected to multiple backends, broken cross-service causality, or a need to govern telemetry at a common exit.

    If the system is one small monolith, an existing backend SDK already answers the required questions, and the team cannot operate a Collector, staying simple can be more reliable. The decision should not be “everyone uses OTel.” It should be whether OTel removes current coupling and correlation gaps and whether the team accepts responsibility for a new data plane.

    Learning path¶

    1. Read Signals, Context, and Pipelines to understand how causality survives process boundaries.
    2. Complete Generate Your First Trace to verify the client, Collector, and debug output path yourself.
    3. Then move to auto-instrumentation, production architecture, security, capacity, and recovery. Do not begin learning by injecting production workloads.

    Official sources and version boundary¶

    • OpenTelemetry Components: specification, API, SDK, Collector, and Operator boundaries.
    • OpenTelemetry Signals: supported signal categories and current status.
    • OpenTelemetry Collector: Collector role, distributions, and component maturity entry point.
    • Collector Configuration: actual pipeline and component activation rules.
