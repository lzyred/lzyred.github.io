    OpenTelemetry Auto-Instrumentation Production Standard¶

    Purpose and scope¶

    This standard defines the minimum production gates for injecting Java, Node.js, Python, .NET, Go, or SDK configuration into Kubernetes workloads through the OpenTelemetry Operator. It decides whether a change may ship. Use the Auto-Instrumentation Canary Rollout Runbook for procedure and the Compatibility and Configuration Reference for runtime-specific facts.

    Auto-instrumentation changes container environment, startup arguments, filesystem content, or sidecars when a pod is created. It also adds CPU, memory, and telemetry egress. It must be managed as a production code change, not as an observability switch for a namespace. The Operator documentation describes pod-creation-time injection and the requirement to create the Instrumentation resource before the application pod: Injecting Auto-instrumentation.

    In this document, MUST is a release blocker, SHOULD is the default and requires a recorded reason when not followed, and MAY is optional.

    Version and fact boundary¶

    • Official-source facts on this page were last verified on 2026-07-12.
    • This page does not claim that arbitrary Operator, Kubernetes, Collector, runtime, and agent versions are compatible.
    • Every change MUST record actual versions, image digests, base-image libc, protocol, Collector distribution, and the most recent pre-production test result. A missing field blocks promotion.
    • Upgrading any component invalidates historical evidence and requires a new canary.

    The version record must include at least the Kubernetes minor version, Operator version, Instrumentation CRD version, auto-instrumentation image digest, application runtime version, application image digest, Collector version, and backend receiver version. Volatile constraints live in the Compatibility and Configuration Reference.

    Release rule matrix¶

    Rule Requirement Minimum evidence If unmet
    OTI-001 Versions and image digests MUST be pinned and verified; “latest” or floating tags are not approval evidence. Change-record version matrix, image digests, and pre-production test date. Block release.
    OTI-002 Injection MUST affect one independently controlled canary workload. Annotating every pod template in the stable Deployment is not a canary. Independent Deployment/ReplicaSet, fixed traffic share or dedicated synthetic traffic, and rollback object. Block release.
    OTI-003 JAVA_TOOL_OPTIONS, PYTHONPATH, NODE_OPTIONS, and runtime startup arguments MUST be inventoried before the change. Review of image configuration, ConfigMaps, pod environment, and actual process arguments. Block when a conflict exists.
    OTI-004 OTLP protocol, endpoint, TLS, and authentication MUST be explicit; NetworkPolicy or equivalent control MUST restrict telemetry egress. Receiver summary, trust-chain test, authentication test, and allowed source/destination rules. Block release.
    OTI-005 The team MUST prove sensitive fields cannot leave the approved boundary. Negative test containing fake tokens, cookies, query parameters, user identifiers, and body markers; zero backend matches. Block and escalate as a data event.
    OTI-006 Collector capacity and availability MUST cover current load and the approved backend-outage budget. Replicas, anti-affinity/PDB, resources, queue, retry, measured capacity, and internal-telemetry dashboard. Block release.
    OTI-007 Admission webhook failurePolicy, timeoutSeconds, CA bundle, certificate expiry, and Service endpoints MUST be reviewed. Webhook configuration snapshot, certificate alert, and failure-drill result. Block release.
    OTI-008 Application health MUST be judged with an SLI independent of the new telemetry path. Baseline and thresholds from load balancer/gateway metrics, synthetic requests, native application metrics, or business success rate. Block release.
    OTI-009 Quantitative pass, stop, and rollback thresholds MUST be defined in advance. Observation window, sample size, p95/p99, error rate, restarts, RSS/CPU, Collector refusal/queue, and cost thresholds. Block release.
    OTI-010 Rollback MUST override namespace injection with explicit "false" and create clean replacement pods. Rollback patch, rehearsal output, and clean-pod evidence. Block release.
    OTI-011 Traces SHOULD be enabled first; metrics and logs should be separate changes. Recorded signal scope. Record deviation and added capacity/data review.
    OTI-012 A clearly named Instrumentation per language and namespace SHOULD be used, with the target container explicit. Resource name and container-name annotation. Record why mis-injection is impossible.

    A real canary¶

    A canary must have independent scaling, rollback, and identifiable traffic. Acceptable patterns include:

    • clone the stable Deployment as example-service-otel-canary, run a small replica count, and route a fixed share or request header through the gateway;
    • expose a dedicated test path and send only non-sensitive synthetic traffic;
    • create an independently controlled canary revision in a progressive-delivery controller and prove that only selected replicas are affected.

    Changing spec.template.metadata.annotations on the stable Deployment rolls every pod in that Deployment. That is a progressive full rollout, not an independent canary, and does not satisfy OTI-002.

    Quantitative change record¶

    This standard must not invent universal thresholds. The service owner fills in absolute or relative limits from the pre-change baseline and records sample size and observation window.

    Input Example format only
    Traffic and sample 5% canary traffic; at least 10,000 requests; observe 30 minutes
    Latency p95 increase below approved value; p99 remains within service SLO
    Correctness 5xx/business-failure delta below approved value
    Resources CPU, RSS, startup time, and GC each have an upper bound
    Stability no new restarts; no increase in readiness failures
    Collector queue/capacity, send_failed, refused, and export latency each have a limit
    Cost estimated and measured spans/s, bytes/s, and retention cost remain in budget
    Stop stop on missing MUST evidence, sensitive-data match, or independent-SLI regression

    Derive the sampling ratio from ingress rate, average trace width, error/slow-request retention policy, Collector capacity, and backend budget. Neither 5% nor 100% is a safe universal default.

    Data-security gate¶

    OTI-004 and OTI-005 must pass before the first injected pod is created:

    1. Application-to-Collector traffic uses organization-approved TLS and authentication; plaintext OTLP is not allowed across trust boundaries.
    2. Credentials reside in the Collector when possible, rather than being copied into every workload.
    3. NetworkPolicy or equivalent controls allow applications to reach only approved Collectors and Collectors to reach only approved backends.
    4. Configuring a Collector filter, transform, or attribute processor is not evidence by itself. Send a request containing fake sensitive markers and prove there are zero matches at Collector egress and in the backend.
    5. If real tokens, cookies, personal data, or tenant identifiers are found, stop export, preserve minimal investigative evidence, and escalate through the organization's data-incident process. Rolling back an agent cannot retract exported data.

    OpenTelemetry's security guidance explains that sensitive-data classification is organization-specific and recommends removing, hashing, or filtering at the SDK or Collector: Handling sensitive data.

    Collector and webhook gates¶

    The Collector MUST at least provide:

    • no single-replica production dependency; replicas, topology distribution, and PDB align with the availability target;
    • sending queue, retry, and persistence choices derived from measured receive rate, average batch size, permitted backend outage, and memory budget;
    • memory_limiter or a validated equivalent, plus container requests and limits;
    • monitoring for otelcol_receiver_accepted_*, otelcol_receiver_refused_*, otelcol_exporter_sent_*, otelcol_exporter_send_failed_*, and exporter queue size/capacity; metric names can change by Collector version, so verify them against the deployed version's internal-telemetry documentation;
    • a backend-outage drill showing alerting before queue exhaustion and bounded recovery backlog.

    Official guidance states that data can still be lost when the downstream is unavailable and the sending queue fills, and recommends capacity planning from queues, memory, and receive rate: Collector resiliency, Scaling the Collector, and Internal telemetry.

    The webhook is part of the pod-creation control path. The team must deliberately choose failurePolicy: Fail or Ignore and record the security/availability trade-off; neither is universally correct. Kubernetes documents that timeout behavior is controlled by failurePolicy and that timeoutSeconds is configurable: Dynamic Admission Control. Certificate expiry, CA-bundle mismatch, missing Service endpoints, and webhook latency must be alerted.

    Independent validation and compliance decision¶

    Approval requires all four evidence classes:

    1. Mutation evidence: only canary pods were injected; target container and agent version are correct.
    2. Telemetry evidence: a known synthetic request creates the expected trace, resource attributes identify it, and Collector refusals/failures do not increase.
    3. Independent health evidence: gateway, synthetic, or native application SLIs meet the change-record thresholds; the OTel path under test cannot certify itself.
    4. Negative evidence: fake sensitive markers cannot be found at Collector egress or in the backend.

    If any of OTI-001 through OTI-010 is unmet, the only decision is noncompliant. A Ready pod, a service name in the backend, or absence of alerts is not sufficient alone.

    Exception mechanism¶

    An exception to a MUST records the rule ID, business reason, risk owner, compensating control, scope, start and expiry, rollback trigger, and approver. Every exception expires; promotion stops if it is not renewed. OTI-005 cannot be waived through a routine change exception when real sensitive data is exposed.

    Rollback boundary¶

    When namespace injection might be enabled, rollback must explicitly set the language annotation to "false" on the workload pod template, not merely remove it. The Operator documentation defines "false" as the valid no-injection value. New pods must then be created and shown to contain no injected init container, sidecar, agent files, or injected environment. See the Auto-Instrumentation Canary Rollout Runbook.
