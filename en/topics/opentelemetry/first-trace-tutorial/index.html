    Generate Your First Trace¶

    This tutorial verifies a minimal trace data path with your own evidence: telemetrygen creates spans and sends them over OTLP to a local Collector, then the Debug Exporter and zPages expose evidence that the Collector received and processed them.

    This is not a production deployment. It has no persistent backend, TLS, authentication, high availability, capacity control, or sensitive-data governance. Every port binds only to 127.0.0.1. Do not copy this Collector setup, open ports, or debug output into production.

    The procedure was checked on 2026-07-12 against the official Collector Quick Start and pins both Collector and telemetrygen to 0.156.0. If that image or Go Module version is unavailable for your platform, stop and select a matching Collector and generator version from official releases. Do not mix arbitrary versions.

    What you will prove¶

    At the end, you should have three separate pieces of evidence:

    1. The generator reports that it produced and sent three traces.
    2. Collector logs contain received spans, including Trace ID, Span ID, and Parent ID relationships within a trace.
    3. The local zPages TraceZ page shows trace activity processed by the Collector.
    flowchart LR T[telemetrygen\ngenerate test spans] -->|OTLP/gRPC\n127.0.0.1:4317| C[Collector 0.156.0] C --> D[Debug Exporter\nterminal logs] C --> Z[zPages\n127.0.0.1:55679] D --> V1{Check IDs and parentage} Z --> V2{Check receive activity}

    The observations are not interchangeable. Generator success does not prove Collector receipt. Collector logs do not prove a real production backend is queryable. zPages is a local troubleshooting UI, not an observability backend.

    Prerequisites¶

    Use a Bash-compatible terminal on macOS, Linux, or WSL with:

    • Docker or a compatible runtime able to run Linux containers;
    • Go installed, with go install able to reach public Go Modules;
    • local ports 4317, 4318, and 55679 free;
    • permission to pull otel/opentelemetry-collector:0.156.0;
    • a home directory where a temporary experiment directory can be created.

    Check the tools and do not ignore errors:

    docker version
go version

    Stop if either command fails. Do not add sudo to bypass a Docker permission problem; fix the local container runtime first.

    Check ports:

    lsof -nP -iTCP:4317 -sTCP:LISTEN
lsof -nP -iTCP:4318 -sTCP:LISTEN
lsof -nP -iTCP:55679 -sTCP:LISTEN

    No output means no process is listening on the corresponding TCP port. If a command returns a listener, stop and release the port. Alternatively, use another set of loopback-only ports only after understanding both Collector and client parameters.

    Create an isolated working directory¶

    export OTEL_LAB_DIR="$(mktemp -d "${TMPDIR:-/tmp}/otel-first-trace.XXXXXX")"
export OTEL_CONTAINER="otel-first-trace-$(date +%Y%m%d%H%M%S)-$$"
mkdir -p "$OTEL_LAB_DIR/bin"
touch "$OTEL_LAB_DIR/.l2do-otel-lab"
cd "$OTEL_LAB_DIR"
export GOBIN="$PWD/bin"

    Confirm that installation will remain in the experiment directory:

    printf 'GOBIN=%s\n' "$GOBIN"

    The output must end in /bin under the random experiment directory. Keep OTEL_LAB_DIR, OTEL_CONTAINER, and every later command in the same shell. Stop if either variable is missing; do not guess a path or container name.

    Install a pinned generator¶

    go install github.com/open-telemetry/opentelemetry-collector-contrib/cmd/telemetrygen@v0.156.0
"$GOBIN/telemetrygen" --help

    The second command should print telemetrygen help, not “file not found.” Pinning keeps the client on the same release line as the tutorial. The official Quick Start may advance to a newer release; check its current image version when running this procedure.

    Pull and identify the Collector image¶

    docker pull otel/opentelemetry-collector:0.156.0
docker image inspect otel/opentelemetry-collector:0.156.0 \
  --format '{{index .RepoDigests 0}}'

    Save the digest from the second command with the experiment record. The digest depends on registry and platform, so this page does not invent a fixed value. If it is empty or the pull fails, stop; do not substitute latest.

    Start the local Collector¶

    if docker container inspect "$OTEL_CONTAINER" >/dev/null 2>&1; then
  printf 'Container-name collision; stop: %s\n' "$OTEL_CONTAINER" >&2
  exit 1
fi

docker run --detach \
  --name "$OTEL_CONTAINER" \
  --label "l2do.otel-lab=$OTEL_LAB_DIR" \
  -p 127.0.0.1:4317:4317 \
  -p 127.0.0.1:4318:4318 \
  -p 127.0.0.1:55679:55679 \
  otel/opentelemetry-collector:0.156.0

    Inspect container state and startup logs:

    docker ps --filter "name=^/${OTEL_CONTAINER}$"
docker logs "$OTEL_CONTAINER"

    The container should be Up. Logs should not show configuration parsing failure, port binding failure, or immediate shutdown.

    For this image version, the official Quick Start configuration exposes:

    • 4317 for OTLP/gRPC;
    • 4318 for OTLP/HTTP;
    • 55679 for zPages.

    These are verified behaviors of the 0.156.0 Quick Start, not permanent defaults for every distribution or custom configuration (official procedure).

    Stop conditions¶

    Do not send data if any of these is true:

    • the container is not Up;
    • Docker reports a port conflict;
    • logs report invalid configuration or Receiver/Exporter startup failure;
    • ports are not bound to 127.0.0.1;
    • the running image tag is not 0.156.0.

    Perform the Cleanup procedure below, then resolve the cause.

    Generate three test traces¶

    With the Collector running, execute:

    "$GOBIN/telemetrygen" traces \
  --otlp-endpoint 127.0.0.1:4317 \
  --otlp-insecure \
  --traces 3

    --otlp-insecure is acceptable only for this loopback, plaintext experiment. It does not verify TLS and must not become a production parameter.

    The generator should report three traces generated and stop its Batch Span Processor. Timestamps, source line numbers, and IDs change with versions and executions; do not compare them character-for-character with official sample output. If the command times out or reports export failure, check the Collector and the 4317 listener instead of repeatedly generating noise.

    Verify that the Collector received spans¶

    Save logs in the experiment directory:

    docker logs "$OTEL_CONTAINER" > collector-output.txt 2>&1
grep -E '^Span|(Trace ID|Parent ID|ID|Name|Kind|Start time|End time|Status code)[[:space:]]*:' \
  collector-output.txt

    Expect one or more Span # sections containing:

    • non-empty Trace ID and ID values;
    • an empty Parent ID for a root span;
    • a child Parent ID equal to another span's ID in the same trace;
    • Name, Kind, start time, and end time;
    • a common Trace ID for spans in the same parent-child group.

    The actual IDs are random on each run. The acceptance criterion is their relationship, not a sample string.

    List unique Trace IDs in this log if useful:

    grep 'Trace ID' collector-output.txt | awk '{print $NF}' | sort -u

    If the generator succeeds but the log has no spans, the path broke at client export, network entry, or the Collector Pipeline. Generator output alone is not Collector receipt evidence.

    Cross-check with zPages¶

    Open this local URL:

    http://127.0.0.1:55679/debug/tracez

    Select trace activity in the table and confirm that the page is not a connection or blank error. zPages exposes a debugging view of trace activity inside the Collector. It provides no durable retention and does not prove an external backend received anything.

    If logs contain spans but TraceZ is unreachable, investigate the 55679 mapping and zPages Extension. Do not discard receipt evidence already established by the Debug Exporter.

    Evaluate the result¶

    Check Passing evidence Failure indicates
    Client generation telemetrygen reports three traces generated Generator, arguments, or local network issue
    Collector receipt and processing Logs contain spans with reasonable ID parentage Broken OTLP entry or Pipeline
    Debug UI TraceZ is reachable and shows trace activity zPages Extension or port mapping problem

    Only when all three pass can you make this limited statement: test spans traveled from the local generator into and through this Collector. It says nothing about production application instrumentation, cross-service propagation, backend persistence, alerting, or capacity design.

    Common failures and diagnostic order¶

    connection refused¶

    Run:

    docker ps --filter "name=^/${OTEL_CONTAINER}$"
docker logs --tail 100 "$OTEL_CONTAINER"

    Confirm the container is alive, then confirm the generator targets 127.0.0.1:4317. Do not replace the endpoint with a container-private IP in this tutorial.

    Collector exits immediately¶

    Read the full logs. Common local causes are an existing container with the same name or an occupied port. This tutorial uses the image's configuration. If you mounted a custom configuration, you have left the experiment boundary; remove the mount and restart.

    Logs are noisy but contain no spans¶

    Save docker logs again after generating traces. Do not inspect only startup-era output. The Debug Exporter is intended to confirm receipt and processing in localized troubleshooting (Collector Troubleshooting).

    Cleanup¶

    Whether the experiment succeeds or fails, remove only the container carrying this experiment's label and the random directory carrying this experiment's marker. If either guard fails, inspect manually; never bypass the guard with an unverified delete:

    if [ "$(docker inspect "$OTEL_CONTAINER" --format '{{ index .Config.Labels "l2do.otel-lab" }}' 2>/dev/null)" = "$OTEL_LAB_DIR" ]; then
  docker rm --force "$OTEL_CONTAINER"
else
  printf 'Refusing to remove an unowned container: %s\n' "$OTEL_CONTAINER" >&2
fi

if [ -n "$OTEL_LAB_DIR" ] && [ -f "$OTEL_LAB_DIR/.l2do-otel-lab" ]; then
  cd "${TMPDIR:-/tmp}"
  rm -rf -- "$OTEL_LAB_DIR"
else
  printf 'Refusing to remove an unowned directory: %s\n' "${OTEL_LAB_DIR:-<empty>}" >&2
fi

    Confirm no experiment container remains:

    docker ps --all --filter "name=^/${OTEL_CONTAINER}$"

    Only the table header should remain. The image stays in local cache by default. To remove it too:

    docker image rm otel/opentelemetry-collector:0.156.0

    Docker will refuse if another container references the image. Do not force deletion and disrupt another experiment.

    Move from the experiment to a real system¶

    The next step is not to copy --otlp-insecure and the Debug Exporter into a cluster. Add real constraints one layer at a time:

    1. Replace telemetrygen with an application SDK or supported auto-instrumentation.
    2. Define stable service.name and Resource rules for each service.
    3. Verify Context Propagation across a real synchronous or asynchronous call.
    4. Add TLS, authentication, and network boundaries to Collector receivers.
    5. Add memory limiting, queueing, retry, capacity, and Collector self-observation.
    6. Connect a test backend and separately verify Collector send and backend query.

    Change one assumption at a time and preserve the prior layer's evidence. That is much easier to diagnose than deploying a complete platform and then facing “there are no traces.”

    Official sources and version boundary¶

    • Collector Quick Start: official basis for the experiment, ports, and observations.
    • Collector Configuration: component and activation rules for a custom Pipeline.
    • Collector Troubleshooting: intended use of the Debug Exporter and zPages.
    • telemetrygen source: generator source at the pinned version.
