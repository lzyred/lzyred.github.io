    OpenTelemetry Compatibility and Configuration Reference¶

    Boundary of this reference¶

    This page is a lookup surface, not a certified version list. Official facts were last verified on 2026-07-12, but the OpenTelemetry Operator, language agents, Deno, and Collector ship independently. Build compatibility evidence from the actual version's official release notes, image digest, and pre-production test. This page never implies arbitrary combinations are supported.

    Complete this record on every upgrade:

    Component Required evidence
    Kubernetes Minor version, node architecture, container runtime
    Operator Version, Helm chart/manifest version, feature gates
    CRD Instrumentation apiVersion and installed schema
    Auto-instrumentation Language, image tag and digest, agent/runtime version
    Application image Digest, CPU architecture, glibc/musl, startup arguments
    Collector Distribution/version, enabled components, receiver/exporter summary
    Backend OTLP receiver version, authentication/TLS, limits, retention
    Validation Environment, date, known request, performance/data-safety result

    Treat any changed field as a new combination. latest, only a chart version, or “it started in test” is not compatibility evidence.

    OTLP protocols and ports¶

    Transport Default when no port is configured Common endpoint form Note
    OTLP/gRPC 4317 https://collector.example:4317 Collector must enable a gRPC receiver.
    OTLP/HTTP 4318 https://collector.example:4318 Signals commonly use /v1/traces, /v1/metrics, and /v1/logs; base/signal URL construction follows the implementation and specification.

    4317 and 4318 are specification defaults when no port is configured, not fixed ports. A Collector can listen elsewhere and a Service can remap ports. Inspect the actual endpoint and receiver; never infer protocol from port alone. OTLP Exporter specification

    The specification defines grpc and http/protobuf, and allows an SDK to optionally support http/json. Therefore:

    • do not assume every language agent implements http/json;
    • do not rely on a language's current protocol default; set it explicitly and test end to end;
    • http:///https:// in an endpoint describes transport security, not whether the payload is OTLP/HTTP rather than gRPC;
    • across production trust boundaries, use approved TLS and authentication; never disable verification as a connectivity test.

    Operator injection entry points¶

    Annotations belong at spec.template.metadata.annotations. Values can be "true" for the default resource in the current namespace, a resource name, namespace/name, or "false" to disable injection. Namespace annotations opt workloads in, so rollback should set explicit "false" at the workload. Operator auto-instrumentation

    Runtime Injection annotation Primary boundary
    Java instrumentation.opentelemetry.io/inject-java Check JAVA_TOOL_OPTIONS overwrite and JVM/library compatibility.
    Node.js instrumentation.opentelemetry.io/inject-nodejs Check NODE_OPTIONS overwrite.
    Python instrumentation.opentelemetry.io/inject-python Check PYTHONPATH and glibc/musl.
    .NET instrumentation.opentelemetry.io/inject-dotnet Check base image, architecture, and runtime identifier.
    Go instrumentation.opentelemetry.io/inject-go eBPF sidecar, elevated privileges, target executable, single-container constraint.
    Deno/SDK-only instrumentation.opentelemetry.io/inject-sdk Injects SDK configuration; it does not install a full language SDK into an ordinary app.

    Multi-container pods require an explicit target. For one language use instrumentation.opentelemetry.io/container-names. With the Operator multi-instrumentation feature gate enabled, use java-container-names, nodejs-container-names, python-container-names, dotnet-container-names, go-container-names, or sdk-container-names. One container cannot receive multiple language instrumentations, and Go does not support multi-container pods. Follow the deployed version's Operator documentation.

    Runtime compatibility boundaries¶

    Java¶

    • Operator injection can overwrite JAVA_TOOL_OPTIONS supplied by an image or ConfigMap; compare actual process arguments. Official troubleshooting
    • A manual Java agent attaches through -javaagent:<path> or dedicated JAVA_TOOL_OPTIONS; it is in-process code, so pin the version and JAR checksum/digest.
    • Java agent 2.0+ currently defaults to http/protobuf, unlike the historical SDK-autoconfigure default. Explicit configuration is safer. Java agent configuration
    • Supported JVMs, libraries, and frameworks vary by agent release. Validate against that release's Supported Libraries page; “Java 8+” is not framework compatibility evidence.

    Python and musl¶

    • Operator injection can overwrite PYTHONPATH supplied by an image or ConfigMap. Official troubleshooting
    • glibc is the default platform. The Operator documentation says that from v0.113.0, musl can be selected with:
    instrumentation.opentelemetry.io/otel-python-platform: "musl"
    • This only proves that the Operator exposes a platform selector. It does not prove compatibility among the application's Python version, C extensions, architecture, and agent image; test the identical base image.

    Node.js¶

    • Operator injection can overwrite NODE_OPTIONS supplied by an image or ConfigMap. Official troubleshooting
    • Preloaders, other APM agents, ESM loaders, or custom --require flags can compete for startup options. Compare actual process.execArgv and environment; do not assume automatic merging.

    .NET¶

    • Record architecture, distribution, glibc/musl, and runtime identifier. A .NET label does not mean an auto-instrumentation binary is portable across base images.
    • Test profiler loading, startup, and native dependencies in a production-equivalent image. Roll back on failure; do not swap unknown binaries in production.

    Go¶

    • The matching Operator feature gate must be enabled; verify its name and default for the deployed version.
    • Go auto-instrumentation uses an eBPF sidecar. The official example requires privileged: true and runAsUser: 0, so obtain separate security approval.
    • Supply the target executable via OTEL_GO_AUTO_TARGET_EXE or instrumentation.opentelemetry.io/otel-go-auto-target-exe.
    • Official documentation currently says multi-container pods are unsupported. A service-mesh or logging sidecar also makes a pod multi-container; use manual instrumentation or redesign rather than bypassing the check. Operator Go auto-instrumentation

    Deno¶

    • The Operator uses inject-sdk and sets OTEL_DENO=true. Current Deno documentation also enables built-in OpenTelemetry with this variable and defaults to localhost:4318 over http/protobuf, so production must override the endpoint. Deno OpenTelemetry
    • Deno capabilities and startup flags evolve quickly. Some Operator-version documentation still requires --unstable-otel, while the current Deno quick start no longer uses it. Follow the actual Deno version's official documentation and startup test; do not copy either behavior across versions.
    • Deno can export console logs as well. Decide OTEL_DENO_CONSOLE behavior and run a sensitive-log negative test before enablement.

    SDK-only injection¶

    inject-sdk supplies common configuration and resource attributes. It does not install a missing language SDK, automatic-instrumentation library, or business spans into an ordinary application. The app must already include and initialize a compatible SDK; otherwise “environment present, no telemetry” is expected rather than a Collector fault.

    Configuration sources and precedence¶

    The common environment-variable specification says implementations may provide these variables; not every implementation must implement all of them, and language agents can add precedence rules. Environment Variable Specification

    Explicitly inspect:

    • OTEL_SERVICE_NAME (normally higher precedence than service.name in OTEL_RESOURCE_ATTRIBUTES);
    • OTEL_EXPORTER_OTLP_PROTOCOL, common endpoint, and signal-specific endpoints;
    • trace/metric/log exporters and sampler;
    • propagators;
    • agent/SDK configuration-file variables and whether they override ordinary environment variables;
    • effective container environment after Operator injection, not only the CR.

    Do not place customers, tenants, emails, tokens, full URL queries, or request-level identifiers in resource attributes.

    Minimum inspection commands¶

    kubectl version
kubectl -n <operator-namespace> get deploy <operator-deployment> -o yaml
kubectl get crd instrumentations.opentelemetry.io -o yaml
kubectl -n <app-namespace> describe otelinst <name>
kubectl -n <app-namespace> get pod <pod> -o yaml
kubectl -n <app-namespace> exec <pod> -c <container> -- env
kubectl -n <app-namespace> exec <pod> -c <container> -- ps -ef

    These commands prove only part of effective configuration. Final compatibility evidence also requires a known request, independent application SLI, Collector internal telemetry, and a sensitive-field negative test.
