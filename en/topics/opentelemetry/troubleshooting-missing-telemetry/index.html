    Troubleshooting Missing OpenTelemetry Data¶

    Trigger and objective¶

    Use this runbook when a backend cannot find a service or known request, a trace is broken, metrics/logs disappear, or an Operator annotation exists but the pod is not injected. The objective is not to make some data appear in a UI. Find the first evidence-backed break, restore one known request end to end, and do not expand data exposure or application risk.

    Official facts were last verified on 2026-07-12. Runtime defaults and Operator behavior can change; record actual versions first, then consult the Compatibility and Configuration Reference.

    Protect the service first¶

    If application errors, latency, startup, or resources have regressed, use the Auto-Instrumentation Canary Rollout Runbook to roll back. Do not delay service recovery to preserve telemetry. If real sensitive fields appear, stop the relevant export and escalate a data event; agent rollback cannot retract exported data.

    Do not jump to OTEL_TRACES_SAMPLER=always_on, full debug logs, disabled TLS/authentication, or console payload output in production. These shortcuts can create capacity or data incidents.

    Incident record¶

    Record first detection, affected signals, service/namespace/version, last known good time, known request ID and window, Operator/agent/Collector/backend versions, recent changes, and independent business SLI. If there is no known request, send one non-sensitive synthetic request first; do not infer from random traffic.

    Locate the break hop by hop¶

    1. Is the backend query correct?¶

    Query with an explicit time range, service.name, environment, version, and known request ID. Verify timezone, tenant/project, signal type, and sampling policy. For a partial trace, record the last upstream span and expected downstream service; the fault may be propagation, downstream instrumentation, or sampling rather than the whole Collector.

    Continue only after excluding a query error. Otherwise correct the query and stop.

    2. Did the request reach the intended instance?¶

    Use gateway access logs, native request counters, or a synthetic probe to prove the request reached the target canary/pod. If it did not, repair routing rather than OTel.

    kubectl -n <app-namespace> get pod -l <workload-selector> -o wide
kubectl -n <app-namespace> logs deploy/<deployment> -c <app-container> --since=<window>

    Logs may be sensitive; retrieve only the approved window and necessary fields.

    3. Was the pod injected as intended?¶

    kubectl -n <app-namespace> get deploy <deployment> -o yaml
kubectl -n <app-namespace> describe pod <pod-name>
kubectl -n <app-namespace> get events --sort-by=.metadata.creationTimestamp
kubectl -n <app-namespace> describe otelinst <instrumentation-name>

    Check that the annotation is in spec.template.metadata.annotations; language, resource name, and container name are correct; Instrumentation existed before the pod; and the created pod contains the expected init container/sidecar/environment. The Operator troubleshooting page documents the same order and checks: Auto-instrumentation troubleshooting.

    When namespace and workload annotations interact, use actual pod mutation as evidence. Deployment YAML alone does not prove injection.

    4. Was startup configuration overwritten?¶

    Compare image declarations, ConfigMaps, pod spec, and the container's effective environment/process arguments:

    kubectl -n <app-namespace> exec <pod-name> -c <app-container> -- env
kubectl -n <app-namespace> exec <pod-name> -c <app-container> -- ps -ef

    Focus on JAVA_TOOL_OPTIONS, PYTHONPATH, and NODE_OPTIONS. The official guide records that injection can currently override these when set in images or ConfigMaps: configuration-source limitation. If required application flags disappeared, roll back immediately; do not concatenate untested values live.

    5. Does the SDK/agent create the signal?¶

    Use a known synthetic request and bounded agent diagnostics/application logs. Raise diagnostic level only on the canary, for a short approved window, and restore it afterward. Check runtime/library support, disabled instrumentation, sampling that may drop the request, proxy removal of propagation headers, and an unexpected service.name.

    If manual SDK initialization and an automatic agent both configure a provider, check the language's official coexistence guidance; never assume they compose safely.

    6. Do protocol, endpoint, and network agree?¶

    Inspect the effective OTEL_EXPORTER_OTLP_PROTOCOL and endpoint; do not infer protocol from port. The OTLP specification defines 4317/4318 as defaults when no port is configured, while endpoints can use other ports. http/json is an optional form an SDK may implement, not a universal capability: OTLP Exporter specification.

    Check DNS, NetworkPolicy, Service endpoints, TLS SNI/CA, client authentication, and Collector receiver. Do not disable certificate verification to prove reachability.

    kubectl -n <collector-namespace> get svc,endpoints,networkpolicy
kubectl -n <collector-namespace> logs deploy/<collector-deployment> --since=<window>

    7. Where does the Collector drop data?¶

    Within the pipeline, compare receiver accepted/refused, processor drops, exporter sent/send_failed, and queue size/capacity. Metric names vary by version; first consult Internal telemetry for the deployed Collector.

    • receiver does not increase: agent, network, authentication, or receiver break;
    • accepted increases but exporter sent does not: processor/filter, memory limiter, routing, or pipeline binding;
    • send_failed or queue grows: backend authentication, throttling, network, or capacity;
    • queue full: stop adding input; a full sending queue still loses data, as documented in Collector resiliency.

    Do not remove sensitive-data filters merely to make data flow. Follow a unique, non-sensitive marker through the pipeline.

    8. Did the backend receive but fail to index/query?¶

    Inspect backend receiver errors, tenant routing, indexing lag, retention, throttling, and clock skew. Compare Collector exporter sent with backend accepted. If the Collector proves successful send but the backend has no record, escalate the evidence to the backend owner instead of repeatedly restarting applications.

    Symptom to action¶

    Evidence Likely break Action
    No pod mutation Resource order, annotation location/name, webhook Correct canary configuration and recreate the pod.
    Pod mutated, application startup bad Version/architecture incompatibility or environment override Roll back with explicit "false" immediately.
    Agent connection/TLS/auth error Endpoint, protocol, trust, NetworkPolicy Correct approved connectivity; do not weaken security.
    Receiver accepted is zero Agent-to-receiver path Validate DNS, Service, NetworkPolicy, TLS, and auth in order.
    Receiver accepted grows, exporter sent does not Processor/pipeline Review filters, routing, memory limiter, and pipeline binding.
    send_failed/queue grows Backend or Collector capacity Stop rollout, protect the queue, escalate backend/platform.
    Root span exists, downstream absent Context propagation or downstream instrumentation Check proxy headers, downstream sampling, and agent.
    Trace exists under wrong service Resource configuration Correct explicit resource attributes; do not delete old data and claim repair.

    Stop, rollback, and escalation¶

    Roll back immediately when independent application SLI crosses a limit, process arguments are overwritten, the pod CrashLoops, unrelated pods are injected, or diagnostics cannot be bounded. Set the language annotation to "false" and create replacement pods; deleting the annotation is insufficient when namespace injection may exist.

    Escalate immediately to data security when real sensitive data or a negative-test marker reaches an unapproved location. Escalate to platform when the webhook blocks unrelated pod creation, certificate/CA fails, or Collector queue approaches exhaustion. Escalate to backend when the Collector has successful-send evidence but data cannot be queried.

    Closure evidence¶

    Before closure, save the first broken hop, root cause, version matrix, known request end-to-end evidence, independent SLI, Collector metrics, repair/rollback diff, and proof that temporary diagnostics were removed. If the data path is still not explainable, status is “mitigated, unresolved,” even if a little data reappears in the UI.
