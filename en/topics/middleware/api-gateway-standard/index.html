    API Gateway Standard¶

    Purpose¶

    This standard defines how to operate API gateways such as APISIX, Kong, Nginx-based ingress gateways, or similar edge and internal traffic gateways.

    The core judgment:

    An API gateway is not a reverse proxy with plugins. It is the point where external traffic, identity, policy, application ownership, and failure behavior meet.

    A gateway route can expose a service, bypass a control, amplify a dependency failure, or silently change a customer path. Treat gateway configuration as a production change surface, not as an afterthought beside application deployment.

    When to Use¶

    Use this standard when:

    • applications are exposed through a shared gateway
    • routes, plugins, or upstream targets are changed through GitOps, an admin API, or a dashboard
    • the gateway terminates TLS, enforces authentication, applies rate limits, or performs traffic splitting
    • multiple teams depend on the same gateway data plane or control plane
    • a failed route could affect customer traffic, internal platform APIs, or partner integrations

    Non-Goals¶

    This standard does not choose a gateway product, teach all plugin options, or replace application API design. It defines the operating contract for publishing and changing routes.

    Production Contract¶

    Before a route receives production traffic, define:

    Contract Required decision Why it matters
    Route owner Who owns the host, path, upstream service, policy, and rollback? A route without an owner becomes shared production risk.
    Exposure class Is the route public, partner-facing, internal, or admin-only? Exposure decides authentication, network boundary, rate limit, logging, and review depth.
    Source of truth Is route config stored in Git, Kubernetes manifests, gateway CRDs, or a controlled admin API? Operators need to know what must be reverted when live state is wrong.
    Admin boundary Who can change routes, plugins, certificates, and upstreams? From which network path? The admin surface can rewrite production traffic.
    Identity policy Which authentication and authorization controls are enforced at the gateway and which remain in the application? A gateway must not create false security by only checking transport.
    Traffic policy What are the timeout, retry, request-size, rate-limit, and canary rules? Defaults can turn a small upstream issue into a gateway-wide incident.
    Failure behavior What happens when the upstream is unhealthy, slow, overloaded, or partially deployed? Users experience this behavior, not the deployment manifest.
    Observability Which route-level metrics, logs, traces, and audit events prove the route is safe? Gateway incidents are hard to debug without per-route evidence.
    Rollback boundary Does rollback remove exposure, restore a route version, shift traffic, or revert the upstream release? Deleting a route is not the same as restoring the application.

    Recommended Standard¶

    Keep the admin surface narrower than the traffic surface¶

    The data plane may be public. The admin plane must not be.

    Required controls:

    • admin API reachable only from approved networks or management workloads
    • dashboard protected by strong identity and role boundaries
    • route and plugin changes audited with actor, time, object, and diff
    • admin credentials stored in an approved secret manager
    • separate roles for route authoring, approval, certificate management, and emergency operation where scale requires it
    • no anonymous admin access
    • no wildcard admin allowlist
    • no admin token, key, or certificate material in public examples or values files

    If a gateway product stores state in etcd, a database, or a controller, protect that state as part of the gateway control plane. A healthy data-plane pod is not enough if the control plane can be rewritten or lost.

    Publish routes as reviewed contracts¶

    Each production route should have a route contract, even if the implementation is an Ingress, Gateway API resource, APISIX route, Kong route, or product-specific object.

    Required route fields:

    • host and path
    • owning team
    • upstream service and namespace
    • exposure class
    • authentication and authorization expectation
    • TLS certificate owner and renewal path
    • timeout and retry policy
    • request-size limit
    • rate-limit or quota behavior where required
    • canary or weight rule when traffic is split
    • expected health check and readiness behavior
    • validation command or synthetic request
    • rollback owner and rollback action

    Do not publish a route because the manifest applies cleanly. Publish it when the route contract can be validated from the user path to the upstream and back.

    Put policy close to the route, not in memory¶

    Gateway policy is easiest to lose when it lives only in a dashboard or someone's memory.

    Prefer route configuration that can be reviewed, diffed, and restored. If a dashboard or admin API is the only supported path, export the current route and plugin state after approved changes and record the recovery source.

    Controls that should be explicit:

    • TLS termination and certificate reference
    • authentication mode
    • authorization boundary
    • rate limit and burst behavior
    • timeout and retry behavior
    • request and response transformation
    • allowed methods and payload size
    • header forwarding rules
    • logging and tracing settings
    • canary or weighted upstream rule

    Treat retry and timeout defaults as failure design¶

    Retries can hide transient faults, but they can also multiply load on an already failing upstream. Timeouts can protect the gateway, but they can also cut off a slow dependency path before the application can degrade gracefully.

    For each high-impact route, decide:

    • client timeout expectation
    • gateway timeout
    • upstream timeout
    • retry count and retryable status codes
    • whether non-idempotent methods can be retried
    • behavior when the upstream has no healthy endpoints
    • fallback or fail-closed expectation

    If the route accepts writes, be conservative with retries. A duplicated mutation is often worse than a visible failure.

    Validate canary and weighted routing as traffic movement¶

    Traffic splitting is a production traffic movement, not a cosmetic release feature.

    Before enabling canary or weight changes:

    • prove both upstream versions are healthy
    • confirm sticky-session or state expectations
    • define the metric that blocks promotion
    • define the threshold that triggers rollback
    • confirm logs and metrics distinguish versions
    • keep a route-level rollback action ready

    Do not promote a weight change when route-level errors are hidden inside aggregate gateway metrics.

    Route Example¶

    Use placeholders in public examples:

    apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: example-api-route
  namespace: example-namespace
  annotations:
    example.gateway/owner: platform-team
    example.gateway/exposure: public
    example.gateway/timeout: "5s"
    example.gateway/rate-limit: "placeholder"
spec:
  ingressClassName: example-gateway
  tls:
    - hosts:
        - api.example.com
      secretName: example-api-tls
  rules:
    - host: api.example.com
      http:
        paths:
          - path: /v1
            pathType: Prefix
            backend:
              service:
                name: example-api
                port:
                  number: 80

    The example is intentionally incomplete. A real production route also needs the route contract, admin boundary, validation evidence, and rollback action.

    Validation¶

    Pass conditions:

    • gateway controller and data-plane pods are healthy
    • control-plane datastore is healthy where the gateway uses one
    • admin API and dashboard are unreachable from public networks
    • only approved roles can change production routes, plugins, certificates, and upstreams
    • route config matches the approved source of truth
    • TLS certificate is valid, renewed through the documented path, and attached to the expected host
    • synthetic request reaches the intended upstream and receives the expected status and headers
    • unauthenticated or unauthorized requests fail in the expected way
    • request-size, timeout, and rate-limit controls behave as designed
    • route-level request rate, error rate, latency percentiles, upstream health, and 4xx/5xx counts are visible
    • access logs and audit logs can identify route, upstream, actor, and decision point without leaking sensitive payloads
    • rollback was tested in the same control path used for production changes

    Failure Modes¶

    Failure mode Result Prevention
    Admin API exposed Attackers or unapproved users can rewrite traffic. Keep admin access private, authenticated, and audited.
    Route without owner Nobody owns failure, stale policy, or rollback. Require owner and rollback fields before publication.
    Dashboard-only change Live state cannot be reviewed or restored. Use reviewed config or export controlled state after approved changes.
    Missing auth boundary Public traffic reaches a private service path. Classify exposure and validate auth behavior before publication.
    Unsafe retry on writes Duplicate mutations or amplified upstream load. Restrict retries by method, status, and idempotency.
    Aggregate-only metrics Route failures disappear inside gateway totals. Collect route-level metrics and version labels for canary paths.
    Certificate lifecycle unknown Expiry becomes an outage. Define certificate owner, renewal path, and alert.
    Route rollback misunderstood Exposure is removed but bad upstream release remains, or the reverse. Define whether rollback acts on route, traffic weight, or application release.

    Rollback and Recovery¶

    Gateway rollback can mean different actions:

    • restore the previous route definition
    • disable a new route
    • shift weight back to the previous upstream
    • remove a plugin or policy change
    • restore a certificate reference
    • revert the application release behind the route
    • temporarily fail closed for an unsafe public path

    Choose the rollback action before the change.

    For route changes, keep the last known-good route configuration and validate that the gateway has reconciled it. For canary changes, rollback should usually move traffic first; application rollback can follow if the bad version must be removed.

    For admin-surface compromise, recovery includes credential rotation, route and plugin diff review, audit review, and confirmation that no unauthorized upstream, certificate, or policy remains.

    Security Notes¶

    • Keep admin APIs, dashboards, and control-plane datastores off public networks.
    • Store credentials and certificate material in approved secret management.
    • Avoid broad route-edit permissions across teams.
    • Do not log sensitive headers, tokens, request bodies, or business identifiers.
    • Do not use gateway authentication as a substitute for application authorization on sensitive operations.
    • Protect metrics and logs when they reveal hostnames, route names, customer paths, or internal topology.
    • Never publish real hostnames, upstream names, credentials, customer paths, private IPs, or business-specific route evidence.

    Review Cycle¶

    Review this standard after gateway incidents, route ownership changes, new exposure classes, plugin changes, certificate incidents, control-plane upgrades, or when a gateway begins serving additional teams or environments.
