    DolphinScheduler Operations Standard¶

    Purpose¶

    This standard defines how to operate DolphinScheduler or a similar distributed workflow scheduler.

    The core judgment:

    A scheduler is not only "where jobs run." It is the system that protects business time windows. When it fails, the impact is not a failed task; it is a delayed process, stale data, missed handoff, or uncontrolled re-run.

    DolphinScheduler needs metadata protection, worker capacity control, dependency isolation, retry discipline, alerting, and recovery rules before critical workflows depend on it.

    When to Use¶

    Use this standard when:

    • batch jobs or data workflows run through DolphinScheduler
    • failed schedules can delay a business process or downstream report
    • multiple teams share projects, workers, data sources, or shell execution
    • workflows call databases, object storage, APIs, queues, or external systems
    • operators need a controlled way to rerun, skip, pause, or recover workflows

    Non-Goals¶

    This standard does not teach:

    • workflow modeling
    • SQL or data pipeline design
    • DolphinScheduler installation steps
    • business SLA design
    • data quality rules inside individual jobs

    It covers the operational boundary after the scheduler has been selected.

    Production Contract¶

    Before critical workflows are onboarded, define:

    Contract Required decision Why it matters
    Workflow owner Who owns success, failure, rerun, and business communication? A failed workflow without an owner becomes an invisible business delay.
    Metadata protection How are project definitions, schedules, state, users, and history backed up and restored? Losing metadata can erase the scheduler's memory.
    Worker classes Which workers run control tasks, compute tasks, data movement, and shell tasks? One noisy or risky job must not consume the whole platform.
    Time-window objective What is the latest acceptable finish time for important workflows? Queue health should be judged against business time, not only task count.
    Retry and idempotency Which tasks are safe to retry, and which need reconciliation first? Blind retries can duplicate writes or corrupt downstream state.
    Dependency ownership Who owns upstream and downstream dependencies? A workflow graph crosses team and system boundaries.
    Alert path Which failures page humans, which create tickets, and which are only reports? Failed jobs are not equal; response must match impact.
    Recovery rule When do operators rerun, skip, disable, restore, or escalate? Recovery without rules can make the failure larger.

    Recommended Standard¶

    Protect metadata first¶

    The metadata database contains project definitions, schedules, task state, users, permissions, and execution history. Treat it as scheduler state, not as disposable application cache.

    Required controls:

    • regular backup
    • restore test
    • documented metadata database owner
    • schema/version compatibility check before upgrades
    • backup retention aligned to workflow recovery requirements
    • recovery runbook for scheduler state loss

    Do not call a scheduler production-ready until metadata restore has been tested.

    Isolate worker capacity by workload class¶

    Separate workers where possible:

    • lightweight control tasks
    • heavy compute tasks
    • data movement tasks
    • shell or host-sensitive tasks
    • high-priority business-window tasks

    Use queues, worker groups, permissions, or deployment separation to prevent a single workflow from consuming all execution capacity.

    Monitor:

    • worker slots
    • task queue backlog
    • delayed starts
    • task duration drift
    • missed workflow windows
    • failed or killed workers

    Make retries depend on idempotency¶

    Each critical task should declare:

    • retry count
    • retry interval
    • timeout
    • idempotency expectation
    • external systems written to
    • upstream and downstream dependency
    • owner
    • manual reconciliation rule

    Do not blindly rerun tasks that write external state. A duplicated payment export, overwritten object, or repeated API mutation may be worse than the original failure.

    Alert by business impact¶

    Alert routing should consider:

    • criticality of the workflow
    • missed time window
    • downstream dependency impact
    • whether retry is still running
    • whether data has already been partially written
    • owner and escalation path

    The dangerous failure is not always the first task that turns red. Sometimes it is a workflow that is still running but has already missed the handoff window.

    Control shared execution rights¶

    Schedulers often become a path to shell execution, database credentials, and cross-team data movement.

    Controls required:

    • project-level permissions
    • named owners
    • protected data source credentials
    • separated worker users where shell tasks exist
    • audit of workflow edits for critical projects
    • review before granting broad script or data-source access

    Validation¶

    Pass conditions:

    • Master, worker, alert, API, and UI services are healthy.
    • Metadata database is reachable, backed up, and restore-tested.
    • A test workflow runs successfully on the intended worker group.
    • A controlled failed workflow sends an alert to the intended owner.
    • A controlled delayed or missed-window workflow routes to the expected owner, ticket, or report path according to criticality.
    • Queue backlog, worker slot usage, delayed starts, and missed windows are visible.
    • A known idempotent task can be rerun safely.
    • A known non-idempotent task has a documented reconciliation step before rerun.
    • Permissions prevent cross-team workflow modification where required.

    Failure Modes¶

    Failure mode Result Prevention
    Metadata database lost Workflow definitions, state, schedules, and history disappear. Back up and test restore.
    Worker saturation Schedules miss business windows while the platform appears "up." Monitor slots, backlog, delay, and worker classes.
    Unsafe retry Duplicate writes, repeated external calls, or corrupted downstream state. Record idempotency and reconciliation rules.
    Shared shell permissions One workflow can affect host or other teams. Isolate workers, users, and project permissions.
    No alert owner Failed workflows wait unnoticed. Route by owner, criticality, and time-window impact.
    Partial output after failure Downstream systems read inconsistent data. Mark commit points and recovery rules.
    Upgrade breaks metadata compatibility Scheduler starts but workflows cannot be trusted. Check version compatibility and restore path before upgrade.

    Rollback and Recovery¶

    Recovery actions include:

    • restart scheduler components
    • disable a bad workflow
    • pause a project or worker group
    • restore metadata database
    • rerun idempotent tasks
    • reconcile non-idempotent task output
    • manually complete or skip a controlled step
    • notify downstream owners when time windows are missed

    Do not blindly rerun a failed workflow that writes external state. First identify the last committed output and whether downstream consumers have observed it.

    For scheduler upgrades, rollback requires:

    • compatible metadata backup
    • previous deployment artifact
    • known worker compatibility
    • validation workflow
    • owner approval when in-flight workflows exist

    Security Notes¶

    • Restrict project and workflow edit permissions.
    • Protect data source credentials in approved secret management.
    • Avoid broad shell execution rights.
    • Separate worker users or worker groups for risky tasks.
    • Audit workflow changes for critical projects.
    • Do not publish real workflow names, database names, data source names, credentials, hostnames, or business schedules.

    Review Cycle¶

    Review this standard:

    • after missed schedules
    • after metadata incidents
    • after worker saturation
    • after unsafe retries or duplicate outputs
    • before major version upgrades
    • when onboarding new teams
    • when adding shell-heavy or high-impact workflows
    • when business time-window requirements change
