    Nacos Operations Standard¶

    Purpose¶

    This standard defines how to operate Nacos as a service discovery and runtime configuration platform.

    The core judgment:

    Nacos is a runtime control plane. A small configuration edit or registration error can change live traffic faster than a deployment pipeline can stop it.

    Nacos needs ownership, namespace boundaries, change review, service-health discipline, backup, monitoring, and rollback. Treating it as a simple key-value UI or service list creates production risk.

    When to Use¶

    Use this standard when:

    • applications depend on Nacos for service discovery
    • runtime configuration changes are delivered through Nacos
    • multiple teams share one Nacos cluster
    • Nacos is on the critical path for deployment, failover, feature flags, traffic routing, or startup
    • operators need a controlled way to restore config, remove bad instances, or recover namespace state

    Non-Goals¶

    This standard does not define application framework usage, every Nacos deployment topology, or product-specific UI steps. It defines the operating contract for a shared discovery and configuration control plane.

    Production Contract¶

    Before production systems depend on Nacos, define:

    Contract Required decision Why it matters
    Namespace boundary Which environment, team, and application family does each namespace or group represent? A shared namespace can turn one team's edit into another team's outage.
    Config ownership Who owns each high-impact config key, default value, validation, and rollback value? Runtime config changes can bypass normal deployment guardrails.
    Discovery truth What makes an instance eligible to receive traffic? Discovery is dangerous when registration does not reflect readiness.
    Change path Which changes require review, window, approval, or emergency path? A UI edit can be faster than incident response.
    Access model Who can read, write, publish, delete, or administer each namespace? Shared admin access removes accountability.
    Backup boundary What is backed up: database, namespace mapping, config exports, access policy, cluster config? Restoring only the database may not restore an operable control plane.
    Client behavior How do applications behave when Nacos is slow, unavailable, or returns stale data? Startup and failover behavior can become the actual outage.
    Rollback rule Is rollback a config revert, namespace/group revert, metadata cleanup, or database restore? Different failures require different recovery actions.

    Recommended Standard¶

    Separate configuration authority from discovery authority¶

    Nacos often carries two different forms of authority:

    • configuration authority: what values applications read at runtime
    • discovery authority: which service instances are allowed to receive traffic

    Do not manage both as an undifferentiated list of keys and instances. A configuration edit changes behavior; a discovery error changes traffic. Each needs an owner, validation path, and rollback action.

    For shared clusters, require namespace and group naming that includes environment and ownership. Production and non-production values must not share a namespace unless access, naming, and approval rules make accidental cross-use unlikely.

    Control runtime configuration as a release-class change¶

    Every high-impact config change should record:

    • config key and namespace
    • application owner
    • reason
    • expected behavior change
    • previous value
    • proposed value
    • validation method
    • rollback value
    • blast radius
    • approval or emergency justification
    • change window for sensitive paths

    Avoid direct production UI edits without a recorded path. If emergency UI changes are allowed, export or record the resulting state after the incident so the source of truth is not lost.

    Protect discovery from false health¶

    Service discovery is useful only when registered instances are eligible to receive traffic.

    Applications should:

    • register only after readiness is true
    • deregister during controlled shutdown
    • avoid advertising an endpoint before dependencies required for serving are available
    • use accurate cluster, weight, and metadata values
    • avoid stale metadata that keeps traffic on the wrong version or zone

    Monitor:

    • stale instances
    • unexpected instance count
    • incorrect weight
    • cluster or metadata mismatch
    • registration churn
    • discovery latency
    • client-side discovery errors

    Discovery validation should ask: "Would the next request be routed to an instance that can really serve it?"

    Make access smaller than namespace scope¶

    Access should follow ownership and environment, not convenience.

    Required controls:

    • authentication enabled
    • no shared production admin account for routine work
    • least-privilege roles by namespace, group, and environment
    • separate read, publish, delete, and admin permissions where possible
    • audit review for production config changes
    • emergency access with expiry and review
    • no raw application secrets stored as plain config values

    If a config item must reference a secret, prefer an approved secret-management integration or an indirection that does not expose the secret value in Nacos.

    Back up state that can actually restore the control plane¶

    Backup should include:

    • Nacos database if externalized
    • configuration exports
    • namespace and group mapping
    • user and access-policy mapping
    • cluster configuration
    • version and schema compatibility notes
    • recovery owner and restore procedure

    A backup that has never been restored is only a file. Restore must be tested in a non-production environment, including whether applications can read config and discover services after restore.

    Validation¶

    Operational checks:

    curl -fsS http://<nacos-endpoint>/nacos/v1/ns/operator/metrics
curl -fsS "http://<nacos-endpoint>/nacos/v1/ns/instance/list?serviceName=<service-name>"

    Pass conditions:

    • Nacos cluster nodes are healthy
    • datastore or database dependency is healthy and backed up
    • applications can read expected config
    • a controlled config change reaches clients and can be reverted
    • service instance count matches expected topology
    • unhealthy or draining instances are not served
    • namespace and group ownership is documented
    • production write access is limited and auditable
    • backup and restore have been tested
    • client-side Nacos errors, discovery latency, and config-refresh failures are observable

    Failure Modes¶

    Failure mode Result Prevention
    Unreviewed config edit Runtime behavior changes without deployment evidence. Require change record, previous value, validation, and rollback value.
    Shared namespace misuse One team breaks another service or environment. Separate namespace/group boundaries and access roles.
    Stale instance remains registered Traffic reaches a dead or draining endpoint. Readiness-aware registration and shutdown deregistration.
    Wrong weight or metadata Traffic shifts to the wrong version, zone, or cluster. Validate discovery metadata and observe route distribution.
    Nacos unavailable during startup Services fail to start or start with unsafe defaults. Define client fallback behavior and alert on client errors.
    No tested restore Mistaken delete or database failure becomes prolonged outage. Back up database, config exports, namespace mapping, and access policy; test restore.
    Shared admin account No reliable accountability for production edits. Use named or role-based access with audit review.
    Secrets stored as plain config Credential exposure through UI, export, or logs. Use secret management or safe indirection.

    Rollback and Recovery¶

    Rollback may mean:

    • restore the previous config value
    • revert namespace or group mapping
    • remove bad service-instance metadata
    • reset incorrect weight
    • re-publish a known-good config export
    • restore database state after destructive change
    • temporarily freeze config publication while clients stabilize

    For high-risk config changes, capture the previous value before change and validate rollback through the same client path used for release validation.

    For discovery failures, prefer targeted metadata or registration correction before broad restarts. Restarting every client can amplify the incident if the root problem is namespace, metadata, or control-plane health.

    Security Notes¶

    • Enable authentication and avoid shared admin credentials.
    • Restrict namespace access by team and environment.
    • Audit production config changes and permission changes.
    • Do not store raw secrets in plain Nacos config.
    • Protect Nacos endpoints from public exposure.
    • Restrict database and backup access.
    • Do not publish real service names, namespace names, hostnames, internal endpoints, config values, credentials, or incident evidence.

    Review Cycle¶

    Review this standard after config incidents, discovery incidents, namespace growth, access changes, version upgrades, restore tests, or when Nacos becomes a dependency for additional teams or more critical startup paths.
