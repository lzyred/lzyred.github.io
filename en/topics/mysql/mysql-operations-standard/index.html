    MySQL Operations Standard¶

    Purpose¶

    This standard defines safe, reusable practices for running self-managed MySQL in production: how to set up an instance, how to back it up so the backup is actually recoverable, how to migrate or upgrade without data loss, and how to recover under pressure.

    The core judgment of this standard:

    A backup you have never restored is a hypothesis, not a backup. A server that missed committed writes is not a rollback target until it has caught up and been validated.

    When to Use¶

    Use this standard when you:

    • Run self-managed MySQL on bare metal, a VM, or a container where you own backup and recovery.
    • Plan a version upgrade, host move, or replication-based migration.
    • Need a defensible account, backup, migration, and recovery posture for an audit or incident review.

    Non-Goals¶

    This standard does not cover:

    • Managed MySQL internals. Use the provider's backup, failover, and point-in-time-recovery contract; this standard's account model and validation principles still apply.
    • Query and schema design, indexing, or application-level performance tuning.
    • Product-specific HA automation procedures.

    Operating Principles¶

    1. Prefer a managed database service when operational maturity or on-call depth is limited.
    2. Separate administrative, migration, replication, and application accounts; grant least privilege.
    3. Never publish initial root passwords, generated credentials, or connection strings.
    4. Prove backups by restoring them on a schedule, not by checking that the backup job exited 0.
    5. Keep exactly one authoritative writer unless the application is explicitly designed for multi-writer conflict handling.
    6. Make upgrades and migrations controlled changes with a write fence, validation boundary, rollback path, and owner.

    Operating Architecture¶

    flowchart LR App[Application] --> Endpoint[Database write endpoint] Endpoint --> Primary[(Authoritative writer)] Primary --> Replica[(Replica or migration target)] Primary --> Binlog[(Binary logs)] Primary --> Backup[Base backup] Binlog --> PITR[Point-in-time recovery] Backup --> Restore[Test restore environment] PITR --> Restore Monitor[Monitoring and audit] --> Primary Monitor --> Replica Catalog[Asset and ownership catalog] --> Endpoint Catalog --> Primary Catalog --> Replica

    The operating model has one authoritative writer, one declared routing point, a separate recovery path, and observable replication. A replica is not automatically a valid failback target; it becomes one only after position and data validation.

    Single-Writer Cutover Sequence¶

    sequenceDiagram participant Change as Change controller participant App as Application participant Source as Old source participant Target as New target participant Evidence as Evidence store Change->>App: Fence new writes Change->>Source: Record final GTID or binlog position Source-->>Target: Replicate through final boundary Change->>Target: Validate threads, position, checksums Target-->>Evidence: Store validation result Change->>App: Route writes to target App->>Target: Resume writes Change->>Source: Keep non-writable during rollback window

    Product Integration¶

    Product capability Managed object Required behavior
    Asset inventory Instance, endpoint, role, version, owner, data criticality Show the authoritative writer and every replica or recovery target.
    Change management Migration plan, write fence, final GTID, approval, rollback window Coordinate one controlled cutover rather than independent operator actions.
    Backup and recovery Base backup, binlog range, restore test, RPO/RTO result Prove recoverability and expose the latest usable recovery point.
    Secret and access management Application, admin, migration, replication accounts Separate identities, limit source hosts, and rotate emergency credentials.
    Observability Replication threads, errors, lag, connection pressure, restore status Detect when a target is not safe to promote.
    Evidence store Checksums, row invariants, GTID boundary, route change, reviewer Preserve the proof used to authorize cutover or failback.

    Installation Baseline¶

    For self-managed MySQL:

    • Use a supported MySQL version with a defined end-of-life date.
    • Run as a dedicated mysql system user; keep data on a dedicated volume.
    • Set character set and collation explicitly (utf8mb4 unless you have a specific reason).
    • Enable slow query logging and a reviewed long_query_time.
    • Manage the service through systemd so restart and boot behavior are predictable.
    [mysqld]
datadir=/var/lib/mysql
socket=/var/lib/mysql/mysql.sock
character-set-server=utf8mb4
collation-server=utf8mb4_unicode_ci
slow_query_log=ON
long_query_time=1
max_connections=500

    Size max_connections to what the host can actually serve; an unreachable connection ceiling just converts a load spike into a different failure. Treat a collation change as an application-compatibility change, not as incidental migration cleanup.

    Account Model¶

    Use least privilege and scope hosts tightly:

    CREATE USER 'app_user'@'<app-host-or-approved-subnet>' IDENTIFIED BY '<from-secret-manager>';
GRANT SELECT, INSERT, UPDATE, DELETE ON example_db.* TO 'app_user'@'<app-host-or-approved-subnet>';
FLUSH PRIVILEGES;
    • The password placeholder must be supplied through a secret manager, never committed.
    • Application users get DML on their schema only — not ALL PRIVILEGES, not *.*.
    • Scope the host part to the narrowest approved source; do not use broad private-network wildcards as a convenience default.
    • Administrative, migration, and replication accounts are separate and used deliberately, not shared with the application.

    Backup Standard¶

    Choose the backup method by dataset size and recovery objective.

    Method Good for Restore characteristic
    Logical (mysqldump / mysqlpump) Small/medium datasets, portability, single-table recovery Slower restore; replays SQL
    Physical (for example Percona XtraBackup) Large datasets, fast full restore Fast restore; version/layout sensitive
    Binary logs Point-in-time recovery on top of a base backup Requires complete retention and tested replay

    Logical backup example for transactional tables:

    mysqldump --single-transaction --routines --triggers --events example_db > example_db.sql

    For every backup, document: type, schedule, retention, encryption, restore-test frequency, binary-log coverage, and the owner/escalation path. Combine a periodic base backup with binary logs when you need point-in-time recovery.

    Prove the backup¶

    On a schedule, restore the latest backup into an isolated instance and validate it:

    • Row counts on critical tables match expectations.
    • Application-level integrity checks or checksums pass.
    • Routines, triggers, events, users, and grants in the recovery scope are present.
    • Point-in-time replay reaches a chosen timestamp when required by the RPO.
    • Record the restore duration and compare it with the RTO.

    Migration and Upgrade¶

    Before:

    • Confirm source and target versions and the supported upgrade path; do not skip unsupported jumps.
    • Check character set and collation compatibility.
    • Estimate data size, restore time, transfer time, and required downtime.
    • Validate network throughput for the transfer.
    • Freeze incompatible schema changes for the window.
    • Write the cutover and failback plan and confirm the owner.

    Replication-based migration is usually safer than a stop-the-world dump: replicate to the target, let it catch up, then cut over during a controlled window.

    Replication validation¶

    SHOW REPLICA STATUS\G   -- MySQL 8.0.22+
SHOW SLAVE STATUS\G     -- older versions
SELECT COUNT(*) FROM important_table;

    Do not use Seconds_Behind_Source = 0 as the only cutover proof. Also confirm:

    • the I/O and SQL/applier threads are healthy;
    • Last_Error is empty;
    • the executed binary-log coordinates or GTID set cover the agreed boundary;
    • no replication filter silently excludes required data;
    • checksums or application invariants match for critical data.

    Single-writer cutover contract¶

    1. Stop or fence application writes to the old source.
    2. Record the final accepted source position or GTID set.
    3. Wait until the target has applied through that boundary.
    4. Complete final integrity checks.
    5. Redirect application writes to the target.
    6. Keep the old source non-writable until the rollback window closes.
    7. Monitor write success, replication state, error logs, and application invariants.

    Do not allow both servers to accept uncontrolled writes merely to shorten the window. Without conflict resolution, that creates divergent histories that cannot be repaired by changing a connection string.

    Break-Glass: Root Password Reset¶

    Root password reset (--skip-grant-tables or equivalent) is a break-glass procedure, not routine maintenance. It removes authentication from the running server, so require:

    • Approval and a maintenance window.
    • Console or host-level access; never expose the bypassed server over an untrusted network.
    • An audit record of who did it and why.
    • Immediate credential rotation and a return to normal authentication afterward.

    Do not publish environment-specific reset passwords or internal access paths.

    Validation¶

    • Service is up via systemd; the error log shows no crash-recovery loop.
    • Replication, if used, has healthy threads, no error, and an understood executed position.
    • Only the intended server accepts application writes.
    • The most recent restore test passed within its scheduled interval and RTO.
    • Slow query logging is being written and reviewed.
    • The application connects with a least-privilege account, not an administrative account.

    Known Failure Modes¶

    Failure mode Symptom Likely cause Correct response
    Backup completes but will not restore Restore fails or data is incomplete Never restore-tested; missing routines, events, users, or binlogs Restore-test the complete recovery contract.
    Replication stops silently Lag grows; Last_Error is set Schema drift, conflicting writes, filter error, or binlog gap Read replication status; fix the root cause; reseed if required.
    Replica appears caught up but data differs Lag is zero but checksum or row count differs Lag metric treated as proof; replication error or excluded data Stop cutover; verify coordinates, GTIDs, filters, and data.
    Split writes after cutover Source and target each contain unique rows Old source stayed writable or clients were not drained Fence one writer; preserve both histories; reconcile before failback.
    Connection storm Too many connections or memory pressure Leaked connections, weak pooling, or an unrealistic limit Fix pooling and size limits to host capacity.
    Character set corruption Mojibake or changed sort behavior Source/target charset or collation mismatch Verify end-to-end compatibility before migrating.
    Upgrade rollback fails Previous binary cannot open upgraded data files In-place downgrade was assumed safe Restore a compatible pre-upgrade backup or use the supported downgrade path.

    Rollback and Failback¶

    Before the target accepts writes, rollback normally means leaving or returning applications to the still-authoritative source after proving that no write crossed the fence.

    After the target has accepted writes, failback is a data-movement operation, not a connection-string operation:

    1. Stop or fence target writes and record the final target position.
    2. If versions and topology permit, make the old source a replica of the new source so it receives the writes it missed.
    3. Validate replication position, checksums, and application invariants.
    4. Promote the old source only through a second controlled cutover.
    5. If reverse replication is not compatible, export and reconcile target-only writes or continue forward on the target; do not silently discard them.

    For a failed in-place upgrade, restore the pre-upgrade backup onto the prior supported version. Do not assume upgraded data files can be downgraded in place.

    Security Notes¶

    • Bind to approved private interfaces; never expose MySQL directly to the public internet.
    • Encrypt backups and restrict who can read or restore them.
    • Keep credentials in a secret manager; do not pass them on a command line where they can leak into process lists or shell history.
    • Audit administrative and replication access; rotate credentials after any break-glass event.

    Review Cycle¶

    • Owner: the database/platform team.
    • Verify a successful restore test at the scheduled interval.
    • Re-confirm the supported upgrade and replication path before each migration.
    • Re-check account privileges when a new service starts using the database.
