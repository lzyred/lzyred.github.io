    Anonymous Object Storage Migration Case¶

    All identifiers in this case are generic. No original bucket names, domains, regions, customer data, access keys, prefixes, or application names are preserved.

    What This Case Is Really About¶

    The migration looked like a data-copy problem. It was really an authority-change problem.

    Before the write cutover, the source bucket was still the system of record. A failed attempt could usually roll back by pointing reads back to the source. After the write cutover, new objects and updates began landing on the target. From that moment, rollback stopped being a routing change and became reconciliation.

    That single boundary shaped the whole plan. The team moved all dangerous uncertainty before the write cutover and made the cutover the last small step, not the celebration after a large copy job.

    Decision Frame¶

    Decision question Why it mattered Required answer
    What is the source of truth at each phase? Without an authority model, rollback language becomes vague. Source is authoritative before write cutover; target becomes authoritative only after explicit approval.
    How will correctness be proven? Object count and total bytes can match while specific objects, metadata, or access behavior are wrong. Manifest, delta ledger, checksum sampling, metadata checks, and application reads.
    What changes after writes move? Write cutover changes recovery from repointing to reconciling. A rollback and forward-reconciliation plan exists before target writes begin.
    How are fresh writes kept visible during read cutover? Reading from target while writing to source can hide newly written objects if replication lag is ignored. Fresh-write paths are fenced, read-through/fallback-enabled, or explicitly outside the read-after-write contract until the final delta closes.

    The case became safe when the team stopped asking "did the sync finish?" and started asking "which store currently has the right to accept truth?"

    Starting Conditions¶

    • The application continuously read and wrote objects on the source provider.
    • Object prefixes carried application meaning.
    • Some objects had metadata and access behavior that mattered to the application, not just bytes.
    • There was no maintenance window long enough for a full freeze-copy-switch migration.
    • Long-lived migration credentials were not acceptable.

    Constraints¶

    • The application had to remain available.
    • Data inconsistency had to be bounded and observable.
    • Credentials used for migration had to be temporary and revocable.
    • Source and target layout differences had to be reviewed before the first bulk copy.
    • The source had to remain available as a rollback path until after the write cutover observation period.

    Migration Model¶

    The team used three ledgers. This was the most important design choice.

    Ledger Purpose Failure it prevents
    Inventory manifest Declares which buckets, prefixes, object classes, metadata, and access rules are in scope. Copying the wrong shape of data and discovering missing prefixes late.
    Sync ledger Records bulk-copy completion, delta windows, retries, mismatches, and unresolved objects. Treating "command finished" as equivalent to "migration correct."
    Application validation ledger Records real application reads, sampled writes, metadata behavior, and permission checks. Migrating bytes that the application still cannot use.

    The ledgers were not bureaucracy. They were the only way to know whether a rollback would lose data or merely change routing.

    Execution Sequence¶

    1. Build the inventory manifest from source buckets, prefixes, object counts, byte totals, metadata requirements, lifecycle policies, and access rules.
    2. Create target buckets, policies, lifecycle behavior, encryption settings, and temporary migration credentials.
    3. Run the initial bulk sync while the application continued reading and writing from the source.
    4. Record mismatches and retry failures in the sync ledger instead of hiding them in tool logs.
    5. Run delta sync for objects changed during the bulk copy, tracking create, update, delete, overwrite, and tombstone or versioning semantics.
    6. Validate counts and sizes, then perform stratified checksum sampling across prefixes, object sizes, object ages, and storage classes.
    7. Run application read tests against the target without moving write authority.
    8. Move reads to the target only after fresh-write visibility is handled through fast enough replication, read-through fallback, write fencing, or an explicit statement that the affected path has no read-after-write requirement.
    9. Freeze, fence, or tightly sequence writes for the final delta window.
    10. Move writes to the target only after unresolved mismatches were accepted or cleared.
    11. Keep the source read-only through the rollback window and revoke temporary credentials after closure.

    Validation Evidence¶

    Evidence Why count and size were not enough
    Prefix-level manifest comparison A global count can hide a missing or misplaced prefix.
    Stratified checksum sampling Total bytes can match while selected objects differ. Sampling by prefix, size, and age catches more than random spot checks alone.
    Metadata comparison Content-Type, cache headers, encryption, and ACL behavior can break applications even when bytes match.
    Mutation semantics Deletes, overwrites, tombstones, object versions, and list consistency decide whether the target represents current truth or only copied history.
    Application reads SDK behavior, signed URLs, range reads, redirects, and permission paths are not proven by storage CLI output.
    Final delta ledger The dangerous objects are the ones changing during the migration, not the cold archive that copied cleanly.
    Write-cutover observation New writes must be readable from the target, visible to downstream jobs, and absent from unexpected source paths.

    Failure Modes That Surfaced¶

    Count and Size Matched While a Sample Failed¶

    The first comparison looked clean at aggregate level. A checksum sample found mismatches in a small set of objects. The issue was not large enough to move the global totals, but it was large enough to prove that the team could not trust aggregate numbers alone.

    Those objects were re-synced and the sample strategy was widened. The team kept the mismatch record instead of erasing it from memory, because that record later justified why the write cutover could proceed.

    Prefix Mapping Was a Product Contract¶

    One prefix assumption did not match how the target layout had been prepared. If the team had discovered it after bulk sync, the cost would have been a large re-copy. If it had discovered it after write cutover, the cost would have included reconciliation and possible application confusion.

    The inventory manifest made prefix mapping a pre-copy decision. This was the right place to find the mistake.

    Metadata Was Part of the Data¶

    Some application behavior depended on metadata and access semantics rather than only object bytes. A byte-perfect migration could still have broken responses, caching behavior, or client downloads.

    The team added application read tests before write cutover, which caught issues a storage-level diff would not have seen.

    Credentials Needed an End Date¶

    Temporary migration access was convenient enough that it could have become permanent by accident. The team treated credential revocation as part of the migration closure, not as a cleanup someday.

    Reversibility Boundary¶

    The migration had two rollback regimes:

    Phase Authority Rollback cost
    Before read cutover Source accepts reads and writes. No customer-facing rollback; target can be rebuilt.
    After read cutover, before write cutover Source still accepts writes; target serves reads. Repoint reads to source and investigate target behavior.
    After write cutover Target accepts new writes, updates, deletes, and overwrites. Rollback requires reverse sync or reconciliation; routing alone is not enough, and source read-only by itself does not preserve target-side mutations.

    The final write cutover was therefore treated as the smallest, most controlled step. Everything possible was proven before it.

    Outcome¶

    The migration completed without extended outage and without detected data loss. More importantly, the team knew why it was safe to continue at each phase. Counts, sizes, checksums, metadata checks, application reads, and delta reconciliation all answered different questions.

    The lasting lesson was that object storage migration should be designed around authority and evidence. Copy tools move bytes; migration plans move responsibility for truth.

    Reusable Review Questions¶

    • Which store is authoritative before, during, and after each phase?
    • Which prefixes or object classes would be expensive to discover late?
    • Are metadata, ACLs, lifecycle rules, encryption settings, and signed URL behavior part of correctness?
    • How are creates, updates, deletes, overwrites, tombstones, and object versions tracked during bulk copy?
    • During read cutover, what guarantees that a newly written object remains visible to the reader?
    • What test proves the application can read from the target, not merely that the target contains bytes?
    • What exactly happens if the write cutover fails after new writes have landed?
    • When are temporary migration credentials revoked?
