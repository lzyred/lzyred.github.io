    SigNoz Observability Standard¶

    Purpose¶

    This standard defines how to use SigNoz or a similar OpenTelemetry-compatible backend as part of a production observability system.

    The core judgment:

    SigNoz is not observability by itself. It is a place where telemetry can be queried. It becomes production observability only when ingestion, labels, storage, access, alerts, ownership, and failure validation are designed as an operating contract.

    A unified traces, metrics, and logs interface is useful. It is also dangerous if the team treats a working UI as proof that failures will be found during an incident.

    When to Use¶

    Use this standard when:

    • Services export OpenTelemetry traces, metrics, or logs.
    • A team needs a trace-first backend for service latency, error, and dependency analysis.
    • The platform wants one controlled telemetry entry point behind OpenTelemetry Collector.
    • The organization accepts the operational cost of running a ClickHouse-backed observability stack.
    • SigNoz is used alongside, not as an accidental replacement for, existing alerting and log standards.

    SigNoz is strongest when the main reader task is "why did this request path become slow or fail?" It is weaker as the only source of infrastructure alerting, long-retention log archive, or compliance audit evidence.

    Non-Goals¶

    This standard does not:

    • Teach OpenTelemetry instrumentation. See the OpenTelemetry and Java instrumentation documents below.
    • Replace Prometheus alerting standards for infrastructure and SLO alerts.
    • Replace Loki or another log platform when the organization needs dedicated log retention and query controls.
    • Approve sending sensitive request bodies, headers, tokens, user identifiers, or customer data into traces or logs.
    • Treat a demo deployment as production-ready.

    Decision Model¶

    Decide the role before deploying. The role decides the operating contract.

    Role Acceptable use Required boundary
    Lab tool Learn OpenTelemetry, validate instrumentation, explore traces from sample services. No production traffic, no sensitive data, disposable storage.
    Team analysis backend Debug selected services and correlate traces with metrics or logs. Collector boundary, access control, retention, owner, and rollback path.
    Production observability service Support incident diagnosis for real services. Storage lifecycle, backup or rebuild decision, alert routing, SLO impact checks, capacity planning, and failure drills.

    If the team cannot name which role it is deploying, keep the deployment out of production paths.

    Production Contract¶

    Before SigNoz is allowed to receive production telemetry, define these contracts:

    Contract Required decision Why it matters
    Telemetry ownership Which team owns collectors, backend, dashboards, alerts, and retention? During an incident, "the UI is down" must have an owner.
    Collector boundary Do applications send to an OpenTelemetry Collector, or directly to SigNoz? The collector is where filtering, batching, retry, memory limits, and export routing should be controlled.
    Attribute policy Which resource attributes are required and which attributes are forbidden? Bad labels make search useless; unsafe attributes leak data.
    Storage lifecycle What retention, volume, backup, and rebuild model does ClickHouse use? A trace backend can quietly become a large stateful database.
    Access model Who can view traces, logs, dashboards, and alerts? Telemetry may contain production metadata and user-adjacent context.
    Alerting model Which alerts belong in SigNoz and which stay in Prometheus or another alert path? Tool overlap creates alert gaps and duplicate noise.
    Self-monitoring How do SigNoz, ClickHouse, and collector health alert through a path outside SigNoz? The observability backend cannot be the only tool that reports its own failure.
    Sampling and cardinality Who owns sampling, which traces must be preserved, and what cardinality is forbidden? A backend that samples away the failure or explodes on labels is not an incident tool.
    Failure validation What real test proves the path from application to collector to backend to human? A green pod only proves a process is running.
    Rollback boundary How do applications stop exporting to this backend without redeploying unrelated changes? Observability rollout must be reversible without breaking applications.

    Recommended Standard¶

    Put OpenTelemetry Collector in front¶

    Applications should send telemetry to an OpenTelemetry Collector service. The collector then exports to SigNoz.

    This gives the platform one controlled place to:

    • Drop or redact sensitive attributes by default.
    • Hash only approved low-risk pseudonymous identifiers after data-classification review; never hash tokens, authorization headers, cookies, request bodies, full URLs, or secrets as a safety substitute.
    • Set memory limits, queues, retry, and batch behavior.
    • Normalize service.name, namespace, environment, team, and version attributes.
    • Route the same telemetry to another backend during migration or rollback.
    • Pause or throttle export without changing application manifests.

    Do not point application agents at localhost, a developer machine, a kubectl port-forward, or an unowned endpoint. The production endpoint must be a stable service name and port owned by the platform.

    Treat ClickHouse as production state¶

    SigNoz commonly depends on ClickHouse for telemetry storage. ClickHouse may not hold business records, but it still holds incident evidence, performance history, and sensitive metadata.

    Production deployment requires:

    • Pinned chart and image versions.
    • Persistent volumes with known expansion behavior and reclaim policy.
    • Retention policy for traces, metrics, and logs.
    • Resource requests and limits for ClickHouse, collector, query service, frontend, and alerting components.
    • Backup or rebuild decision. If the team chooses not to back up telemetry storage, that must be an explicit loss-acceptance decision.
    • Capacity alerts for disk, ingestion rate, dropped spans, query latency, and queue pressure.

    The unsafe pattern is letting telemetry storage grow until the observability system becomes the next incident.

    Separate three signal jobs¶

    SigNoz can display traces, metrics, and logs, but those signals do different work.

    Signal Strong use Guardrail
    Traces Request path, latency breakdown, dependency errors, service topology. Sampling, safe attributes, consistent service names.
    Metrics Service-level latency, error, throughput, and backend health. Keep SLO and infrastructure alert ownership explicit.
    Logs Context around selected failures and trace correlation. Do not use trace backend logging as an unmanaged long-retention log archive.

    Do not enable every signal everywhere just because the UI supports it. Each signal adds ingestion cost, data risk, and operational ownership.

    Make attributes searchable before adding volume¶

    The first production value of SigNoz depends less on chart installation and more on whether telemetry can be filtered during an incident.

    Require safe resource attributes:

    • service.name
    • service.namespace
    • deployment.environment
    • service.version
    • k8s.namespace.name or equivalent platform namespace
    • team or owner label where the organization uses one

    Avoid unsafe or high-cardinality attributes:

    • user IDs, emails, tokens, authorization headers, cookies
    • request bodies, full URLs, raw SQL, customer names
    • order IDs, request IDs, session IDs as indexed dimensions
    • internal hostnames or private topology details in public examples

    Bad attributes are not a cosmetic problem. They decide whether a real incident can be narrowed down quickly and whether telemetry is safe to retain.

    Govern sampling and cardinality before rollout¶

    Sampling is not only a cost setting. It decides which evidence survives an incident.

    Before production traffic enters SigNoz, define:

    • Sampling owner and approval path.
    • Default sampling policy for normal traffic.
    • Preservation rule for error traces, slow traces, and explicitly marked diagnostic traffic.
    • Rollout limit for how many services or how much traffic may be added in one change.
    • Cardinality budget for resource attributes and span attributes.
    • Validation that synthetic slow requests and synthetic error requests are not sampled away.

    Do not tune sampling only after ClickHouse or the collector is under pressure. By then the system may already have dropped the trace that explains the pressure.

    Kubernetes Deployment Standard¶

    Before deployment:

    • Classify the role: lab, team analysis backend, or production observability service.
    • Confirm where the OpenTelemetry Collector runs and which receiver/exporter ports are used.
    • Confirm the default StorageClass supports the required volume expansion and reclaim behavior.
    • Pin the Helm chart and image versions.
    • Create a dedicated namespace.
    • Decide ingress, authentication, TLS, and network access before exposing the frontend.
    • Review resource requests and limits for ClickHouse, collector, Alertmanager or alerting service, frontend, and query service.
    • Decide retention and capacity alerts before production traffic arrives.

    Deployment is not complete until a real instrumented workload can send telemetry through the intended collector path and a real operator can find it through the intended access path.

    Validation¶

    Validate in layers and stop at the first unclear layer.

    Layer Evidence Stop condition
    Backend health SigNoz frontend, query service, ClickHouse, collector, and alerting components are ready. Any core component is unhealthy or repeatedly restarting.
    Storage ClickHouse tables initialize, PVCs bind, retention is configured, disk alerts exist. Storage cannot grow, retention is absent, or disk pressure has no owner.
    Collector path Test workload sends OTLP telemetry to the collector and collector exports to SigNoz without queue saturation or drops. Export errors, queue pressure, memory limiter drops, or protocol mismatch.
    Attribute quality Test traces show safe service.name, environment, namespace, version, and owner attributes. Service appears as unknown, unsafe attributes leak, or labels are too noisy to search.
    Sampling and cardinality Synthetic slow and error requests remain queryable, and high-cardinality attributes are blocked before broad rollout. The test failure is sampled away, or labels create unbounded series and query pressure.
    Access and alerting A real operator reaches the UI through the approved path and a test alert reaches the intended channel. Access depends on port-forward, shared admin access, or unowned alert routing.
    Self-monitoring SigNoz, ClickHouse, and collector health, drops, queue pressure, disk pressure, and query failures alert through a path that does not depend on SigNoz. SigNoz failure can only be noticed inside SigNoz itself.
    Incident usefulness A synthetic slow/error request can be found from service to trace to related logs or metrics. The UI has data, but the failure cannot be located by service, environment, version, or owner.

    Passing pod readiness alone is not enough. The minimum useful proof is a known failure signal traveling from workload to collector to backend to human.

    Rollback and Exit¶

    Rollback depends on where SigNoz sits in the telemetry path.

    Situation Safer rollback
    Applications export to collector, collector exports to SigNoz Remove or disable the SigNoz exporter in the collector; keep application instrumentation unchanged.
    Applications export directly to SigNoz Change application endpoint configuration and redeploy affected workloads. This has higher blast radius.
    SigNoz deployment is unhealthy but telemetry is still useful elsewhere Route collector export to the alternate backend and pause SigNoz ingestion.
    SigNoz is being removed permanently Stop exporters first, wait for no dependent workloads, then uninstall the release. Delete PVCs only after explicit approval.

    Do not delete PVCs as part of a casual rollback. They may contain incident evidence and retention-controlled telemetry.

    If the stack was used only as a lab tool, deletion is acceptable after confirming no production collector exports to it.

    Failure Modes¶

    Failure mode Symptom Correct response
    UI works but no useful diagnosis Traces exist, but service, version, environment, or owner cannot be filtered. Fix resource attributes before adding more services.
    Collector bypassed Applications export directly to backend, with no filtering or routing control. Move export behind OpenTelemetry Collector.
    ClickHouse disk pressure Ingestion slows, queries fail, or pods restart. Reduce ingestion, check retention, expand storage, and review sampling/cardinality.
    Sensitive telemetry leak Traces or logs contain tokens, headers, request bodies, user data, or customer context. Stop the offending export, filter at collector and application layers, and review retention.
    Demo access becomes production access Operators depend on kubectl port-forward or shared admin credentials. Create controlled ingress, authentication, and authorization before production use.
    Alerting split brain Prometheus and SigNoz both alert on similar symptoms, or neither owns them. Define alert ownership and route each symptom to one response path.
    Self-monitoring depends on SigNoz Backend, ClickHouse, or collector failure is only visible inside the failed stack. Alert SigNoz health, ingestion drops, queue pressure, disk pressure, and query failures through an external path.
    Sampling hides the incident Slow or failed synthetic requests cannot be found in traces. Preserve error/slow diagnostic traffic, lower rollout scope, and review sampling before adding services.
    Backend overload after rollout Dropped spans, collector queue pressure, high ClickHouse write rate. Lower sampling, pause rollout, and validate collector memory and queue limits.
    Logs treated as free add-on Cost and retention grow without log ownership. Keep logs in the approved log platform unless SigNoz log retention and access are owned.

    Security Notes¶

    • Treat telemetry as production data export.
    • Restrict UI access; avoid shared admin accounts.
    • Keep backend credentials in collectors or secret management, not in application manifests.
    • Filter request bodies, authorization headers, cookies, tokens, user identifiers, and full URLs.
    • Use TLS and network controls for frontend and collector paths.
    • Limit who can change collector pipelines, retention, and alert routes.
    • Do not publish real service names, namespaces, domains, user identifiers, dashboard screenshots, or query results.

    Review Cycle¶

    Review this standard:

    • before moving from lab to team or production role
    • when enabling a new signal type, such as logs or metrics
    • when adding a high-traffic service
    • after collector drops, storage pressure, or alert-routing failure
    • during OpenTelemetry Collector, SigNoz, ClickHouse, or Kubernetes upgrades
    • when retention, access policy, or data classification changes
