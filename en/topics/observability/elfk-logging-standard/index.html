    ELFK Logging Standard¶

    Purpose¶

    This standard defines how to run an ELFK (Elasticsearch, Logstash, Filebeat, Kibana) logging stack so that it stays available and affordable, and does not become an outage or a data-leak vector itself.

    The core judgment of this standard:

    A logging stack fails in two predictable ways: it fills its own disk and goes read-only (taking your logs down exactly when you need them), or it quietly indexes secrets and personal data into a searchable store. Retention/lifecycle and what-you-index are not tuning — they are the design.

    When to Use¶

    Use this standard when you:

    • Run a self-managed Elasticsearch-based logging stack (ELFK/ELK or OpenSearch-equivalent).
    • Need controlled retention, index lifecycle, and access to logs.
    • Decide what to collect and how to keep the store from filling up.

    Non-Goals¶

    This standard does not cover:

    • Choosing between ELFK and a metrics-style logging system (Loki) — see the Loki logging standard for that trade-off.
    • Full-text search relevance tuning or APM.
    • Cluster sizing math beyond the capacity-safety principles below.

    Core Principles¶

    1. Use structured logs (JSON with consistent fields) so logs are queryable, not just greppable.
    2. Control retention with Index Lifecycle Management (ILM) — every index has a defined rollover and delete policy.
    3. Protect against the disk watermark: Elasticsearch goes read-only when disk fills, so capacity and retention must keep you below it.
    4. Restrict Kibana and the Elasticsearch API; logs are sensitive.
    5. Do not index secrets, tokens, credentials, full request bodies, or personal data.

    Recommended Standard¶

    Structured logs and a consistent shape¶

    Emit JSON logs with stable fields (timestamp, level, service, env, trace id). Do the parsing/enrichment in Filebeat or Logstash, not by indexing free-form text — unstructured logs explode storage and are hard to query.

    Retention via ILM¶

    Define an ILM policy so indices roll over by size/age and are deleted on schedule. Without it, indices grow until the disk watermark trips and the cluster goes read-only.

    PUT _ilm/policy/logs-policy
{
  "policy": {
    "phases": {
      "hot":    { "actions": { "rollover": { "max_size": "50gb", "max_age": "1d" } } },
      "delete": { "min_age": "14d", "actions": { "delete": {} } }
    }
  }
}

    Set retention from a deliberate decision (cost vs. how far back you actually investigate), not "keep everything".

    Capacity and the disk watermark¶

    Elasticsearch has disk watermarks (low/high/flood-stage). At flood stage it sets indices read-only — a logging outage that often coincides with the incident you are trying to debug. Keep headroom: size storage for peak ingest × retention, alert on disk usage well before the watermark, and ensure ILM deletion is actually running.

    Access control¶

    • Put Kibana behind authentication and least-privilege roles; do not expose it openly.
    • Restrict the Elasticsearch HTTP API to trusted networks/clients.
    • Logs frequently contain sensitive data — treat read access as a privilege.

    Validation¶

    curl -sS http://<es-host>:9200/_cluster/health?pretty
curl -sS http://<es-host>:9200/_cat/indices?v
curl -sS http://<es-host>:9200/_cat/allocation?v   # disk usage per node

    Confirm:

    • Cluster health is green (or a justified yellow), not red.
    • ILM is rolling over and deleting old indices on schedule (_cat/indices shows bounded growth).
    • Disk usage is below the high watermark with headroom.
    • Filebeat/Logstash are shipping; no large ingest backlog.
    • Spot-check that logs do not contain secrets or full sensitive payloads.

    Known Failure Modes¶

    Failure mode Symptom Likely cause Correct response
    Cluster read-only Indexing stops; read_only_allow_delete flag Disk hit flood-stage watermark Free space / delete old indices; clear the block; fix retention.
    Unbounded growth Disk fills steadily No ILM, or ILM delete not running Apply ILM with rollover + delete; verify it executes.
    Storage blowup Cost/disk spikes Indexing unstructured/high-cardinality fields Structure logs; drop noisy fields at Filebeat/Logstash.
    Secrets in index Sensitive data searchable Indexed request bodies/headers/tokens Filter at the shipper; purge affected indices; fix the source.
    Kibana exposed Unauthorized log access No auth / open network Put Kibana behind auth; restrict network.
    Red cluster Shards unassigned Node loss, disk, or shard allocation issue Check _cat/allocation and allocation explain; restore capacity.

    Rollback / Recovery¶

    1. To recover a read-only cluster: free disk (delete or move old indices), then clear the read_only_allow_delete block; afterwards fix ILM/retention so it does not recur.
    2. To roll back an ILM/policy change, restore the previous policy; ILM applies going forward, so verify index ages against the intended retention.
    3. Keep snapshot backups of indices you must retain for compliance; restore from snapshot rather than relying on a live cluster alone.

    Security Notes¶

    • Never index secrets, tokens, credentials, authorization headers, full request bodies, or personal data — filter at Filebeat/Logstash before they reach the store.
    • Restrict Kibana and the ES API; use TLS and authentication.
    • Do not publish real hostnames, endpoints, or index contents.
    • Treat snapshots as sensitive (they contain log data).

    Review Cycle¶

    • Owner: the platform/observability team.
    • Re-confirm retention vs. cost and investigation needs on a regular cadence.
    • Verify ILM rollover/delete is actually running and disk headroom holds.
    • Re-check what fields are indexed when a new log source is onboarded.
