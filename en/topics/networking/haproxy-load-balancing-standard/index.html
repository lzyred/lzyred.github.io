    HAProxy Load Balancing Standard¶

    Purpose¶

    This standard defines safe patterns for using HAProxy to distribute TCP/HTTP traffic across backends in production.

    The core judgment of this standard:

    A load balancer is only as good as its health check. A check that passes when the app is not actually ready will route users straight into errors; a check that is too aggressive will flap healthy backends out of rotation. The health check is the design, not a detail.

    When to Use¶

    Use this standard when you:

    • Balance traffic across multiple backends (HTTP or raw TCP).
    • Need a single validated reload procedure and a draining model for safe backend changes.
    • Are deciding between HAProxy (connection/L4-L7 balancing) and an application proxy.

    Non-Goals¶

    This standard does not cover:

    • TLS termination patterns and HTTP header handling in depth — see the Nginx reverse proxy + TLS standard; the validated-reload and certificate-lifecycle principles there apply.
    • Kubernetes Ingress controllers.
    • WAF or DDoS mitigation.

    Core Principles¶

    1. Match the health check to real application readiness, not just "port is open".
    2. Validate config (haproxy -c) before every reload; reload (not restart) to avoid dropping connections.
    3. Choose L4 (mode tcp) vs L7 (mode http) deliberately — you only get HTTP-aware features in L7.
    4. Drain backends before maintenance instead of yanking them.
    5. Restrict the stats/admin socket; never publish real backend addresses.

    Recommended Standard¶

    Frontend, backend, and a readiness-matching check¶

    frontend example_http
  bind *:80
  mode http
  default_backend example_backend

backend example_backend
  mode http
  balance roundrobin
  option httpchk GET /health           # check a real readiness endpoint
  http-check expect status 200
  default-server inter 3s fall 3 rise 2 # fail after 3 bad, recover after 2 good
  server app-a <backend-a-host>:8080 check
  server app-b <backend-b-host>:8080 check
    • option httpchk GET /health plus http-check expect status 200 means a backend is only "up" when its readiness endpoint actually returns 200 — not merely when the TCP port accepts a connection.
    • fall/rise give hysteresis so a single blip does not eject a healthy backend, and a single good response does not prematurely re-add a flapping one.

    L4 vs L7¶

    Mode Use for Note
    mode http (L7) HTTP services needing path/host routing, header logic, HTTP health checks Slightly more overhead; HTTP-aware
    mode tcp (L4) Databases, message brokers, TLS passthrough, non-HTTP protocols No HTTP features; check with option tcp-check

    Balancing algorithm¶

    roundrobin is a fine default. Use leastconn for long-lived connections (databases, websockets) where request duration varies widely. Persistent sessions (cookie/stick-table) only when the app genuinely needs affinity — sticky sessions undermine even balancing and complicate draining.

    Change Procedure¶

    1. Edit the config.
    2. Validate:
    haproxy -c -f /etc/haproxy/haproxy.cfg
    1. Reload (graceful — existing connections finish):
    systemctl reload haproxy
systemctl status haproxy --no-pager
    1. To take a backend out for maintenance, drain it first (set it to MAINT/drain via the admin socket or config) so in-flight requests complete, then remove it. Do not delete a live backend and reload — that drops its connections.

    Validation¶

    haproxy -c -f /etc/haproxy/haproxy.cfg     # config valid
# via stats socket (restricted): show backend/server state
echo "show stat" | socat stdio /run/haproxy/admin.sock | cut -d, -f1,2,18

    Confirm:

    • All intended servers show UP; none stuck DOWN or flapping.
    • The health check targets a real readiness endpoint and reflects app state.
    • After a reload, no connection errors or 5xx spike; traffic is distributed as intended.

    Known Failure Modes¶

    Symptom Likely cause Check / response
    Users hit errors despite "healthy" LB Health check passes before app is ready Point httpchk at a true readiness endpoint; add expect status.
    Backends flap in/out fall/rise/inter too aggressive, or backend overloaded Add hysteresis; investigate backend latency.
    Connections dropped on change Backend removed + reload instead of drain Drain to MAINT first, let connections finish, then remove.
    Uneven load Sticky sessions or roundrobin with long-lived conns Use leastconn; reconsider session affinity.
    TCP service mis-checked L4 service with HTTP check Use mode tcp + option tcp-check.
    Restart caused an outage Used restart, not reload Always reload for config changes.

    Rollback¶

    1. Keep the previous config under version control or as a timestamped copy.
    2. Restore the prior config, haproxy -c, then systemctl reload haproxy.
    3. If a backend change went wrong, re-enable the drained/removed backend and reload; confirm it returns to UP.
    4. If a reload left HAProxy degraded and config won't validate, restore the last-known-good file before debugging further.

    Security Notes¶

    • Restrict the stats page and admin socket to trusted operators/networks; they can change routing.
    • Do not publish real backend addresses, hostnames, or the admin socket path.
    • For TLS termination, use a managed certificate lifecycle (see the Nginx + TLS standard).
    • Keep HAProxy patched; it is on the request path and often internet-facing.

    Review Cycle¶

    • Owner: the platform/networking team.
    • Re-validate that health checks still reflect readiness when an app's startup behavior changes.
    • Review balancing algorithm and affinity when traffic patterns change.
    • Confirm stats/admin access controls on a regular cadence.
