    Kubernetes Cluster Provisioning Standard¶

    Executive summary¶

    A production cluster is not a set of kubeadm init or cloud-console parameters. It is an operated platform product that must deliver version boundaries, networking and identity, failure domains, data protection, access paths, observability, upgradeability, and recovery.

    mindmap root((Kubernetes Platform)) Version Control plane kubelet and kubectl Add-on matrix Cloud and account Subscription or project Quotas Cost ownership IAM Human access Workload identity Break glass Network VPC or VNet Pod and Service CIDR Egress and private endpoints CNI Routing NetworkPolicy IP capacity CSI StorageClass Topology Snapshot and restore Node pools System General Stateful Spot and GPU Failure domains Zones Capacity Disruption budgets Ingress and Gateway Load balancer TLS DNS and WAF Policy Admission Supply chain Quotas Observability Metrics Logs Events and traces Upgrade API deprecation Add-on order Node replacement Recovery etcd or provider boundary Backup restore Workload migration Cost Requests Autoscaling Idle capacity

    The acceptance target is not “the cluster exists.” It is “workloads can onboard safely within explicit ownership and recovery boundaries.”

    Applicable versions and selection¶

    As of July 2026, Kubernetes maintains 1.36, 1.35, and 1.34. A new cluster should use the latest patch that the provider or distribution supports and whose critical add-on matrix is closed.

    Choice Decision
    Latest 1.36 patch Preferred for a new platform when CNI, CSI, ingress/gateway, admission, operators, and observability all support it
    Latest 1.35 patch Conservative choice when ecosystem certification for 1.36 is incomplete
    Latest 1.34 patch Only for a documented dependency blocker, with an upgrade deadline before onboarding
    1.33 or earlier Not for a new production cluster

    Freeze a compatibility matrix as a release artifact:

    Component Current Target Official evidence Owner Order Blocker
    Control plane <version> <version> <link> Platform 1
    kubelet/runtime <version> <version> <link> Platform 4
    CNI/CoreDNS <version> <version> <link> Network 2
    CSI/snapshot <version> <version> <link> Storage 2
    Ingress/Gateway <version> <version> <link> Edge 3
    Admission/Operators <version> <version> <link> Security/App 3
    Observability agents <version> <version> <link> SRE 3

    Do not upgrade when the provider has not opened the target minor, CNI/CSI support is absent, webhooks use removed APIs, CRD conversion is incompatible, or PDB/capacity prevents safe node replacement.

    Scope¶

    This standard applies to managed Kubernetes services, kubeadm-based self-managed HA clusters, production and shared pre-production platforms, and clusters rebuilt after major network, identity, or runtime changes.

    It does not replace application architecture, service-mesh design, regional business disaster recovery, or provider-specific implementation guides. A self-managed control plane also needs dedicated etcd, certificate, and API-endpoint runbooks.

    Preconditions and provisioning chain¶

    flowchart LR A[Organization and account] --> B[Network and address plan] B --> C[Control plane and API endpoint] C --> D[Node pools and failure domains] D --> E[CNI / CoreDNS / kube-proxy] E --> F[CSI / Snapshot] F --> G[Ingress / Gateway / DNS] G --> H[Identity / Security / Admission] H --> I[Observability] I --> J[Workload namespace onboarding] J --> K[Upgrade and recovery exercise]

    Pod and Service CIDRs must be checked against VPC, VPN, peering, and on-premises networks before control-plane creation; identity must be decided before secrets enter the cluster; CSI topology and restore must be proven before stateful onboarding; ingress, DNS, certificates, and observability must be ready before traffic.

    Required inputs include owners and on-call responsibility, target versions and matrix, network/DNS/NTP/registry reachability, zone quotas, data-protection targets, release and recovery objectives, and cost ownership.

    Prohibitions¶

    • Do not treat all Nodes being Ready as production acceptance.
    • Do not allow Pod or Service CIDRs to overlap existing or planned networks.
    • Do not grant permanent personal or pipeline cluster-admin.
    • Do not use static cloud keys as the default workload identity.
    • Do not create the cluster first and improvise CNI, CSI, and ingress versions later.
    • Do not install competing default IngressClasses or StorageClasses.
    • Do not onboard workloads without monitoring, tested restore, and an upgrade path.
    • Do not skip minor releases or treat control-plane downgrade as recovery.

    Mandatory requirements¶

    Prefer managed control planes; prove self-management¶

    Self-management requires three failure-domain-aware control-plane/etcd members, API load balancing, certificate lifecycle, etcd backup and quorum recovery, component upgrades, and continuous on-call ownership. Without people, automation, and exercises, it is not justified.

    Make platform contracts governable¶

    Record every address range and CNI allocation model; separate human, workload, and break-glass identity; make every node pool declare workload class, labels/taints, zones, autoscaling, OS/runtime, and replacement cycle; assign owners to CNI, CSI, ingress, admission, and observability.

    Establish upgrade and recovery on day one¶

    Before onboarding, exercise deprecated APIs, webhooks, add-on order, PDBs, spare capacity, drain, stop conditions, and recovery in non-production.

    Design example: three-zone production cluster¶

    This is a design example, not evidence from a verified production deployment.

    flowchart TB U[Users] --> DNS[DNS and WAF] DNS --> LB[Regional Load Balancer] LB --> GW[Gateway / Ingress] CP[Managed multi-AZ control plane] --> NP1 CP --> NP2 CP --> NP3 subgraph AZ-A NP1[System pool A] APP1[General pool A] ST1[Stateful pool A] end subgraph AZ-B NP2[System pool B] APP2[General pool B] ST2[Stateful pool B] end subgraph AZ-C NP3[System pool C] APP3[General pool C] ST3[Stateful pool C] end GW --> APP1 GW --> APP2 GW --> APP3 ST1 --> CSI[(Regional / zonal CSI)] ST2 --> CSI ST3 --> CSI OBS[Central observability] --- CP OBS --- APP1 OBS --- APP2 OBS --- APP3

    The example uses a managed multi-zone control plane, system and general pools in each zone, storage-aware stateful pools, enough spare capacity for one-zone loss, owned gateway/DNS/certificate controls, independent backup and restore beyond CSI snapshots, and GitOps with isolated production write permissions.

    Risk¶

    Design error Impact
    CIDR overlap Routing black holes and often a rebuild
    Single-zone capacity A zonal fault exhausts scheduling
    Missing add-on matrix CNI, CSI, or webhooks block an upgrade
    Excess default privilege Human or workload escalation
    Untested restore Healthy PVCs but unusable recovery
    Monitoring only Node Ready DNS, network, storage, and business failures stay hidden

    Validation¶

    Retain evidence for cross-node Pods, Services, DNS, egress and NetworkPolicy; least privilege, workload identity and break-glass; zone distribution, autoscaling and node replacement; PVC remount, expansion, snapshot and restore; TLS, DNS, logs and ingress failover; injected DNS/CNI/CSI/scheduling failures and alerts; upgrade order, PDB behavior, drain and stop conditions; node-pool rebuild, data restore and workload migration.

    A green CI run, Ready, or one smoke test is not enough.

    Exception handling¶

    Every exception records rationale, risk, owner, expiry, and exit plan. A single-zone edge site needs spare hardware and rebuild procedures; a blocked minor needs an explicit deadline; absent workload identity requires short-lived brokered credentials; absent CSI snapshots requires application-level backups and a restore exercise.

    Recovery and rollback principles¶

    • Managed control-plane upgrades recover through forward repair and compatible node pools, not downgrade.
    • Node changes recover through a new pool, cordon/drain, and batched migration.
    • CNI, CSI, ingress, and admission upgrades need rollback and configuration backups.
    • Self-managed clusters need tested etcd backup and fresh-control-plane restoration.
    • Workload recovery distinguishes stateless rebuild, PVC restore, and cross-cluster migration.
    • GitOps declarative state must match runtime recovery.
