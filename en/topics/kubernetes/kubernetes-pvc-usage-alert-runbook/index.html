    PVC Usage Alert Runbook¶

    Purpose¶

    This runbook defines how to respond when a Kubernetes PersistentVolumeClaim approaches or reaches its capacity threshold.

    The core judgment is:

    A PVC usage alert is a data-risk alert, not a disk-cleanup ticket.

    The response must identify the owning workload, confirm real filesystem usage, understand whether the data is disposable or business-critical, and choose a recovery path that does not silently delete state.

    Trigger¶

    Use this runbook for alerts such as:

    • KubePersistentVolumeFillingUp
    • KubePersistentVolumeUsageHigh
    • KubeletVolumeStatsUsedBytes threshold alerts
    • NFS or shared filesystem capacity alerts linked to Kubernetes PVCs
    • Application alerts that indicate failed writes caused by full storage

    Recommended threshold model:

    Severity Suggested Threshold Expected Response
    Warning 75 percent used or forecast full within 7 days Review owner and growth pattern.
    High 85 percent used or forecast full within 72 hours Prepare cleanup or expansion.
    Critical 90 to 95 percent used or writes failing Act immediately and escalate if owner is unavailable.

    Adjust thresholds for workload growth rate. A slow archival volume and a rapidly growing upload volume should not share the same operational urgency.

    Preconditions¶

    Before acting, confirm:

    • Namespace, PVC name, PV name, StorageClass, and owning workload.
    • Whether the volume is block, file, local, cloud disk, or NFS-backed.
    • Whether the data is disposable, cache-like, reconstructable, or business-critical.
    • Whether expansion is supported by the StorageClass and backend.
    • Whether backup or snapshot exists for business-critical data.
    • Whether the application owner is available.

    If ownership is unknown and the alert is critical, escalate to the platform owner and incident commander before deleting anything.

    Impact Check¶

    Check whether the application is already affected:

    kubectl -n <namespace> get pvc <pvc>
kubectl -n <namespace> describe pvc <pvc>
kubectl -n <namespace> get pods -o wide | grep <workload-or-pvc-hint>
kubectl -n <namespace> get events --sort-by=.lastTimestamp

    Look for:

    • Failed writes.
    • Pod restarts.
    • Readiness failures.
    • Application errors related to no space left on device.
    • Ingress or API errors caused by the workload.
    • Queue backlog or delayed jobs.

    If user impact exists, treat the alert as an incident, not routine maintenance.

    Confirm Real Usage¶

    Kubernetes PVC capacity and actual filesystem usage are not always the same signal.

    For kubelet volume stats:

    kubectl -n <namespace> get pvc <pvc> -o wide
kubectl -n <namespace> describe pod <pod-using-pvc>

    From inside an approved diagnostic pod or the application pod:

    df -h <mount-path>
du -xh --max-depth=1 <mount-path> 2>/dev/null | sort -h

    For NFS-backed PVCs, confirm usage on the NFS server or the platform monitoring path as well. A PVC may request 100Gi, while the real shared filesystem is a larger pool with many claim directories.

    Record:

    • PVC requested capacity.
    • Filesystem total, used, available, and percent used.
    • Largest directories or file classes.
    • Recent growth rate.
    • Owning workload and owner.

    Decision Tree¶

    Choose the response based on data type and growth behavior:

    Condition Preferred Action
    Disposable cache or temporary files consume space. Clean up through the application-approved path.
    Business-critical data is growing normally. Expand the volume or backend capacity.
    Business-critical data is growing unexpectedly. Expand only if needed for safety, then investigate growth source.
    Logs or exports are written to persistent data path. Stop the writer or change path, then clean up approved files.
    NFS backend pool is filling across many PVCs. Treat as shared storage incident; capacity action belongs to NFS layer.
    StorageClass does not support expansion. Migrate to a larger volume or new storage profile.
    Owner is unknown. Do not delete. Escalate and apply only non-destructive mitigation.

    Procedure¶

    1. Identify owner and data class¶

    Find the workload using the PVC:

    kubectl -n <namespace> get pods -o jsonpath='{range .items[*]}{.metadata.name}{"\t"}{range .spec.volumes[*]}{.persistentVolumeClaim.claimName}{" "}{end}{"\n"}{end}' | grep <pvc>

    Identify labels, deployment, StatefulSet, CronJob, or Helm release:

    kubectl -n <namespace> get pod <pod> --show-labels
kubectl -n <namespace> get pod <pod> -o jsonpath='{range .metadata.ownerReferences[*]}{.kind}{"/"}{.name}{"\n"}{end}'

    If JSONPath output is unclear, inspect the pod owner references and events with:

    kubectl -n <namespace> describe pod <pod>

    2. Stop unsafe growth if needed¶

    If the volume is filling fast, stop the source before cleaning:

    • Pause the job or CronJob writing the data.
    • Disable the feature flag producing files.
    • Scale down a non-critical writer if approved.
    • Redirect logs or exports away from the persistent path.

    Do not scale down stateful systems blindly. For databases, brokers, and queues, follow the service-specific runbook.

    3. Clean only approved data¶

    Cleanup is allowed only when the data class is understood.

    Safe examples:

    • Expired temporary files owned by the application.
    • Rebuildable cache directories approved by the service owner.
    • Completed export files after retention period.
    • Duplicate diagnostic dumps after incident evidence is preserved.

    Unsafe examples:

    • Database directories.
    • Queue segments.
    • User uploads.
    • Object metadata.
    • Files with unknown ownership.

    Before deletion, prefer moving a small sample to a quarantine path when the application owner needs proof.

    4. Expand when data must be retained¶

    Confirm expansion support:

    kubectl get storageclass <storageclass> -o yaml
kubectl -n <namespace> get pvc <pvc> -o yaml

    If supported, request a larger size:

    kubectl -n <namespace> patch pvc <pvc> \
  --type merge \
  -p '{"spec":{"resources":{"requests":{"storage":"<new-size>"}}}}'

    Then verify:

    kubectl -n <namespace> get pvc <pvc> -w
kubectl -n <namespace> describe pvc <pvc>

    Check the mounted filesystem after expansion. For some drivers, filesystem resize completes only after pod restart or remount.

    5. Migrate when expansion is not available¶

    If the StorageClass cannot expand or the backend is wrong for the workload:

    • Provision a new PVC using the approved storage profile.
    • Stop writes.
    • Copy data with an approved tool and checksum or application-level validation.
    • Start the workload against the new PVC.
    • Keep the old PVC retained until validation and rollback window expire.

    Do not delete the old PVC immediately after migration unless the data is explicitly disposable.

    Validation¶

    The response is complete only when:

    Check Pass Condition
    Usage PVC or backend usage is below the recovery threshold and no longer rising unexpectedly.
    Application Writes succeed and readiness is stable.
    Alerts Critical alert cleared or downgraded with an owner and deadline.
    Data safety No business-critical data was deleted without owner approval.
    Monitoring Dashboards show the same capacity reality as the application sees.
    Follow-up Growth cause, owner, and next review date are recorded.

    Rollback and Escalation¶

    Cleanup rollback is only possible if files were moved or backed up before deletion. If data was deleted, recovery depends on backups or application reconstruction.

    Escalate when:

    • User-facing writes are failing.
    • The PVC is above critical threshold and owner is unavailable.
    • Data class is unknown.
    • Expansion fails or stays pending.
    • Backend storage is full even after PVC-level action.
    • Multiple PVCs on the same NFS or shared storage backend are filling.

    Include namespace, PVC, PV, StorageClass, workload, usage percent, growth rate, and suspected largest paths.

    Post-Incident Review¶

    After recovery, answer:

    • Why was the data growth not predicted earlier?
    • Should the threshold or forecast alert change?
    • Does the workload need retention policy?
    • Should logs, exports, or temporary files move to object storage?
    • Is the StorageClass appropriate for this workload?
    • Does the owner mapping need to be added to the platform inventory?
