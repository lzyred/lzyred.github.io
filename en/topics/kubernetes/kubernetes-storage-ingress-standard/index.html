    Kubernetes Storage and Ingress Standard¶

    Purpose¶

    This standard defines how to design and validate Kubernetes storage and ingress before business workloads depend on them.

    The core judgment is:

    Storage and ingress are not add-ons. They are production contracts between the platform and every application team.

    Storage decides whether state survives, expands, moves, and restores. Ingress decides how users reach services, how TLS is renewed, how traffic is logged, and where the external boundary of the cluster begins. If either is treated as a default installed component, production risk becomes invisible until the first outage.

    When to Use¶

    Use this standard when:

    • Provisioning a new Kubernetes cluster.
    • Introducing or changing a CSI driver, StorageClass, IngressClass, or ingress controller.
    • Onboarding stateful workloads.
    • Publishing services through public or private ingress.
    • Reviewing a cluster before it becomes a shared production platform.

    Non-Goals¶

    This standard does not cover:

    • Full database backup architecture.
    • Service mesh traffic policy.
    • Cloud-specific load balancer billing optimization.
    • Application-level data model design.
    • Every possible ingress controller feature.

    Design Principle¶

    A production platform should expose a small number of named choices, not a large pile of implicit defaults.

    Application teams should be able to answer:

    • Which StorageClass should I use for normal persistent data?
    • Which StorageClass supports expansion?
    • Which storage profile supports ReadWriteMany, if any?
    • Who owns backup and restore?
    • Which IngressClass should expose public traffic?
    • Which IngressClass should expose private traffic?
    • Who owns DNS and TLS renewal?
    • What request limits and timeout defaults apply?

    If these answers require reading the platform install history, the platform contract is not clear enough.

    Storage Decisions¶

    Define storage profiles explicitly¶

    At minimum, define storage profiles like this:

    Profile Typical Use Access Mode Expansion Backup Expectation Notes
    standard-rwo Normal application persistence ReadWriteOnce Supported Application or platform policy Default only if it is safe for most apps.
    fast-rwo Latency-sensitive state ReadWriteOnce Supported Required Use only when performance need is proven.
    shared-rwx Shared files, uploads, model artifacts ReadWriteMany Platform-defined Required Often NFS or managed file storage; monitor real filesystem usage.
    ephemeral Disposable cache or temporary state Depends Not guaranteed None Must not hold business-critical data.

    Do not make a StorageClass default until its behavior under provision, mount, expansion, deletion, and failure is known.

    Make reclaim policy a deliberate decision¶

    Delete is convenient but can destroy data when a PVC is deleted. Retain protects data but creates cleanup responsibility.

    Use this rule:

    • Use Delete only for disposable or easily recreated data.
    • Use Retain for business data unless a backup and restore contract says otherwise.
    • Document who can delete retained volumes and how they verify the data is no longer needed.

    Treat expansion as a runbook, not a checkbox¶

    allowVolumeExpansion: true only means Kubernetes accepts the resize request. It does not prove that the underlying filesystem, CSI driver, quota, and application can safely absorb the change.

    Before approving expansion for a profile, verify:

    • The CSI driver supports online or offline expansion.
    • The filesystem reflects the new size.
    • The application tolerates the resize path.
    • Monitoring follows the real filesystem after expansion.
    • Rollback expectations are clear. Shrinking a volume is usually not a safe rollback.

    Separate backup responsibility from storage provisioning¶

    Provisioning a PV is not the same thing as protecting the data in it.

    For every storage profile, document:

    • Whether backups are required.
    • Whether snapshots are crash-consistent or application-consistent.
    • Who owns restore testing.
    • How long recovery is expected to take.
    • Which workloads must not use the profile.

    CSI Validation¶

    Validate CSI before onboarding stateful workloads:

    kubectl get csidrivers
kubectl get storageclass
kubectl get pods -n kube-system
kubectl get events -A --sort-by=.lastTimestamp

    Then run a real PVC test:

    apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: storage-validation
spec:
  accessModes:
    - ReadWriteOnce
  storageClassName: standard-rwo
  resources:
    requests:
      storage: 1Gi
---
apiVersion: v1
kind: Pod
metadata:
  name: storage-validation
spec:
  restartPolicy: Never
  containers:
    - name: writer
      image: busybox:1.36
      command: ["sh", "-c", "date > /data/proof && cat /data/proof"]
      volumeMounts:
        - name: data
          mountPath: /data
  volumes:
    - name: data
      persistentVolumeClaim:
        claimName: storage-validation

    Pass conditions:

    • PVC becomes Bound.
    • Pod schedules on an expected node.
    • File write succeeds.
    • Events show no CSI errors.
    • Deleting the test resources produces the intended reclaim behavior.

    Ingress Decisions¶

    Define ingress classes as exposure contracts¶

    IngressClass names should communicate exposure and ownership.

    Recommended pattern:

    IngressClass Exposure Typical Owner Required Controls
    public-web Internet-facing Platform or edge team TLS, DNS, access logs, WAF or gateway review if required.
    private-web Internal network Platform team Internal load balancer, private DNS, access logs.
    partner-web Restricted external access Platform plus security Allowlist, TLS, audit, explicit approval.

    Avoid leaving multiple ingress controllers active without a defaulting rule. An application should not be exposed through the wrong controller because an annotation was omitted.

    Own TLS as a lifecycle¶

    TLS is not complete when the first certificate is issued.

    Document:

    • Certificate issuer.
    • Renewal mechanism.
    • DNS validation owner.
    • Expiry alert.
    • Secret namespace and access policy.
    • Emergency replacement procedure.

    If TLS renewal is manual, the service must have an expiry alert and a named owner.

    Standardize ingress defaults¶

    Set and document defaults for:

    • Request timeout.
    • Body size.
    • WebSocket support.
    • HTTP to HTTPS redirect.
    • Access log format.
    • Real client IP handling.
    • Health check path.
    • Rate limiting or WAF boundary if used.

    These values affect application behavior. They must not be hidden inside an ingress controller chart with no platform documentation.

    Ingress Validation¶

    Use a small validation service before exposing real traffic:

    apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: ingress-validation
  annotations:
    cert-manager.io/cluster-issuer: letsencrypt-production
spec:
  ingressClassName: public-web
  tls:
    - hosts:
        - example.devops.test
      secretName: ingress-validation-tls
  rules:
    - host: example.devops.test
      http:
        paths:
          - path: /
            pathType: Prefix
            backend:
              service:
                name: ingress-validation
                port:
                  number: 80

    Pass conditions:

    • Ingress is assigned to the intended controller.
    • Load balancer or gateway address appears.
    • DNS resolves to the expected target.
    • TLS certificate is valid and renewable.
    • Access logs and request metrics appear.
    • A 404, 5xx, and slow request can be observed in the expected dashboard or query path.

    Validation Checklist¶

    Area Check
    StorageClass Default is deliberate, documented, and tested.
    CSI Controller and node plugins are healthy across node pools.
    PVC Provision, mount, write, delete, and expansion behavior are known.
    Reclaim Data retention behavior matches workload criticality.
    Backup Restore owner and restore test expectation are documented.
    IngressClass Public and private exposure paths are distinct.
    TLS Issuer, renewal, expiry alert, and emergency replacement are known.
    Logs and metrics Storage and ingress failures are observable.
    Ownership Platform and application responsibilities are separate.

    Known Failure Modes¶

    Failure Mode Symptom Prevention
    Missing default StorageClass PVC stays Pending. Either define a safe default or require explicit storageClassName.
    Wrong access mode Pod cannot mount volume or multi-writer workload corrupts data. Map workloads to approved profiles.
    CSI IAM or identity failure PVC never provisions, events show cloud permission errors. Validate workload identity before onboarding.
    Unclear reclaim policy Data is deleted too early or retained forever. Document reclaim policy and cleanup owner.
    NFS used as invisible capacity pool PVC appears fine while backend filesystem fills. Monitor real filesystem usage and PVC metadata together.
    Wrong IngressClass Service becomes unreachable or publicly exposed unintentionally. Use explicit ingressClassName and admission checks where possible.
    TLS renewal missing Service fails when certificate expires. Add expiry alert and named renewal owner.
    Controller defaults hidden Large uploads, WebSockets, or slow requests fail unexpectedly. Publish default timeout and body-size policy.

    Rollback and Change Boundary¶

    Storage and ingress changes have different rollback properties.

    Usually reversible:

    • Adding a new StorageClass.
    • Adding a new IngressClass.
    • Changing annotations on a validation ingress.
    • Replacing an ingress controller configuration after a canary.

    High-risk or not safely reversible:

    • Changing the default StorageClass after workloads already depend on it.
    • Changing reclaim policy without a migration plan.
    • Shrinking storage.
    • Moving a stateful workload between storage backends without backup and restore proof.
    • Replacing a public ingress path without DNS, TLS, and client cache planning.

    If the rollback path is "restore from backup," the change is not a normal configuration rollback. Treat it as a data recovery operation.

    Security Notes¶

    • Do not publish real hostnames, certificate details, private IPs, or customer domains in examples.
    • Restrict access to TLS secrets.
    • Avoid mounting broad cloud credentials into CSI components.
    • Use least-privilege workload identity for storage drivers.
    • Review public ingress exposure before DNS is delegated.
    • Keep access logs available for incident review, but avoid logging sensitive request bodies.

    Review Cycle¶

    Review this standard:

    • Before a new cluster accepts workloads.
    • Before changing CSI drivers, StorageClasses, ingress controllers, or IngressClasses.
    • After any data-loss, storage saturation, certificate expiry, or public exposure incident.
    • Before onboarding a new class of stateful workload.
