    Shell Automation Standard¶

    Purpose¶

    This standard defines how to write and operate shell scripts used for DevOps automation.

    The core judgment:

    A shell script becomes production automation the moment it changes infrastructure, data, access, or deployment state. Small scripts fail dangerously because they look too small to deserve controls.

    Shell is useful for glue, host automation, packaging, and simple orchestration. It becomes unsafe when it hides target context, parses complex state casually, assumes one-time execution, or cannot explain what it changed.

    When to Use¶

    Use this standard for scripts that:

    • change servers, containers, Kubernetes objects, or cloud resources
    • run in CI/CD jobs
    • rotate files, logs, backups, certificates, or credentials
    • perform cleanup, migration, or repair
    • touch production access, firewall rules, DNS, or storage
    • are shared between operators or teams

    One-off local exploration does not need the full template, but it must not be promoted into shared automation without review.

    Non-Goals¶

    This standard does not force every task into shell. Do not use shell as the primary implementation when the task needs complex data structures, concurrent state, API pagination, robust retries, long-lived business logic, or transactional behavior. Use a language and library better suited to the job.

    Operating Contract¶

    Every shared or production script needs a short contract:

    Contract Required decision Why it matters
    Purpose What operational change does the script perform? A vague script cannot be reviewed safely.
    Target Which host, namespace, cluster, account, path, or service is changed? Implicit current context causes wrong-target incidents.
    Authority Which credential or permission does the script require? Automation should not default to broad admin rights.
    Inputs Which arguments, files, and environment variables are accepted? Hidden inputs make behavior non-reproducible.
    Mutations What resources can be created, changed, deleted, or rotated? Operators need to know the blast radius.
    Idempotency What happens if the script runs twice or resumes after failure? Re-runs are common during incidents.
    Dry-run Can the script show intended actions without changing state? Review needs a preview path.
    Locking Can two runs collide on the same target? Concurrent runs can corrupt state or duplicate actions.
    Validation What proves the intended state was reached? Exit code alone is not enough.
    Rollback Is recovery automatic, backup-based, manual, or not safe? Some scripts are forward-only.

    If the script cannot say what it changes, it is not safe to run unattended.

    Recommended Standard¶

    Require explicit target selection¶

    Avoid scripts that infer production targets from current shell context.

    Required:

    • --target, --namespace, --cluster, --account, or equivalent explicit input
    • printed target summary before mutation
    • refusal to run when target is empty, ambiguous, or matches a dangerous default
    • confirmation flag such as --yes for destructive actions
    • dry-run as the default for high-impact scripts where practical

    Never let an empty variable become /, an empty namespace become "current namespace", or an unset account become the operator's default cloud account.

    Use strict mode, then handle expected failures deliberately¶

    Baseline:

    #!/usr/bin/env bash
set -Eeuo pipefail

trap 'echo "error: line ${LINENO}" >&2' ERR

    Strict mode reduces silent failure, but it does not replace explicit checks. Handle expected non-zero results deliberately, especially commands such as grep, kubectl get, test, diff, and cleanup steps.

    Validate before mutation¶

    Before changing state, validate:

    • required commands exist
    • required files exist and have safe permissions
    • target exists and matches the intended environment
    • current identity is expected
    • destructive paths are not root, empty, wildcard-expanded unexpectedly, or outside approved prefixes
    • external APIs or clusters are reachable
    • backups or snapshots exist when needed
    • dry-run output matches the planned target

    Use structured tools where possible. Avoid parsing human-formatted command output when JSON, YAML, or API output is available.

    Make repeated runs safe¶

    Prefer idempotent operations:

    • check before create
    • compare before replace
    • backup before overwrite
    • skip already-complete steps
    • record progress where long-running work can resume
    • treat missing optional resources as expected only when documented

    If a script cannot be idempotent, it must say why and state what evidence must be checked before re-running.

    Control concurrency and partial failure¶

    Shared scripts should not allow two operators or CI jobs to mutate the same target at the same time.

    Use an appropriate lock for the target:

    • file lock for local host operations
    • Kubernetes Lease or ConfigMap lock for cluster operations
    • remote state lock for infrastructure operations
    • CI concurrency group for pipeline jobs

    On failure, leave enough state to know which steps completed. A script that dies halfway through a credential rotation, firewall change, or migration must not force operators to guess.

    Provide logs that support incident review¶

    Logs should include:

    • timestamp
    • script version or commit
    • target
    • operator or CI job context where available
    • action
    • result
    • validation result
    • rollback hint on failure

    Do not use set -x in shared or CI scripts that may print credentials, tokens, headers, or private paths.

    Validation¶

    Before approving a shared script:

    bash -n script.sh
shellcheck script.sh
./script.sh --dry-run --target <test-target>

    Pass conditions:

    • syntax check passes
    • ShellCheck findings are fixed or justified
    • target cannot be omitted silently
    • dry-run output is understandable
    • test run changes only the intended test target
    • repeated test run is safe or documented as not idempotent
    • failure exits non-zero
    • logs show enough evidence for incident review
    • no secrets, real hostnames, internal paths, IPs, customer names, or credentials are embedded

    Failure Modes¶

    Failure mode Result Prevention
    Empty variable in destructive path Deletes or edits the wrong target. Strict mode, path guards, and explicit target checks.
    Implicit current context Runs against the wrong cluster, namespace, or account. Require target argument and print target before mutation.
    No idempotency Re-running doubles changes or fails midway. Check state before each mutation and record progress.
    Hidden partial failure Pipeline succeeds with a failed command. Use pipefail and explicit expected-failure handling.
    No lock Two runs mutate the same target. Use a target-appropriate lock or CI concurrency guard.
    Human-output parsing breaks Upgrade changes output format and script acts on wrong data. Prefer JSON/YAML/API output and structured parsing.
    No logs No one can reconstruct what happened. Log target, action, result, and validation.
    Secrets in output Credentials leak to CI logs or terminals. Mask sensitive values and avoid debug tracing.

    Rollback¶

    A script must state its rollback boundary:

    • reversible by running another command
    • reversible from backup
    • reversible by restoring a previous config or artifact
    • not safely reversible
    • requires manual recovery

    If a script deletes data, rotates credentials, changes firewall rules, modifies production access, or migrates state, review rollback before first production use. For forward-only scripts, require stronger dry-run, backup, approval, and observation.

    Security Notes¶

    • Do not store credentials in scripts.
    • Do not print tokens, private keys, session cookies, kubeconfigs, or cloud credentials.
    • Avoid curl | bash for production automation.
    • Pin downloaded artifacts and verify checksums where possible.
    • Use least-privilege service accounts in CI.
    • Review scripts that run with sudo, cluster-admin, or cloud admin rights.
    • Do not publish real hostnames, IPs, domains, customer names, internal paths, or production command output.

    Review Cycle¶

    Review shared automation before first use, after incidents, when permissions change, when target platforms or APIs change, when a script becomes part of CI/CD, and when one-off scripts start being reused by more than one operator.
