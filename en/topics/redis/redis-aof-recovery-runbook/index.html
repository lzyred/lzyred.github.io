    Redis AOF Inspection and Recovery Runbook¶

    Never run redis-check-aof --fix against the production directory, the only copy, or the evidence copy. The default path is to find a more complete recovery source, not to repair AOF.

    flowchart TD A[Redis cannot load AOF] --> B{Healthy replica exists} B -- Yes --> C[Verify replication position and data] C --> D[Rebuild from healthy replica] B -- No --> E{Qualified backup within RPO} E -- Yes --> F[Restore and validate backup] E -- No --> G[Read-only AOF inspection] G --> H{Only final tail can be truncated} H -- No --> I[Stop and escalate data-recovery incident] H -- Yes --> J[Approve estimated loss] J --> K[Repair only repair_dir] K --> L[Isolated load and business validation]
    mindmap root((Redis AOF recovery model)) Preferred sources Healthy replica Qualified backup AOF candidate Evidence preservation Read-only source Evidence checksums Logs and configuration Limited repair Read-only recovery copy Repair copy only RPO estimate Isolation boundary Separate directory Separate port Replication and Cluster disabled No production network Acceptance Keys and TTLs Business samples Single write authority Separate cutover plan

    Trigger¶

    Use this runbook only when Redis cannot start because of an AOF, manifest, or format error; a failed node leaves an AOF candidate to assess; a single-file AOF appears truncated at the tail; Redis 7/8 multipart AOF relationships are inconsistent; or an interrupted rewrite leaves the authoritative file set uncertain.

    Use a product-specific recovery process for Redis Cloud/Software, an Operator that hides the file layer, full Redis Cluster disaster recovery, vendor encryption/modules/storage, or any case where a healthy primary/replica or qualified backup is available.

    Risk¶

    This is high-risk data recovery. An unsafe action can truncate valid writes, select an older replica as authority, destroy the only evidence, reconnect an isolated instance to the original topology, misread commands because of version/modules/ACL/TLS differences, or create two writable authorities.

    Every --fix, candidate promotion, and production cutover requires data-owner approval. verification_status: not-verified means this runbook has not been executed in the user's environment.

    Required permissions¶

    Require read-only access to original configuration, logs, and data; access to replica, backup, replication, and business evidence; controlled root/platform access; temporary capacity for at least three copies; an isolated host/container/network namespace; written RPO and cutover approval; and Redis tools matching the source major/minor release.

    Prechecks¶

    Establish role, traffic, and better candidates¶

    redis-cli -h <host> -p <port> ROLE
redis-cli -h <host> -p <port> INFO replication
redis-cli -h <host> -p <port> INFO persistence
redis-cli -h <host> -p <port> CONFIG GET dir
redis-cli -h <host> -p <port> CONFIG GET appendonly
redis-cli -h <host> -p <port> CONFIG GET appendfilename
redis-cli -h <host> -p <port> CONFIG GET appenddirname

    Determine whether the node is primary, replica, or standalone; whether clients still write; whether Sentinel/Cluster/systemd/orchestration will restart it; which replica is more complete; whether a tested backup exists; and the approved maximum RPO. Do not stop an authoritative writer before traffic is contained.

    Record version and file facts¶

    redis-server --version
redis-check-aof --version 2>/dev/null || true
find <source-dir> -maxdepth 3 -type f -printf '%TY-%Tm-%Td %TH:%TM:%TS %s %p\n' | sort
df -h <source-dir>

    Record redis_conf, source_dir, appendfilename, appenddirname, deployment type, and role. Stop when modules, ACL, TLS, encryption, compression, or the file system cannot be reproduced.

    Build four directories and checksum evidence¶

    flowchart LR S[source_dir<br/>original scene] -->|read-only copy| E[evidence_dir<br/>immutable evidence] E -->|complete copy| R[recovery_dir<br/>read-only candidate] R -->|copy after approval| P[repair_dir<br/>only --fix target] R --> I[load unrepaired candidate in isolation] P --> I2[load repaired candidate in isolation]
    case_id="redis-aof-$(date -u +%Y%m%dT%H%M%SZ)"
work_root="/var/tmp/$case_id"
evidence_dir="$work_root/evidence"
recovery_dir="$work_root/recovery"
repair_dir="$work_root/repair"
sudo install -d -m 0700 "$evidence_dir" "$recovery_dir" "$repair_dir"
sudo cp -a --reflink=auto "<source-dir>/." "$evidence_dir/"
sudo cp -a --reflink=auto "$evidence_dir/." "$recovery_dir/"
sudo find "$evidence_dir" -type f -print0 | sort -z | xargs -0 -r sha256sum | sudo tee "$work_root/evidence.sha256" >/dev/null
sudo find "$recovery_dir" -type f -print0 | sort -z | xargs -0 -r sha256sum | sudo tee "$work_root/recovery-before.sha256" >/dev/null

    source_dir, evidence_dir, and recovery_dir are immutable. Only repair_dir, copied after approval, may be changed.

    Procedure¶

    Identify Redis 6.2 and Redis 7/8 structures¶

    flowchart TB A[Redis 6.2] --> A1[appendonly.aof<br/>single file] B[Redis 7 / 8] --> B1[appendonlydir] B1 --> B2[appendonly.aof.manifest] B1 --> B3[base AOF or RDB] B1 --> B4[incremental AOF files] B2 --> B3 B2 --> B4

    Use the actual configuration and manifest. For Redis 7/8, confirm every referenced base and incremental file exists, ordering is plausible, and files from different rewrite cycles are not mixed. Do not hand-splice missing or internally damaged files.

    Run a read-only check with matching tools¶

    redis-check-aof "$recovery_dir/<appendfilename>" 2>&1 | tee "$work_root/redis-check-aof-readonly.log"
redis-check-aof "$recovery_dir/<appenddirname>/<appendfilename>.manifest" 2>&1 | tee "$work_root/redis-check-aof-readonly.log"

    Load a valid recovery copy without repair. Consider repair only when the final command or final incremental tail is truncated. Stop for internal corruption, missing manifest/files, unrecognized format, or candidates with unique writes.

    Estimate loss before repair¶

    The valid offset and file size estimate byte truncation, not the business time window. Correlate Redis logs, replication offset/backlog, application write audit, backup timestamps, and business records. File mtime is supporting evidence only. Record bytes, files, possible commands, whether the time range is known, approver, and maximum RPO. Do not run --fix when loss cannot be bounded.

    Repair only a fresh repair copy¶

    sudo rm -rf "$repair_dir"
sudo install -d -m 0700 "$repair_dir"
sudo cp -a --reflink=auto "$recovery_dir/." "$repair_dir/"
sudo find "$repair_dir" -type f -print0 | sort -z | xargs -0 -r sha256sum | sudo tee "$work_root/repair-before.sha256" >/dev/null
redis-check-aof --fix "$repair_dir/<appendfilename>" 2>&1 | tee "$work_root/redis-check-aof-fix.log"
redis-check-aof --fix "$repair_dir/<appenddirname>/<appendfilename>.manifest" 2>&1 | tee "$work_root/redis-check-aof-fix.log"
sudo find "$repair_dir" -type f -print0 | sort -z | xargs -0 -r sha256sum | sudo tee "$work_root/repair-after.sha256" >/dev/null
diff -u "$work_root/repair-before.sha256" "$work_root/repair-after.sha256" | tee "$work_root/repair-files.diff" || true

    Do not repeatedly experiment on one repair copy. Discard it and recopy from recovery after any failed attempt.

    Start inside a real isolation boundary¶

    flowchart LR P[Original production topology] -. no connection .-> I[Isolated recovery instance] I --> D[Separate data directory] I --> PORT[Separate port 16379] I --> NET[No production egress] I --> REP[Replication disabled] I --> CL[Cluster disabled] V[Validator] -->|localhost or controlled tunnel| I I --> LOG[Separate log and PID]

    Copy the original configuration to preserve module/ACL/encoding behavior, then override bind, port, directory, log, PID, replication, and cluster-enabled no. Loopback binding alone is not isolation; use a disconnected host, container, or network namespace.

    Standalone candidates can load independently. A primary requires checking for a more complete replica. A replica must not become authority without offset/data comparison. Full Cluster recovery is out of scope because slot/node configuration and cross-shard consistency remain unresolved. Reproduce required modules; never delete unknown commands merely to start Redis.

    Expected results¶

    Stage Expected result
    Candidate selection Replica, backup, and AOF completeness/RPO compared
    Preservation Source unchanged and evidence checksums stored
    Read-only inspection Recovery copy unchanged; tool and entry point known
    Approval Truncation and business loss bounded and approved
    Limited repair Only repair copy changed as expected
    Isolated load Separate directory/port with no topology or production connection
    Data validation Keys, TTLs, types, scale, and business samples acceptable
    Cutover readiness Separate change, backup, monitoring, and rollback plan exists

    Stop conditions¶

    Stop when the four directories are confused; authority cannot be determined; tool versions mismatch; a manifest/base/incremental file is missing; corruption is not at the final tail; loss exceeds RPO or cannot be assessed; dependencies cannot be reproduced; the candidate reaches the old topology; critical data cannot be parsed; or unique writes require merging candidates.

    Success validation¶

    redis-cli -p 16379 PING
redis-cli -p 16379 ROLE
redis-cli -p 16379 INFO persistence
redis-cli -p 16379 INFO replication
redis-cli -p 16379 INFO keyspace
redis-cli -p 16379 DBSIZE
redis-cli -p 16379 CONFIG GET dir
redis-cli -p 16379 CONFIG GET appendonly
redis-cli -p 16379 CONFIG GET appendfilename
redis-cli -p 16379 CONFIG GET appenddirname
redis-cli -p 16379 CONFIG GET cluster-enabled

    Confirm no production upstream/downstream, correct directory and AOF entry point, no continuing format/module/permission error, readable keys/TTLs/types/business samples, plausible scale against an authoritative source, approved RPO, application parsing in isolation, and a separate production cutover with one writer.

    Failure handling¶

    For an unrecognized format, recheck release, manifest, and entry point. For excessive loss, discard repair and prefer replica/RDB/backup. For startup failure, retain logs and recopy from recovery. Reproduce missing modules. Do not onboard after failed samples. Escalate multiple unique writers into data merge and business adjudication.

    Rollback and escalation¶

    Rollback here means discarding recovery/repair candidates and returning to immutable evidence, not editing production. Production cutover keeps the old node and evidence frozen, builds a new instance from the accepted candidate, ensures one writable authority, restores clients gradually with monitoring, and returns to a healthy replica or verified backup on failure. Preserve logs, manifests, file listings, checksums, approvals, and estimated loss.
