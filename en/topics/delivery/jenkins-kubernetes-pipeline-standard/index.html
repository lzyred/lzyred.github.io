    Jenkins to Kubernetes Pipeline Standard¶

    Purpose¶

    This standard defines a safe Jenkins pipeline pattern for building container images and deploying them to Kubernetes: how to keep builds reproducible, how to handle credentials, and how to make production deploys gated, verified, and reversible.

    The core judgment of this standard:

    The pipeline's job is not "to deploy" — it is to make every deploy reproducible, attributable, and reversible. The single decision that delivers most of that is using an immutable image tag (the commit SHA), so what you tested is exactly what you ship and exactly what you can roll back to.

    A green Jenkins build is weak evidence unless it preserves custody: source commit, build inputs, image digest, scan result, approval, target environment, rollout result, and rollback artifact. Without that custody chain, the pipeline can be fast and still be unable to answer the only incident question that matters: "What exactly changed?"

    When to Use¶

    Use this standard when you:

    • Build container images and deploy to Kubernetes from Jenkins.
    • Need a consistent stage model, credential model, and rollback path across services.
    • Want CI (this standard) to hand off to GitOps CD, or to deploy directly with verification.

    Non-Goals¶

    This standard does not cover:

    • GitOps reconciliation — if Argo CD owns cluster state, the pipeline pushes the image and updates Git, and Argo CD deploys (see the Argo CD GitOps standard).
    • Jenkins controller/agent infrastructure and scaling.
    • Build-tool specifics for a given language.

    Pipeline Stages¶

    Use a predictable stage model so failures are localized and promotion is explicit:

    checkout → test → build image → scan → push image →
deploy to staging → smoke test → manual approval → deploy to production → verify

    Each stage must fail the build on error; a failing test or image scan must stop promotion, not warn and continue.

    Release Contract¶

    Every production pipeline must emit or link to a compact release record:

    Contract item Required evidence
    Source Git repository, commit SHA, branch or tag, and build number.
    Artifact Immutable image tag and digest, plus the registry location that production can still pull during rollback.
    Quality gates Test result, image scan result, policy exception if any, and who accepted the exception.
    Target Kubernetes context, namespace, deployment, and the account used to deploy.
    Approval Human approval, GitOps PR approval, or change window reference.
    Runtime proof Rollout status, pod image digest, readiness, metrics, and smoke-test result.
    Recovery proof Previous known-good artifact, rollback command or Git revert, and the automation source that must be corrected.

    If the release record cannot prove the artifact identity and target boundary, do not promote to production. A missing record is not a documentation inconvenience; it means the change cannot be reconstructed under incident pressure.

    Recommended Standard¶

    Immutable image tags¶

    Tag images with the Git commit SHA (and optionally a release tag), never a moving tag like latest. A moving tag means "what ran in staging" and "what runs in prod" can silently differ, and rollback has nothing precise to point at.

    Record the image digest after push and validate the deployed pod image against the intended tag or digest. Tags are human-friendly; digests are the stronger proof when a registry or rebuild path is under investigation.

    Credential handling¶

    Store all credentials in Jenkins credentials or an external secret manager — never hardcoded in Jenkinsfile, scripts, or docs:

    • Git, registry, Kubernetes deploy, notification, and cloud credentials.
    • Inject at the stage that needs them via Jenkins credential bindings or a native registry wrapper.
    • Never print secret values. If a CLI requires stdin, disable shell tracing and keep command output free of credential material.

    Verified, gated promotion¶

    • Deploy to staging and run a real smoke test before any production stage.
    • Gate production behind a manual approval (or a GitOps PR), recording who approved.
    • After the production rollout, verify — do not declare success on "kubectl applied".

    Direct deploy or GitOps handoff¶

    Choose one production authority per service:

    • Direct deploy: Jenkins changes the live Kubernetes workload and must own rollout status, rollback, and source-of-truth correction.
    • GitOps handoff: Jenkins builds and pushes the image, then opens or updates the Git change that Argo CD will reconcile. Jenkins must not also mutate the same production Deployment directly.

    Mixing both paths without an explicit emergency rule creates a recovery trap: Jenkins rolls back the live Deployment, then GitOps replays the failed desired state.

    Example Jenkinsfile skeleton¶

    pipeline {
  agent any
  environment {
    IMAGE             = 'registry.example.com/example/app'
    DEPLOYMENT        = 'example-service'
    APP_LABEL         = 'app=example-service'
    STAGING_CONTEXT   = 'staging-context'
    STAGING_NAMESPACE = 'example-namespace-staging'
    PROD_CONTEXT      = 'prod-context'
    PROD_NAMESPACE    = 'example-namespace'
  }
  stages {
    stage('Checkout') { steps { checkout scm } }
    stage('Test')     { steps { sh 'make test' } }
    stage('Build')    { steps { sh 'docker build -t ${IMAGE}:${GIT_COMMIT} .' } }
    stage('Scan')     { steps { sh 'trivy image --exit-code 1 --severity HIGH,CRITICAL ${IMAGE}:${GIT_COMMIT}' } }
    stage('Push') {
      steps {
        script {
          docker.withRegistry('https://registry.example.com', 'registry-credentials') {
            sh 'docker push ${IMAGE}:${GIT_COMMIT}'
          }
        }
        sh '''
          set -eu
          skopeo inspect docker://${IMAGE}:${GIT_COMMIT} --format '{{.Digest}}' > image-digest.txt
        '''
        archiveArtifacts artifacts: 'image-digest.txt', fingerprint: true
      }
    }
    stage('Deploy staging') {
      steps {
        withCredentials([file(credentialsId: 'kubeconfig-staging', variable: 'KUBECONFIG')]) {
          sh '''
            set -eu
            test "$(kubectl config current-context)" = "${STAGING_CONTEXT}"
            kubectl -n "${STAGING_NAMESPACE}" set image deployment/${DEPLOYMENT} app=${IMAGE}:${GIT_COMMIT}
            kubectl -n "${STAGING_NAMESPACE}" rollout status deployment/${DEPLOYMENT} --timeout=5m
            kubectl -n "${STAGING_NAMESPACE}" get pods -l "${APP_LABEL}" -o jsonpath='{range .items[*]}{.metadata.name}{"\t"}{.status.containerStatuses[*].imageID}{"\n"}{end}'
          '''
        }
        sh 'make smoke-test ENV=staging'
      }
    }
    stage('Approve') { steps { input message: 'Promote to production?' } }
    stage('Deploy production') {
      steps {
        withCredentials([file(credentialsId: 'kubeconfig-production', variable: 'KUBECONFIG')]) {
          sh '''
            set -eu
            test "$(kubectl config current-context)" = "${PROD_CONTEXT}"
            kubectl -n "${PROD_NAMESPACE}" set image deployment/${DEPLOYMENT} app=${IMAGE}:${GIT_COMMIT}
            kubectl -n "${PROD_NAMESPACE}" rollout status deployment/${DEPLOYMENT} --timeout=5m
            kubectl -n "${PROD_NAMESPACE}" get pods -l "${APP_LABEL}" -o jsonpath='{range .items[*]}{.metadata.name}{"\t"}{.status.containerStatuses[*].imageID}{"\n"}{end}'
          '''
        }
      }
    }
  }
}

    The image scan (trivy ... --exit-code 1) blocks promotion on HIGH/CRITICAL vulnerabilities — a scan that only logs is theater.

    Pre-Production Deploy Checklist¶

    Before the production stage:

    • Image tag is immutable (SHA) and is the exact artifact tested in staging.
    • Image digest is recorded after push and visible in the release record.
    • The target Kubernetes context and namespace are correct.
    • The Deployment has readiness probes and a sane rollout strategy (maxUnavailable/maxSurge).
    • The rollback command is known and tested.
    • The previous known-good image still exists in the registry and is pullable by the target cluster.
    • Any database migration, feature flag, or config change has an explicit reversibility decision.
    • Monitoring dashboard and alert ownership exist for the service.

    Validation¶

    kubectl -n example-namespace rollout status deployment/example-service --timeout=5m
kubectl -n example-namespace get pods -l app=example-service
kubectl -n example-namespace get pods -l app=example-service -o jsonpath='{range .items[*]}{.metadata.name}{"\t"}{.spec.containers[*].image}{"\t"}{.status.containerStatuses[*].imageID}{"\n"}{end}'

    Confirm:

    • Rollout completed; new pods are Ready with the expected image SHA.
    • Runtime imageID matches the recorded registry digest, or the workload was deployed by digest directly.
    • Smoke test passed in staging; post-deploy checks (error rate, latency) are within threshold in production.
    • No crash-looping or readiness failures after the rollout window.
    • The deployment target is the intended cluster and namespace, not a similarly named environment.
    • If GitOps owns the service, the Git desired state now points at the same artifact that was verified.

    Known Failure Modes¶

    Failure mode Symptom Likely cause Correct response
    Staging ≠ production Works in staging, breaks in prod Moving tag (latest) Use immutable SHA tags end to end.
    Credentials leaked Secret in logs/repo Hardcoded in Jenkinsfile/echoed Use withCredentials; rotate the exposed secret.
    "Deployed" but broken Pipeline green, service failing Declared success on apply, not verified Use rollout status + smoke/post-deploy checks.
    Scan ignored Vulnerable image shipped Scan logs but doesn't fail build Make the scan fail on HIGH/CRITICAL.
    Rollout hangs Stuck at rollout status Bad readiness probe or insufficient capacity Roll back; fix probe/capacity in staging.
    Wrong target changed Unexpected namespace or cluster modified Context injected implicitly or reused from an agent Print and verify context/namespace before production actions; scope credentials.
    Rollback image unavailable ImagePullBackOff during rollback Previous image was garbage-collected or tag was mutable Retain rollback artifacts and validate pull access before promotion.
    GitOps replays failure Manual rollback succeeds, then bad version returns Git desired state still points at failed artifact Revert or patch Git before closing recovery.
    Data change is forward-only Old app starts but corrupts or rejects new state Migration/config/feature flag not covered by release contract Stop rollback and use the release-specific recovery plan.

    Rollback¶

    kubectl -n example-namespace rollout undo deployment/example-service
kubectl -n example-namespace rollout status deployment/example-service --timeout=5m
    1. Because tags are immutable SHAs, rolling back means redeploying the previous known-good SHA (or rollout undo to the prior ReplicaSet).
    2. If the pipeline updates Git (GitOps), roll back by reverting the commit so the controller does not re-apply the bad version.
    3. Confirm the previous image can still be pulled before declaring rollback available.
    4. Verify after rollback exactly as after a deploy — confirm pods, image SHA or digest, source of truth, and service health.

    Security Notes¶

    • Keep all credentials in Jenkins/secret manager; never in Jenkinsfile, scripts, or notifications.
    • Scope deploy credentials to the target namespace; do not use cluster-admin in pipelines.
    • Block promotion on image scans; pin base images.
    • Do not include webhook URLs, tokens, or sensitive logs in notification templates — notify with job, build, commit, environment, status, and a runbook link only.
    • Treat Jenkins logs and artifacts as audit material; mask secrets, restrict retention access, and avoid storing raw kubeconfigs.

    Review Cycle¶

    • Owner: the delivery/platform team.
    • Re-review the gate and scan thresholds when risk tolerance changes.
    • Confirm rollback works (test it) on a cadence, not just on paper.
    • Re-check deploy-credential scope when namespaces or clusters change.
    • Sample recent release records and verify that artifact, approval, target, and recovery evidence can be reconstructed without asking the person who ran the job.
