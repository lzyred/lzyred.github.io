    Argo CD GitOps Standard¶

    Purpose¶

    This standard defines how to operate Argo CD as a GitOps deployment controller for Kubernetes: what belongs in Git, how aggressively to let Argo CD reconcile, and how to recover both an application and Argo CD itself.

    The core judgment of this standard:

    GitOps moves the risk from "who ran the command" to "what is in Git and how fast it reconciles." selfHeal and prune are not convenience flags — they decide whether a bad commit or a manual hotfix repairs itself or gets reverted under you. Choose them per risk tier, deliberately.

    A healthy Argo CD setup has two clocks: the human review clock in Git and the reconciliation clock in the cluster. Most GitOps incidents happen when those clocks are treated as the same thing. A reviewed commit can still be wrong, and a fast reconciler can turn one wrong line into a fleet-wide outage before anyone reads the diff.

    When to Use¶

    Use this standard when you:

    • Deploy to Kubernetes and want Git to be the source of truth for cluster state.
    • Run, or plan to run, Argo CD as the reconciliation controller.
    • Need a defensible sync policy, secret-handling model, and recovery plan for GitOps.

    Non-Goals¶

    This standard does not cover:

    • CI (building and testing artifacts). GitOps is the CD half; see the Jenkins-to-Kubernetes pipeline standard for the CI half.
    • Manifest templating choice (Helm vs Kustomize vs plain YAML) beyond how it interacts with sync.
    • Multi-cluster fleet design and ApplicationSets generators beyond the single-Application model shown here.

    GitOps Principles¶

    1. Git is the desired state. Cluster changes flow through reviewed commits, not kubectl edit.
    2. Secrets are referenced through a secret-management process, never committed in plain text.
    3. Every Application has a deliberate sync policy matched to its risk tier.
    4. Drift is visible and handled by policy, not silently overwritten everywhere.
    5. Notifications expose status, not webhook secrets or tokens.

    Application Release Contract¶

    Before an Application becomes production-managed by Argo CD, write down the release contract it implies:

    Contract item Required decision
    Source of truth Which repo, branch, path, chart, or generated manifest is allowed to define production state.
    Review authority Who may approve changes to this path, and whether the approver also owns production impact.
    Destination boundary Which cluster, namespace, resource kinds, and AppProject allow-list this Application may touch.
    Reconciliation speed Whether sync is manual, automated, self-healing, or allowed to prune resources.
    Emergency path How to stop reconciliation, revert Git, or apply a temporary fix without fighting the controller.
    Secret boundary Which external secret mechanism is authoritative, and what must never appear in Git.
    Recovery evidence Which status, health, metric, and user-path checks prove the new desired state is safe.

    If any row is unknown, the Application is not ready for automatic production sync. GitOps does not remove the need for release ownership; it makes missing ownership faster to expose.

    Installation Model¶

    Use the official manifests or Helm chart. For production:

    • Run Argo CD in high-availability mode.
    • Keep Argo CD in its own namespace, separate from application namespaces.
    • Restrict who can create/modify Application and AppProject resources via RBAC.
    • Scope AppProject to the repos, destinations, and resource kinds each team may deploy — this is the real blast-radius control.
    • Keep cluster credentials, repo credentials, and admin tokens in the approved secret-management path, not in exported examples or shared runbooks.
    • Treat Argo CD upgrades as production delivery-system changes: back up config, check release notes, upgrade outside high-risk release windows, and verify existing Applications after the controller restarts.

    Validation:

    kubectl get pods -n argocd
kubectl get applications -n argocd
argocd app list

    Sync Policy by Risk Tier¶

    This is the central decision. Match the policy to the service, not the convenience of the operator.

    Tier automated selfHeal prune Use for
    Production / stateful off (manual sync) off off High-risk services; a human approves each sync.
    Standard service on on off Most stateless services; drift is auto-corrected, deletions stay manual.
    Mature, well-tested on on on Strong review + rollback; Git fully owns lifecycle including deletion.

    Manual sync is not a sign of immaturity; for data-bearing or shared-infrastructure services it is often the correct way to keep the human approval clock slower than the controller clock. Automated sync is appropriate only when the team has already proven the validation and rollback loop outside the happy path.

    apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: example-service
  namespace: argocd
spec:
  project: example-team
  source:
    repoURL: https://git.example.com/example/example-repo.git
    targetRevision: main
    path: deploy/kubernetes
  destination:
    server: https://kubernetes.default.svc
    namespace: example-namespace
  syncPolicy:
    automated:
      prune: false      # deletions stay manual for this tier
      selfHeal: true    # revert out-of-band drift

    Two flags worth understanding before you enable them:

    • selfHeal: true reverts any change made directly to the cluster. That is the point — but it also means an emergency kubectl hotfix will be undone within minutes. The correct emergency path is a commit, or temporarily disabling auto-sync, not fighting the controller.
    • prune: true lets Argo CD delete resources that disappear from Git. Powerful for lifecycle hygiene, dangerous if someone deletes a manifest by mistake. Reserve it for services with strong review.

    Secret Handling¶

    Plain secrets must never live in Git. Choose one model and apply it consistently:

    • Sealed Secrets (encrypted in Git, decrypted in-cluster), or
    • An external secret operator pulling from a secret manager, or
    • A CSI secrets driver mounting from a vault.

    Argo CD Application manifests themselves contain repo URLs, project names, and cluster references — treat exports as sensitive topology.

    Drift, Rollback, and Recovery¶

    Drift is not always bad. A direct cluster change may be an emergency patch, an attacker, a failed controller, or a human bypass. The operating question is who is allowed to decide whether Git or the live cluster wins.

    For production Applications:

    • Ignore differences only for fields that are known to be controller-owned, noisy, or safely externalized.
    • Alert on unexpected drift in production even when selfHeal will correct it.
    • Record who disabled auto-sync, why, and when it must be restored.
    • Keep a break-glass path narrower than normal access; emergency access should not become a second deployment system.

    Rollback an application¶

    Because Git is the source of truth, rollback is a Git operation:

    # Preferred: revert the commit, let Argo CD reconcile.
git revert <bad-commit> && git push

    For Applications with automated sync or self-heal enabled, do not start with argocd app rollback. Argo CD cannot roll back an Application while automated sync is enabled, and a self-healing controller may move the live cluster back toward the bad Git state. Use one of these paths first:

    • revert or patch Git and let Argo CD reconcile to the recovered state, or
    • suspend automated sync through the platform-approved path, recording owner, reason, and restore deadline.

    Only after the source of truth is controlled should you use a direct rollback as an emergency bridge:

    argocd app history example-service
argocd app rollback example-service <revision>

    A direct argocd app rollback is an emergency bridge, not the final state. Re-enable automation only after Git matches the recovered state and the application has passed service-level validation.

    Disaster recovery of Argo CD itself¶

    Argo CD is a production dependency; plan for losing it.

    The export contains sensitive deployment topology. Create it only into an approved encrypted/internal backup path with restrictive permissions, and do not attach, commit, or share the raw export.

    umask 077
argocd admin export > <encrypted-internal-backup-path>/argocd-export-<yyyy-mm-dd>.yaml
# restore:
argocd admin import - < <encrypted-internal-backup-path>/argocd-export-<yyyy-mm-dd>.yaml

    Because desired state lives in Git, a rebuilt Argo CD plus the Git repos can reconstruct the cluster — but only if the Application/AppProject definitions are also captured (in Git or in the export).

    Do not treat the export as an ordinary backup artifact. It can expose repositories, projects, cluster destinations, and operational topology. Test import in an isolated environment before relying on it for incident recovery.

    Validation¶

    argocd app get example-service
kubectl get applications -n argocd

    Confirm:

    • Sync status is Synced and health is Healthy for changed apps.
    • The synced revision matches the intended Git commit.
    • No unexpected OutOfSync resources (drift) remain after sync.
    • For prune-enabled apps, nothing was deleted that you did not intend.
    • The AppProject allow-list still matches the intended team boundary.
    • The service-level smoke test or synthetic check passes after sync, not only Argo CD health.
    • Notifications route to the team that owns the release and the team that owns the platform controller.

    Known Failure Modes¶

    Symptom Likely cause Check / response
    App stuck OutOfSync Live drift, ignored diffs, or generated-manifest mismatch argocd app diff; reconcile or add ignoreDifferences deliberately.
    Sync fails RBAC, missing namespace, missing CRD, admission policy Read the sync error; create prerequisites; check AppProject allow-lists.
    Emergency fix keeps reverting selfHeal is reverting out-of-band changes Commit the fix, or temporarily disable auto-sync; don't fight the controller.
    Health stays Unknown Custom resource without health assessment Add a resource health customization.
    Repo connection fails Bad credentials, network, or unknown host key Re-check repo credentials and connectivity.
    Unexpected deletion prune: true plus a removed/misgenerated manifest Restore via Git revert; reconsider prune for that tier.
    Manual rollback keeps being undone Git still points at the bad desired state Revert or patch Git before re-enabling automation.
    Controller outage blocks delivery Argo CD dependency not included in release continuity plan Use the documented emergency path, then restore controller state and reconcile Git.

    Rollback (Operational Summary)¶

    1. Prefer git revert of the offending commit; let Argo CD reconcile back to good state.
    2. If automated sync or self-heal is enabled, revert or patch Git first, or suspend automation with owner, reason, and deadline before any direct rollback.
    3. For emergency speed after the source of truth is controlled, argocd app rollback to a known-good revision, then reconcile Git to match.
    4. If Argo CD itself is the problem, disable auto-sync on critical apps to freeze the cluster while you recover the controller.
    5. Restore Argo CD from a protected export; verify apps return to Synced/Healthy.

    Security Notes¶

    • Never commit secrets, webhook URLs, or chat tokens in plain text.
    • Scope AppProject to allowed repos, destinations, and resource kinds — this bounds blast radius.
    • Use SSO and least-privilege RBAC; restrict who can sync production-tier apps.
    • Treat argocd admin export output as sensitive internal topology; store it encrypted.
    • Pin Argo CD versions and patch promptly; it has cluster-wide reconciliation power.
    • Review Application manifests as production-change authority, not as harmless deployment metadata.

    Review Cycle¶

    • Owner: the platform/delivery team.
    • Re-review each Application's risk tier and sync flags on significant change.
    • Test Argo CD disaster recovery (export/import) on a cadence, not just on paper.
    • Re-check AppProject allow-lists when a team's deployment scope changes.
    • Review disabled auto-sync and drift exceptions as a recurring operational debt list.
