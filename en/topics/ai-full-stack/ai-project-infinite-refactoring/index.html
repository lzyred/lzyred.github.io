    Why AI projects get messier as they improve: a real “merged successfully, delivered unsuccessfully” postmortem¶

    The most dangerous AI-project failure is not that the model cannot write code. It is that repositories keep changing, pull requests keep merging, and the user’s problem never closes.

    The three-minute version¶

    What to remember Conclusion
    What infinite refactoring means The definition of done keeps drifting, so each change fails to form a closable delivery loop
    Why AI makes it easier Models discover and generate changes far faster than humans can review and decide
    Where the root cause lives Product, decision, verification, and release systems fail together
    How many kinds of “done” exist Source, rule verification, build, release, and user outcome are five distinct layers
    How to stop One outcome → change boundary → minimum change → five-layer evidence → user acceptance → freeze

    The article reduces to two models:

    1. Four failed systems explain why the project becomes harder to finish.
    2. The six-step delivery loop defines how the next task must stop.

    Incident: the PR merged, but the wrong page stayed online¶

    In July 2026, a DevOps documentation-maintenance task had four goals:

    • remove a versioned entry page that should never have been public;
    • deepen five shallow bilingual article pairs;
    • add diagrams that genuinely improved understanding;
    • make GitHub Actions verify and publish only after a push to main.

    The resulting pull request changed 24 files, added 1,747 lines, deleted 2,406 lines, and merged successfully into main.

    From the source repository, the work looked complete:

    • PR #39 was merged;
    • merge commit 190bf8e existed;
    • the source page for /topics/devops-answers/ had been removed;
    • the publication workflow declared rsync --delete to remove stale files.

    But when the user opened the production site, the page that should have disappeared was still online. It also had a severe layout defect: the cards collapsed into narrow vertical strips and both Chinese and English wrapped one character at a time.

    Confirmed facts, inference, and unknowns¶

    Type Finding
    Confirmed fact The source PR merged into main
    Confirmed fact The old production route still returned the old page
    Confirmed fact At that review point, the publication repository still mapped to the older source commit 73102bb...
    Confirmed fact The AI had reported the merge before verifying publication and production behavior
    Reasonable inference The end-to-end delivery chain had not been verified and closed
    Unknown The exact workflow step that did not run, failed, or failed to finish publication

    That boundary matters. An old production page does not justify inventing a specific CI root cause.

    What the incident proves is narrower and more important:

    Source completion, PR merge, and user delivery are different states.

    sequenceDiagram participant O as Owner participant A as AI participant S as Source repository participant P as Publication repository participant W as Production site O->>A: Remove the wrong entry and publish the fix A->>S: Change 24 files and open a PR S-->>A: PR merged A-->>O: Report merge completion O->>W: Open the target route W-->>O: Old page still returned P-->>O: Latest publication still maps to old source Note over A,W: Build, release, production, and user evidence are missing

    Why this is more than an ordinary release failure¶

    If the incident is labeled only “the release failed,” the likely response is:

    • edit the workflow again;
    • add another check script;
    • open another PR;
    • write another completion report.

    That response misses the deeper question:

    Why did the team consider the task complete without release evidence or production acceptance?

    Core definition: what infinite refactoring really is¶

    Infinite refactoring does not mean that code changed many times, and it does not mean every large refactor is wrong.

    A healthy refactor has a stable boundary:

    Problem confirmed
→ impact confirmed
→ change boundary confirmed
→ acceptance evidence confirmed
→ stop after acceptance

    Infinite refactoring is different:

    The definition of done keeps drifting, so every implementation fails to create a closable delivery loop.

    flowchart LR A[Vague objective] --> B[AI fills in missing requirements] B --> C[Task scope expands] C --> D[Change exceeds human review capacity] D --> E[Team relies on AI summaries and automated checks] E --> F[Merge is treated as completion] F --> G[Production or user acceptance fails] G --> H[Solution is redefined] H --> B I[No stop condition] -.keeps loop alive.-> B J[No stable product truth] -.keeps loop alive.-> H

    The loop is dangerous because every step can look locally reasonable.

    • The issues the model discovers may be real.
    • The additional tests may be useful.
    • The rewritten content may be more complete.
    • The workflow changes may be technically sound.

    But locally correct changes do not automatically combine into a correct delivery.

    Mechanism: four failed systems create infinite refactoring¶

    The complete model¶

    mindmap root((AI infinite refactoring)) Product system failure No single user outcome No explicit non-goals Page count replaces value Technical completeness replaces closure Decision system failure AI expands scope on its own New sessions redefine the project Owners review summaries, not decisions No protected project truths Verification system failure Tests verify form only Build success equals quality Merge equals completion No user-task acceptance Release system failure Source and publication repositories differ Workflow state is unconfirmed Production is not sampled Stale files and cache lack evidence

    1. Product-system failure: the objective is a wish, not an outcome¶

    This cannot be accepted objectively:

    Improve the AI project’s content, UI, architecture, CI, and user experience comprehensively.

    It does not identify:

    • who uses the result;
    • what observable outcome changes;
    • what is explicitly excluded;
    • the state that ends the task.

    When the goal cannot be accepted, the model keeps filling gaps. “More professional,” “richer,” and “more complete” have no natural finish line.

    2. Decision-system failure: the AI defines, implements, tests, and accepts¶

    Many AI projects quietly use this chain:

    AI defines the standard
→ AI implements it
→ AI writes the tests
→ AI runs the tests
→ AI declares completion from its own rules

    That is an internally consistent self-proof system. It may still be wrong for the user.

    A model can choose local implementation details. It should not unilaterally decide:

    • whether product scope expands;
    • whether global architecture changes;
    • whether release risk is accepted;
    • whether content deserves publication;
    • whether the user has accepted the outcome.

    3. Verification-system failure: automation proves rules, not that the rules are right¶

    Automation is good at checking:

    • file presence;
    • syntax;
    • metadata equality;
    • diagram counts;
    • horizontal overflow;
    • zero exit codes.

    It is much weaker at judging:

    • whether the article is worth reading;
    • whether a diagram improves understanding;
    • whether the body fulfills the title;
    • whether the user can finish a real task;
    • whether the page should exist at all.
    flowchart TD A[Automated verification] --> B[Syntax] A --> C[Format] A --> D[Build] A --> E[Basic layout] A --> F[Rule consistency] G[Human acceptance] --> H[Content value] G --> I[Comprehension] G --> J[Real problem solved] G --> K[Structured thinking] G --> L[Title fulfilled] B --> M[Engineering pass] C --> M D --> M E --> M F --> M H --> N[Product pass] I --> N J --> N K --> N L --> N

    4. Release-system failure: the project does not separate five delivery truths¶

    There are at least five layers of “done”:

    flowchart TD A[1. Source complete] --> B[2. Rule verification complete] B --> C[3. Build artifact complete] C --> D[4. Release complete] D --> E[5. User outcome complete] A -.does not prove.-> C B -.does not prove.-> D C -.does not prove.-> E D -.still does not fully prove.-> E
    Layer What it proves What it does not prove
    Source Files, commits, and PRs changed The project runs
    Rule verification Encoded checks passed The checks represent the right goal
    Build A publishable artifact exists The artifact was released
    Release Production files changed The experience and content are correct
    User outcome The target task is actually closed The system needs no future maintenance

    The incident reached the source layer. It lacked release-layer and user-outcome evidence.

    How one layout bug becomes a 24-file project¶

    AI is excellent at finding related problems. It does not naturally own prioritization.

    flowchart LR A[Original issue<br/>mobile page is too narrow] --> B[Direct fix<br/>container and grid] A --> C[AI discovers related issues] C --> D[Typography inconsistent] C --> E[Navigation weak] C --> F[Articles shallow] C --> G[Governance missing] C --> H[CI design weak] D --> I[Design-system refactor] E --> J[Information-architecture refactor] F --> K[Bulk content rewrite] G --> L[New entry and labels] H --> M[Workflow refactor] I --> N[Change scope becomes unreviewable] J --> N K --> N L --> N M --> N

    All of those problems may be real. The crucial rule is:

    A real problem does not automatically belong in the current pull request.

    Decision debt: the hidden cost AI projects accumulate¶

    Technical debt means code is hard to maintain. Decision debt means the repository contains implemented choices that no one fully understands or endorses.

    Examples:

    • Why does this entry page exist?
    • Why was navigation changed?
    • Why does this metadata represent trust?
    • Why does this test represent content quality?
    • Why were UI, content, and CI changed together?
    flowchart LR A[AI generates quickly] --> B[More files change] B --> C[More hidden decisions] C --> D[Human review falls behind] D --> E[Team trusts AI summaries] E --> F[Unexamined decisions merge] F --> G[Future rework] G --> H[Decision debt grows]

    When no one knows why the old design exists, local change feels risky and a full rewrite feels simpler. That starts the next, larger refactor.

    Boundaries: not every large change is infinite refactoring¶

    A broad refactor can be justified when:

    • one architectural cause repeatedly produces the same production defect;
    • a small change inevitably breaks multiple modules;
    • critical logic cannot be tested;
    • the data model cannot support an approved requirement;
    • security boundaries are ambiguous;
    • release and rollback cannot be guaranteed.

    The decision is not based on file count. It is based on four questions:

    1. Which confirmed objective fails without the refactor?
    2. Is a smaller repair available?
    3. Are migration, validation, and rollback defined?
    4. Is the finish condition stable?
    flowchart TD A[Propose refactor] --> B{Which confirmed goal fails without it?} B -- Cannot answer --> C[Stop refactor] B -- Clear answer --> D{Is a smaller repair available?} D -- Yes --> E[Use minimum repair] D -- No --> F{Migration, validation, and rollback defined?} F -- No --> G[Defer refactor] F -- Yes --> H[Approve controlled refactor]

    Counterexample: what a justified large refactor looks like¶

    Assume a monolithic deployment system has confirmed evidence that:

    • every deployment-state change touches the API, database, and UI;
    • failed states are not persisted;
    • audit records cannot reconstruct execution;
    • three production incidents share the same state-model flaw.

    Extracting a proper domain state machine may change dozens of files without becoming infinite refactoring because:

    • the root cause has repeated incident evidence;
    • the objective is stable state transitions, not “cleaner architecture”;
    • migration and rollback boundaries are explicit;
    • acceptance can be demonstrated with a real release task;
    • the version freezes after acceptance.

    These reasons are not enough:

    • a new model suggested a different architecture;
    • the page could look more premium;
    • the directory feels incomplete;
    • the team may need it someday;
    • the AI found more issues while working.

    Actionable next step: the six-step anti-refactoring loop¶

    “Do not overengineer” is not a control. A project needs a stop system that cannot be bypassed.

    flowchart LR A[1. One outcome] --> B[2. Change boundary] B --> C[3. Minimum change] C --> D[4. Five-layer evidence] D --> E[5. User acceptance] E --> F[6. Freeze version] D -->|Explicit failure| C C -->|Non-blocking discovery| G[Backlog] G --> H[Schedule as a separate task]

    Step 1: define one outcome¶

    Bad objective:

    Improve the AI project’s content, UI, architecture, CI, and experience.

    Good objective:

    Remove the wrong public entry and prove the production route no longer returns the old page.

    A task can require several technical actions, but all of them must serve one user outcome.

    Step 2: set a change boundary and budget¶

    change_budget:
  max_files: 6
  allowed_paths:
    - docs/zh/topics/ai-full-stack/
    - docs/en/topics/ai-full-stack/
    - mkdocs.yml
  forbidden_paths:
    - docs/zh/index.md
    - docs/en/index.md
    - docs/assets/stylesheets/global.css
  allow_new_dependencies: false
  allow_unrelated_refactoring: false

    The budget is not a mechanical line-count rule. It forces the AI to explain scope growth:

    Why does removing one page require changing 24 files?

    Step 3: make only the minimum necessary change¶

    flowchart TD A[Discover a new issue] --> B{Does it block the one outcome?} B -- Yes --> C[Include in current task] B -- No --> D[Record in backlog] C --> E{Is a smaller repair available?} E -- Yes --> F[Use minimum repair] E -- No --> G[Request scope expansion]

    Step 4: submit five-layer evidence¶

    Layer Required evidence
    Source Commit, diff, changed-file list
    Verification Test output and static checks
    Build Artifact and target route
    Release Publication commit and source SHA
    User Production response, desktop and mobile screenshots, real task result

    These statements are not evidence:

    • it should be deleted;
    • it is probably published;
    • the tests should cover it;
    • the workflow will handle it automatically.

    Step 5: accept from the user task¶

    Weak acceptance:

    The page built successfully.
All tests passed.

    Strong acceptance:

    The old route no longer returns the old entry page.
Navigation, search, and sitemap no longer expose it.
Desktop and mobile have no layout defect.
The target article fulfills the problem promised by its title.

    Step 6: freeze after acceptance¶

    stateDiagram-v2 [*] --> Defined Defined --> Implementing: Scope approved Implementing --> Verifying: Minimum change complete Verifying --> Fixing: Explicit failure Fixing --> Verifying Verifying --> Publishing: Automated checks pass Publishing --> OnlineChecking: Release complete OnlineChecking --> Accepted: User task passes OnlineChecking --> Fixing: Production fails Accepted --> Frozen Frozen --> [*] Implementing --> Backlog: Non-blocking issue discovered Backlog --> [*]

    Frozen means:

    • “it could be better” is not enough to continue;
    • new findings become separate tasks;
    • each new task receives a new outcome and budget;
    • old tasks cannot accumulate endless additions.

    Ten-minute diagnosis¶

    Score one point for every “yes.”

    1. The current version cannot be described as one outcome.
    2. The last three PRs changed unrelated modules together.
    3. AI discoveries routinely expand current scope.
    4. The project has no explicit non-goals.
    5. One AI defines, implements, tests, and accepts the work.
    6. Humans rely on the AI summary without understanding the main decisions.
    7. New sessions repeatedly redefine product direction or information architecture.
    8. The team checks PR or test status but not production behavior.
    9. No one can demonstrate one complete real user task.
    10. No one knows the state that requires work to stop.
    Score Interpretation
    0–2 Normal development variation
    3–4 Scope-control risk is visible
    5–7 Infinite refactoring is underway
    8–10 Freeze new features and rebuild the delivery loop

    Replaying the incident with the correct chain¶

    sequenceDiagram participant O as Owner participant A as AI participant S as Source repository participant C as CI participant P as Publication repository participant W as Production site O->>A: Approve one outcome and change boundary A->>S: Commit the minimum necessary change A->>S: Open PR with evidence O->>S: Review and approve merge S->>C: Push to main C->>C: Test, build, and visual verification C->>P: Publish built artifact P-->>A: Return publication commit and source SHA A->>W: Check old route, new page, and device layouts A-->>O: Submit complete five-layer evidence O->>O: Accept or return only explicit failures

    The value of the loop is not additional ceremony. It prevents a low-level state from impersonating a high-level completion.

    The structured model to keep¶

    The entire article reduces to two formulas.

    Root-cause model¶

    Product-system failure
+ decision-system failure
+ verification-system failure
+ release-system failure
= infinite refactoring

    Resolution model¶

    One outcome
→ change boundary
→ minimum change
→ five-layer evidence
→ user acceptance
→ freeze version

    AI projects usually do not enter infinite refactoring because the model is too weak. They enter it because generation capacity grows faster than the project’s decision, verification, and stopping systems.

    “Code changed,” “tests passed,” and “PR merged” prove progress. They do not prove delivery.

    Real delivery requires all five:

    Correct source
+ rules passed
+ build succeeded
+ release completed
+ user outcome achieved

    Miss one layer and the task is still open.

    The end of infinite refactoring is not perfection. It is:

    The problem is closed, the evidence holds, the user accepts the result, and the project now stops changing.
