    From Using Codex to Operating an Engineering Agent¶

    Two engineers can use the same Codex model on the same repository and still produce radically different outcomes. The first keeps extending the prompt: ask for code, correct the direction, explain the directory, request tests, and finally warn the agent not to touch one important file. The second asks Codex to read the repository contract, establish the change boundary, and name the acceptance evidence before it edits anything. After the change, the second user requires tests, diffs, rendered behavior, and human judgment before accepting the result.

    The difference is not who is better at chatting with AI. One person is requesting an answer; the other is operating a constrained, evidence-producing engineering system.

    An experienced Codex user can answer five questions consistently:

    1. What context does Codex need now, and what noise should stay out?
    2. Which judgments can be delegated, and which must remain human-owned?
    3. Which tools and state changes are allowed, and where must the agent stop for approval?
    4. What evidence proves completion instead of merely suggesting it?
    5. Should a correction remain in this task, or become a test, AGENTS.md rule, skill, or tool?

    This article develops those practices from first principles. It does not promise that AI can replace the engineering owner. It shows how Codex can grow from an occasional code generator into a reviewable engineering executor.

    Scope and version boundary¶

    This article reflects official OpenAI Codex documentation and practical local workflows available on 2026-07-12. Product surfaces, model names, account entitlements, and feature availability change. Consult current official documentation for the exact entry points for AGENTS.md, skills, MCP, sandboxing, approvals, review, subagents, and worktrees.

    The durable subject here is not a particular button. It is the operating model: task contracts, least privilege, capability boundaries, verification evidence, and human acceptance.

    The problem: treating an engineering agent as a chat box¶

    The most common beginner failure is not a short prompt. It is assigning every responsibility to one vague sentence:

    Improve this project and keep going until it is finished.

    There is no definition of improvement, no change boundary, no verification method, and no stop condition. The model must infer the objective from partial repository signals. It may produce a large amount of work while leaving the user with a harder diff, more maintenance cost, and conclusions that cannot be proven.

    A coding agent differs from a conversational model because it reads files, runs commands, changes state, and calls external tools. The prompt no longer affects only the prose of an answer. It defines the control plane of a real execution process. An underspecified task is therefore missing change management, not merely a few helpful words.

    The core model: an evidence-driven agent loop¶

    A mature workflow is not prompt followed by response. It is a six-state loop:

    flowchart LR A[Observe<br/>repository, constraints, state] --> B[Define<br/>outcome, boundary, acceptance] B --> C[Execute<br/>small reviewable change] C --> D[Verify<br/>tests, builds, runtime evidence] D --> E[Review<br/>risk, judgment, user experience] E --> F[Capture<br/>tests, guidance, skill, tooling] F --> A D -- insufficient evidence --> B E -- result rejected --> B

    The key change is that verification is not an optional activity after generation; it governs generation from the beginning. Tests, screenshots, performance measurements, data reconciliation, and human review determine which implementation is acceptable before editing starts.

    Stage one: turn a wish into a task contract¶

    An executable task needs at least five parts:

    Part Required question Example
    Outcome What visible result must exist? The homepage lists documents by update date
    Scope Which files, systems, or user paths are involved? Homepage, styling, build-time index generation
    Exclusions What must not change? Article bodies, URLs, and publication workflow
    Acceptance Which facts determine success? Twelve items per page, bilingual parity, no mobile overflow
    Handoff Where must Codex stop? Verified temporary branch awaiting human review

    A useful prompt can be compact as long as the result is falsifiable:

    Outcome: replace the homepage with a recently updated document list.
Scope: bilingual homepages, build-time index, pagination logic and styles.
Do not change: article bodies, topic paths, publication workflow.
Ordering: use the last Git content modification; ignore path-only moves.
Acceptance: 12 items per page; numbered, previous and next navigation;
            direct page URLs; no overflow at 390 px; matching locale order;
            strict build passes.
Handoff: keep the work on a temporary branch with rendered evidence.

    This is not a trick for eliciting better prose. It is a small change request. It gives Codex room to choose implementation details without interpreting autonomy as unlimited scope.

    Stage two: establish facts before authorizing action¶

    Experienced users separate diagnosis from implementation. The first pass should answer:

    • What branch, working tree, and base revision are active?
    • Which repository instructions apply?
    • Which source files and generators own the current behavior?
    • Which statements are verified facts, and which are assumptions?
    • What is the smallest coherent change surface?
    • Which steps require network access, external writes, or elevated permission?

    The purpose is not to obtain a ceremonial plan. It is to expose false premises. Rework often begins when an agent works from the wrong base branch, misses nested guidance, edits generated output instead of its source, or treats a successful build as proof of publication.

    Plans should remain proportional. Maintain one when the work contains dependent stages, meaningful risk boundaries, or irreversible state. A one-line documentation fix does not need a ten-step project plan. A plan is useful when it makes state and decisions observable.

    Stage three: put each correction in the right durable layer¶

    A correction made in one conversation disappears easily. The advanced skill is deciding where it belongs.

    Surface Good fit Poor fit
    Current prompt One-off outcome, immediate scope, temporary judgment Rules that apply to every task
    AGENTS.md Repository conventions, build commands, review rules, directory boundaries Detailed implementation plan for one change
    .codex/config.toml Trusted-project settings for sandbox, model, MCP, and hooks Editorial standards or a temporary objective
    Skill A reusable workflow with references, scripts, and acceptance steps A fact that belongs only to one repository
    MCP or connector Current external context and controlled remote actions Static rules that can live in the repository
    Test or validator Invariants that can be checked mechanically Editorial judgment such as whether an article is insightful

    Official guidance recommends keeping AGENTS.md small and using it for recurring build, test, review, and directory-specific expectations. A useful feedback loop looks like this:

    1. Fix the first failure in the current task.
    2. When it repeats, decide whether it is a stable pattern.
    3. Encode mechanically decidable behavior as a test or validator.
    4. Put behavioral repository constraints in the nearest AGENTS.md.
    5. Extract a cross-repository process into a skill.
    6. Add MCP only when the workflow needs current external data or actions.

    Do not turn every disappointment into another rule. Large guidance files crowd the context and hide contradictions. Preserve the small set of constraints that recur, change behavior, and can be reviewed.

    Stage four: move from code generation to capability orchestration¶

    Codex capabilities live in distinct layers:

    • AGENTS.md defines how work happens in a repository.
    • Skills package repeatable workflows, optionally with scripts, references, and assets.
    • MCP connects Codex to external context and tools such as documentation, browsers, design systems, or internal platforms.
    • Plugins distribute related skills, connectors, and capabilities.
    • Subagents isolate genuinely independent work in separate contexts.
    • Hooks and validators enforce mechanical constraints at lifecycle boundaries.

    More tools do not automatically produce better work. Each tool adds a choice branch, a permission surface, and a failure mode. Mature setups use the smallest sufficient capability set:

    1. Prefer repository facts over an external lookup when the repository is authoritative.
    2. Prefer current official documentation over model memory for product behavior.
    3. Prefer read-only access when it can answer the question.
    4. Do not turn a one-off sequence into a skill prematurely.
    5. Do not delegate work that cannot be accepted independently.

    Skills are valuable because they stabilize triggers, steps, references, and evidence—not because they contain more prose. Official documentation explains that Codex initially sees skill metadata and loads the full SKILL.md only after a match. A vague description can therefore make a well-written skill unreliable.

    Stage five: make verification govern generation¶

    “The code is written” is not completion. Require an evidence ladder:

    Layer What it proves What it cannot prove
    Static checks Syntax, formatting, typing, or links obey rules Real behavior works
    Automated tests Encoded behavior has not regressed Unencoded user intent is correct
    Build and runtime The artifact can be produced and started Critical interaction and visual quality are acceptable
    Browser or end-to-end check A user path actually executes Product judgment and maintainability are sound
    Human review Scope, risk, technical judgment, and experience are acceptable The system will never change

    For a paginated document homepage, “the build passed” is insufficient. Evidence should include:

    • total entries, page size, page count, and last-page boundary;
    • direct access to ?page=2 and browser back/forward behavior;
    • disabled state of previous and next controls;
    • proof that the first item is the latest content modification;
    • matching bilingual entry counts and order;
    • desktop, mobile, light, and dark rendering;
    • a real navigation from a list item to the target document;
    • no new browser-console errors;
    • the complete repository quality gate.

    When verification fails, require Codex to classify the failure as implementation, environment, test, or permission before fixing it. Do not accept green results achieved by weakening tests, disabling strict mode, or suppressing actionable errors.

    Stage six: separate author, verifier, and approver¶

    Self-checking by the same model is useful, but it is not independent review. An author naturally searches for evidence along the assumptions used to build the change.

    Codex /review can inspect uncommitted changes, a branch, a commit, or a custom scope and return prioritized findings without editing the working tree by default. A mature flow has three layers:

    1. Author verification runs the tests, build, and target-path checks.
    2. Independent agent review reads the diff and searches for regressions, omissions, and false assumptions.
    3. Human approval decides whether the content, risk, user experience, and publication boundary are acceptable.

    A pull request is not the essential part. The essential part is that approval is not disguised as the author’s own conclusion. For high-risk work, the reviewer should rerun critical evidence rather than only reading the author’s summary.

    Stage seven: parallelize only genuinely independent work¶

    Subagents and worktrees can accelerate large tasks, but parallelism is not decoration.

    Good parallel tasks usually have three properties:

    • the input boundary is clear, such as accessibility audit versus build audit;
    • the result can be accepted independently, such as a finding list or changes to disjoint files;
    • the task does not require frequent shared intermediate state.

    Poor candidates include two agents editing the same page, an architecture decision that invalidates another agent’s entire premise, or work that can only be integrated by trusting a summary.

    Official documentation notes that subagents consume additional tokens but isolate exploration logs and intermediate noise in separate contexts. Worktrees give parallel Git tasks separate file checkouts so they do not disturb the current workspace. Use them with a clear operating model:

    • keep objectives, constraints, and final judgment in the main task;
    • delegate independently executable research, implementation, or review;
    • require evidence and unresolved questions, not only conclusions;
    • isolate shared-file changes in worktrees or coordinate them serially;
    • rerun integrated verification in the main task.

    Stage eight: treat agent permissions like production permissions¶

    Codex can execute shell commands, access networks, modify files, and call remote tools. Permission design cannot depend on the hope that the agent will not make a mistake.

    Sandboxing and approval policy are separate controls: the sandbox defines what is technically possible; the approval policy determines when crossing a boundary must stop for human confirmation. Experienced users do not disable all boundaries to reduce prompts. They build a safe operating area for routine autonomy:

    • write only inside the active workspace by default;
    • keep network access off until a scoped download or lookup needs it;
    • do not grant mutation rights to a diagnostic task;
    • confirm deletion, publication, message sending, permission changes, and production actions at action time;
    • expose secrets only to the shortest stage that needs them, never to prompts, logs, or the repository;
    • approve narrow reusable command prefixes rather than arbitrary shell execution.

    High autonomy does not require broad privilege. Sustainable autonomy comes from clear boundaries: the agent can continue inside the safe region and knows where it must stop.

    End-to-end practice: turning “change a homepage” into a reviewable delivery¶

    Assume the task is to replace a bilingual static documentation homepage with a recently updated list and improve its visual quality.

    1. Observe¶

    Read the repository contract, theme configuration, homepage sources, generation script, and publication workflow. Confirm that the product is a static MkDocs site and reject a database or backend service as unnecessary.

    2. Define the meaning¶

    Define “recently updated” as the last actual content modification in Git, not filesystem mtime. A path-only migration must not make every article appear newly written. This semantic decision matters more than the CSS because it determines whether the homepage is trustworthy.

    3. Fix the boundary¶

    Change only homepage generation, pagination behavior, and visual styling. Preserve article bodies, URLs, publication repository, and topic classification. Keep all work on a temporary branch.

    4. Choose the smallest architecture¶

    Read Git history at build time and generate both localized homepages. Let the browser handle only pagination. This avoids a database, API, admin interface, and manual updated metadata.

    5. Build a degradable interaction¶

    Keep a complete link list without JavaScript. With JavaScript, expose a fixed page size, store the page in the URL, and restore state through browser history.

    6. Measure the visual reference¶

    Do not guess what “premium typography” means. Inspect the reference site’s computed font family, size, line height, color, and row density. Then build a cross-platform fallback and verify accessible contrast.

    7. Verify¶

    Run generator consistency, public-safety checks, and a strict build. Test Chinese desktop, English desktop, Chinese mobile, English mobile, and dark desktop. Exercise page numbers, previous/next, direct URLs, article navigation, and console errors.

    8. Hand off¶

    Report changed files, untouched areas, date semantics, measured browser results, and validation commands. Stop on the uncommitted branch so the maintainer can inspect the rendered product before approval.

    The value is not that Codex wrote CSS and JavaScript. The workflow connects requirement semantics, source authority, architecture, implementation, runtime proof, and approval into one evidence chain. Code is one segment of that chain.

    Common failure modes¶

    Failure Why it fails Correction
    Keep extending the prompt Context grows while objective and acceptance remain vague Rewrite the task contract
    Put every lesson in AGENTS.md Every task pays for irrelevant context Keep only stable recurring constraints
    Create a skill after one repetition An immature workflow becomes permanent Reuse and correct it before abstraction
    Grant every permission Convenience and blast radius expand together Use least privilege and action-time approval
    Spawn many subagents Conflicts, duplicate exploration, and lossy summaries increase Delegate only independently acceptable work
    Merge when CI is green Automation cannot prove editorial or product judgment Preserve rendered review and human approval
    Let Codex declare itself successful The author validates its own assumptions Require external evidence and independent review
    Edit generated output directly The next synchronization overwrites the change Locate and change the source of truth

    Four practice cycles¶

    This is not a capability score. It is a sequence of exercises that gradually increases the control surface.

    Cycle one: require acceptance evidence for every task¶

    Choose five small changes. Before each one, state outcome, scope, exclusions, and validation. Reject summaries that say only “done”; inspect the diff and evidence.

    Cycle two: establish a repository feedback loop¶

    Track repeated Codex mistakes. Encode mechanical failures as tests and stable repository expectations in the nearest AGENTS.md. After one week, delete rules that did not change behavior.

    Cycle three: abstract one real workflow¶

    Choose a process that has succeeded manually several times, such as bilingual documentation publishing, dependency-incident audit, or front-end visual acceptance. Package triggers, inputs, steps, validation, and references as a skill. Test it on a similar but non-identical task.

    Cycle four: run a multi-role delivery¶

    Use one task to implement and an independent task for read-only review. Add subagents only for genuinely independent investigation, and use a worktree when modifications need isolation. Rerun integrated verification in the main task, then leave acceptance to a human.

    Boundary: responsibilities Codex cannot own¶

    Codex can expand an engineer’s ability to observe and execute, but it cannot become the accountable owner. A person must explicitly own:

    • whether the requirement is worth its opportunity cost;
    • whether production data, customer information, and credentials may enter a tool;
    • whether the business impact of an irreversible action is acceptable;
    • whether technical evidence is sufficient for publication or production change;
    • which user, organizational, and long-term maintenance risks automation missed;
    • whether to commit, merge, deploy, send, publish, or delete.

    When nobody owns those decisions, a stronger model only produces unowned decisions faster.

    Operational conclusions¶

    Experienced Codex practice reduces to ten rules:

    1. Provide an outcome, boundary, and acceptance test—not a vague wish.
    2. Establish facts before authorizing mutation.
    3. Separate verified facts, reasonable inferences, and unknowns.
    4. Define verification before implementation.
    5. Capture repeated corrections in the right durable surface instead of growing prompts forever.
    6. Use the smallest sufficient tool set and least privilege.
    7. Parallelize only independent work that can be accepted separately.
    8. Use independent review to counter author bias.
    9. Let automation own mechanical integrity and humans own judgment, risk, and acceptance.
    10. Turn each successful workflow into a shorter and safer starting point for the next task.

    At that point Codex is no longer a chat partner that must be repeatedly steered back toward the task. It is an engineering agent that works inside an explicit contract, produces evidence, stops at permission boundaries, and remains accountable to a human owner.
