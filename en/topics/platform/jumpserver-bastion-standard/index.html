    JumpServer and Bastion Access Standard¶

    Purpose¶

    This standard defines how to operate JumpServer or a bastion platform as the controlled entry point for privileged infrastructure access.

    The core judgment:

    A bastion is not a login shortcut. It is the audit boundary for privileged operations.

    If engineers use it to reach servers, databases, Kubernetes nodes, network devices, or internal consoles, the bastion must preserve the answer to five questions:

    • Who operated?
    • Which asset was touched?
    • What privilege was used?
    • What evidence was recorded?
    • How is access revoked under pressure?

    A bastion that only centralizes SSH traffic but leaves direct paths open, shared accounts active, sessions unreviewed, and temporary users permanent is not an access-control system. It is a more convenient bypass with a nicer login page.

    When to Use¶

    Use this standard when:

    • Engineers need SSH, RDP, database, or web console access to private assets.
    • Direct access to servers should be removed or reduced.
    • Operations require session recording.
    • Vendors or temporary users need controlled access.
    • Break-glass access must exist without destroying accountability.
    • Asset owners need evidence for access review and incident investigation.

    Non-Goals¶

    This standard does not approve shared root access, unmanaged direct SSH, permanent vendor access, or using a bastion as a substitute for least-privilege cloud IAM. It also does not define every JumpServer feature. The standard defines the access-governance contract that a bastion must satisfy before it is trusted for production assets.

    Production Contract¶

    Before onboarding production assets, define:

    Contract Required Decision Why It Matters
    Asset owner Who owns each server, database, cluster, network device, or console? Access approval without an asset owner has no authority.
    Identity source Which identity system is authoritative for users and groups? Local orphan users weaken offboarding and incident response.
    Access class Which access is routine, privileged, vendor, or emergency? Review frequency and evidence requirements differ by class.
    Privilege boundary Which commands, accounts, databases, or consoles can each role use? A login path is not the same as controlled privilege.
    Direct path policy Which direct SSH, RDP, database, or console paths are closed or exception-approved? Open direct paths bypass bastion evidence.
    Session evidence What must be recorded, retained, protected, and reviewable? An audit boundary needs protected evidence, not only logs.
    Temporary access Who approves vendor or time-bound access and how expiry is enforced? Temporary access becomes permanent unless expiry is automatic or reviewed.
    Break-glass path How emergency access works when JumpServer or identity is down? Incidents need access, but the fallback must be narrower than the normal path.
    Revocation owner Who can revoke access, rotate credentials, and disable sessions quickly? Slow revocation turns access governance into paperwork.
    Platform recovery How JumpServer configuration, recordings, keys, and asset mappings recover? Losing the bastion can also lose evidence and access inventory.

    Recommended Standard¶

    Treat assets as governed inventory¶

    Each asset needs:

    • owner
    • environment
    • asset type
    • criticality
    • allowed access method
    • approved user group
    • privileged account mapping
    • emergency owner
    • review cadence
    • direct-path exception status
    • decommission owner

    Unknown-owner assets should not be exposed through the bastion as a convenience.

    Close or classify direct paths¶

    A bastion only works as an audit boundary when ordinary access flows through it.

    For each asset class, decide whether direct access is:

    Path Default Required Evidence
    SSH/RDP from engineer networks Closed Network policy, security group, firewall, or VPN rule.
    Database client access Closed or explicitly scoped Source ranges, users, roles, timeout, and query logging.
    Kubernetes node access Closed except break-glass Owner, approval, incident link, and post-action reconciliation.
    Network-device admin access Restricted Approved management network and emergency path.
    Cloud console or internal console Identity-governed MFA, role mapping, and audit log retention.

    If a direct path must remain open, record the owner, reason, scope, expiration or review date, compensating evidence, and revocation method. A permanent undocumented exception is a bypass, not a control.

    Separate roles and review duties¶

    Minimum role model:

    • Platform administrator.
    • Auditor.
    • Asset owner.
    • Operator.
    • Temporary vendor or emergency user.

    Administrators should not be the only people who can review their own sessions. The person who grants access should not be the only person who decides whether the recorded operation was acceptable.

    Access should map to named identities. Shared privileged accounts may still exist on old systems, but the bastion must bind their use to a named human, an approval context, and a recorded session.

    Record and review sessions¶

    Privileged sessions should be logged and recorded. If law, policy, or tooling prevents full recording for a high-risk path, access should fail closed unless an approved alternate evidence path exists: command audit, database audit log, incident or change ticket, independent reviewer, and retained proof that the operation matched the approval.

    Protect recordings and audit logs as evidence. They should have retention, access control, integrity protection where available, backup, and deletion rules. A recording that any platform admin can edit or delete without evidence is not reliable audit material.

    Review triggers:

    • Production root or sudo access.
    • Database administrative access.
    • Kubernetes node access.
    • Firewall, route, VPN, DNS, or load-balancer changes.
    • Emergency access.
    • Vendor access.
    • Failed login bursts.
    • Access outside maintenance window.
    • Session recording failure.
    • Direct-path exception use.

    Govern temporary and vendor access¶

    Temporary access should have:

    • requester
    • business or incident reason
    • approving asset owner
    • exact asset scope
    • allowed protocol and account
    • start and end time
    • session recording requirement
    • revocation owner
    • review evidence after use

    Do not rely on a calendar reminder as the control. Use expiry, group removal, or a scheduled review where missed or rejected review removes access by default.

    Keep break-glass narrower than normal access¶

    Break-glass exists for incidents, not convenience. It should define:

    • who can invoke it
    • what condition permits it
    • which assets it reaches
    • which credentials or path it uses
    • how the action is recorded if the bastion is unavailable
    • who is notified
    • how credentials rotate afterward
    • how changes are reconciled after recovery

    The emergency path should be smaller than normal standing access. If break-glass gives wider, quieter, or longer-lived access than the bastion, it will become the real access model during stressful work.

    Validation¶

    Before onboarding an asset:

    • Asset owner, criticality, environment, and review cadence are recorded.
    • User can access only approved assets and accounts.
    • Direct access path is closed or exception-approved.
    • Session recording works and the recording is retrievable by an auditor.
    • Command audit works where supported.
    • Revocation removes access from the bastion and from the underlying account where applicable.
    • Temporary access expires without manual cleanup.
    • Emergency path is documented, tested, and reconciled after use.
    • Audit logs and recordings survive platform restart and backup/restore testing.

    Operational tests should cover each approved access method:

    Access Method Test
    SSH or RDP Access through bastion succeeds; direct path is denied or exception-approved; session evidence is retrievable.
    Database access Approved database role works only through the approved path; direct client access is denied or scoped; database audit evidence exists.
    Kubernetes node access Routine node access is denied; break-glass path requires approval, evidence, and reconciliation.
    Network device access Management path is restricted to the approved network or bastion flow; config-changing session evidence is retained.
    Web or cloud console Identity, MFA, role mapping, and audit log entries match the approved access class.
    Revocation Removing the user or group removes bastion access and underlying asset access where applicable.

    Use placeholder hostnames and asset identifiers in public docs. Do not publish real asset names.

    Failure Modes¶

    Failure Mode Result Prevention
    Shared privileged account No accountability. Map access to individual identity.
    Direct SSH remains open Bastion audit is bypassed. Restrict direct network paths.
    Direct database or console path remains open Sensitive operations happen outside recorded sessions. Close paths or record exception scope, owner, and expiry.
    Session recording can be edited or deleted silently Evidence cannot be trusted during incident review. Protect recording storage and audit administrative access.
    No session review Recording exists but risk is not managed. Schedule and trigger reviews with named reviewers.
    Vendor access never expires Temporary access becomes permanent. Enforce expiry and owner approval.
    Break-glass wider than normal access Incidents bypass governance exactly when risk is highest. Keep emergency scope narrow and reconcile afterward.
    Bastion unavailable Operators lose controlled emergency access. Test break-glass and recovery path.
    Asset owner unknown Access approval has no authority. Maintain asset inventory.
    Identity offboarding not connected Former users retain access through local accounts or groups. Integrate identity lifecycle and run revocation tests.
    Platform backup omits recordings or asset mappings Recovery restores login but loses evidence or scope. Back up configuration, mappings, keys, and evidence metadata.

    Rollback and Revocation¶

    To remove access:

    • Remove user from role or group.
    • Revoke asset authorization.
    • Disable active sessions when risk is immediate.
    • Rotate shared asset credentials if they were used or exposed.
    • Disable temporary account.
    • Remove direct-path exceptions if the reason no longer exists.
    • Review past sessions if access was risky or unauthorized.
    • Preserve audit evidence before deleting accounts or mappings.

    For bastion platform failure, use break-glass access only with an incident record. After recovery, reconcile what changed, rotate emergency credentials, attach evidence, and close the fallback path.

    Security Notes¶

    • Enforce MFA for privileged users.
    • Restrict admin access to JumpServer itself.
    • Protect session recordings and audit logs.
    • Avoid storing unmanaged shared credentials.
    • Review sudo and root access separately.
    • Review database administrative access separately from ordinary read access.
    • Keep asset-to-owner mappings current.
    • Do not expose the bastion admin interface publicly without additional controls.
    • Patch the bastion platform and its dependencies.
    • Treat bastion backup data as sensitive because it may contain credentials, recordings, asset names, and access mappings.

    Review Cycle¶

    Review after:

    • staff change or team transfer
    • vendor onboarding or offboarding
    • privileged-access incident
    • failed recording or audit-log gap
    • direct-path exception creation or expiry
    • bastion upgrade
    • identity-provider change
    • asset criticality change
    • emergency access use
    • backup or restore test
