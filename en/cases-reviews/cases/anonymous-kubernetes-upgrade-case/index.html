    Kubernetes Migration Reliability

    Anonymous Kubernetes Upgrade Case¶

    All identifiers in this case are generic. No original service names, domains, regions, accounts, repositories, manifests, or customer details are preserved.

    What This Case Is Really About¶

    The visible change was a Kubernetes minor-version upgrade. The real change was a compatibility-envelope migration.

    The control plane was only one clock. The cluster also had an API-removal clock, an add-on compatibility clock, a worker-node runtime clock, and a workload-behavior clock. The upgrade became safe only after the team stopped asking "can the control plane move?" and started asking "which part of the cluster loses reversibility at each step?"

    That distinction mattered because the platform could upgrade the control plane quickly, but it could not prove that DNS, CNI, ingress, storage, observability, PodDisruptionBudgets, and application assumptions would all keep behaving under the new version.

    Decision Frame¶

    The upgrade was split around three decisions:

    Decision Why it mattered Required evidence
    Treat add-ons as upgrade subjects, not background components. CNI, CoreDNS, kube-proxy, CSI, ingress, and observability agents define whether workloads actually run. Target-version compatibility matrix and health checks for each add-on.
    Replace nodes through new node groups. In-place mutation makes rollback ambiguous and hides capacity problems. New nodes join cleanly, old nodes remain drainable, and workloads can move in batches.
    Keep irreversible changes out of the window. A control-plane upgrade is already hard enough to reason about. No storage migration, schema migration, or credential rotation that would break rollback.

    This made the upgrade a sequence of gates instead of a single maintenance event.

    Starting Conditions¶

    • One production cluster on an older supported minor version.
    • Mixed workloads: stateless services, ingress, monitoring, scheduled jobs, and stateful components using persistent volumes.
    • Core add-ons tied to the current Kubernetes version.
    • Alerting existed, but not every critical alert had an explicit runbook.
    • Some workloads had strict PodDisruptionBudgets that had not been tested against node replacement.
    • Custom resources, admission or conversion webhooks, aggregated APIs, and controllers had to be checked as part of the API surface, not treated as application detail.

    Constraints¶

    • The cluster had to remain available through the change.
    • Worker nodes could not all be replaced at once without capacity pressure.
    • Persistent-volume behavior had to be verified, not assumed.
    • Monitoring and alert delivery had to survive the upgrade.
    • The old node group had to remain a working fallback until the observation window passed.

    Upgrade Model¶

    The team modeled the upgrade as four clocks that had to be synchronized:

    Clock What could break Gate before moving on
    API surface Removed or changed Kubernetes APIs, CRDs, admission/conversion webhooks, aggregated APIs, and controllers. No active manifest, CRD, webhook, APIService, or controller depended on an incompatible API behavior.
    Add-ons DNS, networking, proxying, ingress, storage, metrics, logs. Add-on versions matched the target release and passed health checks.
    Nodes Runtime, kernel, kubelet, image pull, capacity, drain behavior. New node group admitted pods and old nodes drained in small batches.
    Workloads PDBs, probes, startup timing, persistent storage, external callbacks. Batch smoke tests passed and alert noise stayed explainable.

    The team did not let a green control plane count as a green cluster.

    Execution Sequence¶

    1. Freeze unrelated infrastructure and application changes.
    2. Inventory APIs, add-ons, nodes, ingress routes, PVCs, PDBs, critical workloads, alerts, and runbooks.
    3. Scan manifests, CRDs, admission webhooks, conversion webhooks, APIService health, and controllers for deprecated or removed API behavior.
    4. Snapshot critical manifests and platform settings so rollback was not dependent on memory.
    5. Upgrade the control plane through the supported platform path.
    6. Upgrade or pin add-ons to target-compatible versions in a controlled order.
    7. Create a new worker node group at the target version.
    8. Move workloads by cordoning and draining old nodes in small batches.
    9. Validate cluster health, workload health, storage behavior, ingress behavior, and monitoring after each batch.
    10. Keep old nodes until the observation window passed, then remove them deliberately.

    Validation Evidence¶

    Layer Evidence Stop condition
    API Deprecated API scan, CRD versions, webhook readiness, APIService health, controller logs, admission behavior. Any active workload, CRD, webhook, APIService, or controller still needs an incompatible API path.
    Add-ons CoreDNS resolution, CNI connectivity, kube-proxy behavior, CSI attach/mount, ingress routing, metrics/log delivery. A core add-on is unhealthy, unsupported, or unverifiable.
    Node movement New node readiness, image pulls, daemonsets, drain duration, pending pods, PDB blocks. Drain requires force deleting critical pods or creates sustained capacity pressure.
    Workloads Rollout status, readiness, startup latency, error rate, scheduled job completion. Any critical workload cannot complete a batch smoke test.
    Operations Alerts, dashboards, on-call routing, runbook links. A critical failure mode cannot be seen or routed during the window.

    The most important validation habit was stopping at the first unclear layer. If DNS became flaky after add-on movement, the team did not continue draining nodes and hope the symptom would disappear under more change.

    Failure Modes That Surfaced¶

    Add-On Lag Looked Like Application Instability¶

    The control-plane operation completed cleanly, but a version mismatch in networking and DNS components produced intermittent failures. At first those failures looked like workload regressions.

    The compatibility matrix prevented the team from debugging the wrong layer for too long. Add-ons were moved back into the center of the upgrade plan, and each one gained an explicit health check.

    PodDisruptionBudgets Became Capacity Contracts¶

    Some PDBs were written as if disruption were a rare single-pod event. During node replacement they became hard constraints on drain speed and batch size.

    The fix was not to bypass PDBs. The team adjusted batch size, added capacity where needed, and treated "cannot drain safely" as evidence that the workload's availability contract needed review.

    Storage Symptoms Nearly Hid Under the Upgrade¶

    One persistent volume was close to capacity before the window. Its symptoms appeared during the upgrade and could have been misread as a version problem.

    PVC headroom and attach/mount checks were added to the preflight gate. The lesson was simple: storage pressure that already exists will use the upgrade window as camouflage.

    Monitoring Had to Be Upgraded as Part of the System¶

    Observability agents were not "after the upgrade" work. If logs, metrics, or alerts disappeared during the window, the team would lose the ability to decide whether to continue.

    Monitoring components were therefore treated as part of the upgrade path, with their own version checks and smoke tests.

    Reversibility Boundary¶

    The control-plane upgrade was treated as forward-moving. The practical rollback model came from keeping workloads movable:

    • Old node groups remained available until the target node group handled real workload batches.
    • Manifest snapshots preserved the ability to roll back workload definitions.
    • Application rollback remained separate from cluster upgrade rollback.
    • No storage layout change or schema change was allowed in the same window.
    • The team did not remove the old node group until alerts, scheduled jobs, ingress, DNS, and storage behavior had survived the observation period.

    This avoided a false rollback promise. The plan did not pretend the entire cluster could time-travel; it preserved the parts that mattered for service recovery.

    Outcome¶

    The cluster reached the target version without sustained outage. The decisive move was not the control-plane operation. It was building a reversible path for workloads and forcing add-ons, nodes, storage, and observability to pass their own gates.

    After the upgrade, the team's review checklist changed. A Kubernetes upgrade was no longer considered ready because the target version was supported. It was ready only when the compatibility envelope had been made visible.

    Reusable Review Questions¶

    • Which add-ons must move before workloads can be trusted on the target version?
    • Which APIs are removed or changed between the current and target versions?
    • Are CRDs, admission/conversion webhooks, APIService objects, and controllers compatible with the target version?
    • Can every critical workload drain without force deletion?
    • What capacity is required to keep old and new node groups alive at the same time?
    • Which storage checks prove attach, mount, capacity, and application reads?
    • Can the team see DNS, ingress, storage, and workload failures during the window?
    • What exactly remains reversible after the control plane moves?
