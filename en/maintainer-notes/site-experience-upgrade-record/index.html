    Site Experience Upgrade Record¶

    This document records the experience upgrade direction, trade-offs, and rollout order for the DevOps blog site. It is not a reader-facing expert article. It is a maintainer decision record for future changes to theme, navigation, blog features, and the AI assistant entry point.

    Background¶

    The site already has two foundation capabilities:

    • MkDocs Material is used as the documentation site framework.
    • mkdocs-static-i18n is used for bilingual publishing. Chinese is published at the site root, English is published under /en/, and old /zh/ paths are kept as redirect aliases.

    This solves publishing and bilingual switching, but it does not yet solve the larger problem: making the site feel like a long-lived blog from a senior DevOps practitioner. Future upgrades should focus on credibility, readability, navigation efficiency, and content production discipline rather than decoration.

    Design Goals¶

    Site experience upgrades should serve four goals:

    1. Credibility: the homepage, navigation, and article structure should quickly show that the author has handled production DevOps work, not only collected tool commands.
    2. Readability: long articles, standards, reviews, and cases must remain comfortable on desktop and mobile.
    3. Maintainability: theme, language, navigation, blog features, and publishing flow must be understandable and reversible.
    4. Q&A assistant readiness: content should preserve clear headings, stable URLs, explicit context, and retrieval-friendly knowledge blocks.

    Current Foundation¶

    Internationalization¶

    The current site uses plugin-level internationalization instead of hand-written language links.

    • Chinese source files live under docs/zh/.
    • English source files live under docs/en/.
    • Chinese is built to the site root.
    • English is built to /en/.
    • Old /zh/ paths remain as redirects to avoid breaking external links.

    Maintenance principles:

    • New formal content should have Chinese and English versions when possible.
    • If the English version cannot meet the same quality bar yet, do not publish a weak direct translation.
    • Do not add manual [English] or [中文] links at the top of pages. Language switching is handled by the plugin.

    Content Structure¶

    The current site is organized as an expert blog:

    • Start Here
    • Blog
    • Expert Methodology
    • Engineering Systems
    • Standards & Playbooks
    • Technical Knowledge Base
    • Site Operations

    Maintenance principles:

    • New deep articles should go into Expert Methodology and Engineering Systems instead of being added to the technical knowledge base.
    • The Technical Knowledge Base carries sanitized tool, standard, and operation material; it should not be the main narrative of the site.
    • Reader-facing articles and maintainer rules must remain separate.

    Recommended Upgrade Path¶

    Phase 1: Theme And Navigation Quality¶

    The goal is to improve first impression with low-risk configuration changes.

    Recommended additions:

    • Light and dark theme toggle with system preference support.
    • Restrained brand colors without heavy decorative gradients.
    • Top navigation tabs for major sections.
    • Back-to-top, following table of contents, search highlight, and search suggestions.
    • A lightweight logo or icon that fits a technical site.

    Acceptance criteria:

    • The homepage does not feel like a generic template.
    • Long-form reading is comfortable.
    • Mobile navigation does not crowd the article body.
    • Changes are mostly limited to mkdocs.yml and small CSS additions, so rollback is easy.

    Phase 2: Blog Features¶

    The goal is to make the site feel like a continuously maintained expert blog, not only a documentation archive.

    Features to evaluate:

    • Blog plugin.
    • Article dates.
    • Tags and categories.
    • Archive pages.
    • RSS.
    • Author profile.

    Trade-offs:

    • If there is no stable publishing rhythm yet, do not rush into a full blog system.
    • Tags must stay disciplined. Avoid many tags that only contain one article.
    • Archives and RSS become valuable only after the site publishes consistently.

    Phase 3: External Sharing Quality¶

    The goal is to make shared articles look professional.

    Features to evaluate:

    • Social cards.
    • Open Graph metadata.
    • Favicon.
    • Custom article summaries.
    • Lightweight analytics.

    Trade-offs:

    • Analytics must not harm loading speed or reader privacy.
    • Social cards should prioritize deep articles, not every low-value archived document.

    Phase 4: Q&A AI Assistant Entry¶

    The goal is to reserve a natural entry point for the future DevOps Q&A assistant without turning the site into a chat app.

    Recommended approach:

    • Keep an Assistant entry in navigation or Start Here.
    • Assistant answers must cite site document sections.
    • The assistant must not bypass the public knowledge base to answer sensitive, internal, credential, or private questions.
    • When evidence is missing, the assistant should say so instead of inventing an answer.

    Trade-offs:

    • Start with retrieval plus cited answers. Do not rush into complex multi-turn automation.
    • The assistant should support reading. It should not take over the role of deep articles.

    What Not To Do Now¶

    • Large custom theme rewrites, especially complex dark UI, animation-heavy pages, or layout rewrites.
    • Comment systems, membership, or heavy analytics before the content quality is strong.
    • Forcing tags onto every legacy document.
    • Exposing internal maintainer rules as primary reader-facing navigation.
    • Adding heavy decorative images, gradient backgrounds, or visual noise for a false sense of polish.

    These changes may make the site more complex, but they will not increase expert credibility.

    Content Quality Versus Visual Quality¶

    The polish of a senior DevOps blog mostly comes from three things:

    • Articles explain production constraints, failure modes, and trade-offs.
    • Cases preserve anonymized scale, symptoms, decision pressure, and review conclusions.
    • The site structure lets readers enter from problems, not only from tool categories.

    Visual upgrades can amplify credibility, but they cannot replace it. If Deep Dives, Case Studies, and Decision Reviews lack evidence density, a beautiful theme will only make the site look like a better template.

    Change Record Requirement¶

    Every site-experience change should record:

    • Date.
    • Scope.
    • Reason.
    • Files affected.
    • Verification method.
    • Rollback path.

    Example:

    Date: 2026-06-27
Scope: i18n
Reason: Replace manual language links with plugin-managed bilingual routing.
Files: mkdocs.yml, docs/zh, docs/en, tools/create-zh-redirect-aliases.mjs
Verification: make build, make verifyDist, online HTTP status checks
Rollback: Restore previous mkdocs.yml and remove i18n plugin config

    Current Maintenance Priority¶

    The current priority is to complete the formal content structure: Blog entry point, author credibility, article cards, formal navigation, and section landing pages. Comments, analytics, and dynamic social cards should not enter the default publishing path until their configuration boundaries are clear.

    2026-06-27 Execution Record¶

    Scope¶

    This execution covers the first three phases:

    • Phase 1: theme and navigation quality.
    • Phase 2: blog features.
    • Phase 3: external sharing readiness.

    Implemented¶

    • Enabled light/dark theme switching, system preference support, top navigation, sticky tabs, breadcrumb path, previous/next footer navigation, following table of contents, search highlight, and search suggestions in mkdocs.yml.
    • Switched page typography to system fonts to avoid runtime Google Fonts dependency.
    • Switched to mkdocs-material[imaging] to support official social card image generation, using a local static brand-card layout so builds do not depend on Google Fonts downloads.
    • Enabled the Material blog plugin, added bilingual Blog entries, author metadata, and the first site experience upgrade post.
    • Enabled the Material tags plugin and added bilingual Tags pages.
    • Added tools/create-rss-feeds.mjs to generate RSS feeds for Chinese and English blog posts after the MkDocs build.
    • Added a lightweight brand icon and small CSS adjustments for headings, links, blockquotes, tables, and admonitions.

    Trade-offs¶

    • Did not rewrite the Material theme, keeping future upgrades and rollback simpler.
    • Did not add comments, analytics, or membership features because the current priority is content credibility and publishing stability.
    • Added one real change-record blog post instead of filling the archive with weak placeholder posts.
    • Did not keep mkdocs-rss-plugin because the current mkdocs-static-i18n combination triggers repeated date_from_meta.default_time warnings in strict mode. RSS is generated after the build instead, preserving the feature while keeping --strict viable.
    • Social cards use a unified brand card instead of per-page dynamic title cards for now. This keeps builds stable; dynamic title cards can be added later with an open-source font committed to the repository.

    Verification Requirement¶

    After each change, run:

    make build MKDOCS=/private/tmp/devops-kb-mkdocs-venv/bin/mkdocs
make verifyDist

    After publishing, check at least:

    • Home page.
    • Chinese and English Blog index pages.
    • Chinese and English Tags pages.
    • Chinese and English first blog post.
    • RSS file.

    2026-06-27 Blog Optimization Gap Closure¶

    Current State Review¶

    The previous round already delivered theme switching, navigation tabs, breadcrumb paths, brand color tuning, favicon, the Blog plugin, Tags, RSS, author metadata, and Social Cards. Three issues still made the site feel unfinished:

    • /blog/ had a title but no article cards, section explanation, or subscription entry point.
    • Author metadata was too thin to support a credible senior DevOps blog.
    • The navigation placed the outline, maintainer notes, and old knowledge base at the same level, making the main reading path unclear.

    Implemented In This Round¶

    • The Blog homepage is a real entry point with featured articles, RSS, tags, section positioning, and publishing standards.
    • Author metadata now describes production architecture, delivery engineering, reliability, platform engineering, and publication safety ownership, and links to GitHub.
    • Article card, status card, and maintenance card styles were added so Blog and maintainer pages do not feel like blank Markdown.
    • Navigation was reorganized into Home, Start Here, Blog, Expert Methodology, Engineering Systems, Standards & Playbooks, Technical Knowledge Base, and Site Operations.
    • Section landing pages no longer use temporary phrases such as "planned articles" or "first titles"; they now describe official positioning, reading paths, and quality standards.

    Deferred By Design¶

    • Comments are not enabled yet. A comment system requires GitHub Discussions, a Giscus category, public interaction boundaries, and moderation ownership. Adding a broken comment box would reduce trust.
    • Analytics are not enabled yet. Analytics require a privacy decision and a tool choice such as Plausible, Umami, or external log analysis. The current site should not add visitor-tracking scripts without that decision.
    • Social Cards continue to use a unified brand image. Per-page dynamic title cards require font, Chinese line wrapping, and build-stability validation, so they should be handled as a separate task.

    Completion Standard¶

    • The Blog homepage is no longer an empty shell and gives readers direct article and subscription entry points.
    • Navigation no longer uses an archive label as the public structure; older content is folded into a formal technical knowledge base.
    • Maintainer notes clearly state what is implemented, what is deferred, and why.

    2026-06-27 Site Foundation Hardening¶

    Background¶

    A full audit of blog.l2do.com identified the following gaps affecting site credibility and reader experience, addressed in a single round:

    • All technical articles (Standards, Runbooks, Architecture) had no publication date, so readers could not judge content currency.
    • Tag coverage was critically insufficient: only Blog posts had tags; the approximately 54 articles in the technical knowledge base had none, making the Tags page effectively empty.
    • No comment system.
    • No analytics.
    • Social Cards used the same static brand image across all pages — no dynamic title.
    • English RSS /en/feed.xml was unverified on live; verifyDist did not check RSS output.
    • Author metadata existed only on blog posts; technical articles had no credibility signal.

    Implemented¶

    Feature 1: Last-updated timestamp on all articles

    Enabled mkdocs-git-revision-date-localized-plugin with enable_creation_date: true and fallback_to_build_date: true. All pages now automatically show the last-updated date from git history. Plugin is placed after the i18n plugin in mkdocs.yml (required: i18n must inject the locale variable first).

    Feature 2: Tag coverage

    extra.tags expanded from 8 to 20 entries, adding: Kubernetes, Linux, Containers, CI/CD, Databases, Networking, Cloud, Middleware, Automation, IaC, Runbook, Standards.

    All approximately 54 technical articles now have tags and description frontmatter — including all Standards, Runbooks, Document Templates, Migration Cases, and both language versions of the architecture article. Each article carries at least one tag; most carry two or three.

    Feature 3: RSS verification and verifyDist hardening

    Makefile verifyDist target now includes test -f dist/feed.xml and test -f dist/en/feed.xml, so both RSS feeds must exist before any publish run can pass.

    Feature 4: Author metadata

    Decision: no additional author fields on technical articles. The git-revision-date-localized timestamp (Feature 1) provides the needed credibility signal. The blog plugin .authors.yml only applies to blog posts; adding author fields to technical articles would have no reader benefit.

    Feature 5: Giscus comment system

    Created overrides/partials/comments.html using the official MkDocs Material comments extension point. Added custom_dir: overrides to mkdocs.yml. Comments are enabled by default on all pages. To disable on a specific page, add comments: false to its frontmatter.

    Pending user action:

    1. Enable Discussions at https://github.com/lzyred/lzyred.github.io/settings
    2. Visit https://giscus.app, get data-repo-id and data-category-id
    3. Replace REPLACE_WITH_REPO_ID and REPLACE_WITH_CATEGORY_ID in overrides/partials/comments.html

    Feature 6: Umami analytics

    Created overrides/partials/integrations/analytics/custom.html using the MkDocs Material custom analytics provider mechanism. Added extra.analytics.provider: custom to mkdocs.yml.

    Pending user action:

    1. Register at https://umami.is (free), add blog.l2do.com
    2. Get the Website ID (UUID)
    3. Replace REPLACE_WITH_UMAMI_WEBSITE_ID in mkdocs.yml

    Feature 7: Dynamic Social Cards

    Replaced layouts/l2do.yml with a version that adds three typography layers: site name (top-left), page title (main body), page description (footer). Font is Noto Sans SC, covering both Chinese and Latin characters. theme.font: false only disables Google Fonts in HTML pages and does not affect the social plugin font download. The font is cached in .cache/ after the first build. All 150 cards were regenerated with dynamic titles after clearing the old manifest.
