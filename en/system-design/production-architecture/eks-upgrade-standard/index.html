    Kubernetes Migration Reliability

    EKS and Kubernetes Upgrade Standard¶

    Executive Summary¶

    A Kubernetes minor-version upgrade is not one upgrade. It is three coupled upgrades that must happen in order — control plane, then add-ons, then nodes — plus one compatibility check that has to run before any of them. The control-plane version is the most visible part, but the application-impacting failures usually surface through add-on skew, APIs removed in the target version, admission or conversion webhooks, PDBs, or node migration.

    The incidents come from the couplings. An add-on left at the old version after the control plane moves produces flaky DNS, networking, or storage that looks like a control-plane problem and is not. A workload still calling an API the new version removed breaks the moment its controller restarts — which may be hours later, long after the upgrade was declared "done." A PodDisruptionBudget that blocks all eviction stalls the node drain halfway through, with some workloads on the new version and some on the old.

    The last fully reversible point is before the control-plane upgrade starts. A pre-flight compatibility check finds removed APIs, incompatible CRDs, broken webhooks, and skewed add-ons while the old version is still serving traffic. Skip it, and you discover the same facts during the drain, when half the cluster has already moved and the control plane can no longer be rolled back. This standard is mostly about doing the discovery early and keeping node, add-on, and application rollback paths open until the new version has proven itself under real traffic.

    When to Use This Standard¶

    Use this standard when you:

    • Upgrade EKS, self-managed Kubernetes, or a compatible platform across a minor version.
    • Replace worker node groups as part of a platform upgrade.
    • Upgrade core add-ons (CNI, CoreDNS, kube-proxy, CSI drivers, ingress controller, cert-manager, metrics-server).

    This is the upgrade standard, and it assumes a running cluster. Do not use it as:

    • A cluster-creation guide — see the Kubernetes cluster provisioning standard.
    • An application rollout procedure — see the deployment rollback runbook.
    • A substitute for a provider's exact console/CLI steps — use the provider's documented upgrade path inside this sequence.
    • A way to skip minor versions — upgrade one minor version at a time unless the platform explicitly supports a larger jump.

    Assumptions¶

    This standard assumes:

    • A cluster currently in a supported version, upgrading to the next minor version only.
    • Mostly stateless workloads, with state in managed data services or in storage whose CSI driver compatibility you can verify.
    • The ability to add a new node group rather than mutate nodes in place — this is what makes node rollback "stop draining" instead of "un-upgrade a node."
    • An encrypted backup path for cluster state and secrets (the platform's process, or etcd backup for self-managed clusters).
    • An observation window after the upgrade during which old node groups, compatible add-on versions, and application rollback paths remain available. The managed control plane itself should be treated as forward-only.

    If your cluster is several minor versions behind, this is not one upgrade — it is a planned sequence of them, each run end to end, because Kubernetes only supports single-minor-version jumps and skew rules apply at every step.

    Proportional Scaling¶

    The full sequence below is calibrated for a production cluster carrying real traffic. Match the effort to the risk:

    • A compressible cluster has no customer traffic, no critical StatefulSet/PVC dependency, an accepted downtime or rebuild RTO, and a cluster state that can be recreated from code. You may shorten the observation window or drain in larger batches, but only after running the API-compatibility check and keeping the old node group until smoke tests and metrics pass.
    • A stateless cluster can drain more aggressively because node-migration rollback is "stop the drain, uncordon or scale the old node group, and reschedule there" — but only while that old node group still exists. Once you decommission it, rollback becomes an application redeploy on the new version, not a return to the known-good node version. Keep the old group through the observation window before you call statelessness a safety net. The batched, validated drain matters most when stateful workloads are present and a bad node migration cannot be undone by rescheduling.
    • A cluster with stateful workloads or strict SLAs needs the full model: small drain batches, validation between batches, the observation window, and rehearsed restore. Do not skip the batched drain here just because the small-cluster path allows it.

    The check that never scales away is the API-compatibility pre-flight. Everything else flexes with how much an outage would cost.

    Upgrade Principles¶

    1. Upgrade one minor version at a time unless the platform explicitly supports a larger jump.
    2. Treat add-ons as part of the cluster, not optional extras — version skew between control plane and add-ons is a top cause of breakage.
    3. Upgrade strictly in sequence: control plane → add-ons → nodes.
    4. Prefer new node group + drain over in-place node mutation, so node rollback is "stop draining, uncordon or scale the old group," not "un-upgrade a node."
    5. Validate both Kubernetes health and application behavior before declaring done.
    6. Define rollback triggers up front and keep rollback paths open through the observation window.

    Pre-Upgrade Assessment¶

    Capture a baseline and run the compatibility checks. This is where you prevent the outage — while everything is still on the old version and reversible.

    kubectl version
kubectl get nodes -o wide
kubectl get deploy,statefulset,daemonset -A
kubectl get ingress,svc,pvc,storageclass -A
kubectl get pdb -A                 # check ALLOWED DISRUPTIONS — a PDB at 0 will stall the drain
kubectl top nodes; kubectl top pods -A

    Also capture a baseline error rate, latency, and SLO burn rate from your monitoring now, while the cluster is healthy on the old version. The rollback triggers below must be pre-approved numeric thresholds or SLO deltas, not improvised during the incident.

    Area Check
    APIs removed in the target version Find workloads using APIs removed in the target version (e.g. pluto, kubent). Block the upgrade until manifests, Helm charts, GitOps repos, and CI templates stop emitting those APIs.
    CRDs and conversion webhooks Confirm CRD versions, stored versions, and conversion webhooks are compatible with the target version. A broken conversion webhook can make existing objects unreadable or block controller reconciliation.
    Admission webhooks Validate mutating/validating webhooks, certificate expiry, timeout policy, failure policy, and API compatibility before the control plane moves. A failing webhook can block pod creation after the upgrade.
    Aggregated APIs and APIService Check APIService health and extension API compatibility. Broken aggregated APIs can make controllers or CLIs fail even when core Kubernetes looks healthy.
    Operators and controllers Confirm operator versions, controller-runtime compatibility, leader election, and RBAC against the target Kubernetes version. Controllers often surface APIs removed in the target version hours after the upgrade.
    Add-ons CNI, CoreDNS, kube-proxy, CSI drivers, ingress controller, cert-manager, metrics-server compatibility with the target version.
    Workloads PodDisruptionBudgets, probes, resource requests, priority classes, security contexts.
    Storage CSI driver compatibility, PVC binding, snapshot/backup support.
    Networking Ingress class behavior, load-balancer annotations, NetworkPolicies.
    Access IAM/RBAC, service accounts, and a working break-glass path.

    A PDB that blocks all eviction, an API removed in the target version, or an incompatible webhook/controller will stall the upgrade midway — find these now, not during the drain.

    Standard Upgrade Sequence¶

    1. Freeze and announce¶

    Pause unrelated deploys and infra changes for the window. Communicate impact, rollback criteria, and owners.

    2. Back up cluster state¶

    kubectl get all -A -o yaml > cluster-workloads.snapshot.yaml
kubectl get configmap,ingress,svc,pvc,storageclass -A -o yaml > cluster-config.snapshot.yaml

    Do not export Secret values to plaintext. Back up secrets via the platform's encrypted process; for self-managed clusters, back up etcd with the recommended encrypted method (see the etcd standard).

    3. Upgrade the control plane¶

    Upgrade via the provider/cluster tool, and wait for the provider-reported upgrade-complete status (e.g. the managed cluster back to ACTIVE), not only a passing /readyz — a managed control-plane upgrade can report healthy on /readyz while the provider is still rolling it out. Then validate API server discovery, admission/conversion webhooks, aggregated APIs, and core controllers before touching add-ons or nodes.

    kubectl get --raw='/readyz?verbose'
kubectl get --raw='/apis' >/dev/null
kubectl get apiservice
kubectl get nodes

    4. Upgrade core add-ons to target-compatible versions¶

    kubectl get pods -n kube-system
kubectl rollout status deployment/coredns -n kube-system
kubectl get daemonset -n kube-system

    Do this before moving workloads — nodes joining on the new version need compatible CNI/kube-proxy. This is the step most often skipped and the one that produces "the control plane upgrade broke DNS" reports that are really add-on skew.

    5. Add a new node group, drain the old gradually¶

    Create a new node group on the target version, then move workloads in small batches:

    kubectl cordon <old-node>
kubectl drain <old-node> --ignore-daemonsets --delete-emptydir-data --timeout=10m

    Validate application health after each batch before continuing. This is the reversible model: if a batch misbehaves, stop and keep the old (still-healthy) nodes.

    6. Validate, then decommission old nodes¶

    After the observation window, scale down and remove the old node group. Keep snapshots and notes.

    Validation¶

    kubectl get pods -A
kubectl get events -A --sort-by='.lastTimestamp'
kubectl get ingress,pvc -A

    Application validation must include: HTTP health, a core API/login smoke test, background-job processing, database connectivity, external webhooks/callbacks, and that alerting/dashboards still work. Declare success only after these pass and metrics are stable through the observation window — not when the new node group is Ready.

    Known Failure Modes¶

    Failure mode Symptom Likely cause Correct response
    API removed in target version Controllers/CI fail post-upgrade, sometimes hours later Workload, chart, CRD, webhook, or controller still uses an API removed in the target version Find with pluto/kubent before upgrading; migrate manifests and controller dependencies.
    Webhook/APIService incompatibility Pod creation, controller reconciliation, or CLI discovery fails Admission webhook, conversion webhook, or aggregated API is incompatible or unavailable Validate webhook certificates, failure policies, timeouts, and APIService health before node migration.
    Add-on version skew DNS/networking/storage flaky after control-plane upgrade Add-ons not upgraded to match Upgrade add-ons before moving nodes.
    Drain stalls kubectl drain hangs mid-migration PDB blocks eviction / single replica Fix PDB/replicas; never --force blindly.
    Pods won't schedule Pending after node migration Taints/labels/capacity mismatch on new group Match labels/taints/size; check resource requests.
    Storage won't attach StatefulSet pods stuck CSI driver incompatibility Verify CSI compatibility in pre-assessment; roll back nodes.
    Control plane not reversible Can't downgrade after issue Managed upgrades are forward-only Recover via nodes/add-ons/app rollback + backups (below).

    Rollback¶

    Managed control-plane upgrades are usually not directly reversible — so rollback is designed around the layers that still are. This is exactly why principle 4 (new node group + drain) matters: it keeps node migration reversible even though the control plane is not.

    1. Stop node migration immediately; uncordon or scale the old node group while it is still supported.
    2. Roll back add-ons to compatible versions where safe.
    3. Roll back application deployments (kubectl rollout undo).
    4. Restore stateful components from backups if data was affected.

    Define rollback triggers before starting:

    • A core business smoke test fails.
    • A pre-approved SLO trigger is breached for the agreed duration, such as 5xx rate > <threshold>, p95/p99 latency > <threshold>, or burn-rate delta > <threshold> for more than <duration>.
    • Critical pods cannot schedule.
    • Storage attach/mount errors hit stateful workloads.
    • Ingress/load-balancer behavior breaks external traffic.

    Post-Upgrade Review¶

    Record source/target versions, add-on versions before/after, node-group changes, APIs removed in the target version, webhook/APIService findings, incidents and mitigations, and follow-up work.

    Security Notes¶

    • Never export Secret values to plaintext during backup; use encrypted/secret-manager backups.
    • Verify RBAC, IAM, and break-glass access still work on the new version before decommissioning old paths.
    • Do not publish real cluster names, account IDs, endpoints, or node names.

    Review Cycle¶

    • Owner: the platform/cluster team.
    • Re-run checks for APIs removed in the target version, webhook/APIService compatibility, controller compatibility, and add-on compatibility before every upgrade.
    • Re-confirm rollback triggers and the backup/restore path before each upgrade window.
    • Track the Kubernetes version support window so upgrades are planned, not forced by end-of-support.
