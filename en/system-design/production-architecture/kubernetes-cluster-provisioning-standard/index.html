    Kubernetes Cluster Provisioning Standard¶

    Purpose¶

    This standard defines how to provision a Kubernetes cluster that can safely host production workloads.

    The most important judgment is simple:

    Cluster provisioning is not the command that creates the control plane. It is the ordered construction of a platform dependency graph.

    A cluster is not ready because kubectl get nodes returns Ready. It is ready only when workload identity, network boundaries, node pools, storage, ingress, observability, deployment controls, backup expectations, and ownership rules are all explicit and validated.

    This standard applies to managed clusters such as EKS, CCE, ACK, GKE, and AKS, and to self-managed clusters built with kubeadm. The implementation details differ, but the readiness contract is the same.

    When to Use¶

    Use this standard when:

    • Building a new production or staging Kubernetes cluster.
    • Rebuilding a cluster after a major network, identity, or runtime change.
    • Creating a repeatable cluster profile for multiple teams.
    • Reviewing whether an existing cluster is mature enough for business workloads.
    • Converting an experimental cluster into a platform-owned service.

    Do not use this as a quick local development checklist. Development clusters can skip many production controls, but they should not be mistaken for production reference architecture.

    Non-Goals¶

    This standard does not cover:

    • A vendor-specific click-by-click console procedure.
    • Full Kubernetes security hardening.
    • Application architecture inside the cluster.
    • Service mesh design.
    • Disaster recovery design beyond the provisioning readiness boundary.

    Provisioning Judgment¶

    A production cluster must answer seven questions before the first business workload is deployed:

    Question Required Decision
    Who owns the platform? Platform owner, escalation path, change window, and lifecycle policy.
    Where can traffic flow? VPC or VNet, CIDR, service CIDR, pod CIDR, DNS, egress, private endpoints, and ingress exposure.
    Who can act? Human access, workload identity, service accounts, emergency access, and audit logging.
    Where do workloads run? Node pool boundaries, taints, labels, autoscaling, runtime, images, and maintenance policy.
    How is state stored? StorageClass, CSI driver, expansion, backup, snapshot, reclaim policy, and topology constraints.
    How do users enter? Ingress class, load balancer model, TLS ownership, DNS ownership, and certificate renewal.
    How is failure seen and reversed? Metrics, logs, events, alerts, rollout controls, rollback procedure, and restore boundary.

    If any answer is missing, the cluster may exist, but it is not production-ready.

    Decision Inputs¶

    Capture these inputs before provisioning:

    Area Required Input
    Kubernetes version Target version, supported skew, add-on compatibility, upgrade window.
    Network VPC or VNet, CIDR plan, pod/service ranges, DNS, egress NAT, private endpoint requirements.
    Identity Cluster admin model, group mapping, workload identity mechanism, break-glass account.
    Node pools CPU and memory profile, on-demand or spot policy, GPU or special hardware, taints, labels.
    Storage Default StorageClass, stateful workload needs, backup and restore expectation, expansion policy.
    Ingress Public or private exposure, controller, TLS issuer, DNS ownership, WAF or gateway boundary.
    Observability Metrics, logs, events, traces, alert routing, dashboard ownership.
    Delivery GitOps or pipeline ownership, namespace onboarding, rollback path, deployment policy.
    Governance Naming, tagging, cost allocation, resource quota, image policy, platform review cadence.

    The provisioning record should be short, but it must be explicit. A cluster with undocumented defaults becomes an operational surprise during the first incident.

    Reference Provisioning Sequence¶

    1. Define the platform boundary¶

    Decide what the cluster is allowed to own. A cluster should not accidentally become the owner of DNS, secrets, certificates, ingress, CI/CD, databases, and monitoring simply because those things were installed during setup.

    At minimum, define:

    • Production, staging, development, or shared platform scope.
    • Allowed workload classes.
    • Network exposure model.
    • Team onboarding process.
    • Ownership for platform add-ons.
    • Incident escalation path.

    2. Freeze the version and add-on matrix¶

    Select the Kubernetes version together with the add-ons that must run on it. Do not choose a cluster version first and discover compatibility later.

    The matrix must include:

    • CNI plugin and version.
    • CoreDNS.
    • kube-proxy or replacement.
    • CSI drivers.
    • Ingress controller.
    • Metrics server.
    • Autoscaler or Karpenter-equivalent component.
    • Log and metric agents.
    • Policy controllers if used.

    The minimum acceptable output is a table that says which versions are approved, which are blocked, and which upgrade sequence is required.

    3. Provision network before nodes¶

    The network is harder to change than the cluster. Validate it before creating node pools.

    Required checks:

    • Pod and service CIDRs do not overlap with VPC/VNet, peered networks, VPN ranges, or on-premises ranges.
    • DNS resolution works for cloud APIs, internal services, and private endpoints.
    • Nodes can reach required image registries and control-plane endpoints.
    • Egress policy is deliberate, not an accidental open route.
    • Ingress subnets and internal load balancer subnets are separated when needed.

    If the network plan is unclear, stop. Rebuilding a young cluster is cheaper than living with overlapping address space.

    4. Configure identity and access before applications¶

    Human and workload access must be set before application teams arrive.

    Required decisions:

    • Which groups have cluster-admin access.
    • Which groups have namespace-scoped admin access.
    • How service accounts obtain cloud permissions.
    • How emergency access is granted, logged, and revoked.
    • Which audit logs are retained.

    Avoid binding permanent cluster-admin rights to individual users. Use groups, time-bounded emergency access, and documented review.

    5. Build node pools as workload contracts¶

    Node pools are not just capacity. They are workload placement contracts.

    Define at least:

    • General stateless pool.
    • System or platform add-on pool where appropriate.
    • Stateful or storage-sensitive pool if required.
    • Spot or preemptible pool only for tolerant workloads.
    • GPU or hardware-specific pools only when isolated by taints and labels.

    Every pool should have labels, taints where appropriate, autoscaling boundaries, image lifecycle rules, and a patch or replacement strategy.

    6. Install storage before workload onboarding¶

    Install CSI drivers and StorageClasses before application deployment. A cluster without a tested default storage path is not ready for stateful workloads.

    Minimum validation:

    • A PVC can be dynamically provisioned.
    • The mounted filesystem is writable from a pod.
    • Expansion behavior is known.
    • Reclaim policy is deliberate.
    • Backup or snapshot expectation is documented.
    • Zone or topology constraints are visible to schedulers.

    Storage design details belong in Kubernetes Storage and Ingress Standard.

    7. Install ingress and TLS before publishing services¶

    Ingress must be a managed platform capability, not something each team improvises.

    Validate:

    • IngressClass ownership.
    • Public versus private exposure.
    • TLS certificate issuer and renewal.
    • DNS ownership.
    • Request timeout, body size, and WebSocket expectations.
    • Access log and metric availability.
    • WAF or gateway boundary if required.

    8. Install observability before business traffic¶

    Observability is a launch dependency. Do not wait for the first incident to install it.

    Minimum signals:

    • Node health and resource saturation.
    • Pod restart, scheduling, and image pull failures.
    • API server health if exposed by the platform.
    • Ingress request rate, latency, and error rate.
    • PVC usage and expansion risk where stateful workloads exist.
    • Event collection for debugging.
    • Alert routing to the owning team.

    9. Add delivery and rollback controls¶

    The first deployment mechanism should include rollback before it includes speed.

    Required controls:

    • Namespace onboarding pattern.
    • Deployment owner.
    • Image provenance expectation.
    • Rollout strategy.
    • Rollback command or GitOps revert path.
    • Release health check.
    • Change record.

    See Deployment Rollback Runbook for the operational rollback procedure.

    Managed Cluster Profile¶

    For managed Kubernetes services, prefer the provider-managed control plane and avoid self-managing components that the provider can operate more safely.

    Required managed-cluster decisions:

    • Private or public control-plane endpoint.
    • Control-plane access allowlist.
    • Managed node group versus self-managed node group.
    • IAM or workload identity integration.
    • Provider-supported CSI and load balancer controllers.
    • Provider version support window.
    • Upgrade ownership between platform team and provider.

    Do not assume "managed" means "production-ready." The provider operates part of the platform; your team still owns workload access, add-ons, delivery, observability, data protection, and incident response.

    Self-Managed kubeadm Profile¶

    For kubeadm clusters, the team must explicitly own the control plane.

    Additional requirements:

    • Highly available API server endpoint.
    • etcd backup and restore test.
    • Certificate rotation procedure.
    • Control-plane node patching process.
    • CNI installation and upgrade policy.
    • kubelet and container runtime lifecycle.
    • Disaster recovery runbook.

    If these cannot be staffed and tested, use a managed cluster unless there is a hard constraint that prevents it.

    Application Readiness Gate¶

    Before accepting business workloads, run this gate:

    Gate Pass Condition
    Nodes All intended node pools are ready, labeled, tainted as designed, and autoscaling within bounds.
    System pods CoreDNS, CNI, CSI, ingress, metrics, and observability agents are healthy.
    Storage A test PVC provisions, mounts, writes, expands if supported, and follows the intended reclaim policy.
    Ingress A test service is reachable through the intended public or private route with valid TLS.
    Identity A workload can use the intended cloud identity without static secrets.
    Observability Test failures produce searchable logs, metrics, events, and alerts.
    Delivery A test deployment can roll forward and roll back through the approved mechanism.
    Cost and ownership Namespaces, labels, quotas, and escalation routes are visible.

    The cluster is not open for onboarding until every gate has an owner and a result.

    Validation¶

    Run validation from a clean operator workstation or CI runner with normal platform permissions:

    kubectl get nodes -o wide
kubectl get pods -A
kubectl get storageclass
kubectl get ingressclass
kubectl top nodes
kubectl auth can-i create deployments --namespace <test-namespace>
kubectl auth can-i '*' '*' --as <non-admin-user>

    Then create and remove a small validation workload:

    • Stateless deployment with readiness probe.
    • PVC-backed pod.
    • Internal service.
    • Ingress route with TLS.
    • Deliberate failed rollout followed by rollback.

    Record the result in the cluster readiness document. A screenshot is weaker than a repeatable command and expected output.

    Known Failure Modes¶

    Failure Mode Why It Hurts Prevention
    CIDR overlap Peering, VPN, and service routing fail later. Validate all address ranges before cluster creation.
    Add-on version mismatch CNI, CSI, or ingress breaks after upgrade. Freeze and review the version matrix.
    No tested StorageClass Stateful workloads block or lose data. Run PVC provision, mount, write, expand, and reclaim tests.
    Ingress before TLS ownership Services go public without renewal accountability. Define DNS and certificate ownership before exposure.
    App deployment before observability First incident has no useful evidence. Treat observability as a launch dependency.
    Permanent individual admin access Audit and offboarding become weak. Use groups, emergency access, and periodic review.
    Unbounded node autoscaling Cost spikes or scheduling instability. Set min, max, labels, taints, quotas, and alerts.

    Rollback and Rebuild Boundary¶

    Some provisioning mistakes should be fixed by rebuilding the cluster, not patching around them.

    Prefer rebuild when:

    • Pod or service CIDR conflicts with another network.
    • The cluster was created in the wrong VPC, region, or account boundary.
    • Identity integration was built around long-lived static credentials.
    • Add-on ownership is unclear and no team can safely upgrade it.
    • The cluster has accepted business workloads without baseline observability, and the migration cost is still low.

    Prefer in-place correction when:

    • A node pool needs resizing or replacement.
    • An add-on configuration is wrong but version-compatible.
    • A StorageClass or IngressClass needs a new default after migration planning.
    • RBAC needs tightening with an auditable access review.

    The earlier the problem is found, the more aggressive the rebuild option should be.

    Security Notes¶

    • Do not store kubeconfigs, tokens, certificates, or cloud keys in public documentation.
    • Do not document private endpoint addresses, customer domains, or internal CIDR values in public examples.
    • Use symbolic placeholders such as <pod-cidr> or <service-cidr> instead of copied production ranges.
    • Prefer workload identity over static cloud credentials mounted into pods.
    • Keep audit logging enabled before granting team access.
    • Review cluster-admin bindings after every onboarding wave and after every incident.

    Review Cycle¶

    Review this standard:

    • Before each new cluster build.
    • Before each Kubernetes minor version upgrade.
    • After any network, identity, storage, or ingress incident.
    • When a new workload class requires a new node pool or storage profile.
