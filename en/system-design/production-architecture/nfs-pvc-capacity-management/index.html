    NFS PVC Capacity Management¶

    Purpose¶

    This standard defines how to manage capacity for Kubernetes PVCs backed by NFS or NFS-like shared file storage.

    The core judgment is:

    NFS-backed PVC capacity is a shared filesystem problem wearing Kubernetes metadata.

    A PVC request tells Kubernetes what the workload asked for. It does not always tell you how much space exists on the backend filesystem, how many other PVCs share that backend, or whether the backend can enforce per-claim limits. Capacity management must join both layers.

    When to Use¶

    Use this standard when:

    • Kubernetes PVCs are provisioned from an NFS server, managed file service, or shared filesystem.
    • Multiple PVCs map to directories under the same backend export.
    • PVC usage alerts do not match backend filesystem usage.
    • Application teams use shared files, uploads, model artifacts, reports, or batch outputs.
    • The platform needs a standard for capacity ownership and alert thresholds.

    This standard is especially important when NFS is used as a convenient ReadWriteMany solution. Convenience does not remove the need for capacity isolation, monitoring, and cleanup rules.

    Non-Goals¶

    This standard does not cover:

    • Designing high-availability NFS from scratch.
    • Database storage tuning.
    • Replacing object storage for large immutable artifacts.
    • Full backup architecture for shared files.
    • Vendor-specific file service operations.

    Two-Layer Capacity Model¶

    Manage NFS PVC capacity as two related but different layers:

    Layer What It Shows What It Cannot Prove
    Kubernetes PVC metadata Namespace, claim name, requested size, StorageClass, PV, mount relationship. Real backend free space, shared usage, filesystem pressure.
    NFS backend filesystem Actual total, used, available, inode use, per-directory growth. Kubernetes owner, namespace, workload intent, data criticality.

    Neither layer is enough by itself.

    Kubernetes-only monitoring can miss a backend filesystem filling up across many claims. NFS-only monitoring can tell you a directory is large but not which namespace, workload, and team owns it.

    Required Inventory¶

    Maintain an inventory that maps each PVC to backend reality:

    Field Example
    Namespace payments-prod
    PVC uploads-data
    PV pvc-abc123
    StorageClass shared-rwx
    Backend export nfs-shared-01:/exports/k8s
    Backend path /exports/k8s/payments-prod/uploads-data
    Workload payments-api
    Owner Platform or application team alias
    Data class Uploads, cache, reports, artifacts, logs, unknown
    Retention 30 days, owner-defined, or none
    Backup Required, not required, or unknown
    Expansion model Backend pool expansion, directory quota, or migration

    If the platform cannot map a full directory to an owner, it must not delete from that directory.

    Standard Thresholds¶

    Use two sets of thresholds: backend filesystem thresholds and per-PVC or per-directory growth thresholds.

    Signal Warning Critical Action
    Backend filesystem used percent 75% 90% Review growth, plan expansion, or clean approved data.
    Backend inode usage 70% 85% Find small-file sources and owner.
    Single PVC directory growth 20% in 24h 40% in 24h Identify writer and data class.
    Forecast full 7 days 72 hours Prepare capacity action.
    Unknown owner data Any sustained growth Any critical pressure Escalate; do not delete blindly.

    Tune thresholds for backend size and business risk. Large filesystems may need forecast-based alerts earlier than percent-based alerts.

    Monitoring Standard¶

    Collect Kubernetes metadata¶

    Collect:

    • PVC name.
    • Namespace.
    • PV name.
    • StorageClass.
    • Requested capacity.
    • Volume mode.
    • Access mode.
    • Workload using the claim.

    Kubernetes metrics alone do not always expose backend path. If the provisioner writes path information into PV attributes or annotations, ingest it into the inventory.

    Collect backend filesystem usage¶

    Collect from the NFS server, managed file service metrics, or an approved observation host:

    • Filesystem total, used, available, and percent used.
    • Inode total, used, available, and percent used.
    • Per-export usage.
    • Per-directory usage where feasible.
    • Growth rate.
    • Top directories by size and file count.

    For Linux NFS servers, platform-owned collection may use commands like:

    df -h /exports/k8s
df -i /exports/k8s
du -xh --max-depth=2 /exports/k8s 2>/dev/null | sort -h

    Do not run expensive recursive scans during peak hours on a busy file server without understanding the impact.

    Join the two layers¶

    Capacity dashboards and alerts should show:

    • Backend filesystem pressure.
    • Top PVC directories.
    • Namespace and owner.
    • Workload and data class.
    • Growth trend.
    • Last review date.

    The useful question is not "which disk is full?" but "which workload and data class is consuming shared capacity, and what action is safe?"

    Capacity Response Model¶

    1. Classify data before action¶

    Use this classification:

    Data Class Default Action
    Cache Clean by approved application method or recreate.
    Temporary export Apply retention and move long-term artifacts elsewhere.
    User upload Do not delete without owner approval and backup confirmation.
    Application state Treat as business data; expand or migrate.
    Logs Move to logging pipeline or object storage; clean after retention.
    Unknown Escalate; no destructive action.

    2. Choose the capacity action¶

    Situation Action
    Backend has enough capacity but one PVC directory is growing. Work with owner; enforce retention, move data, or resize claim expectation.
    Backend filesystem is filling because many claims share one pool. Expand backend, split tenants, or migrate high-growth claims.
    Inodes are filling before bytes. Find small-file workload, archive or compact, and change application behavior.
    NFS performance degrades under load. Reduce scan pressure, split workload, or move to a more suitable storage backend.
    PVC request does not match actual usage. Update inventory and alerting; do not rely on request size alone.

    3. Expand deliberately¶

    For NFS, expansion may mean increasing the backend filesystem or changing a quota, not patching an individual PVC.

    Before expansion:

    • Confirm backend pool headroom or provider quota.
    • Confirm backup or snapshot posture.
    • Confirm whether the expansion affects all tenants.
    • Confirm alert thresholds after expansion.
    • Record the new capacity and expected forecast.

    4. Migrate when NFS is the wrong abstraction¶

    NFS is often misused for data that belongs elsewhere.

    Consider migration when:

    • Large immutable artifacts would fit object storage better.
    • A database is placed on shared file storage without a strong reason.
    • A single tenant dominates the shared filesystem.
    • Application performance depends on filesystem semantics NFS cannot guarantee.
    • Cleanup requires risky manual file deletion.

    Cleanup Safety¶

    Never delete from NFS solely because a directory is large.

    Minimum deletion requirements:

    • Owner identified.
    • Data class identified.
    • Retention policy or explicit approval confirmed.
    • Backup or non-critical classification confirmed.
    • Exact path reviewed.
    • Command scoped to the intended directory.
    • Result validated.

    Prefer application-level deletion over platform shell deletion. The application usually knows which files are indexed, referenced, or safe to remove.

    Validation¶

    This standard is implemented when:

    Check Pass Condition
    Inventory Each NFS-backed PVC maps to namespace, workload, owner, backend export, and path.
    Metrics Backend bytes, backend inodes, and PVC metadata are visible together.
    Alerts Backend pressure and rapid per-directory growth are both detected.
    Ownership Unknown-owner directories are tracked as risk.
    Response Cleanup, expansion, migration, and escalation paths are documented.
    Safety No destructive cleanup occurs without owner and data classification.

    Known Failure Modes¶

    Failure Mode Result Prevention
    Monitoring only PVC requested size Backend fills while PVCs appear normal. Monitor backend filesystem usage.
    Monitoring only NFS server Large directory has no clear Kubernetes owner. Maintain PVC-to-path inventory.
    Shared pool with no tenant separation One workload consumes capacity for all workloads. Use quotas, separate exports, or separate storage classes.
    Manual deletion of unknown files Data loss or application corruption. Require owner and data class before cleanup.
    Recursive scans during peak load NFS latency worsens during investigation. Schedule scans or use lightweight metrics.
    No inode alerting Millions of tiny files break writes before bytes fill. Alert on inode usage and file count growth.

    Security Notes¶

    • Do not publish real export paths, server names, IPs, or customer directory names.
    • Restrict shell access to NFS servers.
    • Avoid broad read access to shared application files.
    • Treat user uploads and reports as potentially sensitive.
    • Keep cleanup commands out of public docs unless they are generic and placeholder-based.

    Review Cycle¶

    Review this standard:

    • Before onboarding a new NFS-backed StorageClass.
    • After any NFS capacity or performance incident.
    • When a tenant becomes a top capacity consumer.
    • Before increasing backend capacity by a material amount.
    • When application data retention changes.
