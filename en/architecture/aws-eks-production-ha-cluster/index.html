    AWS EKS Production High Availability Cluster Design and Delivery Standard¶

    Executive Summary¶

    Building a production-grade highly available EKS cluster on AWS is not mainly about increasing node count or creating a Kubernetes control plane. The real work is putting control plane access, worker capacity, networking, identity, ingress, storage, delivery, observability, upgrades, and recovery into one verifiable operating model.

    A credible production EKS design should provide:

    • Workload placement across multiple Availability Zones.
    • Scheduling continuity when one AZ loses capacity.
    • Recoverable and scalable worker capacity.
    • Workload-level AWS permissions instead of broad node-level credentials.
    • Clear ownership for ingress, DNS, certificates, logs, metrics, and alerts.
    • Validation and rollback paths for each critical change.

    If a guide stops at eksctl create cluster, it proves that a cluster can be created. It does not prove that the platform can carry production traffic.

    When To Use This Standard¶

    Use this document when:

    • Creating a new production Amazon EKS cluster.
    • Migrating workloads from virtual machines or self-managed Kubernetes to EKS.
    • Hardening an existing EKS cluster for availability, security, observability, and release safety.
    • Building a reusable delivery checklist for platform or DevOps teams.

    Do not use this as:

    • A beginner Kubernetes tutorial.
    • A short-lived proof-of-concept guide.
    • A replacement for multi-Region disaster recovery architecture.
    • A complete regulated-industry compliance blueprint.

    Multi-AZ inside one Region is not the same as multi-Region resilience. Treat those as different architecture problems.

    Assumptions¶

    This standard assumes:

    • One AWS Region with at least three Availability Zones.
    • Mostly stateless services, with state placed in managed data services or carefully designed persistent storage.
    • GitOps or CI/CD based delivery.
    • Public or internal ingress traffic.
    • Least-privilege access, auditability, and rollback requirements.

    For a small low-risk environment, reduce the design: two capacity classes, one ingress path, basic metrics, basic logs, and explicit backup. Do not add every production component just to make the platform look mature.

  - IAM roles for cluster operators

    The key principle: AWS manages the EKS control plane, but application availability still depends on your VPC, subnets, node groups, scheduling rules, Pod replicas, disruption budgets, storage, ingress, and release process.

    Key Design Decisions¶

    1. Multi-AZ Is The Baseline¶

    A production EKS cluster should normally use at least three AZs. Two AZs can be acceptable for lower-risk systems, but capacity recovery and maintenance flexibility are weaker. A one-AZ EKS cluster should not be described as production high availability.

    Check:

    • The VPC spans at least three AZs.
    • Each AZ has private subnets for worker nodes.
    • Node groups or Karpenter provision capacity across AZs.
    • Application replicas are spread with topology spread constraints or anti-affinity.
    • Storage classes are understood in terms of AZ binding.

    Common failure mode: the cluster is configured with three AZ subnets, but actual worker capacity only exists in one AZ. The architecture diagram says multi-AZ; the scheduler experiences a single-AZ platform.

    2. Worker Capacity Should Be Layered¶

    Do not run every Pod on the same worker pool. A minimum production model has:

    • System capacity: CoreDNS, ingress controllers, log agents, monitoring agents.
    • Application capacity: user-facing and internal services.
    • Batch or Spot capacity: retryable jobs and interruptible workloads.

    The purpose is not complexity. The purpose is to prevent business traffic or batch spikes from evicting the components that keep the cluster observable and reachable.

    Recommended principles:

    • Keep system components on stable capacity.
    • Select instance families based on workload resource profiles.
    • Use Spot only for workloads that tolerate interruption and retry.
    • Use PodDisruptionBudgets for critical services.

    3. Pods Should Not Inherit Broad Node Permissions¶

    Production workloads should not rely on the node IAM role for AWS API access. Use EKS Pod Identity or IRSA so each workload gets the minimum IAM role it needs.

    Acceptance criteria:

    • Each workload receives only the AWS permissions it needs.
    • Secrets, object storage, queues, and container registry permissions are separated.
    • Permission changes are auditable.
    • Removing an application also removes or disables its cloud permissions.

    Common failure mode: S3, ECR, CloudWatch, and Secrets Manager permissions are all attached to the node role. It is convenient at first, but any compromised Pod can inherit excessive lateral movement capability.

    4. Ingress Is More Than An Ingress Object¶

    A production ingress path needs decisions for:

    • DNS ownership.
    • TLS issuance and renewal.
    • Public or internal load balancers.
    • WAF requirements.
    • Stable health check paths.
    • Traffic shifting for rollback or canary releases.

    If a design only contains an Ingress YAML file, the ingress layer is not finished.

    5. Default To Managed Services For State¶

    EKS can run stateful systems, but production teams should not default to placing every database, queue, search engine, or cache inside the cluster. Prefer managed services such as RDS, ElastiCache, MSK, OpenSearch, and S3 where they fit.

    Before running stateful services in the cluster, answer:

    • Where are backups stored?
    • How does recovery work across AZ failure?
    • How are failed upgrades rolled back?
    • What happens when a node is lost?
    • Does the team really have operational ownership of this component?

    Delivery Sequence¶

    Phase 1: Network And Account Boundary¶

    Deliver:

    • VPC, subnets, route tables, NAT Gateway or an approved alternative.
    • Cluster IAM role.
    • Node IAM role.
    • Operator and CI/CD access boundaries.
    • Tagging standards.

    Validate:

    aws sts get-caller-identity
aws ec2 describe-subnets --filters "Name=vpc-id,Values=<vpc-id>"
aws eks describe-cluster --name <cluster-name>

    Risks: NAT Gateway cost is often underestimated; private cluster access is often forgotten; overly broad administrator access tends to become the default.

    Phase 2: Create The EKS Control Plane¶

    Deliver:

    • EKS cluster.
    • Kubernetes version selection.
    • API server endpoint access policy.
    • Control plane logging.
    • Encryption configuration.

    Validate:

    aws eks update-kubeconfig --name <cluster-name> --region <region>
kubectl cluster-info
kubectl get nodes
kubectl get ns

    Note: kubectl get nodes can be empty immediately after the control plane is created. A healthy control plane does not mean worker capacity has joined.

    Phase 3: Worker Capacity And System Components¶

    Deliver:

    • Managed Node Groups or Karpenter.
    • Verified versions for CoreDNS, kube-proxy, and VPC CNI.
    • Scheduling separation for system and application components.
    • Cluster Autoscaler or Karpenter scale-out behavior.

    Validate:

    kubectl get nodes -o wide
kubectl -n kube-system get pods -o wide
kubectl describe node <node-name>
kubectl top nodes

    Failure mode: nodes are Ready, but business Pods stay Pending because subnet IP capacity or CNI configuration is exhausted. The broken object is not the Deployment; it is the network capacity model.

    Phase 4: Identity, Permissions, And Secrets¶

    Deliver:

    • EKS Pod Identity or IRSA.
    • Least-privilege IAM policies.
    • Secret management rules.
    • CI/CD deployment permissions.

    Validate:

    kubectl -n <namespace> describe serviceaccount <service-account>
kubectl -n <namespace> exec deploy/<deployment-name> -- aws sts get-caller-identity

    The identity visible from the business Pod should be the workload role, not the node role.

    Phase 5: Ingress, Certificates, And Release Path¶

    Deliver:

    • AWS Load Balancer Controller.
    • IngressClass.
    • DNS records.
    • TLS certificates.
    • Health checks.
    • Canary or blue-green release strategy.

    Validate:

    kubectl -n kube-system get deploy aws-load-balancer-controller
kubectl -n <namespace> get ingress
kubectl -n <namespace> describe ingress <ingress-name>
curl -I https://<service-domain>

    Failure mode: the Ingress object is created, but no load balancer appears. The usual causes are controller permissions, subnet tags, or incorrect annotations.

    Phase 6: Observability And Alerting¶

    Deliver:

    • Node metrics.
    • Pod metrics.
    • Application logs.
    • Kubernetes Events.
    • API server or audit logs.
    • Critical alerts.

    Minimum alert set:

    • Node NotReady.
    • Pod CrashLoopBackOff.
    • Deployment replica shortage.
    • PVC high usage.
    • Ingress 5xx increase.
    • HPA unable to scale for a sustained period.
    • CoreDNS error rate increase.

    Validate:

    kubectl get events -A --sort-by=.lastTimestamp
kubectl top pods -A
kubectl -n <monitoring-namespace> get pods

    Rollback Strategy¶

    EKS rollback cannot rely only on kubectl rollout undo. Roll back by layer.

    Layer Rollback Method
    Application release Revert image tag, Helm release, or GitOps commit
    Ingress Restore old rule, old certificate, or old load balancer
    Worker nodes Revert launch template or create a previous-version node group
    Permissions Revert IAM policy or ServiceAccount binding
    Cluster add-ons Revert to a compatible add-on version or temporarily scale old capacity
    Kubernetes version Usually cannot be downgraded in place; recover through a new cluster migration plan

    The Kubernetes version upgrade path is often underestimated. Before upgrading production, confirm compatibility for add-ons, node AMIs, controllers, and critical CRDs.

    Production Acceptance Checklist¶

    Before production traffic:

    • The cluster spans three AZs.
    • Private subnet IP capacity is sufficient.
    • Worker capacity spans AZs and can recover automatically.
    • System and application components have capacity separation.
    • CoreDNS has enough replicas and is spread across nodes.
    • Critical applications have multiple replicas, PDBs, requests, and limits.
    • Ingress has DNS, TLS, health checks, and rollback.
    • Workloads use least-privilege identity.
    • Secrets are not stored in images, ConfigMaps, or Git.
    • Logs, metrics, events, and alerts are available.
    • CI/CD permissions are auditable.
    • Backup and restore have been tested.
    • Upgrade and failure drills are recorded.

    Common Mistakes¶

    Mistake 1: Treating The Managed Control Plane As Application HA¶

    AWS reduces control plane operational burden, but application Pods, node capacity, ingress, storage, delivery, and monitoring remain your responsibility. Application availability is designed and verified, not purchased by cluster creation.

    Mistake 2: Counting Nodes Without Checking Placement¶

    Six nodes in one AZ still create an AZ-level failure risk. Replica count and node count only matter when combined with topology distribution.

    Mistake 3: Migrating Stateful Services First¶

    Start migrations with stateless services. Use them to prove image standards, release flow, ingress, logging, metrics, and rollback. Stateful services come later because their hardest problems are data consistency, recovery time, and recovery point.

    Mistake 4: Leaving Permissions To The End¶

    If permissions are handled last, temporary exceptions become permanent patterns. Pod identity and deployment access boundaries should exist from the start.

    Q&A Assistant Blocks¶

    Question: Is a Managed Node Group enough for EKS high availability?

    No. Managed Node Groups help with worker lifecycle management, but high availability also depends on multi-AZ subnets, scheduling spread, Pod replicas, PDBs, ingress, storage, permissions, monitoring, and rollback.

    Question: How many AZs should a production EKS cluster use?

    Usually at least three AZs. Two AZs can work for lower-risk systems, but recovery and maintenance flexibility are weaker. A one-AZ cluster should not be called production high availability.

    Question: How should Pods access AWS services?

    Use EKS Pod Identity or IRSA and bind each workload to a least-privilege IAM role. Do not let application Pods inherit broad permissions from the node role.

    Question: Should stateful services run inside EKS?

    Prefer managed AWS services by default. Run stateful systems inside EKS only when the team owns backup, recovery, upgrade, monitoring, and incident handling for that component.
