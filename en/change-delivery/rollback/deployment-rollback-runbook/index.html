    Deployment Rollback Runbook¶

    Purpose¶

    This runbook defines how to roll back a Kubernetes Deployment when a release causes user impact, reliability degradation, or operational risk.

    The core judgment is:

    A rollback is a controlled recovery action, not a panic command.

    kubectl rollout undo is only one step. A safe rollback also freezes further change, captures current state, verifies that the previous version is healthy, prevents the bad version from being redeployed, and leaves enough evidence for a later fix.

    Trigger¶

    Use this runbook when any of the following happens after a deployment:

    • Error rate, latency, saturation, or restart count increases beyond the release guardrail.
    • Readiness or liveness failures affect serving capacity.
    • A critical user journey fails after release.
    • A dependency or configuration change makes the new pods unhealthy.
    • The release owner, incident commander, or service owner decides that recovery speed is more important than continuing the rollout.

    Do not use rollback as a substitute for diagnosis when the current version is not related to the symptom. First confirm that the release is a plausible cause.

    Preconditions¶

    Before executing rollback, confirm:

    • You know the namespace and deployment name.
    • You have permission to update the deployment.
    • The previous ReplicaSet or image revision is known or can be inferred from rollout history.
    • The previous version is expected to be safe.
    • No concurrent automation will immediately reapply the bad version.
    • Someone is watching application health while the rollback runs.

    If the deployment is managed by GitOps, decide whether the rollback will be performed through Git revert, image tag change, or emergency kubectl operation followed by Git reconciliation.

    Impact Check¶

    Record impact before changing the workload:

    kubectl -n <namespace> get deploy <deployment> -o wide
kubectl -n <namespace> get rs -l app=<app-label>
kubectl -n <namespace> get pods -l app=<app-label> -o wide
kubectl -n <namespace> describe deploy <deployment>

    Capture:

    • Current image or release version.
    • Desired, updated, available, and unavailable replicas.
    • Failing pod names.
    • Last deployment time.
    • Primary symptom and affected user path.

    This does not need to delay recovery. A short command transcript is enough.

    Rollback Decision¶

    Choose the action:

    Condition Action
    New version is clearly causing impact and previous version is known good. Roll back immediately.
    Rollout is still progressing and only a small subset is affected. Pause rollout, verify impact, then roll back or resume.
    Bad config or secret caused all versions to fail. Fix config or secret; rollback alone will not recover.
    Database migration is incompatible with the previous app version. Stop and escalate; use the release-specific recovery plan.
    Unknown root cause but user impact is severe. Roll back if previous version is likely safe, then investigate.

    If the rollback could make data incompatible, do not proceed without the service owner or incident commander.

    Procedure¶

    1. Freeze further changes¶

    Pause automation or stop the pipeline that could reapply the bad release:

    kubectl -n <namespace> rollout pause deployment/<deployment>

    For GitOps-managed services, also suspend or update the source of truth according to the platform process. A manual rollback that GitOps immediately overwrites is not recovery.

    2. Review rollout history¶

    kubectl -n <namespace> rollout history deployment/<deployment>
kubectl -n <namespace> rollout history deployment/<deployment> --revision=<revision>

    Confirm the target revision. If rollout history lacks useful change cause, identify the previous image from ReplicaSets:

    kubectl -n <namespace> get rs -o jsonpath='{range .items[*]}{.metadata.name}{"\t"}{.spec.template.spec.containers[*].image}{"\n"}{end}'

    3. Execute rollback¶

    Rollback to the previous revision:

    kubectl -n <namespace> rollout undo deployment/<deployment>

    Rollback to a specific revision:

    kubectl -n <namespace> rollout undo deployment/<deployment> --to-revision=<revision>

    Then watch status:

    kubectl -n <namespace> rollout status deployment/<deployment> --timeout=10m
kubectl -n <namespace> get pods -l app=<app-label> -o wide

    4. Verify service health¶

    Validate both Kubernetes state and user-facing behavior:

    kubectl -n <namespace> get deploy <deployment>
kubectl -n <namespace> get pods -l app=<app-label>
kubectl -n <namespace> logs deploy/<deployment> --tail=100

    Check the service dashboards for:

    • Error rate returning to baseline.
    • Latency returning to baseline.
    • Restart count stabilizing.
    • Successful readiness.
    • Successful business smoke test.
    • Ingress or gateway 5xx rate dropping.

    5. Prevent redeployment of the bad version¶

    After recovery, block the bad version from being applied again:

    • Revert or amend the GitOps commit.
    • Disable the failed pipeline promotion.
    • Mark the failed image or release as blocked in the release record.
    • Notify the service owner and platform owner.

    Do not close the incident while the automation source still points at the bad release.

    Validation¶

    The rollback is successful only when all conditions are met:

    Check Pass Condition
    Deployment status Desired replicas equal available replicas.
    Pods New pods are running the intended rollback image or revision.
    Readiness No sustained readiness failure.
    Logs The release-related error pattern has stopped or returned to baseline.
    Metrics Error rate, latency, saturation, and restart metrics are back within guardrail.
    User path The affected critical path passes a smoke test.
    Automation GitOps or CI/CD will not redeploy the bad version.

    Rollback Is Not Enough When¶

    Rollback alone may fail or be unsafe when:

    • The release included an irreversible or unreviewed database migration.
    • The failure is caused by an external dependency, not the application version.
    • A shared config, secret, or feature flag is broken across versions.
    • Data written by the new version is incompatible with the old version.
    • The previous image has been removed from the registry.
    • The failure is resource exhaustion caused by current traffic, not the release.

    In these cases, escalate to the incident commander and service owner. The recovery path may require config repair, traffic shedding, feature flag disablement, database recovery, or image rebuild.

    Escalation¶

    Escalate immediately when:

    • Rollback does not complete within the expected window.
    • Available replicas stay below the service minimum.
    • Error rate does not recover after the old version is running.
    • Data compatibility is uncertain.
    • Multiple services are affected.
    • You lack permission to update the deployment or source of truth.

    Escalation should include namespace, deployment, failed version, target rollback version, current symptoms, and commands already executed.

    Post-Check¶

    After recovery:

    • Record the failed version, rollback version, time to detect, and time to recover.
    • Attach the command transcript or dashboard links to the incident or change record.
    • Create a follow-up item for the root fix.
    • Add or tighten release guardrails if the failure escaped detection.
    • Review whether the deployment strategy, probes, or canary policy should change.
