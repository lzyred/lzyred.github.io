    etcd Operations Standard¶

    Purpose¶

    This standard defines how to operate etcd safely as the source of truth for Kubernetes and other distributed systems: how quorum determines availability, why disk latency and database size quietly cause outages, and how to take a snapshot you can actually restore.

    The core judgment of this standard:

    etcd is not "a database the cluster uses" — for Kubernetes it is the cluster's memory. Losing quorum stalls the control plane; losing the data loses the cluster. A snapshot you have never restored is not a recovery plan.

    When to Use¶

    Use this standard when you:

    • Run a self-managed etcd cluster (standalone or as the Kubernetes control-plane store).
    • Need a backup/restore and disaster-recovery procedure for cluster state.
    • Plan an etcd or Kubernetes control-plane upgrade or member change.

    Non-Goals¶

    This standard does not cover:

    • Managed Kubernetes, where the provider owns control-plane etcd. You still need application-level backups (Velero etc.); etcd internals are the provider's.
    • Application data modeling on top of etcd.
    • Tuning beyond the production-critical settings below.

    Operating Principles¶

    1. Treat etcd as critical state with strict change control.
    2. Use an odd number of members (3 or 5) so the cluster keeps quorum when one member is lost.
    3. Put etcd on fast, low-latency disks — etcd is fsync-bound, and slow disks cause leader changes and request timeouts.
    4. Take a verified snapshot before any cluster change.
    5. Monitor leader changes, disk fsync/backend-commit latency, database size, and raised alarms.

    Recommended Standard¶

    Quorum and member count¶

    etcd needs a majority (quorum) of members to serve writes. With 3 members it tolerates 1 failure; with 5 it tolerates 2. Even member counts add cost without improving fault tolerance and can split votes. Spread members across failure domains.

    Disk and size are the silent killers¶

    • etcd commits every write to disk; high fsync/backend-commit latency manifests as frequent leader elections and etcdserver: request timed out.
    • The database has a quota (--quota-backend-bytes, default ~2 GiB). When exceeded, etcd raises a NOSPACE alarm and goes read-only — a cluster-wide outage. Stay well under the quota; compact and defragment on a schedule.
    # compaction (keep current revision), then defragment to reclaim space:
ETCDCTL_API=3 etcdctl compact <rev>
ETCDCTL_API=3 etcdctl defrag
# if a NOSPACE alarm is set, clear it after reclaiming space:
ETCDCTL_API=3 etcdctl alarm list
ETCDCTL_API=3 etcdctl alarm disarm

    Defrag is disruptive per member (it blocks that member briefly); do it one member at a time, starting with followers.

    Backup: Snapshot, Proven by Restore¶

    ETCDCTL_API=3 etcdctl --endpoints=<endpoint> \
  --cacert=<ca> --cert=<cert> --key=<key> \
  snapshot save /backup/etcd-snapshot.db

# verify the snapshot is consistent:
ETCDCTL_API=3 etcdctl snapshot status /backup/etcd-snapshot.db --write-out=table
    • Take snapshots on a schedule and before every cluster change (upgrade, member add/remove).
    • Store snapshots encrypted and off the etcd hosts; they contain all cluster state, including Secrets.
    • Restore-test on a schedule into an isolated environment. A snapshot that has never been restored is a hope, not a backup.

    Disaster Recovery (Restore)¶

    Restoring etcd is destructive and must be coordinated — for a Kubernetes control plane, the API server must be stopped while you restore.

    1. Stop the etcd (and dependent API server) processes on the target.
    2. Restore the snapshot into a new data directory on each member with matching peer URLs.
    3. Start etcd as a new cluster from the restored data, then restart the API server.
    4. Validate health and that cluster state matches the snapshot point in time.

    Do this from a runbook, with an owner, never improvised during an incident.

    Validation¶

    ETCDCTL_API=3 etcdctl endpoint health --cluster
ETCDCTL_API=3 etcdctl endpoint status --cluster --write-out=table
ETCDCTL_API=3 etcdctl alarm list

    Confirm:

    • All endpoints are healthy; exactly one leader; no excessive leader changes.
    • Database size is well under quota; no NOSPACE/CORRUPT alarms.
    • For Kubernetes, the API server is responsive and controllers reconcile normally.
    • The latest snapshot passed a restore test within its interval.

    Known Failure Modes¶

    Failure mode Symptom Likely cause Correct response
    Lost quorum Writes fail; control plane stalls Too many members down / even count Restore a member or recover from snapshot; use odd counts.
    NOSPACE / read-only Cluster refuses writes DB exceeded backend quota Compact, defrag, then alarm disarm; raise quota deliberately.
    Frequent leader elections Timeouts, instability Slow disk fsync, network latency Move to fast disks; check network; isolate noisy neighbors.
    Snapshot won't restore Restore fails Never restore-tested / inconsistent snapshot snapshot status to verify; schedule restore tests.
    Cert/peer mismatch Members won't join Wrong peer URLs or certs on restore Restore with matching peer URLs; protect/rotate certs.

    Rollback¶

    1. Always snapshot before a cluster change so you have a known-good point.
    2. To roll back a failed member change, recover the member from the pre-change snapshot and rejoin.
    3. To roll back catastrophic data loss, perform the disaster-recovery restore from the most recent verified snapshot.
    4. Protect and rotate etcd peer/client certificates after any recovery.

    Security Notes¶

    • Never publish real etcd endpoints, certificates, keys, snapshots, or member names.
    • Snapshots contain all cluster state including Kubernetes Secrets — encrypt and access-control them like secrets.
    • Enable mutual TLS between peers and clients; restrict who can reach the client port.
    • Limit who can run snapshot, defrag, alarm, and member operations.

    Review Cycle¶

    • Owner: the platform/cluster team.
    • Verify a successful snapshot restore test each interval.
    • Re-check database size, quota headroom, and disk latency on a regular cadence.
    • Re-confirm the DR runbook before each control-plane upgrade.
