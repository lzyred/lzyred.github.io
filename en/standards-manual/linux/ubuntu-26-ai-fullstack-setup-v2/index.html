    Ubuntu 26.04 AI 全栈开发环境配置手册 v2.0¶

    适用范围与约束声明¶

    适用： - OS：Ubuntu 26.04 LTS Desktop（GNOME，X11 或 Wayland） - 硬件：x86_64，无 GPU，8GB+ RAM - 用途：个人 AI 应用开发工作站（Go + Python + Vue.js + Claude API + 向量数据库） - 执行身份：具备 sudo 权限的普通用户

    不适用： - 服务器 / headless 环境 - NVIDIA GPU 场景（不覆盖 CUDA 安装） - 生产部署（本手册仅覆盖本地开发工作流） - 多人共享机器（secrets 管理策略需调整）

    风险声明： - Claude Desktop Linux 版为社区非官方构建，Anthropic 不提供支持 - Ollama 在纯 CPU 环境下推理速度有限，仅适合开发期流程验证 - 本手册涉及修改 shell 配置、添加第三方 APT 仓库，建议在全新安装的系统上执行

    预计耗时： 2.5–4 小时（含下载时间）

    架构总览¶

    开发工作站组件栈
├── 系统层
│   ├── Ubuntu 26.04 基础配置（禁休眠、中文输入、Swap）
│   └── Shell 环境（Bash + direnv + 统一 env 文件）
├── 工具层
│   ├── Git + SSH + 全局 .gitignore
│   ├── VSCode（官方 .deb）
│   └── tmux（终端复用）
├── 语言层
│   ├── Go（官方 tarball）
│   ├── Python（pyenv + uv）
│   └── Node.js（nvm + pnpm）
├── 容器层
│   └── Docker Engine + Compose（统一 docker-compose 管理本地服务）
├── 服务层（docker-compose 统一编排）
│   ├── Qdrant（向量数据库）
│   ├── PostgreSQL + pgvector（关系型 + 向量）
│   ├── Redis（缓存 / 会话 / 队列）
│   └── Langfuse（LLM 可观测性）
├── AI 客户端层
│   ├── Claude Code CLI（官方 Linux 路径）
│   ├── Claude Desktop（非官方 .deb，可选）
│   └── Ollama（本地 LLM，开发期成本控制）
└── 项目模板层
    ├── Makefile（工作流标准化）
    ├── .envrc（per-project 环境变量）
    └── docker-compose.yml（本地服务编排）

    核心设计原则： - Shell 环境变量单一来源：~/.config/devenv/env.sh，~/.bashrc 只 source 一次 - 本地服务统一 docker-compose 管理，不直接 docker run - 开发期用 Ollama 验证流程，仅在集成测试和生产调用 Claude API，控制成本 - per-project 环境变量用 direnv 隔离，不污染全局

    Phase 0：预检¶

    执行所有后续步骤前，先确认环境满足基础条件。

    # 确认系统版本
lsb_release -a

# 确认是 GNOME 桌面
echo $XDG_CURRENT_DESKTOP

# 检查可用磁盘空间（建议 /home 至少剩余 40GB）
df -h ~

# 检查内存
free -h

# 确认 sudo 可用
sudo whoami    # 应返回 root

# 确认网络（如需代理请提前配置）
curl -fsSL https://go.dev/VERSION?m=text

    继续前提： 磁盘剩余 ≥ 40GB，RAM ≥ 8GB，网络可访问境外资源。

    Phase 1：系统基础配置¶

    1.1 系统更新与基础工具¶

    sudo apt update && sudo apt upgrade -y

sudo apt install -y \
  curl wget git build-essential \
  ca-certificates gnupg lsb-release \
  software-properties-common apt-transport-https \
  unzip zip jq tree htop net-tools \
  tmux xclip libfuse2 imagemagick p7zip-full icoutils

    验证： 

    git --version && curl --version && tmux -V

    1.2 Swap 配置¶

    为什么： 纯 CPU 运行 Embedding 模型或 Ollama 推理时，内存压力显著。没有 Swap 会触发 OOM Killer，进程被强制终止。

    # 检查当前 Swap 状态
free -h && swapon --show

# 如果没有 Swap，创建 8GB Swapfile
if ! swapon --show | grep -q .; then
  sudo fallocate -l 8G /swapfile
  sudo chmod 600 /swapfile
  sudo mkswap /swapfile
  sudo swapon /swapfile
  echo '/swapfile none swap sw 0 0' | sudo tee -a /etc/fstab
  echo "Swapfile created"
fi

# 降低 swappiness（开发机不应频繁使用 Swap，只作保底）
echo 'vm.swappiness=10' | sudo tee /etc/sysctl.d/99-swappiness.conf
sudo sysctl -p /etc/sysctl.d/99-swappiness.conf

    验证： 

    free -h    # SWAP 行应显示 8G 可用
cat /proc/sys/vm/swappiness    # 应返回 10

    1.3 禁用休眠与息屏¶

    # systemd 层屏蔽所有休眠目标
sudo systemctl mask sleep.target suspend.target hibernate.target hybrid-sleep.target

# GNOME 层关闭息屏与自动挂起
gsettings set org.gnome.settings-daemon.plugins.power sleep-inactive-ac-type 'nothing'
gsettings set org.gnome.settings-daemon.plugins.power sleep-inactive-battery-type 'nothing'
gsettings set org.gnome.desktop.session idle-delay 0
gsettings set org.gnome.desktop.screensaver lock-enabled false
gsettings set org.gnome.desktop.screensaver idle-activation-enabled false

    验证： 

    systemctl is-enabled sleep.target    # 应返回 masked
gsettings get org.gnome.desktop.session idle-delay    # 应返回 uint32 0

    回滚（如需恢复休眠）： 

    sudo systemctl unmask sleep.target suspend.target hibernate.target hybrid-sleep.target

    1.4 中文输入法（Fcitx5）¶

    为什么选 Fcitx5 而不是 IBus： Ubuntu 22.04 起 Fcitx5 已是 Linux 桌面首选输入框架，与 Wayland 兼容更好，中文扩展（拼音、五笔、Rime）更完整，社区维护更活跃。IBus 在 GNOME Wayland 下有已知的光标跟随问题。

    sudo apt install -y \
  fcitx5 \
  fcitx5-chinese-addons \
  fcitx5-frontend-gtk4 \
  fcitx5-frontend-qt5 \
  im-config

# 设置为系统默认输入法框架
im-config -n fcitx5

# 设置开机自启
mkdir -p ~/.config/autostart
cat > ~/.config/autostart/fcitx5.desktop << 'EOF'
[Desktop Entry]
Name=Fcitx5
Exec=fcitx5
Terminal=false
Type=Application
Categories=System;Utility;
StartupNotify=false
EOF

    注销并重新登录后： 

    fcitx5 &
fcitx5-configtool    # 在 Input Method 中添加 Pinyin（拼音）

    VSCode 中文输入问题（已知问题）：

    VSCode 从应用菜单启动时不继承 Fcitx5 环境变量，导致无法输入中文。解决方法：

    # 创建 VSCode 启动脚本，从终端启动以继承环境
cat > ~/.local/bin/code-fcitx << 'EOF'
#!/bin/bash
export GTK_IM_MODULE=fcitx
export QT_IM_MODULE=fcitx
export XMODIFIERS=@im=fcitx
exec code "$@"
EOF
chmod +x ~/.local/bin/code-fcitx

# 修改桌面快捷方式使用该脚本
sudo cp /usr/share/applications/code.desktop ~/.local/share/applications/
sed -i 's|Exec=/usr/share/code/code|Exec=/home/'"$USER"'/.local/bin/code-fcitx|g' \
  ~/.local/share/applications/code.desktop

    常见失败模式： - Wayland 下输入法不跟随光标：Fcitx5 → Behavior → 勾选 "Show preedit in application window" - 切换快捷键冲突：默认 Ctrl+Space，可在 Fcitx5 Config → Global Options 中修改

    1.5 Nerd Fonts¶

    为什么： VSCode 和终端（tmux、oh-my-posh 等）使用图标字体，未安装 Nerd Fonts 会显示方块乱码。

    mkdir -p ~/.local/share/fonts/JetBrainsMono

# 下载 JetBrainsMono Nerd Font（推荐：等宽、连字支持好）
wget -q "https://github.com/ryanoasis/nerd-fonts/releases/download/v3.2.1/JetBrainsMono.zip" \
  -O /tmp/JetBrainsMono.zip

unzip -o /tmp/JetBrainsMono.zip \
  "JetBrainsMonoNerdFont-Regular.ttf" \
  "JetBrainsMonoNerdFont-Bold.ttf" \
  "JetBrainsMonoNerdFont-Italic.ttf" \
  -d ~/.local/share/fonts/JetBrainsMono/

fc-cache -fv
rm /tmp/JetBrainsMono.zip

    VSCode 中启用： Settings → editor.fontFamily → 'JetBrainsMono Nerd Font', monospace

    1.6 tmux（终端复用）¶

    为什么： AI 开发同时需要运行：后端服务、前端 dev server、docker compose 日志、LLM 推理进程。tmux 允许在一个终端窗口管理多个会话和窗格，且 SSH 断连后进程不丢失。

    # 基础 tmux 配置（鼠标支持 + 合理快捷键）
cat > ~/.tmux.conf << 'EOF'
# 前缀键改为 Ctrl+a（比默认 Ctrl+b 更顺手）
unbind C-b
set-option -g prefix C-a
bind-key C-a send-prefix

# 鼠标支持
set -g mouse on

# 窗口从 1 开始编号（与键盘布局一致）
set -g base-index 1
setw -g pane-base-index 1

# 快捷键：垂直 / 水平分割
bind | split-window -h -c "#{pane_current_path}"
bind - split-window -v -c "#{pane_current_path}"

# 增大历史记录行数
set -g history-limit 10000

# 256 色支持
set -g default-terminal "screen-256color"
EOF

    常用快捷键参考：

    操作 快捷键
    垂直分割 Ctrl+a |
    水平分割 Ctrl+a -
    切换窗格 Ctrl+a 方向键
    新建窗口 Ctrl+a c
    切换窗口 Ctrl+a 数字
    分离会话 Ctrl+a d
    重连会话 tmux attach

    Phase 2：Shell 环境架构¶

    为什么单独作为一个 Phase： 后续所有工具都需要修改环境变量。如果每个工具各自追加到 ~/.bashrc，会产生重复条目、顺序错乱、幂等性失效等问题。本 Phase 建立统一的环境变量管理架构，后续工具全部写入这个统一文件。

    2.1 建立统一环境变量文件¶

    # 创建目录结构
mkdir -p ~/.config/devenv
mkdir -p ~/.config/dev-secrets

# 创建统一 env 文件（所有工具的 PATH 和 export 集中在此）
cat > ~/.config/devenv/env.sh << 'EOF'
# ============================================================
# Dev Environment - 统一管理，工具安装时追加到此文件
# ============================================================

# Go
export PATH=$PATH:/usr/local/go/bin
export GOPATH=$HOME/go
export PATH=$PATH:$GOPATH/bin

# pyenv
export PYENV_ROOT="$HOME/.pyenv"
[[ -d $PYENV_ROOT/bin ]] && export PATH="$PYENV_ROOT/bin:$PATH"
command -v pyenv > /dev/null && eval "$(pyenv init -)"

# uv
export PATH="$HOME/.local/bin:$PATH"

# nvm（通过 nvm.sh 加载）
export NVM_DIR="$HOME/.nvm"
[ -s "$NVM_DIR/nvm.sh" ] && source "$NVM_DIR/nvm.sh"
[ -s "$NVM_DIR/bash_completion" ] && source "$NVM_DIR/bash_completion"

# direnv hook
command -v direnv > /dev/null && eval "$(direnv hook bash)"

# 本地 bin
export PATH="$HOME/.local/bin:$PATH"

# Secrets（API Keys 等，不纳入 Git）
[ -f "$HOME/.config/dev-secrets/keys.env" ] && source "$HOME/.config/dev-secrets/keys.env"
EOF

# ~/.bashrc 只添加一行 source（幂等，添加前检查是否已存在）
grep -q 'devenv/env.sh' ~/.bashrc \
  || echo '[ -f "$HOME/.config/devenv/env.sh" ] && source "$HOME/.config/devenv/env.sh"' >> ~/.bashrc

source ~/.bashrc

    2.2 API Key 管理¶

    # 创建 secrets 文件（权限 600，仅当前用户可读）
cat > ~/.config/dev-secrets/keys.env << 'EOF'
# Anthropic
export ANTHROPIC_API_KEY="<ANTHROPIC_API_KEY>"

# OpenAI（如需）
# export OPENAI_API_KEY="<OPENAI_API_KEY>"

# 本地服务（无需保密，但统一放这里方便项目引用）
export QDRANT_HOST="localhost"
export QDRANT_PORT="6333"
export REDIS_URL="redis://localhost:6379"
export DATABASE_URL="postgresql://devuser:<LOCAL_DEV_POSTGRES_PASSWORD>@localhost:5432/devdb"
EOF

chmod 600 ~/.config/dev-secrets/keys.env
source ~/.bashrc

    验证： 

    echo $ANTHROPIC_API_KEY | cut -c1-10    # 显示 Key 前 10 位

    2.3 direnv（per-project 环境变量）¶

    为什么： 全局 secrets 文件适合通用 Key，但不同项目有不同的模型选择、不同的服务端口、不同的 DB 名。direnv 实现进入目录自动加载、离开自动卸载，互不污染。

    sudo apt install -y direnv
# direnv hook 已在 env.sh 中配置，直接生效
source ~/.bashrc

    每个项目的用法： 

    cd ~/projects/my-ai-app

cat > .envrc << 'EOF'
# 项目专属变量（覆盖全局）
export ANTHROPIC_API_KEY="<PROJECT_ANTHROPIC_API_KEY>"
export MODEL="claude-sonnet-4-6"
export QDRANT_COLLECTION="my-app-vectors"
export LOG_LEVEL="debug"
EOF

direnv allow .    # 首次允许加载
echo '.envrc' >> .gitignore    # 不提交 secrets

    Phase 3：Git 与 SSH 配置¶

    # Git 基础配置
git config --global user.name "Your Name"
git config --global user.email "you@example.com"
git config --global core.editor "code --wait"
git config --global init.defaultBranch main
git config --global pull.rebase false
git config --global push.autoSetupRemote true

# SSH Key 生成（如已有可跳过）
[ -f ~/.ssh/id_ed25519 ] || ssh-keygen -t ed25519 -C "you@example.com" -f ~/.ssh/id_ed25519

# SSH config 模板（多账户 / 多服务器场景）
mkdir -p ~/.ssh && chmod 700 ~/.ssh
cat > ~/.ssh/config << 'EOF'
# GitHub 主账户
Host github.com
  HostName github.com
  User git
  IdentityFile ~/.ssh/id_ed25519
  AddKeysToAgent yes

# 开发服务器（按需填写）
# Host dev-server
#   HostName 192.168.1.xxx
#   User ubuntu
#   IdentityFile ~/.ssh/id_ed25519
#   ServerAliveInterval 60
EOF
chmod 600 ~/.ssh/config

# 输出公钥，复制到 GitHub → Settings → SSH Keys
echo ""
echo "=== 你的 SSH 公钥（复制到 GitHub）==="
cat ~/.ssh/id_ed25519.pub

    验证（添加 SSH Key 到 GitHub 后）： 

    ssh -T git@github.com    # 应返回 "Hi username! You've successfully authenticated"

    Phase 4：VSCode¶

    为什么用官方 .deb 而不是 Snap： Snap 版 VSCode 运行在沙箱中，对 /home 之外的路径、系统 socket、Go/Python 工具链的访问有限制，会导致 LSP 路径解析失败。官方 .deb 无此限制。

    curl -fsSL https://packages.microsoft.com/keys/microsoft.asc \
  | gpg --dearmor \
  | sudo tee /usr/share/keyrings/packages.microsoft.gpg > /dev/null

echo "deb [arch=amd64 signed-by=/usr/share/keyrings/packages.microsoft.gpg] \
  https://packages.microsoft.com/repos/code stable main" \
  | sudo tee /etc/apt/sources.list.d/vscode.list

sudo apt update && sudo apt install -y code
code --version

    VSCode 核心配置（Ctrl+Shift+P → "Open User Settings JSON"）：

    {
  "editor.fontFamily": "'JetBrainsMono Nerd Font', 'Courier New', monospace",
  "editor.fontSize": 14,
  "editor.lineHeight": 1.6,
  "editor.tabSize": 2,
  "editor.formatOnSave": true,
  "editor.minimap.enabled": false,
  "editor.bracketPairColorization.enabled": true,

    Phase 5：Go 开发环境¶

    为什么用官方 tarball： Ubuntu apt 仓库的 Go 版本通常落后官方 1–2 个版本。Go 的更新频率约每 6 个月一个大版本，工具链更新也依赖具体版本，建议直接用官方包。

    6.2 安装 uv¶

    为什么 uv 而不是 pip/poetry： uv 用 Rust 实现，依赖解析和安装速度比 pip 快 10–100 倍，兼容 pip 语法，内置 venv 管理，2024 年后已成为 Python 工具链新标准。

    Phase 9：本地服务编排（docker-compose）¶

    核心原则： 所有本地依赖服务通过一个 docker-compose.yml 统一管理，不手动 docker run。

    mkdir -p ~/dev-services/data/{qdrant,postgres,redis,langfuse}
cd ~/dev-services

    创建 ~/dev-services/docker-compose.yml：

    # ~/dev-services/docker-compose.yml
# 本地开发服务编排：向量库 + 关系型DB + 缓存 + LLM可观测性
# 使用：docker compose up -d / docker compose down

services:

  # ── 向量数据库 ──────────────────────────────────────────
  qdrant:
    image: qdrant/qdrant:latest
    container_name: dev-qdrant
    restart: unless-stopped
    ports:
      - "6333:6333"    # HTTP / REST API
      - "6334:6334"    # gRPC
    volumes:
      - ./data/qdrant:/qdrant/storage
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:6333/healthz"]
      interval: 30s
      timeout: 5s
      retries: 3

  # ── PostgreSQL + pgvector ────────────────────────────────
  postgres:
    image: pgvector/pgvector:pg16
    container_name: dev-postgres
    restart: unless-stopped
    environment:
      POSTGRES_USER: devuser
      POSTGRES_PASSWORD: <LOCAL_DEV_POSTGRES_PASSWORD>
      POSTGRES_DB: devdb
    ports:
      - "5432:5432"
    volumes:
      - ./data/postgres:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U devuser -d devdb"]
      interval: 10s
      timeout: 5s
      retries: 5

  # ── Redis ────────────────────────────────────────────────
  redis:
    image: redis:7-alpine
    container_name: dev-redis
    restart: unless-stopped
    ports:
      - "6379:6379"
    volumes:
      - ./data/redis:/data
    command: redis-server --appendonly yes
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]
      interval: 10s
      timeout: 3s
      retries: 3

  # ── Langfuse（LLM 调用可观测性）────────────────────────
  langfuse:
    image: langfuse/langfuse:latest
    container_name: dev-langfuse
    restart: unless-stopped
    depends_on:
      postgres:
        condition: service_healthy
    ports:
      - "3000:3000"
    environment:
      DATABASE_URL: postgresql://devuser:<LOCAL_DEV_POSTGRES_PASSWORD>@postgres:5432/langfuse
      NEXTAUTH_SECRET: <LOCAL_DEV_NEXTAUTH_SECRET>
      NEXTAUTH_URL: http://localhost:3000
      SALT: <LOCAL_DEV_SALT>

    操作命令：

    cd ~/dev-services

    Phase 11：Ollama（本地 LLM）¶

    为什么： 开发期间反复调用 Claude API 有成本。Ollama 在本地运行开源小模型，提供 OpenAI 兼容接口。用本地模型验证业务流程，用 Claude API 做集成测试和生产调用。

    CPU 环境的现实预期： 7B 模型在 8 核 CPU 上推理速度约 2–8 token/s（明显慢于 GPU），适合用于流程验证，不适合交互式使用。

    Phase 13：Langfuse 配置（LLM 可观测性）¶

    为什么必须有可观测性： AI 应用的调试和优化依赖对每次 LLM 调用的追踪：prompt 内容、响应时间、token 消耗、检索命中率、向量相似度分布。没有这一层，debug 靠 print，优化靠猜。

    Langfuse 已在 Phase 9 的 docker-compose 中包含，启动后访问 http://localhost:3000 注册本地账号。

    Python 接入：

    uv add langfuse

    # 在项目入口处初始化
from langfuse import Langfuse
from langfuse.callback import CallbackHandler
import os

langfuse = Langfuse(
    public_key=os.environ["LANGFUSE_PUBLIC_KEY"],
    secret_key=os.environ["LANGFUSE_SECRET_KEY"],
    host="http://localhost:3000"
)

# 与 LangChain 集成
langfuse_handler = CallbackHandler(
    public_key=os.environ["LANGFUSE_PUBLIC_KEY"],
    secret_key=os.environ["LANGFUSE_SECRET_KEY"],
    host="http://localhost:3000"
)

# 使用
chain.invoke({"input": "..."}, config={"callbacks": [langfuse_handler]})

    接入后可追踪： - 每次 LLM 调用的完整 prompt 和 response - Token 消耗（输入/输出分别统计） - 端到端 RAG pipeline 延迟分解 - 向量检索命中的 chunk 内容
