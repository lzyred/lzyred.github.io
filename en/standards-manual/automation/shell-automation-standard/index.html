    Shell Automation Standard¶

    Purpose¶

    This standard defines how to write and operate shell scripts used for DevOps automation.

    The core judgment:

    A shell script is production automation when it changes infrastructure, data, access, or deployment state.

    Small scripts often become operational control planes. They need the same discipline as other automation: clear scope, safe inputs, idempotency, logs, validation, and rollback.

    When to Use¶

    Use this standard for scripts that:

    • Change servers, containers, Kubernetes objects, or cloud resources.
    • Run in CI/CD jobs.
    • Rotate files, logs, backups, or certificates.
    • Perform cleanup or migration.
    • Are shared between operators or teams.

    One-off local exploration does not need the full template, but it must not be promoted into shared automation without review.

    Non-Goals¶

    This standard does not force every task into shell. If the logic needs complex state, parsing, API retries, or long-term maintainability, use a language and library better suited to the job.

    Recommended Standard¶

    Start every script with an operating contract¶

    The top of the script or adjacent README must define:

    • Purpose.
    • Target environment.
    • Required permissions.
    • Inputs and defaults.
    • What resources it changes.
    • Dry-run behavior.
    • Validation command.
    • Rollback or recovery boundary.

    If the script cannot say what it changes, it is not safe to run unattended.

    Use strict mode deliberately¶

    Baseline:

    #!/usr/bin/env bash
set -Eeuo pipefail

trap 'echo "error: line ${LINENO}" >&2' ERR

    Strict mode reduces silent failure, but it does not replace explicit checks. Handle expected non-zero results deliberately.

    Validate inputs before changing state¶

    Validate:

    • Required commands exist.
    • Required files exist.
    • Target namespace, host, cluster, or account is explicit.
    • Destructive flags require explicit confirmation or --yes.
    • Output paths do not point to root or empty variables.

    Avoid scripts that infer production targets from current shell context.

    Make repeated runs safe¶

    Prefer idempotent operations:

    • Check before create.
    • Compare before replace.
    • Backup before overwrite.
    • Skip already-complete steps.
    • Treat missing optional resources as expected only when documented.

    Provide dry-run and audit output¶

    For shared scripts, support:

    ./script.sh --dry-run
./script.sh --target <target> --yes

    Dry-run should show intended actions without modifying state. Logs should include timestamp, target, action, result, and operator context when available.

    Validation¶

    Before approving a script:

    bash -n script.sh
shellcheck script.sh
./script.sh --dry-run --target <test-target>

    Pass conditions:

    • Static syntax check passes.
    • ShellCheck findings are fixed or justified.
    • Dry-run output is understandable.
    • Test run changes only the intended test target.
    • Failure exits non-zero.
    • Logs show enough evidence for incident review.

    Failure Modes¶

    Failure Mode Result Prevention
    Empty variable in destructive path Deletes or edits the wrong target. Use strict mode and explicit path guards.
    Implicit current context Runs against wrong cluster or account. Require target argument and print target before action.
    No idempotency Re-running doubles changes or fails midway. Check state before each mutation.
    Hidden partial failure Pipeline succeeds with failed command. Use pipefail and explicit checks.
    No logs No one can reconstruct what happened. Log action and result.
    Secrets in output Credentials leak to CI logs. Mask sensitive values and avoid set -x in shared scripts.

    Rollback¶

    A script must state its rollback boundary:

    • Reversible by running another command.
    • Reversible from backup.
    • Not safely reversible.
    • Requires manual recovery.

    If a script deletes data, rotates credentials, changes firewall rules, or modifies production access, the rollback plan must be reviewed before first production use.

    Security Notes¶

    • Do not store credentials in scripts.
    • Do not print tokens or private keys.
    • Avoid curl | bash for production automation.
    • Pin downloaded artifacts and verify checksums where possible.
    • Use least privilege service accounts in CI.
    • Review scripts that run with sudo, cluster-admin, or cloud admin rights.

    Review Cycle¶

    Review shared automation before first use, after incidents, when permissions change, and when target platforms or APIs change.
