    Nginx Reverse Proxy and TLS Standard¶

    Purpose¶

    This standard defines safe patterns for deploying Nginx or OpenResty as a reverse proxy that terminates TLS in front of application upstreams.

    The core judgment of this standard:

    A reverse proxy is a live production dependency on the request path. Every config change must be validated before it is applied, and reversible after, because a bad reload takes down everything behind it — not one service.

    When to Use¶

    Use this standard when you:

    • Terminate HTTPS for one or more upstream services behind Nginx/OpenResty.
    • Need a consistent, reviewable proxy configuration pattern across services.
    • Manage TLS certificates and their renewal.

    Non-Goals¶

    This standard does not cover:

    • Ingress controllers in Kubernetes (use the Kubernetes ingress standard; the TLS and header principles still apply).
    • WAF rule design or DDoS mitigation.
    • L4/TCP load balancing as the primary concern — see the HAProxy standard for connection-level balancing.
    • CDN or edge-cache strategy.

    Core Principles¶

    1. Separate the public listener configuration from upstream/service configuration so each can be reviewed and changed independently.
    2. Source TLS certificates from a controlled lifecycle (automated issuance/renewal where possible).
    3. Never commit real certificates, private keys, domains, or upstream addresses.
    4. Always nginx -t before reload; never apply an unvalidated config to a live proxy.
    5. Keep access logs useful while avoiding sensitive request data (no auth headers, no full query strings with secrets).

    Recommended Standard¶

    Reverse proxy with explicit upstream¶

    Define upstreams separately, with health-related limits, so a single bad backend does not blackhole traffic.

    upstream example_upstream {
    server <backend-a-host>:8080 max_fails=3 fail_timeout=30s;
    server <backend-b-host>:8080 max_fails=3 fail_timeout=30s;
}

server {
    listen 80;
    server_name service.example.com;

    # Redirect all plaintext to HTTPS; do not serve the app over :80.
    return 301 https://$host$request_uri;
}

    TLS listener¶

    server {
    listen 443 ssl;
    http2 on;
    server_name service.example.com;

    ssl_certificate     /etc/nginx/certs/service.example.com.crt;
    ssl_certificate_key /etc/nginx/certs/service.example.com.key;
    ssl_protocols       TLSv1.2 TLSv1.3;
    ssl_ciphers         HIGH:!aNULL:!MD5;
    ssl_prefer_server_ciphers on;
    ssl_session_cache   shared:SSL:10m;

    client_max_body_size 16m;   # size to the largest legitimate upload

    location / {
        proxy_pass http://example_upstream;
        proxy_http_version 1.1;
        proxy_set_header Host              $host;
        proxy_set_header X-Real-IP         $remote_addr;
        proxy_set_header X-Forwarded-For   $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_connect_timeout 5s;
        proxy_send_timeout    60s;
        proxy_read_timeout    60s;
    }
}

    Certificate and key paths are illustrative. The private key file must have restrictive permissions and must never be committed.

    Forwarded headers must be consistent¶

    X-Forwarded-Proto is the most common source of redirect loops: if the app builds HTTPS redirects from a header the proxy sets wrong, the browser bounces forever. Set it to $scheme on the listener that actually terminated TLS, and make sure the app trusts the proxy.

    Change Procedure¶

    Treat every change as a controlled change on the request path.

    1. Edit the relevant config fragment (listener or upstream), not a monolith.
    2. Validate syntax and referenced files:
    sudo nginx -t
    1. Reload (graceful; in-flight requests finish on old workers):
    sudo systemctl reload nginx
sudo systemctl status nginx --no-pager
    1. Validate behavior (see below) before considering the change done.
    2. If validation fails, roll back immediately — do not iterate live.

    Certificate Lifecycle¶

    Automate issuance and renewal where possible; manual renewal is a recurring outage waiting for the day someone forgets.

    Check expiry of a live endpoint:

    echo | openssl s_client -servername service.example.com -connect service.example.com:443 2>/dev/null \
  | openssl x509 -noout -dates

    Check a local certificate file:

    openssl x509 -in /etc/nginx/certs/service.example.com.crt -noout -dates
    • Test renewal in staging before production.
    • Alert on certificate expiry well ahead of the deadline (see the Prometheus alerting / blackbox TLS-expiry check).
    • Reload Nginx after renewal so it picks up the new certificate.

    Validation¶

    nginx -t                                  # config valid
curl -I https://service.example.com       # status line + headers
curl -sS https://service.example.com/health

    Confirm:

    • TLS handshake succeeds with the expected protocol and a non-expired certificate.
    • The expected status code returns and X-Forwarded-Proto reaches the app correctly (no redirect loop).
    • Upstream is reachable from the proxy host; both backends serve traffic.
    • Error and access logs show no spike of 5xx after reload.

    Known Failure Modes¶

    Symptom Likely cause Check / response
    502 Bad Gateway Upstream down or wrong port curl the upstream from the proxy host; fix proxy_pass/upstream.
    504 Gateway Timeout Upstream slow; proxy timeout too low Raise proxy_read_timeout only if the upstream is legitimately slow; otherwise fix the upstream.
    TLS warning / handshake failure Expired or wrong certificate, missing chain openssl x509 -noout -dates; ensure full chain is served.
    Infinite redirect loop App builds HTTPS redirects from a wrong X-Forwarded-Proto Set X-Forwarded-Proto $scheme; ensure app trusts the proxy.
    Large upload rejected (413) client_max_body_size too small Raise to the largest legitimate upload.
    Reload "succeeded" but old config served Reload skipped or failed silently nginx -t then systemctl reload; check systemctl status.

    Rollback¶

    1. Keep the previous config (version control or a timestamped copy) so you can restore it in one step.
    2. Restore the prior config fragment, run nginx -t, then systemctl reload nginx.
    3. If a reload left Nginx serving errors and the config won't validate, restore the last-known-good file and reload; do not leave a broken proxy while debugging.
    4. For a certificate rollback, restore the previous valid certificate/key pair and reload.

    Security Notes¶

    • Store private keys with restrictive permissions (owner-only); never commit them.
    • Do not log Authorization headers, cookies, or full query strings that may carry tokens.
    • Do not expose admin or internal routes through a public listener unless explicitly required and access-controlled.
    • Disable old TLS versions (keep TLS 1.2+); review cipher suites on a schedule.
    • Keep the proxy and OpenResty/Lua modules patched; the proxy is internet-facing.

    Review Cycle¶

    • Owner: the platform/networking team.
    • Review cipher and protocol settings on a regular cadence as recommendations change.
    • Confirm certificate renewal automation and expiry alerting still work each quarter.
    • Re-validate forwarded-header handling whenever an upstream app changes its redirect logic.
