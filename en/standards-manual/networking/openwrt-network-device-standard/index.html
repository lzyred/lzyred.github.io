    OpenWrt Network Device Standard¶

    Purpose¶

    This standard defines how to handle OpenWrt-based network devices in lab, edge, or small-site environments.

    The core judgment:

    An edge router is infrastructure, even when it is small enough to sit on a desk.

    OpenWrt devices can affect routing, VPN, DNS, firewall policy, and wireless access. Their configuration must be backed up, reviewed, and recoverable.

    When to Use¶

    Use this standard for:

    • Lab routing.
    • Small-office or edge routing.
    • VPN gateway experiments.
    • Network testing.
    • Custom firmware evaluation.

    Do not use unmanaged OpenWrt devices as hidden production network dependencies.

    Non-Goals¶

    This standard does not recommend OpenWrt as the default production router platform, nor does it document ISP-specific settings, Wi-Fi passwords, VPN profiles, or full firewall policies.

    Recommended Standard¶

    Separate firmware, config, and secrets¶

    Keep distinct records for:

    • Firmware version.
    • Package list.
    • Network configuration.
    • Firewall policy.
    • VPN configuration.
    • Wireless settings.
    • Secrets and keys.

    Secrets must not be committed or published.

    Back up before change¶

    Before changing firmware or network policy:

    uci export network
uci export firewall
uci export dhcp

    Store backups in an approved private location.

    Control exposure¶

    Document:

    • Management interface.
    • Allowed admin networks.
    • WAN exposure.
    • VPN peers.
    • Firewall zones.
    • DNS behavior.

    Validation¶

    uci show network
uci show firewall
logread | tail

    Pass conditions:

    • Management access works from approved network.
    • Unapproved WAN admin access is blocked.
    • Required routes work.
    • DNS behavior matches plan.
    • Firewall zones match intended trust boundary.
    • Backup is current.

    Failure Modes¶

    Failure Mode Result Prevention
    No config backup Device change is hard to recover. Export config before changes.
    WAN admin exposed Device can be attacked remotely. Restrict management interfaces.
    VPN key published Private network exposed. Keep keys out of docs and repos.
    Firmware drift Behavior differs from expected. Record firmware and packages.
    Firewall zone mistake Networks over-trust each other. Validate zones and denied paths.

    Rollback¶

    Rollback options:

    • Restore previous UCI configuration.
    • Revert firewall rules.
    • Restore previous firmware if supported.
    • Factory reset and reapply backup.

    For remote devices, ensure out-of-band recovery or onsite support before risky network changes.

    Security Notes¶

    • Do not publish Wi-Fi passwords, VPN profiles, keys, ISP accounts, MAC lists, or real routes.
    • Change default passwords.
    • Restrict SSH and LuCI access.
    • Keep firmware updated.
    • Disable services that are not needed.

    Review Cycle¶

    Review after firmware upgrades, VPN changes, firewall changes, remote-access changes, and network incidents.
